# 3.7 완전 자동화 팀 설계: /loop vs Cowork Scheduled Tasks

## 개요

2026년 Anthropic이 제공하는 자동화 도구는 **두 가지 철학**을 대표한다:

1. **`/loop`** (Claude Code): 개발자 중심, Cron 기반 스크립트화
2. **Scheduled Tasks** (Cowork): PM 중심, GUI 기반 드래그 드롭

**이 장은** 각 방식으로 완전히 자동화된 팀을 어떻게 설계하는지 단계별로 보여준다.

---

## Part A: `/loop` 기반 자동화 팀 (Claude Code)

### 개념: "백그라운드 워커 팀"

```
기존:
PM이 매일 08:00에 손으로 /plan 실행
    ↓ 결과 확인
    ↓ 14:00에 /qa 실행
    ↓ 결과 확인
    ↓ 18:00에 /ship 실행

새로운 (/loop):
일 1회 설정 → 365일 자동 실행
PM은 모니터링만 함
```

### /loop 문법 (빠른 참고)

```bash
# 기본 문법
/loop <시간> <커맨드> [옵션]

# 예시
/loop 08:00 /plan
/loop 14:00 /qa
/loop 18:00 /ship

# 모니터링
/loop --list              # 예약된 작업 확인
/loop --logs --task-id X  # 특정 작업 로그 보기
/loop --disable           # 모든 작업 일시정지
/loop --enable            # 모든 작업 재개
```

### 예제 1: 스타트업 (1일 1회 배포)

#### 목표
```
PM의 수동 작업 0%
자동화 팀: Oracle(계획) → Forge(명세) → Argus(테스트) → 배포
```

#### 설정

```bash
# 아침 8시: 오늘의 할일 계획
/loop 08:00 /discover --scope daily

# 오전 10시: 명세 작성
/loop 10:00 /prd --from-discovery

# 오후 2시: QA 테스트
/loop 14:00 /qa --comprehensive

# 저녁 6시: 배포
/loop 18:00 /ship --auto-approve

# 자정: 일일 리포트 저장
/loop 23:59 /archive-daily
```

#### 실제 실행 흐름

```
Day 1 (월요일)
08:00 → /discover
       생성: oday-plan.md (할일 리스트)

10:00 → /prd
       입력: today-plan.md
       출력: prd-Mon.md (명세)

14:00 → /qa
       입력: prd-Mon.md
       테스트 실행 (자동)
       결과: test-Mon-passed.json

18:00 → /ship
       입력: test-Mon-passed.json
       배포 실행
       자동으로 Slack에 "배포 완료" 알림

23:59 → /archive-daily
       오늘의 모든 산출물 아카이브
       내일을 위한 상태 정리
```

#### PM의 역할

```
┌─ 월요일 아침 (1분)
│  "배포가 잘 되었나요?" /loop --logs
│  → 저녁에 배포 완료 확인
│
└─ 주간 회고 (금요일 오후 1시)
   "이번 주 배포 7회, 성공률 100%"
   → /loop --report --week
```

**총 PM 개입: 주 5분** (기존: 주 35시간)

### 예제 2: 성장기 팀 (주 1회 배포, 복합 자동화)

#### 목표
```
주간 계획 → 일일 빌드 → 수요일 검증 → 금요일 배포
자동화 수준: 90% (복잡한 의사결정만 PM 개입)
```

#### 설정

```bash
# ========== 월요일 아침 (주간 계획) ==========
/loop 08:00 "Monday" /plan-weekly --team

# ========== 매일 (일일 빌드) ==========
/loop 09:00 /build --daily
/loop 11:00 /test --unit
/loop 14:00 /security-scan

# ========== 수요일 오후 (중간 검증) ==========
/loop 15:00 "Wednesday" /review-progress --checkpoint

# ========== 목요일 오후 (배포 준비) ==========
/loop 16:00 "Thursday" /staging-deploy --test-env

# ========== 금요일 (최종 배포) ==========
/loop 10:00 "Friday" /pre-deploy-checklist
/loop 17:00 "Friday" /deploy-production --blue-green
```

#### 토요일 자정: 주간 리포트

```bash
/loop 23:59 "Saturday" /weekly-report \
  --include-metrics \
  --include-blockers \
  --include-next-week-preview
```

#### PM 개입 지점

```
월요일 08:30  → /plan-weekly 결과 검토 (5분)
              → 우선순위 조정 필요하면 /plan-weekly --update

수요일 15:30  → /review-progress 체크 (3분)
              → 블로커 있으면 /resolve --blocker-id

목요일 16:30  → /staging-deploy 테스트 (10분)
              → 문제 있으면 /rollback-staging

금요일 10:30  → /pre-deploy-checklist 리뷰 (2분)
              → 체크리스트 통과하면 배포 자동 진행

금요일 17:30  → /deploy-production 최종 모니터링 (5분)
```

**총 PM 개입: 주 25분** (기존: 주 20시간)

### 예제 3: 엔터프라이즈 (24시간 운영, 글로벌 팀)

#### 목표
```
시간대별 에이전트 자동화
- 아침 (KST 08:00): Oracle (계획)
- 오전 (KST 10:00): Atlas (아키텍처)
- 정오 (KST 12:00): Forge (명세)
- 오후 (KST 14:00): Argus (테스트)
- 저녁 (KST 18:00): 배포

각 에이전트가 전 날의 결과를 기반으로 독립적으로 실행
```

#### 설정 (KST 기준)

```bash
# ========== 아침 08:00: Oracle 에이전트 ==========
/loop 08:00 /discover \
  --scope "new-opportunities" \
  --analyze-market \
  --check-customer-feedback \
  --output-format json

# ========== 오전 10:00: Atlas 에이전트 ==========
/loop 10:00 /architecture-review \
  --input-from-discovery \
  --check-performance \
  --identify-bottlenecks \
  --propose-improvements

# ========== 정오 12:00: Forge 에이전트 ==========
/loop 12:00 /write-prd \
  --from-discovery \
  --from-architecture \
  --include-okr-mapping \
  --auto-prioritize

# ========== 오후 14:00: Argus 에이전트 ==========
/loop 14:00 /comprehensive-qa \
  --from-prd \
  --include-security-scan \
  --include-performance-test \
  --include-code-review \
  --report-json

# ========== 저녁 18:00: 배포 ==========
/loop 18:00 /pre-flight-check
/loop 18:15 /deploy-staging --wait-for-approval
/loop 19:00 /deploy-production --canary 10%
/loop 20:00 /canary-health-check

# ========== 자정: 일일 아카이브 ==========
/loop 23:59 /daily-archive \
  --collect-all-outputs \
  --summarize-metrics \
  --prepare-next-day
```

#### 모니터링 & 개입

```bash
# 에이전트 상태 대시보드
/loop --status-dashboard  # 모든 에이전트의 실시간 상태

# 에러 처리 자동화
/loop --on-error "notify-slack" \
  --channels "#ai-pm-alerts" \
  --escalate-after "2-consecutive-failures"

# 긴급 개입 (PM이 하는 유일한 작업)
/loop --pause "deploy" --reason "critical-bug-found"
/loop --resume "deploy" --after-fix-confirmed
```

#### PM의 역할

```
매일 아침 08:05: 대시보드 확인 (1분)
- Oracle 실행 완료? → 다음 단계로 진행
- 문제 있으면 /loop --pause --task oracle

매일 오후 14:05: Argus 테스트 결과 확인 (2분)
- QA 실패했으면 /loop --rollback
- 통과했으면 배포 진행

매일 저녁 18:05: 배포 모니터링 (5분)
- Canary 10% 정상?
- 문제 있으면 /loop --pause-deploy
- 정상이면 100% 배포 자동 진행

주 1회 (금요일): 주간 리포트 (10분)
- 7일 x 5 = 35회 자동 실행 결과 분석
- 성공률, 병목, 개선안 도출
```

**총 PM 개입: 일 8분, 주 58분** (기존: 주 40시간)

**효과: 시간 98% 절감 + 품질 향상**

## Part B: Cowork Scheduled Tasks (GUI 기반)

### 개념: "PM이 보면서 조정하는 팀"

```
/loop:       PM이 설정 후 손 떼고 있음
Scheduled:   PM이 UI에서 드래그하며 세밀 조정

Cowork이 좋은 이유:
- 코딩 불필요 (드래그 드롭)
- 실시간 모니터링 (UI에 진행상황 표시)
- 팀원과 공유 (모두가 볼 수 있음)
- 조정 쉬움 (시간, 조건 변경 가능)
```

---

### Cowork Scheduled Tasks 사용법

#### 기본 UI 흐름

```
1. Cowork 앱 열기
   ↓
2. "Automations" 탭
   ↓
3. "+ Create Automation" 클릭
   ↓
4. 다음 내용 입력:
   - 이름: "Daily Planning"
   - 시간: 08:00 (드래그 슬라이더)
   - 빈도: Daily (드롭다운)
   - 작업: /plan-daily (커맨드 선택)
   - 조건: 월~금요일만 (체크박스)
   ↓
5. "Save" 클릭
   ↓
6. 매일 08:00에 자동 실행
```

#### 조건부 실행 (고급)

```
예: "테스트가 통과한 경우에만 배포"

1. Task: /qa 완료 후
2. 조건: success == true
3. 다음 Task: /deploy 자동 실행
4. 조건 실패: /notify-slack "#failed-qa"
```

### 예제 1: PM 친화적 설정 (비기술자용)

#### 목표
```
매주 월요일 오전: 주간 계획 회의 자동 준비
비기술자도 이해할 수 있는 UI
```

#### Cowork UI 설정

```
Automation 1: "주간 계획 자료 준비"
├─ 시간: 월요일 07:00
├─ 작업: /discover-weekly
├─ 출력: weekly-plan.md
├─ 알림: Slack #pm-channel
│         "주간 계획 자료 준비 완료!"
│
└─ Automation 2: "계획 자료 시각화"
   ├─ 조건: weekly-plan.md 준비 완료
   ├─ 시간: 자동 (위 작업 완료 후 5분)
   ├─ 작업: /ppt-from-plan
   │         (weekly-plan.md → PowerPoint 자동 생성)
   └─ 알림: Slack
            "슬라이드 준비 완료! [보기]"
```

#### PM이 UI에서 하는 작업

```
월요일 07:30에 Cowork 앱 열기
↓
Automation 대시보드 확인
- "주간 계획 자료 준비" ✅ 완료
- "계획 자료 시각화" ⏳ 진행 중
↓
완료 후 [Weekly Planning] 슬라이드 클릭
↓
슬라이드 내용 검토
↓
필요하면 드래그 드롭으로 우선순위 변경
↓
저장
↓
팀에 "월 08:00 회의 자료 준비됨" 공유 (UI에서 1클릭)
```

## Part C: 선택 기준

### `/loop` vs Cowork Scheduled Tasks

| 항목 | `/loop` | Cowork |
|------|--------|--------|
| **사용자** | 기술자 (개발자, 기술 PM) | 비기술자 (PM, 마케터) |
| **설정 난도** | 높음 (커맨드라인) | 낮음 (UI 드래그) |
| **속도** | 빠름 (경량) | 보통 (UI 오버헤드) |
| **팀 공유** | 없음 (로그만) | ✅ Projects와 통합 |
| **실시간 모니터링** | `/loop --logs` (터미널) | UI 대시보드 (시각적) |
| **조정 난도** | 중간 (커맨드 수정) | 낮음 (드래그 드롭) |
| **에러 처리** | 코드로 구현 | UI 조건부 분기 |
| **팀원 승인** | 없음 | Dispatch (UI) |
| **규제 감시** | Audit log 수동 | Cowork Enterprise (자동) |

### 의사결정 플로우

```mermaid
flowchart TD

    B -->|No, UI가 좋음| D[Cowork Scheduled Tasks]

    F -->|Yes| G[/loop + Cowork<br/>하이브리드]

    D --> H["✅ Cowork 100%"]
```

## Part D: 실전 체크리스트

### `/loop` 기반 자동화 시작

- [ ] 반복되는 작업 3개 식별
- [ ] 각 작업의 소요 시간 측정
- [ ] 자동화 우선순위 정렬 (시간 절감 기준)
- [ ] `/loop` 문법 학습
- [ ] 첫 자동화 작업 설정
- [ ] 1주일 모니터링 (로그 확인)
- [ ] 조정/최적화

### Cowork Scheduled Tasks 시작

- [ ] Cowork 앱 설치 (또는 웹 열기)
- [ ] 팀원 초대
- [ ] Projects 보드 설정
- [ ] 반복 작업 1개 선택
- [ ] UI에서 드래그 드롭으로 설정
- [ ] 1주일 실행 모니터링
- [ ] 조정/최적화

### 하이브리드 설정

- [ ] Claude Code `/loop` 설정 (복잡한 분석)
- [ ] Cowork Projects 설정 (팀 공유용)
- [ ] 연동 방식 정의 (결과를 Projects에 자동 저장)
- [ ] 1주일 실행 테스트
- [ ] PM/개발자의 역할 분담 확정
