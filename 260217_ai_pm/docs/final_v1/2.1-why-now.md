# 왜 지금, AI 네이티브 PM인가

> *"PM의 가치는 문서를 만드는 것이 아니라, 올바른 결정을 내리는 것이다. AI는 전자를 자동화하고, 후자를 증폭시킨다."*

## 🔗 하네스 엔지니어링과의 연결

이 Part는 하네스 엔지니어링의 **헌법(Constitution)**을 정의하는 철학적 기반입니다.

### 연결 지점

- **헌법 — 기본 철학**: "자동화/증강/직접판단" 프레임워크가 프로젝트의 근본 원칙이 되며, CLAUDE.md에 명시하면 모든 작업이 이 원칙에 정렬됩니다.
- **Repository Harness**: "AI를 도구가 아닌 팀원으로 대하는" 철학적 기반을 제공합니다.

**참고**: [dependency-map.md](./dependency-map.md)의 헌법(Constitution) 매핑 섹션을 먼저 읽으면 이 Part의 내용이 더 명확해집니다.

---

## 숫자로 보는 변화

2025년, PM의 업무 환경은 극적으로 달라졌다.

McKinsey의 조사에 따르면 지식 노동자의 **60\~70%가 AI로 자동화 가능한 반복 업무**에 시간을 쓰고 있었다. PM도 예외가 아니었다. 회의록 정리, 데이터 취합, 문서 초안 작성—이런 작업이 하루의 대부분을 차지했다.

그런데 2025~2026년, 상황이 바뀌었다.

- **GitHub Copilot 사용 개발자**: 코드 작성 속도 55% 향상 (GitHub 공식 데이터)
- **AI 도구 사용 기업**: 제품 출시 사이클 평균 30~40% 단축 (Gartner, 2025)
- **Claude Code 등장**: 터미널에서 파일 분석, 문서 생성, 데이터 시각화, 프로토타입까지 — PM이 직접 실행 가능

질문은 더 이상 *"PM이 AI를 써야 하나?"*가 아니다.
**"AI를 쓰지 않는 PM이 어떻게 경쟁할 것인가?"**다.

## PM의 역할은 변하고 있다

### 어제의 PM: 문서 작성자

기존 PM의 일과는 다양했다. 스프레드시트를 만들고, 요구사항 문서(PRD)를 작성하고, 엑셀로 로드맵을 정렬하고, 회의 노트를 정리했다. 이 모든 것이 *문서 생산*이었다. 문서는 엔지니어가 '뭘 만들어야 하는지'를 알기 위한 지시서였다.

문제는 명확했다. 시간이 걸렸다. 피드백 루프가 느렸다. 문서가 수정될 때마다 여러 사람이 다시 읽어야 했다.

### 오늘의 PM: 문제 정의자 + 검증 설계자 + 실행 오케스트레이터

AI가 문서 작성을 자동화하면서, PM의 진정한 가치는 다른 곳에 있음이 드러났다.

**문제 정의 (Problem Definition)**
무엇을 해결할 것인가? 왜 우리가 할 것인가? 고객이 진짜 원하는 것과 말하는 것의 차이를 구분하는 것. 이것은 AI가 대신할 수 없다.

**검증 설계 (Validation Design)**
우리가 맞다는 걸 어떻게 알 것인가? 어떤 데이터가 성공을 증명하는가? 실험의 설계와 해석에서 PM의 맥락적 판단이 필요하다.

**실행 오케스트레이션 (Execution Orchestration)**
팀 전체가 움직일 때, 각자 다른 속도, 다른 리스크를 가진다. 엔지니어링, 디자인, 마케팅이 동시에 전진할 수 있도록 조율하는 것. 이것은 조직의 맥락을 아는 사람만이 할 수 있다.

이 세 가지는 AI가 대체할 수 없다. PM의 판단이 필요하다.

## "자동화 / 증강 / 직접판단" 프레임워크

PM이 AI를 어떻게 사용할지를 분류하면:

### 자동화 (Automation) — AI가 혼자 한다

반복적이고, 규칙이 명확하고, 판단이 불필요한 작업이다.

| 작업 | AI가 하는 것 | PM이 하는 것 |
| --- | --- | --- |
| 회의록 정리 | 녹취 → 요약 → 액션 아이템 추출 | 최종 확인 |
| 데이터 포맷 변환 | CSV → Markdown 테이블, JSON → 차트 | 없음 |
| 문서 포맷팅 | 템플릿 적용, TOC 생성, 넘버링 | 없음 |
| 일일 브리핑 생성 | Slack/Linear/Calendar 취합 | 우선순위 조정 |

### 증강 (Augmentation) — AI가 초안, PM이 판단

PM의 판단이 있지만, AI가 속도를 높이거나 옵션을 생성할 때 유용하다.

| 작업 | AI가 하는 것 | PM이 하는 것 |
| --- | --- | --- |
| 경쟁 분석 | 데이터 수집, 비교 매트릭스 생성 | 포지셔닝 해석, 전략적 시사점 |
| 유저 리서치 정리 | 인터뷰 전사, 패턴 클러스터링 | 인사이트 검증, 핵심 문제 선택 |
| PRD 초안 | 구조 생성, 유저 스토리 작성 | 범위 결정, 트레이드오프 판단 |
| A/B 테스트 분석 | 통계 검증, 시각화 | Go/No-Go 결정 |

### 직접판단 (Direct Judgment) — PM만이 할 수 있다

이것은 AI에게 맡길 수 없는 영역이다.

| PM의 판단 | 왜 AI가 못하는가 |
| --- | --- |
| 전략 방향 결정 | 조직의 정치적 맥락, 이해관계자 역학 |
| 트레이드오프 결정 | 비즈니스 제약과 기술 부채의 균형 |
| 팀과의 어려운 대화 | 감정, 신뢰, 관계의 맥락 |
| 고객의 진짜 문제 식별 | 말하지 않은 것을 읽는 능력 |
| 실패 후 방향 전환 | 조직의 준비도와 타이밍 판단 |

AI를 잘 쓰는 PM은 이 세 영역의 경계를 명확히 안다.

> 💡 **이 프레임워크는 가이드 전체를 관통합니다.** 각 챕터에서 *"이 작업은 자동화/증강/직접판단 중 어디에 해당하는가?"*를 계속 질문하게 됩니다. Appendix [A.2-level3-exercises.md](./A.2-level3-exercises.md)에서 이 프레임워크로 실제 프로젝트에 적용하는 연습을 할 수 있습니다.

## AI 네이티브 PM이란

**AI를 보조 도구가 아닌 '팀원'으로 다루는 사고방식이다.**

보조 도구는 '필요할 때 꺼내는 것'이다. 스프레드시트처럼.

팀원은 다르다. 상시 접근 가능하고, 이전 대화를 기억하고, 당신이 한 말을 바탕으로 다음 제안을 만든다.

AI 네이티브 PM은:

1. **매일 Claude와 대화한다.** 생각을 정리하듯이. "이 기능을 왜 만들어야 하지?"라고 물으면, Claude가 반문한다. "어떤 유저 세그먼트가 가장 큰 고통을 겪고 있나요?"
2. **데이터를 주면, 분석 초안을 받는다.** 판단은 내가 한다. AI의 분석에 🟢/🟡/🔴 신뢰도를 매기고, 직접 검증할 부분을 식별한다.
3. **시뮬레이션을 돌린다.** "만약 우리가 프리미엄 가격을 20% 올리면?" — Claude가 시나리오별 영향을 계산하고, PM이 시장 맥락에서 판단한다.
4. **프로젝트 메모리를 유지한다.** CLAUDE.md에 제품 컨텍스트, 팀 구조, 이전 결정을 기록해두면, 매번 처음부터 설명하지 않아도 된다.
5. **Human-in-the-Loop를 설계한다.** 어떤 작업은 자동으로 실행하고, 어떤 작업은 PM의 승인을 거치도록 루프를 설계한다. 이것이 AI 파트너십의 핵심이다.

> 💡 **Human-in-the-Loop 사고방식**은 이 가이드의 핵심 철학입니다. [3.6-human-in-the-loop.md](./3.6-human-in-the-loop.md)에서 HITL/HOTL/HOOTL 세 가지 협업 모델과 루프 깊이 프레임워크를 깊이 다룹니다.

## Before & After: 하루의 변화

### 기존 PM의 하루 (8시간)

```
회의 노트 정리         ████░░░░  45분
스프레드시트 업데이트   ████████░  1시간
요구사항 문서 작성     ████████████░  1.5시간
메일 & 슬랙           ████████░  1시간
회의 (의사결정 없음)   ████████████████░  2시간
─────────────────────────────────────
실제 문제 해결 & 전략  ████████░  1시간  ← 가치 창출
```

**실제 가치 창출: 1/8 (12.5%)**

### AI 네이티브 PM의 하루 (8시간)

```
아침 스탠드업 (Claude 브리핑)  ██░░  15분
경쟁 분석 초안 검토 & 수정    ████░  30분  ← 증강
유저 리서치 정리 & 인사이트    ██████░  40분  ← 증강
트레이드오프 시뮬레이션       ████████░  1시간  ← 증강+판단
팀 미팅 (의사결정 포함)       ████████░  1시간  ← 판단
메일 & 슬랙                 ███░  20분  ← 자동화
─────────────────────────────────────
전략 & 방향성 정의           ████████████████░  2시간  ← 가치 창출
버퍼                        ████░  30분
```

**실제 가치 창출: 5/8 (62.5%)**

차이는 5배다. 생각하는 시간이 늘었다.

핵심은 '더 빨리 일하는 것'이 아니다. **더 중요한 일에 시간을 쓰는 것**이다.

## 왜 Claude Code인가

PM을 위한 AI 도구는 많다. ChatGPT, Copilot, Gemini… 그런데 왜 Claude Code인가?

|  | ChatGPT / Web AI | Cursor / Copilot | **Claude Code** |
| --- | --- | --- | --- |
| 인터페이스 | 웹 채팅 | IDE 내장 | **터미널 (CLI)** |
| 파일 접근 | 업로드만 | 코드 파일 | **프로젝트 전체** |
| 컨텍스트 유지 | 대화별 리셋 | 파일 단위 | **CLAUDE.md 프로젝트 메모리** |
| 외부 도구 연동 | 플러그인 | 제한적 | **MCP (Slack, Linear, Notion, GitHub)** |
| 자동화 | 수동 반복 | 코드 제안 | **슬래시 커맨드, 스킬, n8n** |
| PM 적합도 | 범용 | 개발자 중심 | **PM 워크플로우 최적화 가능** |

Claude Code의 핵심 차별점은 **프로젝트 메모리**와 **도구 연동**이다. PM의 제품 컨텍스트를 기억하고, Slack/Linear/Notion과 직접 연결되어, 터미널 하나에서 PM의 전체 워크플로우를 실행할 수 있다.

> 📖 Claude Code가 무엇인지, CLI의 4가지 장점이 궁금하다면 → [2.2-what-is-claude-code.md](./2.2-what-is-claude-code.md)
> 🛠️ 바로 설치하고 싶다면 → [2.3-install-and-first-run.md](./2.3-install-and-first-run.md)

## 이 가이드의 구조

이 과정은 PM의 일과에 맞춰 설계되었다. 도구 기능 중심이 아니라, **워크플로우 중심**이다.

```
Part 2: 시작하기          왜 지금 AI PM인가, Claude Code 소개, 설치
   ↓
Part 3: 기본기            파일/모드/메모리/서브에이전트/에이전트 팀/HITL
   ↓
Part 4: 고급 설정         MCP 연동, CLAUDE.md 심화, 슬래시 커맨드, 스킬, 자동화
   ↓
Part 5: Discovery         유저 리서치 분석, 경쟁사 분석
   ↓
Part 6: Definition        PRD 작성, 전략 문서
   ↓
Part 7: Delivery          바이브 코딩, 비주얼 에셋, GitHub 배포
   ↓
Part 8: Growth            실험 분석, KPI 대시보드, AI 옵저빌리티
   ↓
Part 9: 전략과 성장       AI 제품 전략, PM 성장 로드맵
   ↓
Appendix                  러닝 시나리오, 실전 연습, 유스케이스 10선
```

각 Part는 **실제 Claude Code 터미널 입력 → 응답 → PM 판단 포인트** 워크스루를 포함한다. 읽는 것이 아니라, 따라 하면서 배운다.

## 레벨별 학습 경로

이 가이드는 세 레벨을 염두에 두고 작성되었다. 자신의 레벨을 파악하고, 맞는 경로를 선택하자.

### J (Junior PM) — "반복에서 해방되기"

PM 경력 0~2년. AI 도구 경험이 적다.

**목표**: 반복적인 작업에서 해방되어, 질문하는 법을 배우기

**추천 경로**:
1. Part 2 전체 (설치부터 시작)
2. Part 3.1~3.3 (파일, 모드, 메모리 기초)
3. Part 3.6 (Human-in-the-Loop 철학 — 모든 레벨 필수)
4. Part 5.1 (유저 리서치 데이터 분석 따라하기)
5. Part 7.1 (바이브 코딩으로 프로토타입 만들기)
6. Appendix A.4~A.5 (일일 브리핑, 상태 보고서 자동화)

**도전 과제**: `samples/user-survey-results.csv`로 설문 데이터 분석 워크스루 완주

### P (Practitioner PM) — "AI와 함께 생각하기"

PM 경력 3~7년. AI 도구를 일부 사용하고 있다.

**목표**: AI와 '함께 생각하기' — 증강 영역에서 AI를 파트너로 활용

**추천 경로**:
1. Part 2 훑기 → Part 3 전체 (특히 3.4 서브에이전트, 3.6 HITL)
2. Part 4 (MCP 연동, 슬래시 커맨드, 스킬 — 자동화 인프라 구축)
3. Part 5~6 (Discovery → Definition 전체 흐름)
4. Part 8.1 (실험 분석)
5. Appendix A.1 (러닝 시나리오 전체 워크스루)

**도전 과제**: `templates/CLAUDE-md-starter.md`를 자신의 프로젝트에 맞게 커스터마이징

### L (Lead PM) — "팀의 사고를 확장하기"

PM 경력 7년+. 팀/조직을 리딩하고 있다.

**목표**: AI를 조직의 사고 확장 도구로 활용. 팀 전체의 PM 생산성을 높이기.

**추천 경로**:
1. Part 2 훑기 → Part 3.4~3.6 집중 (서브에이전트, 에이전트 팀, HITL)
2. Part 4 전체 (팀 공유용 인프라 설계)
3. Part 6.2 (전략 문서 — Diagnosis/Guiding Policy/Coherent Actions)
4. Part 8 전체 (Growth — 실험, KPI, 옵저빌리티)
5. Part 9 (AI 제품 전략, 팀 도입 가이드)
6. Appendix A.6~A.8 (배틀 카드, 페르소나, 투자 메모)

**도전 과제**: `templates/commands/` 슬래시 커맨드를 팀 표준으로 배포하기

## 이 가이드를 읽는 방법

1. **순서대로 읽지 않아도 된다.** 위의 레벨별 경로를 참고하되, 지금 당장 필요한 챕터부터 시작해도 좋다.

3. **CLAUDE.md를 먼저 만든다.** Part 3.3에서 프로젝트 메모리를 설정하면, 이후 모든 챕터의 실습이 더 풍부해진다.
