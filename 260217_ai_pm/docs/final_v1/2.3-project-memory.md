# CLAUDE.md: 프로젝트의 두뇌를 만들다

## 개요
CLAUDE.md는 Claude Code와 함께 일할 때 **세션이 바뀌어도 유지되는** 프로젝트 컨텍스트입니다.
일회성 지시가 아니라, "이 프로젝트에서는 항상 이렇게 일해"라는 영구적 설정을 만들 수 있습니다.

예를 들어, PM이 CLAUDE.md에 "항상 수치에는 출처를 붙여", "마케팅 가정을 사실처럼 쓰지 말아" 같은 규칙을 써두면,
이후 모든 Claude 대화에서 자동으로 그 규칙이 적용됩니다.

---

## 1. CLAUDE.md 메모리 계층 구조

Claude Code는 4단계 계층 구조로 CLAUDE.md를 찾고 적용합니다:

```
┌─────────────────────────────────────────────────────┐
│ 1. Global (~/.claude/CLAUDE.md)                     │
│    └─ 조직 전체 공통 규칙 (모든 PM이 공유)          │
│       • 문서 작성 스타일                             │
│       • 회사 보안 정책                              │
│       • 브랜드 톤 & 보이스                           │
│       • 단위 표준 (USD 사용, UTC 시간대)            │
│                                                      │
│ 2. Project (/project/root/CLAUDE.md)               │
│    └─ 프로젝트별 규칙 (이 프로젝트만)               │
│       • 제품명, 타겟층, 지표                        │
│       • Persona 정의                                │
│       • 기술 스택 정보                              │
│       • 프로젝트 문화                               │
│                                                      │
│ 3. Directory (하위 폴더별 CLAUDE.md)               │
│    └─ 모듈/섹션별 세분화 규칙                       │
│       • /docs/ : 문서 작성 스타일                   │
│       • /features/ : 기능별 컨텍스트                │
│       • /marketing/ : 마케팅 톤                     │
│                                                      │
│ 4. Personal (CLAUDE.local.md) ⭐ 개인만 적용       │
│    └─ 개인 선호도 (개인 파일로 추적 안 함)          │
│       • 깊이 선호도 (think vs ultrathink)           │
│       • 응답 길이 (간결 vs 상세)                    │
│       • 코드 스타일 (Python vs JavaScript)          │
└─────────────────────────────────────────────────────┘

적용 순서: Local > Directory > Project > Global
(더 구체적인 규칙이 일반적 규칙을 덮어씀)
```

**예시:**
```
Global CLAUDE.md: "모든 내용을 영어로 작성"
Project CLAUDE.md: "TaskFlow 프로젝트는 한국어로 작성"
→ TaskFlow 프로젝트에서는 한국어 사용 (Project가 Global 덮어씀)

Directory CLAUDE.md (/marketing/): "마케팅 카피는 캐주얼 톤"
Project CLAUDE.md: "모든 문서는 정중한 톤"
→ /marketing/ 폴더에서는 캐주얼 톤 (Directory가 Project 덮어씀)
```

## 2. 실제 CLAUDE.md 작성 예시 (PM 프로젝트용)

### 2.1 Project 레벨 CLAUDE.md

**파일 위치:** `/your-project-root/CLAUDE.md`

**예시 내용:**

```markdown
# TaskFlow - CLAUDE.md
# 제품: 팀 프로젝트 관리 SaaS

## 📌 제품 정의
- **제품명**: TaskFlow
- **설명**: B2B 프로젝트 관리 도구 (Asana/Monday.com 경쟁)
- **타겟 시장**: Series A-C 스타트업, 팀 규모 50-500명
- **홈페이지**: taskflow.io
- **현재 버전**: v2.3.1

## 🎯 북극성 지표 (True North Metric)
- **메인 지표**: Weekly Active Teams (WAT)
  - 현재: 850개 팀
  - 월간 성장률: +12%
  - 목표 (2026 연말): 1,200개 팀
  - 정의: 지난 7일 내 1명 이상이 활동한 팀

- **보조 지표**:
  - Monthly Recurring Revenue (MRR): $47K → $75K (목표)
  - Net Revenue Retention: 105% (업계 기준 100%)

## 👥 Personas

### Persona 1: Sarah - Enterprise Admin
- **역할**: HR/IT 담당자, 회사 인프라 책임
- **나이**: 38세
- **회사**: 500명 스타트업 (Series B)
- **Pain Points**:
  - SSO, SAML 2.0 필수 (보안)
  - 감사 로그 (compliance)
  - 사용자 권한 관리 (RBAC)
- **Success Metrics**:
  - 회사 전체 팀 100개 모두 활동 (WAT +100)
  - 월 결제 $3,500 (팀당 $35)
- **선호 커뮤니케이션**: 이메일, 정기 체크인 콜
- **결정 속도**: 느림 (2-3주 구매 결정)

### Persona 2: Mike - IC Engineer
- **역할**: 백엔드 엔지니어
- **나이**: 28세
- **회사**: 70명 스타트업 (Series A)
- **Pain Points**:
  - GitHub 연동 (작업 자동 연동)
  - 키보드 단축키 (빠른 작업)
  - 빠른 UI (Figma 수준의 응답성)
- **Success Metrics**:
  - TaskFlow에서 1일 평균 12개 작업 처리
  - GitHub pull request와 자동 매칭
- **선호 커뮤니케이션**: Slack, 커뮤니티 포럼
- **결정 속도**: 매우 빠름 (직접 사용 후 자동 채택)

### Persona 3: Alex - Team Lead (Product)
- **역할**: PM/팀 리드
- **나이**: 34세
- **회사**: 180명 스타트업 (Series A 후반)
- **Pain Points**:
  - 팀 가시성 (모든 프로젝트 한눈에)
  - 워크로드 밸런싱 (팀원 과부하 감지)
  - 리포팅 자동화 (월간 보고서)
- **Success Metrics**:
  - TaskFlow로 주간 스탠드업 시간 50% 단축
  - 팀 벌나우트 지표 감지 (과부하 팀원 신속 파악)
- **선호 커뮤니케이션**: 화상 회의, 문서 협력
- **결정 속도**: 중간 (팀과 협의 후 1주일 내)

## 📝 Writing Style (작문 스타일)
- **언어**: 한국어 기본, 기술 용어는 영문 병기
  - 예: "대시보드(Dashboard)를 커스터마이징할 수 있습니다"

- **문서 구조**: 항상 "문제→근거→제안→검증" 순서
```
  ❌ "대시보드 기능을 추가해야 합니다"
  ✅ "팀 리드 3명이 월간 리포트 작성에 4시간씩 소요 중입니다(문제).
     경쟁사(Asana) 대비 리포팅 자동화가 없는 상황입니다(근거).
     대시보드 자동화 기능을 Q2에 추가하면 월 4시간/팀원 절감
     가능합니다(제안). 3개 고객사 파일럿에서 확인됨(검증)."
```
- **수치 사용**: ALWAYS 출처 명시
```
  ❌ "대부분의 팀이 원하는 기능입니다"
  ✅ "고객 인터뷰 5/7명(71%)이 요청한 기능입니다"
  ✅ "2026년 Gartner 보고서에서 RBAC가 구매 결정 상위 3개
     기능으로 선정됨"
```
- **가정 vs 사실**: NEVER 가정을 사실처럼
```
  ❌ "사용자들은 이 기능을 좋아할 것입니다"
  ✅ "가정: 사용자들이 이 기능을 좋아할 것으로 예상됩니다.
     검증 방법: 고객 인터뷰 또는 A/B 테스트"
```
- **Tone & Voice**:
  - 대상이 Sarah(Admin)일 때: 정중하고 신뢰감 있는 톤
  - 대상이 Mike(Engineer)일 때: 직설적, 기술 중심
  - 대상이 Alex(PM)일 때: 데이터 기반, 비즈니스 임팩트 강조

## 🛠️ 기술 정보
- **데이터베이스**: PostgreSQL 14
- **백엔드**: Node.js + Express
- **프론트엔드**: React 18 + TypeScript
- **API**: REST (GraphQL 검토 중)
- **인프라**: AWS (EC2, RDS, S3)
- **모니터링**: DataDog, Sentry

### API Rate Limits (외부 참조)
- Slack: 60 requests/minute
- GitHub: 5000 requests/hour (authenticated)
- Stripe: 100 requests/second

## 📋 기본 규칙

### DO (필수)
- 모든 제안에 근거(데이터/인터뷰/경쟁사분석) 포함
- 수치는 항상 출처 명시
- 기능 영향도는 "구현 복잡도" + "사용자 영향" 2가지 평가
- 새 기능 제안 시 우선순위 근거 제시 (RICE: Reach/Impact/Confidence/Effort)

### NEVER (금지)
- 고객 이름, 연락처 외 개인정보 포함 금지
- 스프린트 목표, 팀 역할 변경 등 휘발성 정보 금지
- "나중에", "아마도" 같은 모호한 표현 금지
- 내부 임금, 펀딩 정보, M&A 희설 포함 금지

## 🔄 검토 주기
- **분기별 리뷰**: 이 파일 업데이트 (Q1/Q2/Q3/Q4)
- **팀 확대 시**: Persona 추가/수정
- **제품 피봇**: 새로운 북극성 지표 정의

## 🏷️ 태그 (참고: Claude가 검색할 때 사용)
#product-context #personas #taskflow #saas-pm
```

### 2.2 Directory 레벨 CLAUDE.md (선택사항)

**파일 위치:** `/project/docs/CLAUDE.md`

```markdown
# TaskFlow - 문서 작성 가이드

## 이 폴더의 역할
/docs/ 폴더의 모든 문서는 **대내외 공식 문서**입니다.
따라서 Global + Project CLAUDE.md 규칙을 따르되,
문서 특화 규칙을 추가합니다.

## 문서 타입별 구조

### PRD (Product Requirements Document)
1. 개요 (한 문단, 30초 읽기)
2. 문제 정의 (데이터/인터뷰 근거)
3. 목표 & 성공 기준
4. 사용자 스토리 (Persona별)
5. 기술 요구사항 (엔지니어 리뷰용)
6. 위험 & 의존성
7. 로드맵 & 마일스톤

### 회의록 (Meeting Notes)
- 참석자 (이름 + 직책)
- 결정사항 (Action item 형식)
- 데이터/근거 (논의된 수치)
- 다음 단계 (Owner + 마감일)
- 반박/이견 명시 (의견 불일치 기록)

## 금지 사항 (문서)
- ❌ 개인 의견 (근거 없는)
- ❌ 미래 추측 ("다음달에는...")
- ❌ 경쟁사 루머
- ❌ 임직원 성과 평가 언급

## 외부 공개 가능 여부
- PRD: 요약본만 (마케팅, 고객 공개 불가)
- 회의록: 공개 불가 (결정사항만 별도 공지)
- 리포트: 공개 가능 (고객 사례 제외)
```

## 3. 임시 규칙 테스트: 세션 내 1회성 규칙

CLAUDE.md를 수정하지 않고 **현재 세션에서만** 규칙을 시험하려면,
입력 시작에 `#` prefix를 붙입니다:

**사용 문법:**
```
# [임시 규칙]
> [질문]
```

**예시:**

```
Terminal 입력:

# 다음 분석에서는 모든 수치를 "예상치" vs "실제" 2가지로 구분해줘

> @q1-sales-report.csv 이번 분기 판매 트렌드를 분석해줘

Claude Code 응답:

📊 Q1 판매 분석 (예상 vs 실제)

| 월 | 예상 | 실제 | 차이 |
|----|------|------|------|
| 1월 | $250K | $245K | -2% |
| 2월 | $260K | $267K | +3% |
| 3월 | $270K | $289K | +7% |

---

다음 세션에서는:

> @q1-sales-report.csv 이번 분기 판매 트렌드를 분석해줘

Claude Code 응답:

📊 Q1 판매 분석
- 월 성장률: 평균 +9% ...

(# 임시 규칙이 적용되지 않음 - 세션 종료 시 사라짐)
```

**임시 규칙의 용도:**
- 새로운 스타일 테스트 (실제 CLAUDE.md 수정 전)
- 일회성 분석 특수성 처리
- 팀원별 선호도 차이 조정

## 4. CLAUDE.md에 포함되면 안 되는 것

❌ **금지 사항:**

| 항목 | 이유 | 대안 |
| --- | --- | --- |
| API 키, 토큰 | 보안 위험 | 환경 변수 사용 |
| 개인정보 (직원 주소, 전화) | 개인정보보호법 | "Persona" 이름만 사용 |
| 실시간 지표 (일일 매출) | 빠르게 변함 | 월간/분기별 스냅샷만 |
| 스프린트 목표, 할당 | 휘발성 정보 | 프로젝트 로드맵만 |
| 경영진 급여, 펀딩 정보 | 기밀 | 전혀 포함하지 말 것 |
| 내부 의견 불일치 | 민감도 높음 | 공식 결정만 |

## 6. CLAUDE.md 작성 체크리스트

### 첫 작성 시 (필수 섹션)

- [ ] 제품 정의 (이름, 설명, 타겟)
- [ ] 북극성 지표 (1개 메인 + 2-3개 보조)
- [ ] 3-5개 Persona (이름, 역할, Pain Point, Success Metric)
- [ ] Writing Style (톤, 구조, 수치 규칙)
- [ ] DO/NEVER 규칙 (5개씩)
- [ ] 기술 스택 (백엔드, 프론트엔드, 데이터베이스)

### 유지보수 (정기 리뷰)

- [ ] 분기별 1회 (북극성 지표 업데이트?)
- [ ] Persona 변화 감지 (새로운 타겟층?)
- [ ] 기술 스택 변경 기록
- [ ] 금지 항목 추가 (회의록 가정 금지 등)

### 팀 공유

- [ ] 신입 온보딩 시 CLAUDE.md 먼저 읽기
- [ ] 분기별 전체 팀 리뷰 (1시간)
- [ ] Persona 변경 시 모두에게 공지

## 7. 실습 과제

1. **자신의 프로젝트 CLAUDE.md 작성 (30분)**
  - 제품명, 북극성 지표 정의
  - Persona 3개 최소 작성
  - Writing Style 규칙 3개 최소

2. **사전 vs 사후 비교 (10분)**
  - CLAUDE.md 없이 1개 질문 → Claude에게 요청
  - 같은 질문을 CLAUDE.md 작성 후 반복
  - 응답 품질 차이 관찰

3. **임시 규칙 테스트 (5분)**
  - 한 가지 특수 분석 스타일에 `#` rule 적용
  - 다음 세션에서 규칙이 사라지는지 확인

| 항목 | 설명 |
| --- | --- |
| **목적** | 세션 간 지속되는 프로젝트 컨텍스트 |
| **위치** | `/project/CLAUDE.md` 또는 `~/.claude/CLAUDE.md` |

| **리뷰** | 분기별 1회 (3개월마다) |
| **포함** | 제품정의, Persona, Writing Style, DO/NEVER |
| **미포함** | API키, 개인정보, 실시간 지표, 급여 정보 |
| **임시 규칙** | `#` prefix로 세션 내 1회만 적용 |
| **핵심 효과** | "제품을 모르는 Claude"에서 "제품을 아는Claude"로 변신 |

## Part 3: Claude Memory (2026년 03/02 출시)

### 개요

**Claude Memory**는 CLAUDE.md의 보완 시스템입니다.

- **CLAUDE.md**: PM이 **수동으로** 작성하는 체계적 컨텍스트 (불변 규칙, Persona 정의)
- **Claude Memory**: Claude가 **자동으로** 학습하는 대화 맥락 (패턴, 선호도, 암묵적 규칙)

둘이 함께 작동하면, PM의 작업 방식을 Claude가 점점 더 잘 이해하게 됩니다.

---

### 1. Claude Memory의 3가지 계층

```
┌─────────────────────────────────────────────────┐
│ Level 1: Session Memory (1회 대화)              │
│ └─ 현재 대화 내에서만 유효                      │
│    • 5분 전에 PM이 제시한 요구사항              │
│    • 이전 메시지에서 언급한 문서명              │
│    • 현재 진행 중인 작업                        │
│                                                  │
│ Level 2: Project Memory (프로젝트별)            │
│ └─ 같은 프로젝트에서 반복되는 패턴               │
│    • "이 프로젝트는 항상 마일스톤 기반으로"    │
│    • "팀원 Alice는 항상 수치를 먼저 요청"      │
│    • "이 부서는 주간 보고를 목요일에 원함"     │
│                                                  │
│ Level 3: Organization Memory (조직 전체)        │
│ └─ 모든 프로젝트에서 공유하는 문화              │
│    • "항상 데이터 기반 의사결정"                │
│    • "한국어로 커뮤니케이션"                    │
│    • "보안과 개인정보는 최우선"                 │
└─────────────────────────────────────────────────┘
```

---

### 2. CLAUDE.md vs Claude Memory 비교

| 항목 | CLAUDE.md | Claude Memory |
|------|-----------|---------------|
| **작성자** | PM (수동) | Claude (자동) |
| **포함 내용** | 명시적 규칙 | 암묵적 패턴 |
| **보관 위치** | 파일 시스템 | Claude 서버 |
| **지속 기간** | 영구 | 구성 가능 (기본 90일) |
| **입력 방식** | 텍스트 편집 | 대화 학습 |
| **정확도** | 100% (명확함) | 70-90% (추론) |
| **예시** | "수치는 출처 필수" | "이 PM은 항상 비용 절감을 강조" |
| **변경 빈도** | 분기별 | 매 대화마다 갱신 |

**결합 효과:**
```
CLAUDE.md (명시적) + Claude Memory (암묵적)
     ↓
"제품을 정확히 이해하는 Claude"
```

---

### 3. Claude Memory가 학습하는 것들

#### 예시 1: 작업 스타일

```
대화 1:
PM: "이번 분기 목표를 5줄로 정리해줘"
Claude: "...5줄 정리..."

대화 2:
PM: "경쟁사 분석을 해줘"
Claude가 학습: "이 PM은 항상 간결하게 원한다" → 자동으로 5줄 형식 사용

CLAUDE.md 기반 규칙이 아님 (파일에 없음)
Claude Memory가 패턴으로 학습한 것
```

#### 예시 2: 의사결정 기준

```
대화 1:
PM: "새 기능 우선순위 제안해줘"
Claude: "사용자 영향도와 구현 복잡도를 고려하면..."

대화 2-10: (일관되게 "영향도 + 복잡도" 기준으로 분석)

대화 11:
PM: (아무 기준 제시 없음) "우선순위 결정해줘"
Claude가 학습: "이 PM은 영향도 + 복잡도로 판단한다"
→ 자동으로 같은 프레임워크 사용
```

#### 예시 3: 커뮤니케이션 톤

```
대화 1-5:
PM: "이것 좀 검토해줘", "저것도", "그리고 이것도"
→ 짧고 직접적인 요청

Claude가 학습: "이 PM은 빠른 피드백을 원한다"
→ 장황한 설명 없이 핵심만 제시
→ 대화 6부터 자동으로 간결하게 응답
```

### 4. Claude Memory 관리 (프로젝트 수준)

#### 4.1 메모리 유지 (자동)

Claude는 다음을 자동으로 기억합니다:

```
✅ 학습됨 (자동 저장):
- 프로젝트 이름, 제품, 타겟
- 팀 구조와 역할
- 의사결정 프레임워크 (ROI, RICE, 등)
- 선호하는 문서 형식
- 반복되는 용어와 약어
- PM의 질문 패턴
```

#### 4.2 메모리 리셋

메모리가 부정확하거나 오래되었다면:

```bash
# Claude와 대화 시작:
> 메모리 리셋: 이전 대화들이 부정확하므로 이 프로젝트에 대해
  처음부터 배워줘. 새로운 전제:
  - 타겟층: Enterprise (기존: SMB)
  - 모델: 마진율 기반 (기존: 사용자 수 기반)
```

Claude는 이후 대화에서 새로운 기준을 적용합니다.

#### 4.3 메모리 검사

메모리가 올바른지 확인:

```bash
> 내가 어떤 프로젝트에서 일하고 있다고 생각하는지 정리해줘
> (내 작업 스타일을 어떻게 파악했는지 설명해줘)

Claude 응답:
"프로젝트: TaskFlow (B2B 프로젝트 관리 SaaS)
 타겟: Series A-C 스타트업 (50-500명)
 북극성 지표: Weekly Active Teams

 학습한 작업 스타일:
 - 항상 RICE 프레임워크로 우선순위 결정
 - 수치는 출처와 함께 제시 필요
 - 일반 조언보다 데이터 기반 제안 선호
 - 응답은 3줄 이상 10줄 미만 선호"
```

### 5. CLAUDE.md + Claude Memory 통합 워크플로우

#### 시나리오: TaskFlow 프로젝트 온보딩

**Week 1: CLAUDE.md 설정**

```
PM이 실행:
1. /project/CLAUDE.md 작성 (제품명, Persona, Writing Style)
2. Claude와 공유
   > @CLAUDE.md 를 참고해서 앞으로 이 프로젝트에 대해 배워줄 준비해
```

**Week 2-8: 대화 반복 (Claude Memory 축적)**

```
대화들:
PM: "Q2 로드맵 제안해줘" (RICE 기준 사용)
Claude가 학습: "이 PM은 RICE를 선호한다"

PM: "경쟁사 Asana의 최신 기능을 분석해" (비용 영향도 중심)
Claude가 학습: "이 PM은 비용 절감 가능성을 중시한다"

PM: "다음 분기 KPI는?" (항상 근거 요청)
Claude가 학습: "데이터 기반 논의만 진행한다"
```

**Week 9 이후: 시너지 발생**

```
PM: (특정 지시 없이) "우리 로드맵은?"

Claude가 응답:
1. CLAUDE.md의 "북극성 지표"와 "Persona" 기준으로
2. Claude Memory의 "RICE 프레임워크" + "비용 절감" 중심으로
3. 자동으로 맞춤형 제안

→ 시간 절감, 정확도 향상
```

### 6. CLAUDE.md vs Claude Memory 선택 가이드

**CLAUDE.md에 써야 할 것:**

```
❌ "이 PM은 항상 간결한 응답을 원한다"
✅ "최대 200줄 범위 내에서 답변해주세요 (보고서 제외)"

❌ "우리 팀은 데이터를 좋아한다"
✅ "모든 주장에 출처(데이터, 인터뷰, 경쟁사분석) 필수"

❌ "팀 문화는 빠른 실행"
✅ "스프린트는 2주 단위, 배포는 금요일 18:00"
```

**Claude Memory가 자동으로 배우는 것:**

```
❌ "매번 CLAUDE.md에 작성" (과도)
✅ 자동 학습:
   • 이 PM의 질문 패턴
   • 반복되는 피드백 스타일
   • 선호하는 분석 프레임워크
   • 암묵적 가치관 (성장 vs 수익성)
```

### 7. 프라이버시 & 보안

#### Memory Scope (기본값)

- **개인 프로젝트**: Memory는 본인만 접근
- **팀 프로젝트**: Memory는 팀 내에서만 공유
- **공개 프로젝트**: Memory는 프라이빗 (외부 접근 불가)

#### 메모리에서 제외되는 정보

```
❌ 자동으로 삭제:
- API 키, 토큰, 비밀번호
- 개인정보 (직원 이름, 연락처, 급여)
- 기밀 정보 (펀딩, M&A, 내부 갈등)
- 실시간 민감 데이터 (일일 재무, 고객명)

✅ 학습되는 것:
- 제품 개념 및 전략
- 공개된 Persona
- 팀 구조와 역할 (직책까지만)
- 공개 기술 스택
```

Timeline:
Day 1 ────────────────────────── Day 90 ────────────────────────── Day 180
 ↓                                  ↓                                 ↓
Memory 시작                      Memory 보관됨              Memory 자동 리셋
(패턴 학습 중)                   (최대 효율)              (신규 패턴 시작)

대책:
- CLAUDE.md를 정기적으로 업데이트 (변화된 규칙 기록)
- 매 분기마다 "메모리 체크"로 확인
- 중요한 패턴은 CLAUDE.md로 옮기기
```

### 9. Best Practice: CLAUDE.md + Claude Memory 하이브리드

#### 패턴 1: 규칙은 CLAUDE.md, 패턴은 Memory

```
CLAUDE.md (파일에 고정):
```markdown
## Writing Style
- 모든 제안에 근거(데이터/인터뷰) 포함 필수
- 수치는 항상 출처 명시
- 가정은 절대 사실처럼 표현 금지
```

Claude Memory (자동 학습):
→ "이 PM은 항상 RICE 프레임워크로 우선순위 결정한다"
→ "비용 절감 관점의 제안을 많이 요청한다"
→ "일반적 조언보다 구체적 데이터 분석을 선호한다"
```

#### 패턴 2: 3개월마다 Memory를 CLAUDE.md로 이관

```
Quarter 1:
- Claude Memory가 "이 PM은 분석 중심"으로 학습

Quarter 2 검토:
- CLAUDE.md에 추가:
  ```
  ## Analysis Preference
  - 항상 최소 3가지 대안과 비교 분석 제시
  - 각 대안의 Risk/Benefit 명시
  ```

- 이제 새로운 팀원이 봐도 알 수 있음
```

#### 패턴 3: 온보딩 가속화

```
새 팀원이 올 때:

1단계: CLAUDE.md 공유 (30분)
   "우리 프로젝트는 이렇게 진행됩니다"

2단계: 이전 대화 요약 공유 (30분)
   "지난 분기 Claude Memory 학습 내용"
   = 암묵적 규칙들을 명시화

3단계: Claude와 대화 시작
   Claude가 이미 "프로젝트 문화"를 알고 있음
   → 신규 팀원도 빠르게 적응
```

### 10. 실전 체크리스트

#### 프로젝트 시작 시

- [ ] CLAUDE.md 작성 (제품, Persona, Style)
- [ ] Claude와 공유: "이 CLAUDE.md 학습해줄래?"
- [ ] 첫 주 대화 10-15개 진행 (Memory 축적 시작)

#### 월간 (1회)

- [ ] Claude에게: "현재까지 나를 어떻게 파악했는지 정리해줄래?"
- [ ] 피드백: 맞으면 OK, 틀리면 수정 지시
- [ ] CLAUDE.md 업데이트 (변화된 규칙)

#### 분기별 (3개월)

- [ ] 지난 분기 Memory 체크
- [ ] 학습된 패턴 중 중요한 것 → CLAUDE.md로 이관
- [ ] 새로운 Persona, 전략 추가 (있으면)
- [ ] Memory 리셋 필요 여부 판단

#### 연간 (12개월)

- [ ] CLAUDE.md 대폭 리뷰 (변화된 제품, 시장, 팀)
- [ ] Memory 마이그레이션 계획 (중요 패턴 보존)
- [ ] 팀 전체 공유 & 새로운 사이클 시작
