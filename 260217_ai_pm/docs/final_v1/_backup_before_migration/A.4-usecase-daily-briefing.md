# 유즈케이스: PM 일일 브리핑 자동화

> **PM 역량**: 운영 효율 / 정보 종합 / 의사결정 속도
> **난이도**: J → P
> **관련 문서**: [4.1-mcp-integration.md](./4.1-mcp-integration.md), [4.5-automation-n8n.md](./4.5-automation-n8n.md)

---

## 왜 PM에게 필요한가?

PM의 아침은 보통 이렇게 시작됩니다: Slack에서 밤사이 스레드 확인, Linear에서 어제 변경된 이슈 확인, Notion에서 금일 미팅 아젠다 확인, 이메일에서 외부 파트너 회신 확인. 이 4가지를 훑어보는 데만 30~45분이 걸립니다.

Claude Code + MCP를 사용하면 이 과정을 **5분짜리 브리핑 문서 하나**로 압축할 수 있습니다.

---

## 상황

당신은 AI Avatar 서비스의 PM입니다. 3개 스쿼드를 동시에 리딩하고 있고, 매일 아침 다음을 파악해야 합니다:

- **Slack**: #product-avatar, #bug-report, #cs-escalation 채널의 밤사이 주요 스레드
- **Linear**: 어제 상태 변경된 이슈 (Blocked, Done, In Review)
- **Notion**: 오늘 미팅 아젠다와 관련 문서
- **GitHub**: 어제 머지된 PR과 오늘 리뷰 대기 중인 PR

---

## 기존 방식 vs Claude 방식

```
기존 (30~45분, 매일 반복):
1. Slack 앱 열기 → 3개 채널 × 밤사이 스레드 읽기 (15분)
2. Linear 보드 열기 → 필터 변경 → 상태 변화 확인 (5분)
3. Notion 캘린더 확인 → 미팅 문서 클릭 (5분)
4. GitHub PR 탭 확인 (5분)
5. 머릿속에서 오늘 우선순위 정리 (5~10분)

Claude (5분):
1. 하나의 명령 → 4개 소스 자동 수집 → 브리핑 문서 생성
```

## 워크스루

### Step 1: MCP 연결 확인

브리핑 자동화의 전제 조건은 MCP 서버 연결입니다.

```
Terminal 입력:
> 현재 연결된 MCP 서버 목록을 보여줘

Claude Code 응답:
연결된 MCP 서버:
1. ✅ slack — Slack (3개 워크스페이스)
2. ✅ linear — Linear (product-avatar 팀)
3. ✅ notion — Notion (Product Wiki)
4. ✅ github — GitHub (avatar-platform 리포)

4개 서버 모두 정상 연결 상태입니다.
```

### Step 2: 일일 브리핑 요청

```
Terminal 입력:
> 오늘 아침 브리핑을 만들어줘. 다음 소스에서 정보를 수집해:

  1. Slack: #product-avatar, #bug-report, #cs-escalation에서
     어제 오후 6시 이후 메시지 중 스레드 3개 이상 달린 것

  2. Linear: product-avatar 팀에서 어제 상태가 변경된 이슈
     (특히 Blocked, Done으로 바뀐 것 위주)

  3. Notion: 오늘 날짜의 미팅 일정과 관련 문서 링크

  4. GitHub: avatar-platform 리포에서
     어제 머지된 PR + 오늘 리뷰 대기 PR

  브리핑 형식:
  - 🔴 즉시 확인 필요 (블로커, 에스컬레이션)
  - 🟡 오늘 중 확인 (상태 변경, 리뷰 대기)
  - 🟢 참고 사항 (완료된 것, FYI)

Claude Code 응답:
4개 소스에서 데이터를 수집하고 있습니다...

✅ Slack: 12개 관련 메시지 수집
✅ Linear: 8개 상태 변경 이슈 확인
✅ Notion: 3개 미팅 일정 확인
✅ GitHub: 5개 PR 확인
```

## PM 판단 포인트

### ❓ "Claude가 '권장 액션'을 제안하는데, 그대로 따라도 되나?"

이것이 이 유즈케이스의 가장 중요한 판단 지점입니다.

```
Claude가 잘하는 것:
- 여러 소스의 정보를 시간순/우선순위로 정리
- 관련 이슈와 미팅 간의 연결점 찾기 (예: A사 이슈 ↔ 오후 데모)
- 정량적 상태 요약 (PR 대기 시간, 스프린트 완료율)

PM이 직접 판단해야 하는 것:
- "핫픽스 우선순위"를 올릴지 vs 데모를 연기할지
- 블로커가 있는 이슈의 에스컬레이션 경로 결정
- A/B 테스트 결과에서 +12% 전환 vs +3% 이탈의 트레이드오프 판단
```

### ❓ "브리핑의 정확도를 어떻게 검증하나?"

```
검증 체크리스트:
1. [건수 확인] Slack 스레드 수가 실제와 일치하는가?
2. [시간 범위] "어제 오후 6시 이후"가 정확히 반영되었는가?
3. [누락 확인] 중요한 채널이나 이슈가 빠지지 않았는가?
4. [연결 정확도] 이슈와 PR의 매핑이 맞는가?

→ 처음 1주일은 수동 크로스체크를 병행하세요.
   정확도가 90% 이상으로 안정되면 자동화 단계로 전환합니다.
```

## 자동화: 매일 아침 자동 실행

1주일간 수동으로 검증한 후, 스케줄 태스크로 전환합니다.

```
Terminal 입력:
> /schedule-task

태스크 내용:
매일 평일 오전 8:30에 아침 브리핑을 생성해서
/docs/daily-briefing/ 폴더에 YYYY-MM-DD.md 파일로 저장해줘.

수집 소스:
1. Slack: #product-avatar, #bug-report, #cs-escalation
   (전일 18:00 이후 메시지)
2. Linear: product-avatar 팀 상태 변경 이슈
3. Notion: 금일 미팅 일정
4. GitHub: avatar-platform 리포 PR

형식:
🔴 즉시 확인 / 🟡 오늘 중 / 🟢 참고
각 항목에 관련 링크와 권장 액션 포함

Claude Code 응답:
스케줄 태스크를 생성합니다.

태스크명: daily-pm-briefing
스케줄: 0 8 30 * * 1-5 (평일 08:30)
출력: /docs/daily-briefing/YYYY-MM-DD.md
```

```

  추가 포함사항:
  - 지난 주 스프린트 완료율 트렌드 (최근 4주)
  - 미해결 블로커 누적 현황
  - 이번 주 예정된 릴리즈
  - 지난 주 고객 피드백 요약 (NPS 코멘트 포함)
```

### 이해관계자별 브리핑 분기

```
Terminal 입력:
> 같은 데이터로 3가지 버전의 브리핑을 만들어줘:

  1. PM용 (전체 상세): 현재 형식 유지
  2. 엔지니어링 리드용: 기술 이슈/PR 위주, 비즈니스 맥락 축소
  3. 경영진용: 핵심 지표 + 리스크 요약만 (A4 반 페이지 이내)
```

## 실습 과제

### Level 1 (J): 기본 브리핑 생성
자신의 Slack/Linear/Notion 데이터로 일일 브리핑을 생성하세요. Claude 응답에서 정보 누락 여부를 수동으로 확인하세요.

### Level 2 (P): 자동화 + 커스텀
스케줄 태스크를 설정하고, 1주일간 매일 브리핑의 정확도를 기록하세요. 누락/오류 패턴을 발견하면 프롬프트를 개선하세요.

### Level 3 (L): 팀 브리핑 시스템
팀 전체가 사용하는 브리핑 시스템을 설계하세요. 역할별 뷰(PM, 엔지니어, 디자이너), 에스컬레이션 자동 감지, 주간 트렌드 분석을 포함하세요.
