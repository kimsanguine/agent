## 왜 PM에게 필요한가?

PM이 작성하는 콘텐츠는 다양한 채널로 배포됩니다: 릴리즈 노트는 앱 내 공지, 블로그, 이메일, 소셜미디어로 나가고, 제품 업데이트는 세일즈 덱, 내부 위키, 파트너 포털에 각각 다른 형태로 전달됩니다.

같은 정보를 5개 채널에 맞게 바꾸는 작업은 단순하지만 시간이 많이 들고, 채널별 톤/포맷/길이 제약이 다르기 때문에 실수가 잦습니다. Claude Code는 **하나의 소스 콘텐츠에서 다채널 변형을 한 번에 생성**합니다.

---

## 상황

당신은 AI Avatar 서비스의 PM입니다. "실시간 아바타 스트리밍" 기능을 출시합니다. 다음 채널에 동시에 콘텐츠를 배포해야 합니다:

1. **인앱 공지**: 150자 이내, CTA 버튼 포함
2. **블로그 포스트**: 800~1,200단어, SEO 최적화
3. **이메일 뉴스레터**: 300단어, 개인화된 톤
4. **트위터/X 스레드**: 5~7개 트윗
5. **세일즈 원페이저**: A4 1장, 고객 가치 중심
6. **내부 Slack 공지**: 팀원 대상, 기술 세부사항 포함

## 기존 방식 vs Claude 방식

```
기존 (1~2일):
1. 릴리즈 노트 원본 작성 (1시간)
2. 블로그 초안 (2시간) → 마케팅팀 리뷰 (반나절)
3. 이메일 작성 (30분) → 디자인 검수 (2시간)
4. 소셜미디어 카피 (30분) → 마케팅팀 승인
5. 세일즈 원페이저 (1시간) → 세일즈팀 피드백
6. Slack 공지 (15분)
→ 총 6개 콘텐츠, 여러 팀 간 핸드오프 반복

Claude (2시간):
1. 소스 문서 작성 (30분)
2. 6개 채널 동시 변환 (10분)
3. PM 리뷰 + 채널별 미세 조정 (1시간)
4. 배포 (20분)
```

## PM 판단 포인트

### ❓ "6개 채널을 한 번에 만들면 품질이 떨어지지 않나?"

```
핵심은 "소스 문서의 품질"입니다.

Claude가 잘하는 것:
- 소스 정보를 채널 포맷에 맞게 변환
- 글자 수 제약 준수
- SEO 키워드 밀도 조절
- 톤 변환 (공식 ↔ 캐주얼 ↔ 기술적)

PM이 점검해야 하는 것:
1. [메시지 일관성] 6개 채널에서 핵심 메시지가 동일한가?
   → "500ms 이하, 60% 비용 절감, 10개 언어"가 모든 채널에 있는가?
2. [채널 적합성] 각 채널의 오디언스에 맞는 톤인가?
   → 트위터 ≠ 세일즈 원페이저 ≠ Slack
3. [사실 확인] 모든 수치가 소스와 일치하는가?
   → "60% 절감"이 검증된 수치인지
4. [법무 리뷰] "업계 최초", "보장" 같은 표현이 적절한지
```

### ❓ "어떤 채널부터 발행해야 하나?"

```
릴리즈 콘텐츠 발행 순서:

1. 내부 Slack (D-1) — 팀 사전 준비
2. 인앱 공지 (D-Day 09:00) — 기존 사용자 즉시 노출
3. 블로그 (D-Day 10:00) — SEO 인덱싱 시작
4. 이메일 (D-Day 11:00) — 기존 고객 도달
5. 소셜미디어 (D-Day 12:00) — 블로그 링크 포함
6. 세일즈 원페이저 (D-Day) — 세일즈팀 무장 완료

이 순서는 "내부 → 기존 고객 → 외부" 원칙입니다.
팀이 모르는 상태에서 고객 문의가 들어오면 안 됩니다.
```

## 실습 과제

### Level 1 (J): 3채널 적응
자신의 제품 업데이트 1건을 블로그, 이메일, 소셜미디어 3개 채널에 맞게 변환하세요. 채널별 톤과 포맷 차이를 비교하세요.

### Level 2 (P): 6채널 + 발행 계획
릴리즈 1건에 대해 6개 채널 콘텐츠를 생성하고, 발행 순서/시간 계획을 세우세요. 각 채널의 성과 지표(CTR, 오픈율 등)를 정의하세요.

### Level 3 (L): 콘텐츠 운영 시스템
릴리즈 → 콘텐츠 생성 → 리뷰 → 발행 → 성과 측정의 전체 파이프라인을 설계하세요. SKILL.md로 자동화하고, 릴리즈 유형별(Major/Minor/Hotfix) 채널 전략을 다르게 설정하세요.
