# PRD 작성: AI와의 소크라틱 대화

## 개요
Definition 단계에서 PM의 가장 어려운 작업은 **"올바른 문제에 올바른 해결책을 설정하는 것"**입니다.

Claude Code는 PRD를 "혼자 쓰는 문서"가 아닌 **"AI와의 소크라틱 대화로 만드는 문서"**로 변환합니다:
- AI가 PM의 가정을 질문으로 도전
- 여러 접근법을 병렬로 생성 및 비교
- 멀티 관점(엔지니어, 경영진, 사용자 리서처)의 동시 검토

이를 통해 PM은 **더 빠르고 더 견고한 PRD**를 작성할 수 있습니다.

## 소크라틱 질문법이란 무엇인가

### 정의

**소크라틱 질문법(Socratic Questioning)**은 고대 그리스 철학자 소크라테스가 사용한 대화법으로, **답을 직접 알려주는 대신 체계적인 질문을 통해 상대방 스스로 진리를 발견하도록 유도하는 방법**이다.

소크라테스는 스스로 "나는 아무것도 모른다"고 말하면서, 질문을 통해 상대방의 무의식적 가정, 논리적 모순, 검증되지 않은 믿음을 드러냈다. 이것이 2,400년이 지난 지금, PM의 제품 정의 과정에서 놀라울 만큼 유효하다.

### 소크라틱 질문의 6가지 유형

Richard Paul과 Linda Elder가 정리한 소크라틱 질문의 6가지 카테고리는 PRD 작성 과정에 직접 대응된다.

| 질문 유형 | 소크라틱 원형 | PM의 PRD 맥락 예시 |
| --- | --- | --- |
| **명확화 질문** | "그 말이 정확히 무슨 뜻인가?" | "온보딩 완료"의 정의가 정확히 뭔가? 어떤 액션을 기준으로? |
| **가정 탐색** | "당신은 무엇을 전제하고 있는가?" | 이 기능이 이탈율을 낮출 거라는 근거가 있나? |
| **근거 질문** | "그렇게 생각하는 이유는?" | 고객이 이 기능을 원한다는 데이터는 무엇인가? |
| **관점 전환** | "다른 관점에서 보면 어떤가?" | 엔지니어/경영진/유저 리서처는 이것을 어떻게 볼까? |
| **결과 탐색** | "그렇게 하면 무슨 일이 일어나는가?" | 이 기능을 출시하면 어떤 2차 효과가 생기는가? |
| **메타 질문** | "왜 이 질문이 중요한가?" | 이 PRD에서 가장 중요한 판단은 무엇이고, 왜 그런가? |

### 왜 PRD에 소크라틱 방식이 필요한가

대부분의 PRD 실패는 **"잘못된 답"이 아니라 "묻지 않은 질문"**에서 온다.

기존 PRD 작성 방식의 문제를 보자.

**기존 방식: 독백형 PRD**
```
PM이 혼자 생각 → 문서 작성 → 리뷰 요청 → 피드백 대기 → 수정
```
이 과정에서 PM의 가정은 **검증 없이 문서에 녹아든다.** "이 기능이 필요하다"는 전제 자체가 틀렸을 수 있지만, 누구도 그 질문을 하지 않는다. 리뷰어들은 이미 작성된 문서의 디테일에만 피드백하고, 근본적인 방향을 의심하지 않는 경우가 많다.

**소크라틱 방식: 대화형 PRD**
```
PM이 컨텍스트 제공 → AI가 가정을 질문 → PM이 답변하며 사고 정리
→ AI가 반론 제시 → PM이 검증 계획 수립 → 견고한 PRD 탄생
```
AI가 소크라테스 역할을 한다. PM의 가정에 끊임없이 "왜?"를 물으면서, PM 스스로 자신의 논리를 재검증하게 만든다.

### PM에게 소크라틱 방식이 특별히 효과적인 3가지 이유

**1. 확증 편향(Confirmation Bias) 차단**

PM은 자신의 가설을 좋아한다. "이 기능을 만들면 지표가 오를 것이다"라는 믿음이 생기면, 이를 뒷받침하는 데이터만 보게 된다. AI의 질문은 이 편향을 깨뜨린다. "그 데이터가 정말 이 가설을 지지하나요? 다른 해석은 없나요?"

**2. 이해관계자 시뮬레이션**

실제 팀에서 엔지니어, 경영진, 디자이너의 시간을 리뷰에 쓰기 전에, AI가 각 관점을 미리 시뮬레이션한다. PM은 "이 질문은 실제 리뷰에서도 나올 것이다"라는 사실을 미리 알고 대비할 수 있다.

**3. "묻지 못한 질문"의 발견**

PM이 혼자 생각하면 자신이 모르는 것을 모른다(unknown unknowns). AI의 체계적 질문은 PM이 **"아, 이 부분은 생각도 못했는데"**라고 말하게 만든다. 이것이 PRD 품질의 결정적 차이를 만든다.

### Claude Code에서의 소크라틱 대화 구현

Claude Code에서 소크라틱 대화를 활성화하는 핵심은 **프롬프트 설계**다.

**비효과적 프롬프트 (독백 유도):**
```
> 온보딩 개선 PRD를 작성해줘.
```
→ Claude가 일방적으로 PRD를 생성. PM의 가정 검증 없음.

**효과적 프롬프트 (대화 유도):**
```
> 온보딩 개선 PRD를 작성하려고 해.
  먼저 내가 놓치고 있을 수 있는 질문 5-7개를 해줘.
  특히 내 가정이 틀렸을 가능성이 있는 부분을 집중적으로.
```
→ Claude가 소크라테스 모드로 전환. PM의 가정을 드러내고 도전.

**CLAUDE.md에 소크라틱 모드를 기본값으로 설정:**
```markdown
## PRD 작성 규칙
- PRD 초안 작성 전에 반드시 PM에게 5-7개 질문을 먼저 한다
- 각 질문에는 "이 질문이 중요한 이유"를 함께 제시한다
- PM의 답변 후, 추가 반론/보완 질문을 2-3개 더 한다
- PM이 "이제 작성해줘"라고 할 때까지 질문 단계를 유지한다
```

> 💡 **핵심 원칙**: "좋은 PRD는 좋은 답이 아니라, 좋은 질문에서 시작된다." Claude를 '작성 도구'가 아닌 '질문 파트너'로 활용할 때 PRD 품질이 극적으로 높아진다.

## 섹션 1: 컨텍스트 기반 질문 생성

### 상황
PM이 온보딩 개선 PRD를 작성하려고 합니다.
현재 상황:
- 온보딩 완료율: 45%
- 목표: 미정
- 문제: 구체적으로 어디서 떨어지는지 불명확

기존 방식: 혼자 고민하며 PRD 초안 작성 (2-3일)
Claude Code 방식: AI의 질문으로 가정 점검 후 빠르게 작성

### Claude Code 워크플로우: 1단계 - 컨텍스트 풀 로딩

```bash
cc claude code

> @company-context.md @user-research/ @existing-prd-template.md
  우리 제품의 온보딩 개선 PRD를 작성하려고 해.
  먼저 내가 놓치고 있을 수 있는 질문 5-7개를 해줘.
```

### Claude의 응답 (실제 터미널 아웃풋)

```
Claude: 온보딩 개선 PRD 작성 전에 확인하고 싶은 질문들입니다:

## 핵심 질문들

### 1. 목표 설정 (가장 중요)
Q1: 현재 온보딩 완료율이 45%인데, 목표치는 몇 %인가요?
  → 이게 없으면 성공/실패 판단 불가
  → 만약 55%를 목표라면 노력 대비 효과가 작을 수 있음
  → 업계 벤치마크(보통 60-75%)와 비교했나요?

### 2. 정의의 명확성
Q2: "온보딩 완료"의 정의는 어떤 액션 기준인가요?
  → 예: 첫 프로젝트 생성? 팀원 초대? 첫 workflow 실행?
  → 이 정의에 따라 개선 방향이 완전히 달라짐

### 3. 플랫폼/세그먼트 우선순위
Q3: 모바일과 웹 중 우선순위는? 또는 사용자 세그먼트별 우선순위는?
  → 모바일 온보딩 완료율(35%) vs 웹(55%)이라면 전략이 달라짐
  → 소규모팀(5명) vs 대규모팀(50명) 온보딩도 다를 수 있음

### 4. 실패 분석
Q4: 기존에 시도했던 온보딩 개선 방안과 그 결과는?
  → 왜 이전 시도들이 성과를 내지 못했는가?
  → "비디오 튜토리얼 추가"를 이미 했다면, 그 효과는?
  → 배우기: 어떤 접근법이 우리 고객에게 작동하는가?

### 5. 연쇄 효과
Q5: 이 개선의 성공이 다른 지표(Retention, ARPU 등)에 미치는 영향은?
  → 온보딩 완료율 +10%가 정말 비즈니스 임팩트인가?
  → 기존 데이터: 온보딩 완료자 vs 미완료자의 30일 retention 차이는?
  → 이게 없으면 우선순위 투표에서 지를 수 있음

### 6. 비용-편익 제약
Q6: 이 개선에 투자할 수 있는 엔지니어링 자원은?
  → 3명-스프린트 vs 10명-스프린트로 완전히 스코프가 달라짐
  → 그에 맞춘 솔루션을 만들어야 함

### 7. 사용자 세그먼트별 전환 데이터
Q7: 온보딩의 어느 단계에서 주로 떨어지는가?
  → 가정: "UI가 복잡해서" vs 실제: "2단계(팀 초대)에서 70% 이탈"
  → 정확한 데이터가 있으면 솔루션을 훨씬 타이트하게 만들 수 있음

## 권장 다음 스텝

1. 위 질문 중 Q1, Q2, Q7에 먼저 답해주세요 (30분)
2. 그 다음 가설 기반 접근법 3가지를 생성하겠습니다
3. 각 접근법별 예상 효과, 복잡도, 리스크를 비교하겠습니다
```

### 💡 PM이 주목해야 할 포인트

**이 질문들의 목적은 "PM을 틀렸다고 지적"하는 것이 아닙니다.**
오히려 이런 질문 과정을 통해:
1. 내가 놓친 가정을 명확히 한다
2. 팀과 논의할 때 더 명확한 질문을 던진다
3. PRD가 더 견고해진다

## 섹션 2: 소크라틱 질문 - 상황 설정

### 상황
PM이 위 질문들에 답했습니다:
```
Q1 응답: 목표는 60% (현 45% → +15%)
Q2 응답: 온보딩 완료 = "첫 workflow 실행 후 저장"
Q3 응답: 웹 우선, 모바일은 나중
Q7 응답: 3단계(workflow 생성)에서 68% 떨어짐
```

이제 Claude Code는 이 컨텍스트 기반으로 더 깊이 있는 질문을 할 수 있습니다.

### Claude Code 워크플로우: 2단계 - 깊이 있는 탐색

```bash
> Q1-Q7 답변을 기반으로 온보딩 개선을 위한 구체적인 접근법을
  제안해줄 수 있어?

  각 접근법에 대해:
  - 온보딩 완료율 예상 향상 (45% → ?)
  - 구현 복잡도 (낮음/중간/높음)
  - 주요 리스크
  - 예상 소요 엔지니어링 비용

  을 비교해줘.
```

### Claude의 응답 (실제 터미널 아웃풋)

```
Claude: 온보딩 68% 이탈 지점이 "workflow 생성"이라는 데이터 기반으로
3가지 접근법을 제안합니다.

## 3가지 접근법 비교

### 접근법 A: "인터액티브 가이드" (추천도: 높음)

**개요:**
step-by-step 모달을 통해 사용자가 첫 workflow를 만들도록 유도.
실제 UI 위에서 하이라이트를 통해 각 버튼을 가리킨다.

**예상 효과:**
- 완료율: 45% → 58% (68% 이탈 중 ~50% 전환)
- 근거: Intercom, Pendo 벤치마크에서 가이드 추가 시 +10-15%

**구현 복잡도:** 중간
- 새로운 UI 추가 X (기존 UI 활용)
- 모달 로직 + 상태 추적 (1-2주)
- A/B 테스트 필요 (2주)

**주요 리스크:**
- 모달이 "성가신 팝업"으로 인식될 수 있음 → 스킵율 높음
- 모바일에서 UX 저해 (향후 모바일 개선 시 구조 변경 가능성)

**엔지니어링 비용:**
- 2명 엔지니어 × 2-3주 = 약 80 엔지니어링 시간
- 디자인: 40시간
- QA/배포: 20시간

**다음 검증 포인트:**
- "첫 workflow는 대부분 테스트용인가, 실무용인가?"
  → 만약 테스트용이면 이 유도가 의미 없을 수 있음

---

### 접근법 B: "템플릿 기반 빠른 시작" (추천도: 높음)

**개요:**
"workflow 처음이세요?"라는 화면에서 3가지 템플릿 제공.
사용자가 템플릿 하나를 선택하면 workflow가 자동 생성.
실제로 수정할 수 있고, 즉시 실행 가능.

**예상 효과:**
- 완료율: 45% → 62% (68% 이탈 중 ~70% 전환)
- 근거: 템플릿은 "blank page problem" 해결 → 더 높은 전환율

**구현 복잡도:** 낮음
- 기존 workflow 생성 UI를 "템플릿 선택" 스텝으로 변경 (1주)
- 템플릿 3개 생성/테스트 (1주)
- 배포: 1주

**주요 리스크:**
- 템플릿이 고객 use case와 안 맞으면 혼란 초래
- "템플릿에 내 프로세스를 맞춰야 한다"는 느낌 → 낮은 만족도

**엔지니어링 비용:**
- 1명 엔지니어 × 2-3주 = 약 60-80 엔지니어링 시간
- 디자인: 30시간
- 템플릿 기획/콘텐츠: 20시간

**다음 검증 포인트:**
- "고객들이 만들고 싶은 workflow의 TOP use case 3개는?"
  → 사용자 리서치에서 추출 필요

---

### 접근법 C: "교육 + 지원 강화" (추천도: 중간)

**개요:**
- 이메일 자동화 시리즈: 가입 후 3일, 7일, 14일에 workflow 튜토리얼 발송
- 헬프 센터 개선: workflow 생성 관련 FAQ/비디오 추가
- 라이브 챗 지원 강화: 처음 7일간 이메일 지원 우선순위 상향

**예상 효과:**
- 완료율: 45% → 52% (68% 이탈 중 ~20% 전환)
- 근거: 교육만으로는 UI/UX 문제를 해결하지 못함
- 주로 "이미 결정한" 고객만 도움이 됨

**구현 복잡도:** 낮음
- 이메일 시리즈 제작: 1주 (마케팅)
- 비디오 제작: 2주 (콘텐츠팀)
- 헬프 센터 업데이트: 1주 (기술 라이팅)

**주요 리스크:**
```

```
   - 목표: 45% → 60%

2. **다른 우선순위와 비교했는가?**

3. **체류율(Retention)에 미치는 영향**

   - 만약 차이가 30% vs 32%라면 사실 큰 임팩트 아님

4. **숨겨진 비용**
   - 마케팅팀의 템플릿 기획 비용? (사용자 리서치 기반)
```

**원인:**
- Claude가 기본적으로 사용자의 의견에 동의하려는 성향을 가짐
- "좋은 PRD를 작성하도록 도와달라"는 요청에 대해 Claude는 "지지" 모드로 작동할 수 있음
- 소크라틱 방식의 질문이 아니라 일방적 요청이었을 경우

**해결 방법:**

Claude에게 명시적으로 devil's advocate 역할을 요청합니다:

```bash
> 지금 내 PRD에 대해 이 관점에서 반론 3가지를 만들어줘:
  "이 PRD의 가정과 접근법이 틀렸다면?"

  각 반론마다:
  1. "왜 틀렸을 가능성이 있는가"
  2. "그럴 경우의 영향"
  3. "검증 방법"

  을 포함해줘.
```

더 구체적으로는:

```bash
> 이 PRD에서 가장 위험한 가정 TOP 3을 찾아줘.
  그리고 각각:
  - "이 가정이 틀렸을 확률"은?
  - "틀렸을 경우 이 전체 PRD가 실패할 확률"은?
  - "이 가정을 가장 빠르게 검증하는 방법"은?
```

**PM의 판단:**
- Claude의 초기 응답("좋은 방향입니다")에 만족하면 안 됨
- 항상 "비판적 피드백"을 별도로 요청해야 PRD 품질이 높아짐
- 팀 미팅 전에 "누군가는 이 PRD를 반대할 것"을 대비해야 함

---

### **상황 2: PRD가 너무 길어져서 핵심이 흐려지는 경우**

**증상:**
- Claude가 20페이지 분량의 상세한 PRD 작성
- 섹션: Context (3p), Problem (4p), Solution (5p), Implementation (4p), Success Metrics (4p)
- PM 자신도 이 문서를 다 읽기 힘들고, 팀도 핵심을 못 찾음
- "엔지니어들이 실제로 이 문서를 다 읽을까?"라는 의문

**원인:**
- Claude가 "완벽한" PRD를 만들려고 모든 상황을 고려하고 설명
- PM이 초기 요청에서 "상세한 PRD"를 요청하면 Claude는 모든 내용을 포함
- 현실의 팀은 "충분히 이해할 수 있는 최소 수준의 문서"를 원함

**해결 방법:**

PRD를 1페이지 버전으로 축약 요청:

```bash
> 이 PRD를 1페이지 요약으로 줄여줄 수 있어?
  "만약 엔지니어가 5분만 읽을 수 있다면 알아야 할 것"만 남겨.

  포함해야 할 것:
  - 문제가 뭔가?
  - 우리가 하려는 게 뭔가?
  - 왜 이게 중요한가?
  - 성공이 뭔가?

  제외해도 되는 것:
  - 배경 설명, 시장 분석
  - 여러 옵션 비교
  - 상세 기술 스펙
```

더 나아가 단계적 공개 문서 구조:

```bash
> PRD를 3단계로 만들어줄 수 있어?
  1. Executive Summary (1페이지): 위 내용
  2. Detailed PRD (5페이지): 완전한 버전
  3. Technical Appendix (별도): 엔지니어용 상세 스펙
```

**PM의 판단:**
- "좋은 PRD = 긴 문서"라는 오해 제거
- 실제로는 "핵심을 1페이지에 설명할 수 있는 것"이 좋은 PRD
- 상세 내용은 첨부 문서로 분리하되, 누구나 핵심은 5분 내에 이해 가능해야 함

### **상황 3: 멀티에이전트 리뷰 결과가 서로 완전히 상충하는 경우**

**증상:**
- 🔧 엔지니어 의견: "이 기능은 기술적으로 불가능합니다. 아키텍처 변경 필요."
- 💼 경영진 의견: "고객 요청이 많으니 빨리 배포해야 합니다. 3주 내 출시 필수."
- 🔍 사용자 리서처: "이건 실제 사용자 통증이 아닙니다. 먼저 검증하세요."
- PM: "이 3가지 의견을 어떻게 조화시키지?"

**원인:**
- 각 역할이 다른 우선순위를 가지고 있음 (기술성 vs 비즈니스 vs 사용자 가치)
- 이건 실은 "Claude의 문제"가 아니라 "조직의 현실"
- 하지만 이 상황을 제대로 이해하고 의사결정을 내리는 것이 PM의 핵심 역할

**해결 방법:**

먼저 Claude에게 명확히 상충을 정리하도록 요청:

```bash
> 세 리뷰어의 의견이 완전히 다르네.

  각 입장의 근거를 정리해줄 수 있어?
  1. 엔지니어가 "기술적으로 불가능"이라고 하는 이유 TOP 3
  2. 경영진이 "3주 내 배포"를 주장하는 이유 TOP 3
  3. 리서처가 "검증부터"를 권고하는 이유 TOP 3

  그리고 각각 "만약 우리가 이 의견을 무시하면 어떤 리스크가 있을까?"도 함께.
```

다음 단계로 의사결정 프레임워크:

```bash
> 이 상황에서 PM이 취할 수 있는 의사결정 옵션 4가지를 만들어줘:

  옵션 1: 엔지니어 의견 우선
  → 어떤 가정 하에 가능한가? 비용은?

  옵션 2: 경영진 의견 우선
  → 엔지니어 반발을 어떻게 처리할 것인가?

  옵션 3: 리서처 의견 우선
  → 경영진을 납득시킬 방법은?

  옵션 4: 하이브리드 (절충)
  → 각 그룹의 최소 요구사항은 무엇인가?
```

**PM의 의사결정:**
- 이 상황은 "데이터로 해결할 수 없고", 조직의 우선순위와 위험을 판단해야 함
- 일반적으로:
  - 기술적으로 불가능하면 불가능 (엔지니어 우선)
  - 하지만 "불가능"이 정말인지 확인 (대체 기술 있나?)
  - 경영진의 "3주" 요청이 정말 hard deadline인지 확인
  - 그 사이에 "최소 검증" (사용자 5명 인터뷰 등)을 병렬로 진행 가능한가?
- 최종 결정: 팀 합의 기반이 아니라 PM이 "이 리스크를 감수합니다"라고 선언하고 결정
