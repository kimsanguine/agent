# 전략 문서 작성: Rumelt의 전략 커널

## 개요
Definition 단계의 최고 수준 작업은 **전략 수립**입니다.

이 챕터에서는 Richard Rumelt의 **전략 커널(Strategy Kernel)** 프레임워크를 사용합니다. 이것은 좋은 전략의 핵심 구조이며, Claude Code와 결합하면 PM이 데이터 기반의 견고한 전략을 1일 내에 수립할 수 있습니다.

---

## Rumelt의 전략 커널이란

### 배경: "Good Strategy Bad Strategy"

Richard Rumelt은 UCLA Anderson School의 전략 교수로, 2011년 저서 *"Good Strategy Bad Strategy"*에서 전략에 대한 근본적 재정의를 제안했다. 그의 핵심 주장은 이렇다.

> *"대부분의 전략이라고 불리는 것은 전략이 아니다. 그것은 목표의 나열이거나, 동기 부여의 구호이거나, 뜬구름 같은 비전일 뿐이다."*

Rumelt이 정의한 **나쁜 전략(Bad Strategy)**의 4가지 특징은 PM이라면 반드시 경계해야 할 것들이다.

| 나쁜 전략의 특징 | PM 맥락의 예시 |
| --- | --- |
| **허울 좋은 언어 (Fluff)** | "AI-first 플랫폼으로 혁신적 경험 제공" — 구체적으로 뭘 한다는 건가? |
| **문제 직시 회피** | 이탈율이 올라가고 있는데, "새 기능 출시"로 덮으려는 로드맵 |
| **목표를 전략으로 착각** | "올해 ARR 2배 성장" — 이건 목표지, 어떻게 도달할지의 전략이 아니다 |
| **나쁜 전략적 목표** | 서로 충돌하는 10개 이니셔티브를 "모두 중요"하다고 나열 |

### 전략 커널의 3가지 구성 요소

Rumelt이 제시한 **좋은 전략의 핵심 구조**가 바로 **전략 커널(Strategy Kernel)**이다. 세 가지 요소로 이루어진다.

```
┌─────────────────────────────────────────────────────┐
│                   전략 커널 (Strategy Kernel)          │
│                                                       │
│  ┌─────────────┐                                     │
│  │  Diagnosis   │  현실을 직시하고, 핵심 문제를 정의한다   │
│  │  (진단)      │  "우리가 직면한 도전의 본질은 무엇인가?" │
│  └──────┬──────┘                                     │
│         ↓                                             │
│  ┌─────────────┐                                     │
│  │  Guiding     │  문제를 어떻게 접근할지의 원칙을 세운다  │
│  │  Policy      │  "어떤 방향으로, 무엇을 하고 무엇을      │
│  │  (안내 정책)  │   하지 않을 것인가?"                    │
│  └──────┬──────┘                                     │
│         ↓                                             │
│  ┌─────────────┐                                     │
│  │  Coherent    │  정책을 실행하는 구체적이고 일관된 행동   │
│  │  Actions     │  "실제로 누가, 무엇을, 언제 하는가?"     │
│  │  (일관된 행동) │                                     │
│  └─────────────┘                                     │
└─────────────────────────────────────────────────────┘
```

**1. Diagnosis (진단)**

진단은 "현실을 정확히 보는 것"이다. 현재 상황의 복잡한 요인들을 걸러내고, **핵심 도전(critical challenge)**을 식별하는 과정이다.

좋은 진단의 조건:
- 문제의 **인과관계 체인**을 드러낸다 (A → B → C 형태)
- **통제 가능한 것**과 **통제 불가능한 것**을 분리한다
- 데이터에 기반하되, **데이터가 말해주지 않는 것**도 인식한다
- "아무것도 안 하면 어떻게 되는가?"라는 질문에 답한다

나쁜 진단의 예: "시장이 커지고 있으니 성장 기회가 있다" (누구에게나 적용 가능 → 전략적 의미 없음)
좋은 진단의 예: "우리의 신규 고객 획득이 3개월 연속 감소하고 있으며, 주 원인은 포지셔닝 모호함으로 인한 차별화 실패다"

**2. Guiding Policy (안내 정책)**

안내 정책은 "방향을 선택하는 것"이다. 이것은 구체적 행동이 아니라, **행동을 결정하는 원칙**이다.

좋은 안내 정책의 조건:
- **선택과 포기**가 명확하다 ("우리는 A를 한다"는 곧 "B, C는 하지 않는다"를 의미)
- **레버리지 포인트**를 활용한다 (작은 행동 → 큰 효과)
- 조직의 **강점을 약점이 아닌 곳에 집중**시킨다
- 경쟁자와 **다른 경로**를 택한다

나쁜 정책의 예: "최고의 제품을 만들자" (모든 회사가 이렇게 말함)
좋은 정책의 예: "우리는 분산팀의 비동기 신뢰에 집중하고, 실시간 협업과 전사 통합 플랫폼은 하지 않는다"

**3. Coherent Actions (일관된 행동)**

일관된 행동은 "정책을 현실로 바꾸는 것"이다. 핵심은 **"일관된(coherent)"**이라는 단어다. 각 행동이 서로를 보강하고, 안내 정책과 정렬되어야 한다.

좋은 일관된 행동의 조건:
- 각 행동이 **서로를 강화**한다 (시너지)
- 안내 정책과 **직접 연결**된다
- **순서와 타이밍**이 설계되어 있다 (어떤 것을 먼저?)
- 자원 배분이 **집중**되어 있다 (모든 곳에 조금씩이 아닌, 핵심에 집중)

나쁜 행동 목록의 예: "10개 이니셔티브를 동시에 진행하고, 각각에 1명씩 배치"
좋은 행동 목록의 예: "Q1에 온보딩 개선에 3명 집중 → Q2에 API 확장으로 개발자 생태계 구축 → Q3에 프리미엄 티어 출시"

### 전략 커널의 PM 적용: 왜 이것이 효과적인가

PM이 전략을 수립할 때 가장 흔한 실수 3가지와 전략 커널의 해결 방식:

| PM의 흔한 실수 | 전략 커널의 해결 | Claude Code 활용 |
| --- | --- | --- |
| 목표 나열을 전략으로 착각 | Diagnosis가 목표보다 앞에 온다 — 먼저 현실을 직시 | 데이터를 입력하면 Claude가 인과관계 체인을 시각화 |
| "모두 중요하다"는 무결정 | Guiding Policy가 선택/포기를 강제 | 3가지 전략 옵션을 병렬 생성하고, 각각의 trade-off 비교 |
| 실행 단계의 비일관성 | Coherent Actions가 상호 보강을 검증 | Devil's Advocate 모드로 행동 간 모순을 찾아냄 |

### Rumelt 전략 커널 vs 다른 전략 프레임워크

| 프레임워크 | 핵심 특징 | Rumelt 커널과의 차이 |
| --- | --- | --- |
| SWOT | 상황 분석 4분면 | SWOT은 진단의 **입력 도구**일 뿐, 전략 자체가 아님 |
| Porter의 Five Forces | 산업 구조 분석 | 산업 분석에 강하지만, "그래서 우리가 뭘 해야 하나?"에 약함 |
| Blue Ocean Strategy | 새 시장 창출 | 시장 창출에 집중하지만, 기존 시장 대응에는 부족 |
| OKR | 목표-핵심결과 | OKR은 **실행 관리 도구**이며, 전략 수립 도구가 아님 |
| **Rumelt 전략 커널** | 진단→정책→행동 | **"왜 이것을 하는가"부터 시작**, 선택/포기가 명확, 일관성 검증 |

> 💡 **전략 커널은 다른 프레임워크를 대체하는 것이 아니라, 통합하는 상위 구조다.** SWOT은 진단 단계의 입력으로, OKR은 일관된 행동의 실행 관리 도구로 함께 사용할 수 있다.

## Claude Code와 전략 커널의 결합

Claude Code는 전략 커널의 각 단계를 다음과 같이 가속한다:

| 전략 커널 단계 | 기존 방식 | Claude Code 방식 |
| --- | --- | --- |
| **Diagnosis** | 여러 소스 수동 취합 (3-5일) | @파일 참조로 데이터 통합 → 인과관계 자동 추출 (30분) |
| **Guiding Policy** | 팀 워크숍 2-3회 (1-2주) | 3가지 전략 방향 병렬 생성 + trade-off 비교 (1시간) |
| **Coherent Actions** | 개별 팀별 실행 계획 (1주) | Devil's Advocate로 행동 간 모순 검증 + 순서 설계 (1시간) |

단, 각 단계에서 **PM의 직접판단**이 필수다:
- Diagnosis: "이 인과관계가 우리 조직의 맥락에서 맞는가?"
- Guiding Policy: "이 선택이 이해관계자들에게 받아들여질 수 있는가?"
- Coherent Actions: "우리 팀이 이 순서로 실행할 수 있는가?"

## 섹션 1: 진단 단계 (Diagnosis)

### 상황
PM이 새로운 제품/기능의 전략을 수립하려고 합니다.
현재 수집된 자료:
- 시장 데이터: 경쟁사, 시장 규모 추이
- 내부 지표: 사용 현황, 수익, retention
- 고객 피드백: 인터뷰, 설문, 지원팀 노트

문제: 이 자료들을 보면서 "정말 핵심 문제가 뭐지?"라는 혼란

### Claude Code 워크플로우: 1단계 - 데이터 기반 진단

```bash
cc claude code

> @market-data.md @internal-metrics.csv @customer-feedback/ @competitor-analysis.md
  우리 제품의 전략을 수립하려고 해.

  먼저 이 자료들을 종합해서 우리가 직면한 핵심 문제를 진단해줘.

  특히:
  1. 우리의 현재 상황 (강점/약점/기회/위협)
  2. 통제 가능한 원인 vs 외부 요인 분리
  3. 만약 이 상황을 놔두면 향후 6-12개월 어떻게 될까?
```

### Claude의 응답 (실제 터미널 아웃풋)

```
Claude: 당신의 비즈니스를 종합 분석했습니다.

## 현 상황 분석

### 수량적 신호
- 현재 MRR: $250K (전년 대비 +15% YoY 증가)
- 신규 고객 수: 월 12건 (3개월 전 18건에서 감소)
- 이탈율: 5% 월 (초기 3%에서 상승)
- 평균 체류 기간: 14개월 (이전 18개월에서 감소)

### 정성적 신호
- 고객 피드백: 만족도 3.8/5 → 주요 불만 "기능 부족", "대체 도구 사용 중"
- 경쟁사 분석: 2개 신규 경쟁사 진입, 기존 경쟁사 가격 인하
- 팀 피드백: "뭘 해야 할지 모르겠다", "우리의 차별점이 뭐지?"

## Diagnosis: 우리는 "성장 정체(Growth Plateau)" 단계에 있습니다

### 핵심 문제의 체인
```
        신규 고객 획득 ↓ (18 → 12)
               ↓
        이탈율 ↑ (3% → 5%)
               ↓
        포지셔닝 모호 (고객이 우리를 뭐라고 생각하는가?)
               ↓
        경쟁 환경 변화에 대응 불가
               ↓
        6개월 후: MRR 정체 또는 하락 예상
```
### 각 요인 분석

**1. 신규 고객 획득 감소 (18 → 12, -33%)**

*외부 요인 (통제 불가):*
- 시장 성숙: CAC(고객 획득 비용) 상승 (업계 평균)
- 경쟁 증가: 마케팅 비용 상승

*내부 요인 (통제 가능):*
- ❌ 마케팅 메시지가 명확하지 않음
  (고객이 "우리가 누구인지" 모름 → 경쟁사와 구분 못 함)
- ❌ 제품-시장 적합 약화
  (새로운 세그먼트와 안 맞음)
- ❌ 콘텐츠 마케팅 부족
  (교육 기반 획득 전략 없음)

**2. 이탈율 상승 (3% → 5%)**

*직접 원인:*
- 고객 만족도 하락: 3.9/5 → 3.8/5 (작지만 의미 있는 감소)
- 주요 불만: "기능 부족" (43%) + "경쟁사가 더 나음" (28%)

*근본 원인:*
- ❌ 제품 로드맵이 고객 니즈와 정렬되지 않음
- ❌ 경쟁사가 더 빠르게 진화 중
- ❌ 고객에게 "왜 우리를 써야 하는가"의 명확한 이유 부족

**3. 포지셔닝 모호 (숨겨진 원인)**

*현재 우리의 포지셔닝:*
"협업 도구" (너무 광범위 → 누구나 그렇게 말함)

*고객이 우리를 봤을 때:*
- "협업 도구 중 하나? 그냥 Slack/Asana 쓰면 되지 않나?"
- 명확한 "왜 나?"가 없음

## 전략적 의미: 우리는 "차별화" 재정의가 필요합니다

### 6-12개월 시나리오 (현재 추세 유지 시)

| 지표 | 현재 | 6개월 후 (예상) | 12개월 후 (예상) |
|------|------|------------|------------|
| 월 신규 고객 | 12 | 10 | 8 |
| 월 이탈율 | 5% | 6% | 7% |
| MRR | $250K | $240K | $220K |
| 경쟁력 | 약함 | 더 약해짐 | 위험 |

**"아무것도 안 하면" → 쇠락의 길**

## 근본 질문: 우리는 누구이고, 누구를 위한가?

### 현재 (모호)
- 누구: "협업 도구 회사"
- 누구를 위해: "팀? 기업? 개인?"

### 필요한 선택 (명확해야 함)
- 누구: "OOO를 해결하는 도구"
- 누구를 위해: "특정 고객 (예: 원격 팀, 마케팅팀 등)"

[진단-분석.md 파일이 생성되었습니다]
```

### 💡 PM이 주목해야 할 포인트

1. **"Growth Plateau"라는 진단의 신뢰도**
  - 정말 이것이 핵심 문제인가?
  - 혹은 단순히 계절성/순환인 건 아닌가?

2. **인과관계 검증**
  - "포지셔닝 모호 → 신규 고객 감소"라는 가정이 맞는가?
  - 아니면 다른 원인 (예: 마케팅 예산 감소)은 없었나?

3. **다음 단계**
  - 이 진단을 토대로 어떤 전략 방향을 탐색할 것인가?

## 섹션 2: 전략 옵션 생성 (Guiding Policy)

### 상황
진단이 명확합니다: "포지셔닝 재정의를 통한 차별화 필요"

이제 PM이 선택해야 할 것은:
- 우리의 새로운 포지셔닝은?
- 누구를 target으로 할 것인가?
- 어떤 기능/특성을 강조할 것인가?

### Claude Code 워크플로우: 2단계 - 전략 방향 생성

```bash
> 위 진단에 기반해서 3가지 전략 방향을 제안해줘.

  각 방향에 대해:
  - 핵심 선택 (우리는 OOO를 잘 하는 회사가 될 것이다)
  - 핵심 포기 (우리는 OOO를 하지 않을 것이다)
  - 이 선택으로 예상되는 trade-off
  - 우리가 이걸 이길 수 있는 이유 (sustainable advantage)

  을 각각 설명해줘.
```

### Claude의 응답 (실제 터미널 아웃풋)

```
Claude: 3가지 전략 방향을 생성했습니다.

## 전략 방향 1: "분산팀의 신뢰 구축 도구"

### 핵심 선택
**우리는: 원격/분산팀이 비동기로 일하면서도 투명성과 신뢰를 유지하는 도구가 될 것이다**

- 시간대 다른 팀원들이 "지금 누가 뭘 하고 있는가"를 신뢰하고 확인할 수 있도록
- 진행 상황이 자동으로 기록되고, 나중에 "뭐 됐어?"라고 물을 필요 없도록
- 마이크로매니징 없이도 팀장이 팀의 건강을 알 수 있도록

### 예시 고객
- 리모트 팀 (5-30명)
- 글로벌 분산 팀
- 프리랜서 협력 환경

### 핵심 포기
- "실시간 협업" (Slack, Teams 경쟁 X)
- "복잡한 프로젝트 관리" (Asana, Jira 경쟁 X)
- "전사 통합 플랫폼" 목표 (일단 하나 분야 깊게)

### 이 선택으로 이기는 이유
- **강점 1**: 기존 고객들이 "비동기 문제"로 불만 (43% 불만)
  → 우리가 이미 이 고통을 이해함

- **강점 2**: "신뢰 기반 관리"는 마이크로매니징의 반대
  → 원격팀 리더들의 근본적 니즈 (미충족)

- **강점 3**: 구현 간단 (복잡한 PM 도구 아님)
  → 빠른 도입, 높은 만족도

### 예상 Trade-off

| 항목 | 강점 | 약점 |
|------|------|------|
| 시장 크기 | 중간 (원격팀 증가) | 상대적 작음 |
| 경쟁 강도 | 낮음 (틈새) | 틈새라 대기업 관심 X |
| 개발 복잡도 | 낮음 | - |
| 고객 수익성 | 높음 (충성도 높음) | - |

## 전략 방향 2: "마케팅팀의 워크플로우 자동화"

### 핵심 선택
**우리는: 마케팅팀의 반복적 협업을 자동화하고, 캠페인-실행-측정을 한곳에서 하는 도구가 될 것이다**

- 캠페인 아이디어 → 실행 → 성과 측정까지 끊김 없이
- 마케팅팀 내에서 "누가 뭘 해야 하는가"가 자동으로 흘러나옴
- 반복적인 캠페인 구조를 재사용 가능하도록

### 예시 고객
- 마케팅팀 (특히 5-20명 규모)
- SaaS 마케팅팀
- 이커머스 마케팅팀

### 핵심 포기
- "전사 협업 도구"로서의 위치
- "기술팀/영업팀/기타"와의 통합 (마케팅 전용)
- "초대형 엔터프라이즈" 고객 (지금은 범위 외)

### 이 선택으로 이기는 이유
- **강점 1**: 마케팅팀은 "무언가가 반복되는" 일이 많음
  → Workflow 도구가 가장 효과적

- **강점 2**: 마케팅팀은 "자동화에 민감"
  → 시간 절감 = 수익 증가

- **강점 3**: Vertical 집중하면 "산업별 템플릿" 만들기 쉬움
  → 높은 도입율

### 예상 Trade-off

| 항목 | 강점 | 약점 |
|------|------|------|
| 시장 크기 | 크다 (모든 회사 마케팅팀) | 매우 경쟁 심함 (HubSpot, Mailchimp 등) |
| 경쟁 강도 | 높음 | - |
| 개발 복잡도 | 높음 (산업별 맞춤 필요) | - |
| 고객 수익성 | 중간 (price-sensitive) | - |

## 전략 방향 3: "중소기업의 첫 협업 도구"

### 핵심 선택
**우리는: 중소기업(10-50명)이 구글 독스, 엑셀, 메일에서 벗어나
처음으로 "일을 체계적으로 관리"하도록 도와주는 도구가 될 것이다**

- 매우 간단함 (온보딩 30분)
- "이미 하고 있는 일"을 도구로 옮기는 것이 주 목적
- 기업의 성장 단계에 맞춰 진화 (SMB → Mid-market)

### 예시 고객
- 스타트업 (10-30명)
- 중소 사이드 비즈니스
- 가족 사업

### 핵심 포기
- "엔터프라이즈급 기능"
- "높은 커스터마이징"
- "고도의 통합"

### 이 선택으로 이기는 이유
- **강점 1**: 새 고객 초기 비용 낮음 (free tier 쉬움)
  → CAC 회수 빠름

- **강점 2**: SMB는 "어떤 도구가 필요한지" 모름
  → 우리의 가이드에 따를 가능성 높음

- **강점 3**: 일단 도입 → 성장할 때까지 함께 성장
  → 높은 LTV

### 예상 Trade-off

| 항목 | 강점 | 약점 |
|------|------|------|
| 시장 크기 | 매우 크다 (모든 SMB) | 경쟁 intense (Asana, Monday 등 SMB 타겟) |
| 경쟁 강도 | 높음 | - |
| 개발 복잡도 | 낮음 | - |
| 고객 수익성 | 낮음 (낮은 ARPU) | - |
```

```
## 5가지 실패 시나리오

### 시나리오 1: "분산팀의 원격근무 트렌드가 역행한다"

**상황:**
주요 경제권(US, EU)이 "RTO(Return to Office)"로 돌아가면서
분산팀의 비중이 감소하는 경우

**발생 확률:** 중간 (30%)
- 이미 일부 대기업 (Amazon, Google 등)이 RTO 시작
- 하지만 일부 업계(테크, 마케팅)는 계속 원격

**영향도:** 높음
- 시장 규모 축소
- 우리의 "비동기 신뢰" 가치 감소

**조기 신호:**
- 분기별 신규 고객의 "원격" 비율 추적
  (예: Q1은 80% 원격 고객 → Q2는 65% → Q3은 45%로 감소)
- 기존 고객의 "RTO로 전환" 안내

**대응 방안:**
- 하이브리드 팀 용도로 마케팅 메시지 조정
- "비동기 신뢰"를 "전 시간대 투명성"으로 재정의
  (원격/하이브리드 둘 다 적용 가능하도록)

---

### 시나리오 2: "Slack/Teams가 우리 기능을 복제한다"

**상황:**
Slack이 "Slack Canvas"나 "Status Update" 같은
workflow 기능을 강화해서 우리와 경쟁하는 경우

**발생 확률:** 높음 (60%)
- Slack은 매년 새로운 기능 추가 (history)
- 통신 도구 × workflow = 자연스러운 통합

**영향도:** 매우 높음
- "왜 Slack 플러그인을 쓰면 되는데 우리 도구를?"
- 우리 고객의 통합비용 증가

**조기 신호:**
- 대형 경쟁사(Slack, Teams)의 제품 로드맵 모니터링
- 고객 이탈 사유에서 "Slack으로 충분하다"는 언급 빈도

**대응 방안:**
- 우리의 "전문화" 강조: 우리는 Slack보다 비동기 신뢰에 깊게 최적화됨
- API 통합 강화: Slack과 경쟁이 아니라 complement하도록
  (우리의 데이터 → Slack 채널에 자동 푸시)

---

### 시나리오 3: "고객 수익성이 예상보다 낮다"

**상황:**
분산팀은 보통 작은 팀(5-15명)이고, 이들의 ARPU는
우리가 기대했던 것보다 낮은 경우

**발생 확률:** 중간 (40%)
- 분산팀 = 보통 스타트업/작은 회사
- "가격 민감" 고객군

**영향도:** 높음
- CAC 회수 기간 길어짐
- 성장 속도 느려짐

**조기 신호:**
- 새 고객의 평균 ARPU 추적
  (계획: $200/mo → 실제: $100/mo)
- Churn 분석에서 "비용" 사유 증가

**대응 방안:**
- 프리미엄 티어 설계: "분산팀" 맞춤형 고급 기능
  (예: 글로벌 시간대 관리, 언어별 협업)
- Upsell 전략: 팀 성장하면서 자연스럽게 상향
- 산업 집중: 높은 ARPU 산업(금융, 법무)의 분산팀 타겟

---

### 시나리오 4: "우리의 차별성이 '분산팀'이 아니라 '간단함'이었다"

**상황:**
고객 인터뷰를 통해 깨달은 것:
고객들이 우리를 선택한 이유는 "비동기 신뢰"가 아니라
"Asana보다 훨씬 쓰기 간단하다"는 것

**발생 확률:** 중간-높음 (50%)
- 정성 인터뷰는 편향될 수 있음
- "진정한" 선택 이유는 다를 수 있음

**영향도:** 매우 높음
- 우리의 전략 기초 자체가 흔들림
- "분산팀"으로 집중 → 틀린 방향일 수 있음

**조기 신호:**
- 신규 고객 획득 이유 추적 (정성 + 정량)
  설문: "우리를 선택한 이유는? (복수선택)"
  - 비동기 신뢰: 30%
  - 간단한 UI: 60% ← 이게 더 크다!

- 고객 만족도: "간단함" 관련 점수 vs "신뢰" 관련 점수

**대응 방안:**
- 전략 재수립: "분산팀을 위한 간단한 도구"로 수정
- 마케팅 메시지 재정렬: "간단함"을 전면에
- 제품 로드맵: "기능 추가"보다 "심플함 유지"에 집중

---

### 시나리오 5: "1년 후 우리가 할 일이 없다 (Product-Market Fit 도달 후)"

**상황:**
우리의 "분산팀" 포지셔닝이 너무 성공해서,
1년 후 우리가 목표 시장을 포화시킨 경우

**발생 확률:** 낮음-중간 (25%)
- 시장이 생각보다 작을 수 있음
- 하지만 우리가 "분산팀용 신뢰 도구"에서 정말 성공하면 가능

**영향도:** 중간 (좋은 문제, 하지만 문제)
- 성장이 멈춤
- 다음 확장 방향이 모호

**조기 신호:**
- 신규 고객 획득 감소 추세
- 기존 고객 중 "성장했는데 우리를 outgrow" 하는 사례
- 팀의 "다음이 뭐지?"라는 의문

**대응 방안:**
- 일찍부터 다음 전략 준비:
  "분산팀 + 30-100명" 세그먼트로 확장?
  "분산팀" → "분산팀이 성장한 회사" support?

- 인접 시장 탐색:
  비동기팀 외 "프로젝트 기반 팀" (에이전시 등)

- M&A 준비: 우리의 위치가 강해지면, 큰 회사가 인수 제안할 수 있음
```

## PM 역할 확장 포인트

### Before: 기존 워크플로우
```
Day 1-3: 회의실 몇 번 → "이렇게 가자" 정도의 방향만 설정
Day 4-7: CEO와 논의 → 불명확한 채 진행
Day 8-14: "아, 이게 문제네" 하면서 중간에 방향 바꿈
Day 15+: 낭비된 시간, 불명확한 우선순위

총 소요 시간: 불명확함 (계속 의심)
PM의 역할: 의사결정자라기보다 "방향 표류"
```

### After: Claude Code 활용 워크플로우
```
시간 0: PM이 데이터 준비 (시장, 내부 지표, 고객 피드백)
시간 1: Claude가 진단 → PM이 합의 (30분)
시간 2: Claude가 3가지 전략 옵션 → PM이 선택 (1시간)
시간 3: Claude가 선택의 약점 도출 → PM이 모니터링 계획 수립 (1시간)
시간 4: 팀 회의 (전략 결정 + alignment)

총 소요 시간: 약 1일 (집중 작업)
PM의 역할: 데이터 기반 의사결정자 → 리더
```

### 핵심 변화
- **속도**: 불명확(2주+) → 1일
- **신뢰도**: "이게 맞을까?" → "이게 맞는 이유 + 틀릴 수 있는 이유"
- **팀 정렬**: "방향 모호" → "명확한 선택과 포기"

### 필요한 스킬 전환
- **이전**: "어떤 전략이 맞을까 생각하기"
- **이후**: "이 전략이 틀렸을 때 뭘 할 것인가" 준비하기
