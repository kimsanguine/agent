# 슬래시 커맨드: 반복 워크플로우를 원클릭으로

> **핵심 개념**: PM의 반복되는 업무를 `.claude/commands/` 폴더의 마크다운 파일로 정의하고, Claude Code에서 `/명령어`로 실행하는 가장 강력한 생산성 도구

---

## 슬래시 커맨드란?

### 정의
```
.claude/commands/ 폴더 내 마크다운 파일 → /파일명 으로 원클릭 실행
```

슬래시 커맨드는 Claude Code의 가장 강력한 워크플로우 자동화 기능입니다. 매번 같은 프롬프트를 반복 입력하는 대신, 표준화된 명령어로 즉시 실행할 수 있습니다.

### PM이 얻는 것
- **시간 절감**: 15분 프롬프트 작성 → 10초 커맨드 실행
- **일관성**: 같은 절차로 동일한 품질의 결과 생성
- **추적성**: 어떤 커맨드로 생성된 결과인지 명확
- **팀 공유**: 팀 전체가 같은 워크플로우 사용 가능

---

## PM을 위한 8가지 핵심 슬래시 커맨드 (2026년 버전)

### 1. `/today` — 일일 브리핑

**목적**: 아침 출근 후 5분 안에 오늘의 우선순위 파악

```markdown
# .claude/commands/today.md

오늘의 브리핑을 생성해줘.

절차:
1. tasks/ 폴더에서 "오늘 마감" 또는 "과거 미완료" 태스크 목록화
2. status/ 에서 최근 주간 보고서 읽고 이번 주 맥락 파악
3. research/digest/ 에서 어제 뉴스 다이제스트 요약
4. content/ 에서 작성 중인 콘텐츠 상태 확인

출력 형식:
## 🎯 오늘의 우선순위 (최대 3개)
- [우선순위 1]: 마감일시 및 기대효과
- [우선순위 2]: 마감일시 및 기대효과
- [우선순위 3]: 마감일시 및 기대효과

## 📋 마감 임박 태스크 (2일 이내)
- [태스크 1]: D-1 | 진행률 | 블로커 있는가?

## 📊 이번 주 진행 현황
- 계획 대비 진행률
- 이번 주 주요 변수

## 📰 어제 핵심 뉴스 (Top 3)
- 뉴스 제목 | 왜 PM으로서 중요한가?
```

**사용 예시**:
```bash
$ /today
```

**PM 판단 포인트**:
- 우선순위 3개가 현실적인가? (5개 이상 = 방향 재정렬 신호)
- 블로커가 있는 태스크는 즉시 처리해야 하는가?
- 어제 뉴스 중 이번 주 전략에 영향을 주는 것이 있는가?

### 2. `/prd` — PRD 자동 생성

**목적**: 새 기능 또는 프로젝트의 Lean PRD 1시간 안에 초안 작성

```markdown
# .claude/commands/prd.md

기능명: $ARGUMENTS

Lean PRD를 생성해줘. 다음 구조로 작성:

## 1️⃣ 왜? (Why) - 1줄 정의
[시장 상황] 때문에 [대상 사용자]는 [문제]를 겪는다.
→ 해결책: [솔루션]

## 2️⃣ 성공 기준 (Metrics - OMTM)
One Metric That Matters: [지표명] → [현황] → [목표]
예: 월간 활성사용자 | 현황: 5,000명 | 목표: 10,000명 (8주)

## 3️⃣ 핵심 시나리오 (3개만)
### 시나리오 A: [사용자 직무]
- 입력: [사용자가 하는 일]
- 기대 결과: [우리 솔루션 이후]

## 4️⃣ 제약 (Constraints)
- 기술 제약:
- 비즈니스 제약:
- 시장 제약:

## 5️⃣ POC (Proof of Concept)
최소 검증 범위: [가장 핵심적인 1개 시나리오만]

---

이 PRD를 Notion에 게시할까요?
```

**사용 예시**:
```bash
$ /prd "AI 기반 자동 보고서 생성"
```

**PM 판단 포인트**:
- OMTM이 명확한가? (여러 지표 = 전략 흔들림)
- 3개 시나리오가 사용자의 진짜 문제를 해결하는가?
- 이 PRD를 엔지니어에게 보낼 수 있을 정도인가?

### 3. `/status` — 주간 상태 보고서

**목적**: 금요일 오후, 이번 주 성과를 정량적으로 정리

```markdown
# .claude/commands/status.md

이번 주 상태 보고서를 생성해줘.

데이터 수집:
1. tasks/ 에서 완료된 태스크 + 진행 중 태스크 + 미완료 태스크 분류
2. prd/ 에서 이번 주 생성된 PRD 및 업데이트된 PRD 목록
3. content/ 에서 작성/게시된 콘텐츠 목록
4. metrics.md 에서 주간 KPI 변화

출력 형식:

## 📈 주간 성과 (Week of 2026-02-XX)

### ✅ 완료 (주간)
- [완료된 태스크들: 총 X개]

### 🔄 진행 중 (이월 예정)
- [진행 중인 태스크들: 총 X개]

### 🚨 블로커
- [블로커 1]: 원인 | 영향 범위 | 해결 예정일
- [블로커 2]: 원인 | 영향 범위 | 해결 예정일

### 📊 핵심 지표
| 지표 | 지난주 | 이번주 | 변화 |
|------|--------|--------|------|
| MAU | 50K | 52K | +4% |
| 콘텐츠 발행수 | 8 | 12 | +50% |

### 🎯 다음 주 계획
1. [우선순위 1]
2. [우선순위 2]
3. [우선순위 3]

---

이 보고서를 Slack #status-report 채널에 공유할까요?
```

**사용 예시**:
```bash
$ /status
```

**PM 판단 포인트**:
- 완료 대비 진행 중 비율이 건강한가? (완료 < 진행 중 = 일정 위험)
- 블로커가 2개 이상이면 다음 주 계획 재조정 필요한가?
- 핵심 지표가 목표 방향과 일치하는가?

### 4. `/competitor` — 경쟁사 5축 분석

**목적**: 경쟁 전략 회의 전, 90분 안에 경쟁사 분석 완료

```markdown
# .claude/commands/competitor.md

경쟁사명: $ARGUMENTS

웹 검색을 통해 경쟁사를 5축으로 분석해줘.

## 🔍 5축 분석

### 1️⃣ Product (제품)
- 핵심 기능 Top 3
- 최근 업데이트 (지난 3개월)
- 차별화 포인트

### 2️⃣ Pricing (가격)
- 가격 정책 (Free/Pro/Enterprise)
- 타겟 고객층별 CAC
- 예상 ARR (Available data)

### 3️⃣ Positioning (포지셔닝)
- 마케팅 메시지
- 타겟 고객
- Brand Tone

### 4️⃣ Performance (성과)
- 사용자 규모 (추정)
- 성장률 추세
- 시장 점유율 (가능한 범위)

### 5️⃣ People (사람)
- CEO/Founder 배경
- 주요 투자자
- 팀 규모

## 💡 전략적 인사이트

### 가설 3개
1. [가설]: 근거
2. [가설]: 근거
3. [가설]: 근거

### Moat 분석 (경쟁 해자)
- 이 경쟁사가 지킬 수 있는 것: [기술/네트워크/브랜드/규제]
- 우리가 공략할 수 있는 것: [시장 틈새/새로운 기술/다른 세그먼트]
```

**사용 예시**:
```bash
$ /competitor "OpenAI"
$ /competitor "Anthropic"
```

**PM 판단 포인트**:
- 이 경쟁사가 우리의 진정한 경쟁자인가? (시장/고객/가격이 겹치는가?)
- 가설 3개 중 우리가 검증해야 할 것은? (리서치 계획 수립)
- 해자 분석이 우리의 전략에 영향을 주는가?

### 5. `/linkedin` — LinkedIn 포스트 초안

**목적**: 주간 1-2회 콘텐츠 마케팅 포스트, 30분 안에 3가지 옵션 생성

```markdown
# .claude/commands/linkedin.md

주제: $ARGUMENTS

LinkedI 포스트 초안 3가지를 생성해줘. 각 포스트는:
1. 톤앤매너는 llm-context/brand/voice.md 를 참고
2. 구조: 질문/상황 → 경험 기반 관찰 → 인사이트 → 소프트 CTA
3. 길이: 150-300자 (LinkedIn 알고리즘 최적화)

---

## 옵션 1: [타입] 포스트

[개인 경험/관찰을 시작으로 하는 질문]

"..." [짧은 스토리라인]

→ [우리의 인사이트]

→ [호출 액션] (댓글/공유 유도)

---

## 옵션 2: [타입] 포스트

[다른 앵글의 콘텐츠]

---

## 옵션 3: [타입] 포스트

[또 다른 앵글]

---

이 중 어느 포스트를 게시할까요?
```

**사용 예시**:
```bash
$ /linkedin "AI 시대의 PM 역할 변화"
$ /linkedin "데이터 기반 의사결정의 함정"
```

**PM 판단 포인트**:
- 3개 옵션 중 우리 브랜드 톤과 가장 맞는 것은?
- 이 포스트가 타겟 고객(CEO/PM)의 공감을 얻을까?
- CTA가 자연스럽고 강압적이지 않은가?

### 6. `/agent` — Vibe Agenting 프로토타입

**목적**: 새로운 AI Agent 아이디어를 TDD로 빠르게 검증

```markdown
# .claude/commands/agent.md

Agent 아이디어: $ARGUMENTS

다음 순서로 Agent 프로토타입을 구축해줘.

## Step 1: 문제 정의
- 이 Agent가 해결해야 할 문제는?
- 성공의 정의는? (1-2개의 구체적인 테스트 케이스)

## Step 2: 테스트 케이스 (Eval First)
```
테스트 케이스 1:
- 입력: [Agent에 주는 명령]
- 기대 결과: [이상적인 아웃풋]
- 성공 기준: [어떻게 성공을 판단할 것인가]
```
## Step 3: Agent Instructions (마크다운)
Agent의 역할, 목표, 사용할 도구, 제약을 명확히 정의

## Step 4: 프로토타입 코드
가장 간단한 형태의 작동하는 코드 (최소 50줄, 최대 100줄)

## Step 5: 테스트 & 결과
Step 2의 테스트 케이스들을 프로토타입에 실행 → 통과/실패 기록

## 다음 스텝
- 통과: 실제 구현 로드맵 수립
- 실패: 어느 부분이 막혔는가? → Agent Instructions 수정 → 재테스트
```

**사용 예시**:
```bash
$ /agent "경쟁 분석을 자동으로 수행하는 에이전트"
```

**PM 판단 포인트**:
- 이 Agent가 정말 필요한가? (수동으로는 불가능한 규모인가?)
- 테스트 케이스를 통과하는 프로토타입이 구축되었는가?
- 이것을 실제 제품으로 확대하려면 어떤 투자가 필요한가?

### 7. `/simplify` — 코드 자동 최적화 (2026년 신규)

**목적**: 복잡한 코드를 간결하고 효율적으로 리팩토링

```markdown
# .claude/commands/simplify.md

대상 파일: $ARGUMENTS

다음을 수행해줘:
1. 코드를 읽고 복잡도 분석 (순환 복잡도, 중복 코드, 비효율적 패턴)
2. 다음 기준으로 개선:
   - 단순성: 같은 기능을 더 적은 줄 수로
   - 가독성: 변수명, 함수명, 주석 최적화
   - 성능: 불필요한 반복, 메모리 누수 제거
3. Before/After 코드 비교 + 개선 효과 설명
4. 개선된 코드는 기존 파일을 대체

## 개선 체크리스트
- [ ] 함수는 단일 책임 원칙을 따르는가?
- [ ] 복사-붙여넣기 코드는 제거했는가?
- [ ] 변수명은 자기 설명적인가?
- [ ] 불필요한 중첩은 제거했는가?
```

**사용 예시**:
```bash
$ /simplify "src/components/Dashboard.tsx"
$ /simplify "lib/api/client.py"
```

**PM 판단 포인트**:
- 코드가 정말 더 간단해졌는가?
- 성능 개선이 측정 가능한가?
- 기존 테스트를 모두 통과하는가?

### 8. `/batch` — 배치 작업 묶음 실행 (2026년 신규)

**목적**: 여러 개의 유사한 작업을 한 번에 처리

```markdown
# .claude/commands/batch.md

작업 목록: $ARGUMENTS

다음 배치 작업을 한 번에 처리해줘:
1. 작업 목록 파싱 (몇 개의 작업인가?)
2. 각 작업의 입력/출력 규격 통일
3. 병렬 처리 가능한가 순차 처리가 필요한가 판단
4. 각 작업 실행 + 결과 통합

## 배치 작업 예시
- 10개 경쟁사에 대해 /competitor 실행
- 5개 새 기능에 대해 /prd 초안 생성
- 20개 파일에 대해 /simplify 자동 리팩토링

## 출력 형식
- 각 작업별 진행 상황
- 실패한 작업 (있다면 원인)
- 통합 결과 레포트
```

**사용 예시**:
```bash
$ /batch "competitors.csv"  # CSV 파일에 경쟁사 목록
$ /batch "features-list.md" # 마크다운 파일에 기능 목록
```

**PM 판단 포인트**:
- 배치 처리가 완료되었는가?
- 개별 처리보다 품질이 일관성 있는가?
- 에러 처리는 충분한가?

## 슬래시 커맨드 작성 팁

### 1. 구체적인 출력 형식 지정
```markdown
❌ 나쁜 예: "PRD를 작성해줘"

✅ 좋은 예: "다음 마크다운 구조로 PRD를 작성해줘:
## 왜? (Why)
## 성공 기준
## 핵심 시나리오 (3개)
..."
```

### 2. 파일 경로 명시
```markdown
✅ "tasks/ 폴더의 모든 .md 파일을 읽고..."
✅ "status/ 의 최근 주간 보고서를 찾아서..."
```

### 3. $ARGUMENTS 활용 (동적 입력)
```bash
# $ARGUMENTS 없음: /today 는 항상 고정된 결과
# $ARGUMENTS 있음: /prd "기능명" 으로 유연하게 대응
```

### 4. 후속 액션 질문 포함
```markdown
✅ "이 보고서를 Slack에 공유할까요?"
✅ "이 PRD를 팀에 공개할까요?"

→ PM이 최종 판단하는 구조 (자동화 ≠ 무조건 실행)
```

## 고급 기능: 플래그와 옵션 (2026년 신규)

### `--bare` 플래그 — 스크립트화된 호출

**목적**: 슬래시 커맨드를 자동화된 파이프라인이나 cron 스크립트에서 호출

```bash
# 기존: 대화형 모드
$ claude /prd "새 기능"
# → Claude가 대화형으로 질문하며 PRD를 작성

# 신규: --bare 플래그로 스크립트화
$ claude --bare /prd "새 기능" < feature-brief.txt
# → Claude가 대화 없이 PRD만 출력 (STDOUT)

# 실제 활용: cron 스크립트
*/6 * * * * claude --bare /today >> logs/daily-briefing.log
```

**사용 사례**:
- 매일 아침 자동으로 `/today` 브리핑을 로그에 저장
- CI/CD 파이프라인에서 자동으로 `/competitor` 분석 실행
- 주간 자동 보고서 생성 스크립트

**PM 판단 포인트**:
- 어떤 작업이 정말 자동화되어야 하는가?
- 대화형 피드백이 필요한 작업은 제외하는가?

### `--channels relay` — 휴대폰 승인 프롬프트

**목적**: 중요한 배포나 의사결정 실행 전 휴대폰으로 승인을 받기

```bash
# 예: 배포 준비
$ claude --channels relay "배포 준비: 모든 테스트 통과, 승인 필요"
# → Claude가 휴대폰의 claude.ai로 메시지 전송
# → PM이 휴대폰에서 "승인" 또는 "거부" 클릭
# → 승인 시에만 배포 진행

# 실제 활용: 배포 파이프라인 (배포 직전)
if ./tests/run-all.sh; then
  claude --channels relay "모든 테스트 통과. 배포할까요?"
  if [ $? -eq 0 ]; then
    ./deploy.sh
  fi
fi
```

**사용 사례**:
- 배포 직전 최종 승인 (휴대폰에서 즉시 OK/취소)
- 긴급 이슈 해결 후 라이브 반영 승인
- 대금 결제 또는 중요 자산 삭제 전 확인

**PM 판단 포인트**:
- 휴대폰 승인이 정말 필요한가?
- 승인 시간 초과는 어떻게 처리할 것인가?

## PM 역할 확장 포인트

### 슬래시 커맨드는 "자동화"가 아니라 "확장"

| 구분 | Before (전통 PM) | After (슬래시 커맨드) |
| --- | --- | --- |
| 시간 투입 | 매주 3-4시간 | 매주 30분 |
| 판단 횟수 | 적음 (바쁘면 스킵) | 많음 (빨라서 자주) |
| 데이터 기반 | 약함 (메모리 의존) | 강함 (모든 결과 기록) |
| 팀 정렬 | 약함 (일관성 없음) | 강함 (표준화) |

### PM이 판단해야 할 것

```
❌ 무조건 자동화할 것
- 민감한 의사결정 (전략, 인원 조정, 배포 시점 등)
- 정성적 판단이 필요한 것 (고객 피드백 해석 등)

✅ 자동화해야 할 것
- 반복적인 정보 수집 (/today, /status)
- 표준화된 산출물 (/prd, /competitor, /linkedin)
- 빠른 프로토타이핑 (/agent)

🔀 슬래시 커맨드가 돕는 것
- "판단을 위한 정보 수집"을 빠르게
- PM이 "판단 자체"에 더 많은 시간을 쓸 수 있게
```

### 성숙도 단계

```
Level 1 (이번 주): /today, /prd 를 사용해보기
Level 2 (다음 주): 팀의 피드백을 받아 커맨드 개선
Level 3 (한 달): 팀 전체가 같은 커맨드 사용 (일관성 확보)
Level 4 (분기): 새로운 워크플로우 발견 → 새 커맨드 추가
```

## `/loop` — 백그라운드 자동화 팀 만들기 (2026년 최고 임팩트 기능)

> **핵심**: `/loop`는 Claude Code를 24시간 백그라운드 워커로 변환. 매일 정해진 시간에 자동으로 PM의 반복 작업을 실행한다.

### 개념

```
기존 (수동):
08:00 - PM이 /today 입력
14:00 - PM이 /qa 입력
18:00 - PM이 /ship 입력

신규 (/loop 자동화):
/loop 08:00 /today  # 매일 아침 8시에 자동 실행
/loop 14:00 /qa     # 매일 오후 2시에 자동 실행
/loop 18:00 /ship   # 매일 저녁 6시에 자동 실행
```

**결과**: PM은 한 번만 `/loop` 명령어를 설정하면, 이후 매일 자동으로 실행된다.

### 문법

```bash
/loop <시간> <커맨드> [옵션]

# 예시
/loop 08:00 /today
/loop 14:00 /qa --targets=critical
/loop 18:00 /ship --notify=slack
/loop 09:30 /competitor "monthly-check"

# 매시간 실행
/loop 00 /status  # 매 시간의 정각에 실행

# 매주 특정 요일
/loop 09:00@monday /weekly-planning
/loop 17:00@friday /weekly-review
```

### 실제 사례 1: 일일 자동화 팀 (완전 자동화 워크플로우)

**시나리오**: 스타트업 PM이 매일 정해진 시간에 다음을 실행

```bash
# 아침 8시: 오늘 우선순위 파악
/loop 08:00 /today --output=slack

# 오전 10시: 고객 피드백 자동 분류 + 분석
/loop 10:00 /analyze-feedback

# 오후 2시: QA/테스트 자동 실행
/loop 14:00 /qa --notify=blocked-tasks

# 오후 5시: 일일 메트릭 체크
/loop 17:00 /metrics-check

# 저녁 6시: 배포 가능 여부 자동 판단
/loop 18:00 /deploy-readiness

# 저녁 6시 30분: 배포 + 모니터링
/loop 18:30 /ship --auto-canary
```

**효과**:
- PM의 수동 작업: 0분 (1일 1-2시간 단축)
- 자동화 팀이 백그라운드에서 돌아감
- 긴급 블로커 발생 시만 PM이 개입

### 실제 사례 2: 주간 자동화 팀 (계획 수립)

```bash
# 월요일 9시: 주간 계획
/loop 09:00@monday /weekly-planning

# 수요일 2시: 주간 진행 상황 체크 (중간 점검)
/loop 14:00@wednesday /mid-week-checkpoint

# 금요일 5시: 주간 회고
/loop 17:00@friday /weekly-review

# 금요일 5시 30분: 자동 보고서 생성 + Slack/이메일 전송
/loop 17:30@friday /weekly-report --send=slack,email
```

### 실제 사례 3: Vibe Agent Team 완전 자동화

**Oracle 팀** (아침 8시 - 무엇을 할 것인가?):
```bash
/loop 08:00 /discover --market-updates
/loop 08:30 /validate --last-week-decisions
```

**Atlas 팀** (오전 10시 - 어떻게 할 것인가?):
```bash
/loop 10:00 /plan-strategy --for-week
/loop 10:30 /architecture-review --vs-competitors
```

**Forge 팀** (오후 12시 - 명세 및 출시):
```bash
/loop 12:00 /write-prd --from-backlog
/loop 12:30 /set-okr --q1
```

**Argus 팀** (오후 2시 - 구현 검증):
```bash
/loop 14:00 /qa --critical-paths
/loop 14:30 /security-audit --auto-scan
```

**배포 팀** (저녁 6시 - 릴리스):
```bash
/loop 18:00 /ship --canary=10%
/loop 18:30 /monitor --alert=p0
/loop 19:00 /postmortem --if-incident
```

**결과**: 가상 팀 5명이 매일 24시간 자동으로 움직이고 있는 것처럼 보임

### 문제 해결 & 모니터링

```bash
# /loop 상태 확인
/loop --status
# 출력: "08:00 /today [✅ 성공]", "14:00 /qa [⚠️ 실패]" 등

# /loop 로그 확인
/loop --logs --since=7days
# 최근 7일간 실행 로그

# 특정 /loop 비활성화 (일시 정지)
/loop --disable 14:00

# 특정 /loop 다시 활성화
/loop --enable 14:00

# 모든 /loop 중지
/loop --stop-all
```

### `/loop` 실패 처리

**시나리오**: 14:00 /qa가 자동 실행됐는데 실패

```bash
# 자동 알림 설정 (선택)
/loop 14:00 /qa --on-fail=notify:slack,ceo

# 또는 자동 롤백
/loop 18:00 /ship --on-fail=rollback-to-previous

# 또는 수동 개입 대기
/loop 14:00 /qa --on-fail=wait:ceo-approval
```

### `/loop` 비용 고려사항

1M Context가 제공되면서, `/loop` 자동화는 이제 비용 효율적이다:

```
일일 자동화 팀 (5개 커맨드):
- 각 호출: 평균 100K tokens (전체 컨텍스트 로드)
- 월 비용: 5 × 30 × $0.10 = $15

기존 수동 방식 (PM 1시간 × 30일):
- PM 시간 비용: 1시간 × 30일 × $50/시간 = $1,500
- 절감: $1,485 (99% 절감!)
```

### PM 판단 포인트

- [ ] 어떤 작업이 정말 매일 반복되는가? (일일 /today, 주간 /status)
- [ ] 그 작업이 자동화되면 품질 손실이 없는가? (QA 테스트 ✅, 전략 결정 ❌)
- [ ] 실패했을 때 자동 롤백이 가능한가? (배포 ✅, 보고서 작성 ❌)
- [ ] 모니터링과 알림은 충분한가? (블로커 발생 시 PM에게 즉시 알림)

## 자주 묻는 질문 (FAQ)

### Q: 슬래시 커맨드가 항상 좋은 결과를 주나요?
**A**: 아니요. 입력이 정확할수록 좋은 결과가 나옵니다.
- `/prd "새 기능"` ← 불명확 (실패율 높음)
- `/prd "AI 기반 자동 보고서 생성 기능 (타겟: 마케팅팀, 8주 POC)"` ← 명확 (성공율 높음)

### Q: 팀원들과 다른 커맨드를 쓰면?
**A**: 산출물의 형식과 품질이 달라집니다. 팀 내 표준화가 중요합니다.
- 권장: 팀 리드가 1-2개 커맨드를 정의 → 팀 전체 사용
- 금지: 각자 다른 프롬프트 사용 (일관성 깨짐)

### Q: 커맨드를 수정해야 할 때는?
**A**: 매주 피드백을 받아 개선하세요.
- 월요일: 지난주 커맨드 결과 리뷰
- 화요일: 개선사항 적용 (마크다운 수정)
- 수요일부터: 새로운 커맨드로 실행
