## 1) 목적
- 병렬 에이전트 활용 기준을 익혀 대량 분석/정리 업무의 리드타임을 단축한다.
- 에이전트 실행과 최종 합성(synthesis)을 분리 설계해 결과 품질을 안정화한다.

## 2) 원문 핵심 요약 (ccforpms)
- 에이전트는 서로 상태를 공유하지 않는 독립 Claude 인스턴스로 동작한다.
- 적합한 사용 사례가 명확하다: 대량 문서 처리, 경쟁사 다건 리서치, 소스별 병렬 분석, 마감 임박 작업.
- 비적합한 경우도 명시된다: 단일 소작업, 선행 의존성이 큰 순차 작업, 매우 짧은 작업.
- 프롬프트 패턴은 3가지다: agent 수 명시형, 병렬 의도 위임형, 소스별 전문 역할 분업형.
- 대표 오케스트레이션 패턴으로 fan-out, specialized roles, parallel research, batch document generation, validation pipeline이 제시된다.
- 병렬 실행 후 반드시 최종 문서로 synthesis를 지시해야 실무 산출물 품질이 올라간다.
- 시작 명령은 `/start-1-4`, 병렬 대상 예시로 `/meetings`, `/interviews`, `@survey-results.csv`가 제공된다.
- 문제 발생 시 전체 재실행보다 실패한 agent만 재실행하는 방식이 권장된다.

### Claude Code 핵심 기능 연결 (official 반영)
- Agents 모듈은 Claude Code의 sub-agent/agent team 병렬 실행 강점을 실무에 매핑하는 핵심 구간으로, PM 오케스트레이션 역량을 직접 훈련한다.
- multi-surface 환경에서 병렬 결과를 즉시 비교/합성하면 대량 분석 업무의 병목을 빠르게 제거할 수 있다.
- MCP를 붙이면 외부 소스(협업 도구, 이슈, 문서)별 병렬 수집 후 단일 결론으로 통합하는 패턴을 운영할 수 있다.
- `CLAUDE.md`에 에이전트 출력 형식과 검증 규칙을 고정하면 병렬 결과의 품질 편차를 줄일 수 있다.
- custom sub-agent와 ad-hoc agent를 구분해 쓰면 반복 업무와 일회성 분석을 분리 운영할 수 있다.
- git/PR 흐름과 결합하면 synthesis 결과를 팀 검토 가능한 의사결정 문서로 축적할 수 있다.
- CLI composability(병렬 요청 → 실패 agent 재실행 → synthesis)를 표준 루프로 만들면 대규모 PM 리서치 대응 속도가 높아진다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **병렬 대상 업무를 먼저 선별한다.**
   - 실행 입력(프롬프트): `이번 주 업무 중 병렬 처리 가능한 작업과 순차 작업을 분리해줘.`
   - 기대 출력: 병렬/순차 분류표.
   - 체크포인트: 선행 의존성이 큰 작업이 병렬 목록에 섞이지 않았는지 확인.

2. **에이전트 역할을 명시해 fan-out한다.**
   - 실행 입력(프롬프트): `3 agents로 나눠서 VOC를 채널별(앱리뷰/CS/영업콜) 분석해줘.`
   - 기대 출력: 채널별 분석 결과 초안 3개.
   - 체크포인트: 각 agent 출력 형식이 동일한지 확인.

3. **역할 특화 지시를 추가한다.**
   - 실행 입력(프롬프트): `Agent1은 문제 정의, Agent2는 원인, Agent3은 실행 우선순위 중심으로 작성해줘.`
   - 기대 출력: 상호 보완되는 결과 세트.
   - 체크포인트: 내용 중복/충돌 지점 표시 여부 확인.

4. **중간 점검으로 실패 agent를 선별한다.**
   - 실행 입력(프롬프트): `각 agent 출력에서 근거 부족 항목만 표시해줘.`
   - 기대 출력: 재실행 필요 목록.
   - 체크포인트: 전체 재실행 대신 실패 agent만 골라내는지 확인.

5. **실패 agent만 재실행한다.**
   - 실행 입력(프롬프트): `Agent2 결과만 재실행해서 근거 출처를 추가해줘.`
   - 기대 출력: 보정된 결과 1개.
   - 체크포인트: 수정 범위가 필요한 부분으로 제한되는지 확인.

6. **최종 synthesis를 별도 지시한다.**
   - 실행 입력(프롬프트): `3개 결과를 통합해 우선순위 Top5 실행안으로 합성해줘.`
   - 기대 출력: 의사결정용 통합 문서.
   - 체크포인트: Top5마다 근거·영향·다음 액션이 포함되는지 확인.

7. **회의용 요약본으로 변환한다.**
   - 실행 입력(프롬프트): `임원 공유용 1페이지 요약으로 축약해줘. 숫자 근거를 유지해줘.`
   - 기대 출력: 회의 즉시 사용 가능한 요약본.
   - 체크포인트: 상세본과 요약본 결론이 일치하는지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션 + Level 2**: PM이 직접 작성보다 역할 분리·우선순위·합성 설계를 담당하도록 학습 구조를 맞춘다.
- **P4 속도-검증 균형 + Level 2**: 병렬 속도 이점을 취하되 synthesis와 재실행 규칙으로 품질 리스크를 통제한다.
- **P6 자동화 운영 기반 + Level 2**: fan-out 및 검증 파이프라인을 반복 가능한 운영 패턴으로 표준화한다.
- **P7 Automation vs Augmentation + Level 2**: 초안 수집은 자동화하고, 최종 판단·우선순위는 PM이 수행하는 경계를 명확히 한다.
- **P8 Connected Hub + Level 2~3**: 병렬 분석 결과를 이후 상태보고/자동화 파이프라인으로 연결 가능한 형태로 출력하게 설계한다.

## 4) 실무 적용 시나리오
시니어 PM이 분기 말 VOC 1,200건을 일주일 안에 정리해야 한다. 에이전트를 채널별(앱리뷰/CS/세일즈콜/설문)로 분리해 병렬 분석하고, 공통 템플릿으로 결과를 합성한다. 이후 PM이 상위 5개 문제를 우선순위화해 다음 분기 로드맵 입력으로 연결한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 병렬 처리 후보 업무를 선정하고 에이전트 역할(데이터 정리/원인 분석/리스크 요약)과 입력 데이터 범위를 명확히 나눕니다.
- **수~목 실행**: 역할별 에이전트를 동시 실행해 중간 결과를 수집하고, 출력 포맷(문제 정의·근거·영향도)을 통일합니다.
- **금요일 검증**: synthesis 문서 1부를 만들어 리더 리뷰를 받고, 우선순위 Top5가 실행 가능한 액션으로 연결되는지 확인합니다.
- **Done 기준**: 에이전트별 출력 품질이 기준(근거 출처 포함, 형식 일관성) 충족, 최종 통합본이 의사결정 회의에 사용됩니다.
- **리스크/완화**: 병렬 결과가 서로 충돌할 수 있으므로, 최종 합성 단계에서 공통 기준표(영향도/긴급도/실행비용)로 재정렬합니다.

### PM Use Case 확장 (Claude Code 기능 매핑)
| PM 상황 | Claude Code 기능 | 실행 예시 | 검증 포인트 |
|---|---|---|---|
| VOC/인터뷰 데이터가 많아 분석 지연 | agent 병렬 fan-out | 채널별 agent 동시 분석 후 통합 | 리드타임 단축, 누락 항목 점검 |
| 병렬 결과가 제각각이라 합치기 어려움 | `CLAUDE.md` 출력 템플릿 규칙 | 문제/근거/영향/액션 형식 강제 | 결과 형식 일관성 확보 |
| 일부 결과만 품질이 낮아 전체 재작업 발생 | 실패 agent 선택 재실행 | 근거 부족 agent만 재실행 지시 | 전체 재실행 비율 감소 |
| 외부 툴 데이터까지 포함한 분석 필요 | MCP 소스별 병렬 수집 | 이슈/문서/피드백 소스를 agent별 처리 | 소스 출처 명시 여부 확인 |
| 의사결정 회의용 결과물이 필요 | synthesis + git/PR 리뷰 | Top5 실행안 통합 문서를 PR로 검토 | 결론/근거/담당/기한 포함 |

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 시간 | 행동 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 월 | 09:30~10:30 | 병렬 대상 업무 선정 | 병렬 후보 리스트 | 순차 의존 작업 제외 완료 |
| 화 | 14:00~15:00 | agent 역할 설계(3~4개) | 역할 분담표 | 역할별 입력/출력 규격 정의 |
| 수 | 10:00~11:30 | 병렬 실행 1차 | 채널별 결과 초안 | 출력 형식 일관성 90% 이상 |
| 목 | 15:00~16:00 | 실패 agent 선별 및 부분 재실행 | 보정 결과 | 전체 재실행 없이 복구 성공 |
| 금 | 11:00~12:00 | synthesis + Top5 확정 | 통합 의사결정 문서 | Top5에 근거/영향/액션 포함 |

- **초보자 실행 팁 1**: 병렬 시작 전에 "공통 출력 템플릿"을 먼저 고정합니다.
- **초보자 실행 팁 2**: 중간 점검에서 실패 agent를 식별하지 않으면 품질 편차가 커집니다.
- **초보자 실행 팁 3**: synthesis는 반드시 별도 명령으로 요청합니다(자동 병합 기대 금지).
- **롤백/복구 액션**: 금요일 통합 결과에서 결론 충돌이 2건 이상 나오면, 수요일 원본 출력으로 롤백하고 공통 기준표(영향/긴급/비용)를 먼저 합의한 뒤 재합성합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 병렬로 돌렸는데 결과가 제각각이라 합치기 어려움**
   - 해결: 실행 전 출력 템플릿(문제/근거/영향/제안) 4칸을 고정.

2. **막힘: 실패 agent 때문에 전체를 다시 돌림**
   - 해결: 실패 agent만 재실행하는 절차를 체크리스트에 명시.

3. **막힘: 합성 문서가 길기만 하고 의사결정에 안 쓰임**
   - 해결: Top5 우선순위와 즉시 실행 액션(담당/기한)까지 포함.

4. **막힘: PM이 병렬 결과를 그대로 채택해 판단 품질 저하**
   - 해결: 최종 승인 단계에서 PM 검증 질문 3개(근거 충분성/사업 적합성/실행 가능성)를 의무화.
