## 1) 목적
- 병렬 에이전트 활용 기준을 익혀 대량 분석/정리 업무의 리드타임을 단축한다.
- 에이전트 실행과 최종 합성(synthesis)을 분리 설계해 결과 품질을 안정화한다.

## 2) 원문 핵심 요약 (ccforpms)
- 에이전트는 서로 상태를 공유하지 않는 독립 Claude 인스턴스로 동작한다.
- 적합한 사용 사례가 명확하다: 대량 문서 처리, 경쟁사 다건 리서치, 소스별 병렬 분석, 마감 임박 작업.
- 비적합한 경우도 명시된다: 단일 소작업, 선행 의존성이 큰 순차 작업, 매우 짧은 작업.
- 프롬프트 패턴은 3가지다: agent 수 명시형, 병렬 의도 위임형, 소스별 전문 역할 분업형.
- 대표 오케스트레이션 패턴으로 fan-out, specialized roles, parallel research, batch document generation, validation pipeline이 제시된다.
- 병렬 실행 후 반드시 최종 문서로 synthesis를 지시해야 실무 산출물 품질이 올라간다.
- 시작 명령은 `/start-1-4`, 병렬 대상 예시로 `/meetings`, `/interviews`, `@survey-results.csv`가 제공된다.
- 문제 발생 시 전체 재실행보다 실패한 agent만 재실행하는 방식이 권장된다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션 + Level 2**: PM이 직접 작성보다 역할 분리·우선순위·합성 설계를 담당하도록 학습 구조를 맞춘다.
- **P4 속도-검증 균형 + Level 2**: 병렬 속도 이점을 취하되 synthesis와 재실행 규칙으로 품질 리스크를 통제한다.
- **P6 자동화 운영 기반 + Level 2**: fan-out 및 검증 파이프라인을 반복 가능한 운영 패턴으로 표준화한다.
- **P7 Automation vs Augmentation + Level 2**: 초안 수집은 자동화하고, 최종 판단·우선순위는 PM이 수행하는 경계를 명확히 한다.
- **P8 Connected Hub + Level 2~3**: 병렬 분석 결과를 이후 상태보고/자동화 파이프라인으로 연결 가능한 형태로 출력하게 설계한다.

## 4) 실무 적용 시나리오
시니어 PM이 분기 말 VOC 1,200건을 일주일 안에 정리해야 한다. 에이전트를 채널별(앱리뷰/CS/세일즈콜/설문)로 분리해 병렬 분석하고, 공통 템플릿으로 결과를 합성한다. 이후 PM이 상위 5개 문제를 우선순위화해 다음 분기 로드맵 입력으로 연결한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 병렬 처리 후보 업무를 선정하고 에이전트 역할(데이터 정리/원인 분석/리스크 요약)과 입력 데이터 범위를 명확히 나눕니다.
- **수~목 실행**: 역할별 에이전트를 동시 실행해 중간 결과를 수집하고, 출력 포맷(문제 정의·근거·영향도)을 통일합니다.
- **금요일 검증**: synthesis 문서 1부를 만들어 리더 리뷰를 받고, 우선순위 Top5가 실행 가능한 액션으로 연결되는지 확인합니다.
- **Done 기준**: 에이전트별 출력 품질이 기준(근거 출처 포함, 형식 일관성) 충족, 최종 통합본이 의사결정 회의에 사용됩니다.
- **리스크/완화**: 병렬 결과가 서로 충돌할 수 있으므로, 최종 합성 단계에서 공통 기준표(영향도/긴급도/실행비용)로 재정렬합니다.
