## 1) 목적
- 설치 실패로 학습이 중단되지 않도록 PM 기준의 최소 기술 셋업 절차를 표준화한다.
- 설치 완료 상태를 명령어로 검증해 팀 내 환경 편차를 줄인다.

## 2) 원문 핵심 요약 (ccforpms)
- 필수 전제는 Claude Pro 또는 Max 구독이며 무료 플랜으로는 Claude Code를 사용할 수 없다.
- 런타임 최소 버전이 명확하다: Node.js v18 이상, 지원 OS는 macOS/Windows 10+.
- 설치 순서가 구체적이다: `node --version` 확인 → 필요 시 Node 설치 → 터미널 재시작 → Claude Code 설치.
- macOS 설치 명령은 `curl -fsSL https://claude.ai/install.sh | bash`, Windows PowerShell 설치 명령은 `irm https://claude.ai/install.ps1 | iex`다.
- 설치 검증은 `claude --version`, 계정 인증은 `claude` 실행 후 "Claude account with subscription" 선택으로 진행한다.
- `/doctor` 명령으로 설치 상태, 구독 연결, 시스템 설정, 일반 오류를 일괄 진단한다.
- 대표 장애 패턴이 정리돼 있다: `claude` 미인식, 구독 미연결, Node 버전 불일치, 권한 부족.
- PATH 이슈 시 `~/.bashrc` 또는 `~/.zshrc`에 `export PATH=$HOME/.local/bin:$PATH` 추가가 필요할 수 있다.

### Claude Code 핵심 기능 연결 (official 반영)
- 설치 단계부터 Claude Code는 CLI를 중심으로 IDE/에디터와 함께 쓰는 multi-surface 워크플로우를 전제하므로, 실행 환경과 검증 환경을 함께 맞춰야 한다.
- PM 관점에서 설치는 단순 개발 셋업이 아니라 반복 자동화의 시작점이므로, `/doctor` 같은 진단 루틴을 운영 절차로 표준화해야 한다.
- MCP 사용 예정 팀은 초기 설치 직후 연결 가능성(권한/네트워크/정책)을 확인해 이후 연동 장애를 줄여야 한다.
- `CLAUDE.md`에 설치/복구 기본 규칙을 남겨두면 팀 온보딩 시 동일 오류를 빠르게 재현·해결할 수 있다.
- 향후 sub-agent/agent team 병렬 작업을 안정적으로 돌리려면 런타임/권한 일관성이 먼저 확보되어야 한다.
- git/PR 기반 운영에서는 설치 가이드 자체도 버전 관리해 OS별 실패 패턴과 복구 히스토리를 축적할 수 있다.
- CLI composability 관점에서 `node --version → claude --version → /doctor`를 하나의 운영 체인으로 고정하면 초반 불확실성을 크게 낮춘다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **OS와 Node 기준선을 먼저 확인한다.**
   - 실행 입력(명령): `node --version`
   - 기대 출력: `v18.x` 이상 버전 문자열.
   - 체크포인트: v18 미만이면 설치 진행 중단 후 Node 업그레이드 선행.

2. **설치 명령을 OS별로 정확히 분기한다.**
   - macOS 실행 입력: `curl -fsSL https://claude.ai/install.sh | bash`
   - Windows 실행 입력: `irm https://claude.ai/install.ps1 | iex`
   - 기대 출력: 설치 완료 로그 + 바이너리 경로 안내.
   - 체크포인트: 중간에 권한 오류가 나면 관리자 권한/보안 정책 먼저 점검.

3. **터미널 재시작 후 버전 검증을 진행한다.**
   - 실행 입력(명령): `claude --version`
   - 기대 출력: Claude Code 버전 번호.
   - 체크포인트: `command not found`면 PATH 설정 단계로 즉시 이동.

4. **계정 인증을 완료한다.**
   - 실행 입력(행동): `claude` 실행 후 `Claude account with subscription` 선택.
   - 기대 출력: 인증 완료 후 대화 입력 가능 상태.
   - 체크포인트: Pro/Max 구독 계정인지 확인, 팀 계정 혼동 방지.

5. **진단 명령으로 설치 상태를 일괄 점검한다.**
   - 실행 입력(명령): `/doctor`
   - 기대 출력: 설치/권한/연결 상태 리포트.
   - 체크포인트: 치명 오류(critical) 0건 여부 확인.

6. **PATH 이슈를 복구한다.**
   - 실행 입력(명령): `~/.zshrc` 또는 `~/.bashrc`에 `export PATH=$HOME/.local/bin:$PATH` 추가 후 쉘 재실행.
   - 기대 출력: `claude --version` 정상 출력.
   - 체크포인트: PATH 수정 후에도 실패하면 설치 경로 자체를 재검토.

7. **팀 공통 설치 로그를 남긴다.**
   - 실행 입력(프롬프트): `설치 성공/실패 기록 템플릿을 만들어줘. 항목은 OS, 오류코드, 해결시간 포함.`
   - 기대 출력: 복구 가능한 설치 이력 템플릿.
   - 체크포인트: 다음 온보딩에서 그대로 재사용 가능한 수준인지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 2**: 설치를 설명형 문서가 아니라 명령 실행과 검증 체크 중심으로 구성해 즉시 가동 상태를 만든다.
- **P4 속도-검증 균형 + Level 2**: 빠른 설치 후 `/doctor`로 검증 게이트를 넣어 초기 오류를 조기 차단한다.
- **P6 자동화 운영 기반 + Level 2**: 환경 진단을 수동 감으로 처리하지 않고 표준 명령 기반 루틴으로 운영한다.
- **P8 Connected Hub + Level 1**: 구독/CLI/OS 권한을 연결된 실행 조건으로 다뤄 이후 MCP·자동화 확장의 토대를 만든다.

## 4) 실무 적용 시나리오
주니어 PM이 주간 회의 전에 PRD 자동화 실습을 해야 하는데 CLI가 동작하지 않는 상황이다. 0.1 문서 절차대로 Node 버전부터 재확인하고, 설치 명령 재실행 후 `/doctor` 결과를 캡처해 팀 채널에 공유한다. 동일 이슈가 반복되면 공통 설치 스크립트와 FAQ를 만들어 다음 온보딩에서 재사용한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 팀원 전원이 `node --version`, `claude --version`을 점검하고 OS별 설치 경로(macOS/Windows)를 표준 문서로 통일합니다.
- **수~목 실행**: 설치 실패 케이스를 의도적으로 1건 재현해 복구 절차(PATH/권한/재설치)를 검증하고 스크린샷과 명령 로그를 수집합니다.
- **금요일 검증**: `/doctor` 결과를 일괄 리뷰해 공통 오류 패턴을 추출하고 다음 온보딩에 반영할 FAQ 우선순위를 확정합니다.
- **Done 기준**: 전원 계정 인증 완료, `/doctor` 치명 오류 0건, 설치/복구 절차 문서가 팀 위키에 게시된 상태입니다.
- **리스크/완화**: 개인 PC 권한·보안 정책 차이로 지연될 수 있으므로, 사전 관리자 권한 체크와 대체 장비(공용 머신) 플랜을 준비합니다.

- **초보자 실행 규칙 1**: 설치 명령은 복붙 전에 OS를 다시 확인합니다.
- **초보자 실행 규칙 2**: 오류 메시지는 축약하지 말고 원문 그대로 저장합니다.
- **초보자 실행 규칙 3**: 설치 성공 후에도 반드시 `/doctor`를 실행해 숨은 오류를 찾습니다.
- **롤백/복구 액션**: 금요일까지 `claude --version` 실패가 지속되면 기존 쉘 설정 파일 변경분을 되돌리고(백업본 복원), 공식 설치 명령을 깨끗한 새 터미널 세션에서 재실행한 뒤 `/doctor`로 재검증합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: `claude: command not found`**
   - 해결: PATH 변수에 `~/.local/bin` 추가 후 쉘 재시작, 그래도 실패하면 설치 경로 재확인.

2. **막힘: 계정 인증은 됐는데 구독 오류가 뜸**
   - 해결: 로그인 계정이 Pro/Max인지 확인하고, 개인/업무 계정 전환 상태를 점검.

3. **막힘: `/doctor`에서 권한 오류 반복**
   - 해결: 관리자 권한으로 재시도하고, 사내 보안 정책 때문에 차단된 항목은 IT 지원 요청 티켓 발행.

4. **막힘: 팀원마다 설치 단계가 달라 가이드가 불일치**
   - 해결: OS별 표준 절차 문서를 분리하고, 각 단계 완료 시점에 공통 체크박스를 사용.
