## 1) 목적
- Dennis baseline(Cursor+MCP)을 기준점으로 삼고, 이든 확장(Claude.ai + Claude Code + n8n)의 **실행 차이**를 명확히 제시한다.
- Level1~4 운영을 도구 소개가 아니라 **산출물 중심 운영 체계**로 정의한다.
- 팀이 도입 단계별 실패 패턴과 전환 기준을 사전에 이해하도록 한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 코스는 PM이 Claude Code로 실무 산출물을 직접 만들도록 설계되어 있으며, 단순 읽기가 아닌 인터랙티브 실습이 중심이다.
- 핵심 역량은 파일 작업, 병렬 에이전트, 커스텀 서브에이전트, 프로젝트 메모리(CLAUDE.md), 고급 PRD/분석/전략 작성이다.
- 실전 모듈(2.x)은 PRD 작성, 데이터 분석, 전략 수립을 통해 AI를 사고 파트너로 사용하는 패턴을 제시한다.
- Vibe Coding(4.x)은 계획→구현→반복→버전관리→배포 루프로 산출물을 공개 가능한 형태까지 완결한다.
- Company Context 문서군은 모든 의사결정이 회사 목표/KPI와 연결되도록 기준 맥락을 제공한다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview 기준으로 생태계 설계의 출발점은 도구 나열이 아니라 **문제-작업-산출물** 단위의 실행 흐름 정의다.
- PM은 Claude AI(전략/초안)와 Claude Code(실행/검증)를 역할 분리해 운영하면 회의 결과가 실제 파일/티켓/리포트로 빠르게 전환된다.
- Tutorials Hub의 공통 패턴처럼, 각 단계는 “시작 프롬프트”보다 “검증 가능한 결과물”을 기준으로 완료 판단해야 한다.
- Opus 4.6 튜토리얼 관점에서 다단계 과제는 Plan 기반으로 분해 후 실행할 때 품질 편차와 재작업률을 줄일 수 있다.
- Chrome 튜토리얼에서 제시된 탐색 확장은 외부 근거 수집 단계에 유효하며, 수집 결과는 Claude Code 문서 체계로 환원해야 운영 가치가 생긴다.
- Cowork 튜토리얼의 파일 협업 관점을 반영해 비개발 PM도 폴더 단위 산출물 운영에 참여할 수 있도록 표준 포맷을 정의해야 한다.
- 금융 서비스 튜토리얼의 신뢰성 원칙처럼 변경 이력, 근거 링크, 승인 로그를 남기면 생태계 확장 시 컴플라이언스 리스크를 낮출 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일, 팀 리드가 현재 운영 수준을 Level1~4로 자가진단하고 “이번 주 목표 레벨”을 1단계만 상향하는 원칙을 선언한다.
2. 월요일 오후, PM이 각 레벨에서 반드시 남겨야 할 산출물 목록(PRD 링크, 티켓, 자동화 로그, 에이전트 결과물)을 체크리스트로 배포한다.
3. 화요일, Level1 Quick Win 4종 중 최소 2개를 실행해 즉시 체감 성과(리드타임, 누락률) 데이터를 수집한다.
4. 수요일, Claude Code Hub(Level2) 기준으로 CLAUDE.md/명령어/템플릿이 실제 실행에 재사용되는지 점검한다.
5. 목요일, n8n 후보 업무를 점수화해 Level3에서 자동화할 플로우 1~2개를 선정하고 실패 복구 정책까지 설계한다.
6. 금요일 오전, `/agent` 기반 Level4 파일럿 과제를 1건 선정해 PRD→Eval 경로를 축약 리허설한다.
7. 금요일 오후, OMTM 리뷰에서 레벨 상승 여부를 결정하고, 게이트 미충족 항목은 다음 주 보강 백로그로 이월한다.
8. 주간 종료 시 생태계 운영 보드에 “현재 레벨/다음 레벨/차단 이슈/담당자”를 업데이트해 팀 공통 시야를 맞춘다.

#### 검증/복구 흐름
1. 월요일 진단에서 레벨 불일치가 발견되면 상향 계획을 즉시 보류하고 하위 레벨 안정화부터 수행한다.
2. 화~수 실행에서 산출물 링크가 누락되면 해당 단계는 완료로 인정하지 않고 같은 날 내 재제출한다.
3. 목요일 자동화 설계에서 재시도/알림/fallback이 정의되지 않으면 배포 금지 상태를 유지한다.
4. 금요일 리뷰에서 OMTM 개선 근거가 부족하면 레벨 상승을 승인하지 않고 1주 추가 검증을 부여한다.
5. 반복 실패 항목은 운영 규칙 문서에 금지/필수 체크로 반영해 재발을 차단한다.

### 시나리오: 4주 내 팀 생태계 도입
- Week 1 (Level1): PRD 게시, 댓글 분류, 티켓 생성, 상태 보고 4개 Quick Win 실행
- Week 2 (Level2): CLAUDE.md, Slash Commands, 운영 템플릿 정착
- Week 3 (Level3): n8n으로 뉴스/피드백/CS 자동화 2개 이상 가동
- Week 4 (Level4): `/agent`로 프로토타입-평가-핸드오프 루프 실행
- 매주 금요일: OMTM 대시보드로 성과/실패 원인 리뷰

### 이번 주 완료형 실행 플랜
- **월~화:** 팀 현재 레벨을 진단하고 Level1~4 전환 기준표 초안을 작성해 오너별 책임을 배정한다.
- **수~목:** Level1 즉시 과제 2개와 Level2 허브 세팅 1개를 병행 실행해 실제 운영 마찰을 기록한다.
- **금 검증:** OMTM 기준으로 이번 주 실행 결과를 리뷰하고 다음 주에 올릴 단계(Level3 준비 여부)를 의사결정한다.
- **Done 기준:** 레벨 진단표, 단계 전환 기준표, 실행 로그, 차주 진입 계획이 한 문서에 통합된다.
- **리스크·완화:** 단계 과점프(준비 없이 Level3/4 진입) 위험이 있으므로 진입 게이트 미충족 시 보강 태스크를 우선 배치한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 레벨을 한 번에 올리려다 운영 붕괴**
  - 증상: Level1 미정착 상태에서 Level3/4를 동시에 추진한다.
  - 해결: `한 주 한 단계` 원칙을 적용하고, 선행 게이트 미충족 시 상향 시도를 차단한다.
- **막힘 2: 도구는 도입했지만 산출물 기준이 없음**
  - 증상: 실행 로그는 많은데 의사결정에 쓸 결과물이 없다.
  - 해결: 단계별 필수 산출물 템플릿(목표/근거/지표/owner/due)을 공통 스키마로 강제한다.
- **막힘 3: 주간 리뷰가 활동 보고로 끝남**
  - 증상: 무엇을 멈추고 무엇을 확대할지 결론이 없다.
  - 해결: 리뷰 아젠다를 OMTM 변화→원인 가설→다음 액션 3개 순서로 고정한다.
- **막힘 4: 책임 공백으로 차단 이슈 장기화**
  - 증상: 자동화/허브/에이전트 이슈가 서로 떠넘겨진다.
  - 해결: 단계별 owner 1명과 에스컬레이션 대상 1명을 고정해 24시간 내 처리 책임을 명확히 한다.
