## 1) 목적
- Dennis baseline(Cursor+MCP)을 기준점으로 삼고, 이든 확장(Claude.ai + Claude Code + n8n)의 **실행 차이**를 명확히 제시한다.
- Level1~4 운영을 도구 소개가 아니라 **산출물 중심 운영 체계**로 정의한다.
- 팀이 도입 단계별 실패 패턴과 전환 기준을 사전에 이해하도록 한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 코스는 PM이 Claude Code로 실무 산출물을 직접 만들도록 설계되어 있으며, 단순 읽기가 아닌 인터랙티브 실습이 중심이다.
- 핵심 역량은 파일 작업, 병렬 에이전트, 커스텀 서브에이전트, 프로젝트 메모리(CLAUDE.md), 고급 PRD/분석/전략 작성이다.
- 실전 모듈(2.x)은 PRD 작성, 데이터 분석, 전략 수립을 통해 AI를 사고 파트너로 사용하는 패턴을 제시한다.
- Vibe Coding(4.x)은 계획→구현→반복→버전관리→배포 루프로 산출물을 공개 가능한 형태까지 완결한다.
- Company Context 문서군은 모든 의사결정이 회사 목표/KPI와 연결되도록 기준 맥락을 제공한다.

## 3) 이든 철학 반영 포인트 (v2.4)
### Dennis baseline vs 이든 확장 비교
| 비교 축 | Dennis baseline | 이든 확장(v2.4) | 운영 영향 |
|---|---|---|---|
| 허브 구조 | Cursor 단일 허브 | Claude.ai(전략) + Claude Code(실행) + n8n(자동화) | 역할 분리로 재현성 향상 |
| 문서 흐름 | 수동 정리 비중 높음 | `/prd`→Notion 게시→Linear 티켓 자동 연계 | 리드타임 단축 |
| 상태 공유 | 수동 보고 중심 | OMTM 기반 자동 리포트 + Slack 공유 | 운영 가시성 강화 |
| 자동화 깊이 | 제한적 트리거 | 스케줄/이벤트 기반 n8n 파이프라인 | 반복 업무 감소 |
| 확장 방식 | 개인 생산성 최적화 | 팀 운영 표준 + `/agent` 체계화 | 조직 확장성 확보 |

- **P8 Connected Hub**를 핵심 원칙으로 적용한다.
- **P10 Validation-First Discovery**: 검증 공백(Validation Gap)을 줄이기 위해 가설 검증 주기를 주/스프린트 단위에서 일/시간 단위로 단축하는 운영을 도입한다.
- **FR-016~FR-019**(Level1~4)를 모듈별 필수 요구사항으로 취급한다.
- **P7**에 따라 자동화 가능한 영역과 사람 판단 영역을 문서에 분리 기록한다.

## 4) 실무 적용 시나리오
### 시나리오: 4주 내 팀 생태계 도입
- Week 1 (Level1): PRD 게시, 댓글 분류, 티켓 생성, 상태 보고 4개 Quick Win 실행
- Week 2 (Level2): CLAUDE.md, Slash Commands, 운영 템플릿 정착
- Week 3 (Level3): n8n으로 뉴스/피드백/CS 자동화 2개 이상 가동
- Week 4 (Level4): `/agent`로 프로토타입-평가-핸드오프 루프 실행
- 매주 금요일: OMTM 대시보드로 성과/실패 원인 리뷰

### 이번 주 완료형 실행 플랜
- **월~화:** 팀 현재 레벨을 진단하고 Level1~4 전환 기준표 초안을 작성해 오너별 책임을 배정한다.
- **수~목:** Level1 즉시 과제 2개와 Level2 허브 세팅 1개를 병행 실행해 실제 운영 마찰을 기록한다.
- **금 검증:** OMTM 기준으로 이번 주 실행 결과를 리뷰하고 다음 주에 올릴 단계(Level3 준비 여부)를 의사결정한다.
- **Done 기준:** 레벨 진단표, 단계 전환 기준표, 실행 로그, 차주 진입 계획이 한 문서에 통합된다.
- **리스크·완화:** 단계 과점프(준비 없이 Level3/4 진입) 위험이 있으므로 진입 게이트 미충족 시 보강 태스크를 우선 배치한다.
