## 1) 목적
- 로컬 프로토타입을 실제 URL로 배포해 사용자/이해관계자 검증이 가능한 상태로 전환한다.
- 배포를 이벤트로 끝내지 않고, 자동 재배포·모니터링·핫픽스까지 포함한 운영 루프를 구축한다.
- Vibe Coding의 마지막 단계에서 “만드는 팀”을 “운영하는 팀”으로 전환한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 목표는 코드를 외부 서버에 배포해 공유 가능한 공개 URL(`*.vercel.app`)을 확보하는 것이다.
2. 배포는 로컬 결과를 인터넷 서비스로 전환하는 과정이고, hosting은 인프라 운영을 플랫폼이 담당하는 구조다.
3. Vercel은 Next.js와 연동성이 높고 GitHub 연결 시 자동 배포 설정이 쉽다.
4. 기본 운영 플로우는 GitHub push → 변경 감지 → 재빌드/재배포 → 수분 내 반영이다.
5. 필요 시 Vercel CLI(`npm i -g vercel`)로 터미널에서 직접 배포를 수행할 수 있다.
6. 인증/실행 오류는 `vercel login` 재실행, 프로젝트 경로 재확인, build logs 점검이 우선 대응이다.
7. 배포 후에는 모바일/다양한 환경 테스트와 빠른 개선-재배포 루프가 중요하다.
8. 전체 과정은 Plan → Build → Iterate → Save → Go Live로 닫히며, 다른 제품에도 재사용 가능하다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **배포 후보 커밋을 먼저 고정한다.**
   - 입력 예시: "main의 특정 커밋 SHA를 이번 릴리스 기준으로 지정".
   - 실행 액션: 기능 추가를 중단하고 배포 대상 범위를 문서로 잠근다.
   - 기대 출력: 배포 직전 범위 변경으로 인한 장애 가능성 감소.
2. **`/start-4-5`로 Go Live 절차를 시작한다.**
   - 입력 예시: "이번 목표는 베타 사용자 5명에게 URL 공유".
   - 실행 액션: 연결/배포/검증 순서대로 체크리스트를 수행한다.
   - 기대 출력: 공개 URL 발급과 기본 상태 확인 완료.
3. **Vercel 프로젝트를 GitHub와 연동한다.**
   - 입력 예시: "push 시 자동 재배포".
   - 실행 액션: 배포 설정과 환경변수를 점검하고 첫 배포를 실행한다.
   - 기대 출력: `*.vercel.app` URL과 빌드 성공 로그.
4. **핵심 사용자 시나리오를 30분 안에 점검한다.**
   - 입력 예시: "가입→핵심 행동→결과 확인".
   - 실행 액션: 모바일/데스크탑에서 핵심 흐름을 빠르게 테스트한다.
   - 기대 출력: Go/No-Go 판단 가능한 초기 품질 리포트.
5. **장애 대응 채널과 담당자를 고정한다.**
   - 입력 예시: "핫픽스 담당 1명, 공지 채널 1개".
   - 실행 액션: 이슈 접수, 우선순위, 커뮤니케이션 루트를 문서화한다.
   - 기대 출력: 장애 시 의사결정 지연 없이 즉시 대응 가능한 체계.
6. **자동 재배포 동작을 검증한다.**
   - 입력 예시: "사소한 문구 수정 후 push".
   - 실행 액션: 빌드 성공과 반영 시간(분)을 기록한다.
   - 기대 출력: 운영 중 수정-배포 루프 신뢰성 확보.
7. **48시간 모니터링 지표를 시작한다.**
   - 입력 예시: 오류율, 응답속도, 핵심 행동 완료율.
   - 실행 액션: 지표 추적표와 에스컬레이션 조건을 활성화한다.
   - 기대 출력: 다음 스프린트 우선순위로 바로 연결되는 운영 데이터.

### Claude Code 핵심 기능 연결 (official 반영)
- **배포 전 체크리스트 게이트 운영**: Go Live 직전 필수 검증 항목을 계획으로 잠가 배포 실패를 줄인다.
- **GitHub 연동 배포 루프 강화**: 코드 변경→push→재배포 흐름을 기준으로 운영 속도를 높인다.
- **로그/오류 확인 중심 복구**: 배포 실패 시 빌드 로그와 설정을 먼저 점검해 원인 추정을 줄인다.
- **자연어 기반 핫픽스 지시**: PM이 장애 현상·영향·우선순위를 자연어로 전달해 즉시 수정 작업으로 연결한다.
- **운영 문맥 지속 관리**: 48시간 모니터링 지표를 다음 스프린트 계획(4.2)으로 환류한다.
- **반복 배포 친화 구조**: 작은 수정을 빠르게 배포하고 검증하는 운영 루프를 기본값으로 만든다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 문서 완성보다 실사용 URL 확보를 우선 성과로 본다.
- **P4 속도-검증 균형**: 배포 속도와 운영 안정성(로그/알림/롤백)을 함께 설계한다.
- **P6 자동화 운영 기반**: push 기반 자동 배포를 기본값으로 만들어 반복 릴리스 비용을 줄인다.
- **P8 Connected Hub**: GitHub(코드)↔Vercel(배포)↔운영 알림을 연결해 의사결정 지연을 줄인다.
- **Level4 연결 명시**: `/agent`로 만든 Prototype을 배포하고 Eval 결과를 다음 PRD/TDD 갱신으로 환류한다.
- **P9 Words-to-Workflow**: 배포 체크리스트 문장이 실제 Go/No-Go 게이트로 작동하도록 운영 규칙을 고정한다.
- v2.4 관점에서 Go Live 이후는 n8n 알림/리포팅과 결합해 운영 자동화(Level3)로 확장한다.

## 4) 실무 적용 시나리오
**상황: 파일럿 고객 5명 대상 베타 공개**
1. GitHub main 기준으로 배포 후보를 고정한다.
2. Vercel에 연결해 첫 공개 URL을 발급한다.
3. 모바일/데스크탑 핵심 시나리오를 30분 내 점검한다.
4. 피드백 접수 채널(폼/슬랙)과 핫픽스 담당자를 지정한다.
5. 버그 수정 후 push만으로 자동 재배포가 되는지 검증한다.
6. 48시간 운영 데이터를 모아 다음 스프린트 우선순위를 재정렬한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 배포 대상 브랜치를 확정하고 환경변수/도메인/권한 설정을 사전 점검해 첫 배포 실패를 예방한다.
- **수~목 실행**: 스테이징/프로덕션 배포를 순차 진행하고, 자동 재배포와 로그 확인 루틴을 운영 문서로 고정한다.
- **금요일 검증**: 핵심 사용자 시나리오, 디바이스 호환성, 오류 로그를 기준으로 Go/No-Go를 최종 판정한다.
- **Done 기준**: 공개 URL이 안정 동작하고, 48시간 모니터링 항목·담당자·에스컬레이션 경로가 운영표에 명시되어 있다.
- **리스크/완화**: 배포 직후 장애가 발생하면 롤백 기준을 사전에 정의하고, 핫픽스 브랜치-담당자-공지 채널을 즉시 가동한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 파일럿 고객 5명에게 베타를 공개하는 주간, PM이 Go Live와 초기 운영을 동시에 책임진다.

- **월요일**
  - Owner: PM, 개발 리드
  - Action: 배포 대상 브랜치/커밋 확정, 환경변수·권한 사전 점검
  - Artifact: 배포 준비 체크리스트, 환경변수 검증 로그
  - Check: 필수 설정 누락 0건, 배포 대상 범위 확정
- **화요일**
  - Owner: 배포 담당
  - Action: Vercel 연결 및 첫 스테이징 배포 수행
  - Artifact: 스테이징 URL, 빌드 로그, 기본 테스트 결과
  - Check: 핵심 시나리오 5개 중 4개 이상 성공
- **수요일**
  - Owner: PM, QA
  - Action: 모바일/데스크탑 호환성 점검, 오류 로그 확인
  - Artifact: 디바이스별 테스트 표, 오류 분류표
  - Check: 치명 오류 0건, major 오류 대응 계획 확정
- **목요일**
  - Owner: PM, CS/운영 담당
  - Action: 파일럿 고객에게 URL 공유, 피드백 수집 채널 운영 시작
  - Artifact: 공유 메시지 템플릿, 피드백 취합 문서
  - Check: 피드백 응답률 목표(예: 60%) 달성 여부 점검
- **금요일**
  - Owner: PM, 팀 리드
  - Action: 48시간 운영 지표 리뷰, 차주 개선 우선순위 확정
  - Artifact: 운영 리포트, 차주 백로그
  - Check: Go/No-Go + 핫픽스 필요 여부 최종 판정

**리스크 트리거(발동 조건)**
- 배포 후 2시간 내 치명 오류 2건 이상 발생하거나, 핵심 행동 완료율이 기준 대비 30% 이상 하락하면 즉시 롤백 트리거로 본다.

**Rollback/Mitigation 흐름**
1. 신규 배포를 중지하고 직전 안정 배포 버전으로 즉시 롤백한다.
2. 핫픽스 브랜치를 생성해 원인 수정 후 스테이징에서 재검증한다.
3. 사용자 공지 채널에 장애/조치/예상 복구시간을 투명하게 공유한다.
4. 동일 이슈 재발 방지를 위해 배포 체크리스트에 사전 검증 항목을 추가한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 배포는 성공했는데 화면이 깨지거나 일부 기능이 동작하지 않음**
   - 해결: 빌드 로그에서 실패/경고를 먼저 확인하고, 환경변수 누락 여부를 우선 점검한다.
   - 확인: 동일 커밋을 스테이징에서 재배포해 재현 여부를 비교한다.
2. **막힘: 자동 재배포가 작동하지 않아 수동 배포가 반복됨**
   - 해결: GitHub 연동 상태와 브랜치 감지 설정을 재확인하고, 테스트 커밋으로 트리거를 검증한다.
   - 확인: push 후 반영 시간(분)을 기록해 정상 범위인지 확인한다.
3. **막힘: 피드백 채널은 열었지만 우선순위 판단이 늦어짐**
   - 해결: 피드백을 blocker/major/minor로 즉시 태깅하고 SLA를 함께 정의한다.
   - 확인: 금요일 리뷰 전에 blocker 처리율 100%를 목표로 관리한다.
4. **막힘: 장애 공지가 늦어 신뢰가 하락**
   - 해결: 공지 템플릿(현상/영향/조치/예상시간)을 사전 작성해 즉시 게시한다.
   - 확인: 장애 인지 후 15분 내 1차 공지 발행 여부를 KPI로 관리한다.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)
- [ ] 이 문서의 목표를 **문제-행동-성과지표** 1문장으로 적었다.
- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.
- [ ] `CLAUDE.md`의 프로젝트 규칙(금지사항/품질게이트/보고형식)을 확인했다.
- [ ] `@` 또는 첨부 자료로 **맥락 입력(문서/데이터/정책) 3개 이상** 연결했다.
- [ ] 필요한 경우 **sub-agent/병렬 작업** 범위와 합성(synthesis) 책임자를 지정했다.
- [ ] **MCP/외부도구 연동 필요 여부**와 접근 권한 상태를 점검했다.
- [ ] 산출물에 **검증 기준(정확성/재현성/지표/테스트)** 을 명시했다.
- [ ] 결과를 **Git/문서 링크/로그**로 추적 가능하게 남겼다.
- [ ] 다음 액션 3개를 **담당자·기한·검증포인트**와 함께 확정했다.

- [ ] 출시 후 48시간 모니터링 항목(오류율/응답속도/핵심행동완료율)을 정의했다.
- [ ] 핫픽스 책임자와 커뮤니케이션 경로를 명확히 했다.
