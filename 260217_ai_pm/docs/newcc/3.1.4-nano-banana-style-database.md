## 1) 목적
- 일회성 이미지 생성 결과를 팀 자산으로 전환하는 **스타일 라이브러리 운영 체계**를 구축한다.
- PM/디자인/마케팅이 동일한 스타일 자산을 재사용해 제작 속도와 일관성을 동시에 높인다.
- “좋은 결과 기억”이 아닌 “검색·호출·재적용 가능한 시스템”으로 전환한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 핵심은 이미지 생성 기술이 아니라 **저장→재사용→축적**이라는 메타 스킬이다.
2. 시작 명령은 `/start-3-1-4`, 선행 권장은 3.1.3이며 모듈 분량은 약 20분이다.
3. `style-library.html`에는 ID/썸네일/이름/카테고리/태그/프롬프트/활용 예시가 저장된다.
4. 활용 방식은 `Generate [subject] using style #X`처럼 **번호 기반 호출**로 단순화된다.
5. 라이브러리 확장 경로는 3가지다: 실무 중 저장, 외부 프롬프트 검증 후 편입, 레퍼런스 이미지에서 스타일 추출.
6. `style_extract.py`와 Gemini를 통해 색감·조명·구도·질감·분위기·타이포·효과를 텍스트 스타일로 구조화할 수 있다.
7. 카테고리/태그 체계를 잘 설계할수록 탐색 속도와 재현성이 함께 올라간다.
8. 저장 반영 이슈는 브라우저 새로고침/저장 완료 여부 확인으로 해결되는 경우가 많다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 새로운 프롬프트를 매번 새로 쓰지 않고, 검증된 style ID를 즉시 재사용한다.
- **P2 맥락 우선**: 스타일 카드에 용도/채널/금지 조건을 함께 기록해 맥락 없는 재사용을 방지한다.
- **P6 자동화 운영 기반**: 스타일 등록·검수·폐기 주기를 정의하면 품질 관리가 가능해진다.
- **P8 Connected Hub**: 스타일 자산을 문서/캠페인/프로토타입 작업과 연결해 분절된 제작을 줄인다.
- **P9 Words-to-Workflow**: “style #12로 생성” 같은 짧은 언어 명령을 표준 실행 트리거로 사용한다.
- Week 3~6 운영 관점에서, 스타일 DB는 콘텐츠 자동화·리뷰 루프·채널 확장 시 재사용률을 높이는 핵심 인프라다.

## 4) 실무 적용 시나리오
**상황: 제품 업데이트마다 비슷한 채널 자산을 반복 제작하는 팀**
1. 이번 릴리스에서 성과가 좋았던 비주얼 10개를 스타일 카드로 등록한다.
2. 카드마다 적용 채널(랜딩/SNS/뉴스레터)과 금지 조건(텍스트 밀도, 색상 충돌)을 기록한다.
3. 다음 캠페인에서 style ID 기반으로 초안을 빠르게 생성한다.
4. 품질 낮은 카드는 월간 리뷰에서 아카이브 처리하고, 우수 카드는 추천 태그를 강화한다.
5. 신규 팀원도 DB를 통해 동일 톤의 자산을 1일 내 재현할 수 있게 한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `style-library.html` 카드 스키마(ID, 태그, 용도, 금지 조건)를 확정하고 기존 우수 결과물 10건을 1차 등록한다.
- **수~목 실행**: style ID 호출 테스트를 채널별로 수행해 재현성(랜딩/SNS/뉴스레터)을 점검하고, `style_extract.py` 추출 결과를 카드 설명에 반영한다.
- **금요일 검증**: 월간 운영 리허설(등록, 검수, 폐기, 추천)을 1회 돌려 DB가 실제 제작 속도를 높이는지 측정한다.
- **Done 기준**: 카드 10개 이상이 검색 가능 상태로 정리되고, style ID 기반 생성 결과가 최소 3개 채널에서 일관되게 재현된다.
- **리스크/완화**: 태그 난립 리스크는 승인된 태그 사전으로 통제하고, 저품질 카드 누적은 월간 폐기 기준(재현성/전환기여/브랜드 적합성)으로 완화한다.

## 5) 실습/적용 체크리스트
- [ ] `style-library.html` 기반 스타일 카드 템플릿을 만들었다.
- [ ] 기존 결과물 10건 이상을 ID/태그 포함 카드로 등록했다.
- [ ] style ID 호출 방식으로 신규 주제 생성 테스트를 수행했다.
- [ ] 외부 프롬프트 유입 시 검증 절차(테스트 후 편입)를 문서화했다.
- [ ] 월간 운영 규칙(등록/검수/폐기/추천)을 팀 규칙으로 합의했다.
- [ ] 추출 스타일(`style_extract.py`)의 정확도를 샘플로 점검했다.

## 6) 산출물
- 스타일 DB(카드 10개 이상) 및 운영 규칙 문서
- style ID 호출 예시 모음(채널별)
- 외부 프롬프트 검증 체크리스트
- 월간 품질 리뷰 로그
- 스타일 카드 운영 대시보드(등록 수, 재사용률, 폐기율, 추천 Top 5 style ID)

### 통합/삭제 판단 기준
- 아래 조건 중 2개 이상 충족 시 3.1.3 또는 3.2.x 운영 문서로 통합 검토:
  1. 팀 규모가 작아 스타일 DB 유지 비용 대비 활용도가 낮다.
  2. 실제 제작이 대부분 단발성이라 재사용 가치가 제한적이다.
  3. 3.2.x 채널별 에셋 운영 문서에서 동일 내용을 이미 강하게 다루고 있다.
- 반대로, 반복 제작/다채널 운영/팀 확장이 빈번한 조직은 독립 모듈 유지가 유리하다.
