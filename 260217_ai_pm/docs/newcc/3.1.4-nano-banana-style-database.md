## 1) 목적
- 일회성 이미지 생성 결과를 팀 자산으로 전환하는 **스타일 라이브러리 운영 체계**를 구축한다.
- PM/디자인/마케팅이 동일한 스타일 자산을 재사용해 제작 속도와 일관성을 동시에 높인다.
- “좋은 결과 기억”이 아닌 “검색·호출·재적용 가능한 시스템”으로 전환한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 핵심은 이미지 생성 기술이 아니라 **저장→재사용→축적**이라는 메타 스킬이다.
2. 시작 명령은 `/start-3-1-4`, 선행 권장은 3.1.3이며 모듈 분량은 약 20분이다.
3. `style-library.html`에는 ID/썸네일/이름/카테고리/태그/프롬프트/활용 예시가 저장된다.
4. 활용 방식은 `Generate [subject] using style #X`처럼 **번호 기반 호출**로 단순화된다.
5. 라이브러리 확장 경로는 3가지다: 실무 중 저장, 외부 프롬프트 검증 후 편입, 레퍼런스 이미지에서 스타일 추출.
6. `style_extract.py`와 Gemini를 통해 색감·조명·구도·질감·분위기·타이포·효과를 텍스트 스타일로 구조화할 수 있다.
7. 카테고리/태그 체계를 잘 설계할수록 탐색 속도와 재현성이 함께 올라간다.
8. 저장 반영 이슈는 브라우저 새로고침/저장 완료 여부 확인으로 해결되는 경우가 많다.

### Claude Code 핵심 기능 연결 (official 반영)
- **재사용 중심 자산화(Overview)**: 생성 결과를 대화에서 끝내지 않고 ID 기반 라이브러리로 운영해 누적 효율을 만든다.
- **입문자도 쓰는 규칙 설계(Tutorial: Getting Started)**: 카드 필드(ID/카테고리/태그/용도)를 단순화해 팀 확장 시 학습 비용을 줄인다.
- **고급 분류/요약 품질(Tutorial: Claude Opus 4.6)**: 카드 설명을 추상어가 아닌 실행 가능한 스타일 지시문으로 정제한다.
- **외부 참고 유입(Tutorial: Claude for Chrome)**: 레퍼런스 스타일을 빠르게 조사해 검증 후 DB에 편입하는 파이프라인을 만든다.
- **협업 검수 루프(Tutorial: Claude Cowork)**: PM/디자이너/마케터가 등록·검수·폐기를 분담해 DB 품질을 유지한다.
- **거버넌스 기반 운영(Tutorial: Financial Services)**: 변경 이력, 승인 책임자, 폐기 기준을 명시해 라이브러리 신뢰도를 확보한다.
- **명령형 호출 표준화(Claude Code)**: `Generate [subject] using style #X` 패턴으로 제작 시간을 단축하고 재현성을 높인다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 스타일 카드 스키마 정의
- 입력 아티팩트: 기존 우수 이미지 10개, 채널 목록
- 프롬프트 조각: "style card에 필수로 들어갈 필드(ID, 카테고리, 태그, 용도, 금지조건, 대표 프롬프트)를 제안해줘."
- 기대 출력: `style-card-schema.md`
- 검증 체크포인트: 검색/재사용에 필요한 필드가 누락되지 않는다.

2) 1차 카드 등록
- 입력 아티팩트: 우수 결과물 10건
- 프롬프트 조각: "각 결과물을 카드 형태로 요약하고 style ID를 부여해줘."
- 기대 출력: `style-library.html` 1차 데이터
- 검증 체크포인트: 카드마다 ID와 대표 사용 채널이 존재한다.

3) 태그 분류 체계 고정
- 입력 아티팩트: 팀 용어 사전, 기존 태그 목록
- 프롬프트 조각: "태그 난립을 막기 위해 승인 태그 사전(상위 15개)을 만들어줘."
- 기대 출력: `approved-tags.md`
- 검증 체크포인트: 유사 태그(예: clean/minimal/minimalistic)가 통합된다.

4) style ID 호출 테스트
- 입력 아티팩트: style #1~#5, 신규 주제 3개
- 프롬프트 조각: "Generate [subject] using style #X 형태로 채널별 재현 테스트를 설계해줘."
- 기대 출력: `style-reuse-test-plan.md`, 테스트 결과 이미지
- 검증 체크포인트: 동일 style ID로 채널 3곳에서 톤 재현 가능.

5) 스타일 추출 자동화 보조
- 입력 아티팩트: 참조 이미지 세트, `style_extract.py` 출력
- 프롬프트 조각: "추출된 스타일 설명에서 재사용 가치가 높은 문장만 카드 설명으로 정제해줘."
- 기대 출력: 카드 설명 고도화 버전
- 검증 체크포인트: 추출 문장이 추상어보다 실행 가능한 지시문 형태다.

6) 월간 검수/폐기 규칙 수립
- 입력 아티팩트: 사용 로그, 성과 지표(재사용률/전환 기여)
- 프롬프트 조각: "저품질 카드를 정리하기 위한 폐기 기준과 유지 기준을 수치로 제안해줘."
- 기대 출력: `style-governance.md`
- 검증 체크포인트: 등록/검수/폐기 주기와 책임자가 명시된다.

7) 신규 팀원 온보딩 검증
- 입력 아티팩트: 스타일 DB, 호출 가이드
- 프롬프트 조각: "신규 팀원이 style ID 기반으로 1일 내 같은 톤을 재현하도록 온보딩 절차를 작성해줘."
- 기대 출력: `style-db-onboarding.md`
- 검증 체크포인트: 신규 인원이 style ID 3개를 성공적으로 재현한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 새로운 프롬프트를 매번 새로 쓰지 않고, 검증된 style ID를 즉시 재사용한다.
- **P2 맥락 우선**: 스타일 카드에 용도/채널/금지 조건을 함께 기록해 맥락 없는 재사용을 방지한다.
- **P6 자동화 운영 기반**: 스타일 등록·검수·폐기 주기를 정의하면 품질 관리가 가능해진다.
- **P8 Connected Hub**: 스타일 자산을 문서/캠페인/프로토타입 작업과 연결해 분절된 제작을 줄인다.
- **P9 Words-to-Workflow**: “style #12로 생성” 같은 짧은 언어 명령을 표준 실행 트리거로 사용한다.
- Week 3~6 운영 관점에서, 스타일 DB는 콘텐츠 자동화·리뷰 루프·채널 확장 시 재사용률을 높이는 핵심 인프라다.

## 4) 실무 적용 시나리오
**상황: 제품 업데이트마다 비슷한 채널 자산을 반복 제작하는 팀**
1. 이번 릴리스에서 성과가 좋았던 비주얼 10개를 스타일 카드로 등록한다.
2. 카드마다 적용 채널(랜딩/SNS/뉴스레터)과 금지 조건(텍스트 밀도, 색상 충돌)을 기록한다.
3. 다음 캠페인에서 style ID 기반으로 초안을 빠르게 생성한다.
4. 품질 낮은 카드는 월간 리뷰에서 아카이브 처리하고, 우수 카드는 추천 태그를 강화한다.
5. 신규 팀원도 DB를 통해 동일 톤의 자산을 1일 내 재현할 수 있게 한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `style-library.html` 카드 스키마(ID, 태그, 용도, 금지 조건)를 확정하고 기존 우수 결과물 10건을 1차 등록한다.
- **수~목 실행**: style ID 호출 테스트를 채널별로 수행해 재현성(랜딩/SNS/뉴스레터)을 점검하고, `style_extract.py` 추출 결과를 카드 설명에 반영한다.
- **금요일 검증**: 월간 운영 리허설(등록, 검수, 폐기, 추천)을 1회 돌려 DB가 실제 제작 속도를 높이는지 측정한다.
- **Done 기준**: 카드 10개 이상이 검색 가능 상태로 정리되고, style ID 기반 생성 결과가 최소 3개 채널에서 일관되게 재현된다.
- **리스크/완화**: 태그 난립 리스크는 승인된 태그 사전으로 통제하고, 저품질 카드 누적은 월간 폐기 기준(재현성/전환기여/브랜드 적합성)으로 완화한다.

- 운영 팁 1: 월요일 등록은 '많이'보다 '검색 가능한 카드'를 만드는 데 집중한다.
- 운영 팁 2: 화요일 태그 정리는 초기 강하게 잡아야 이후 유지보수 비용이 줄어든다.
- 운영 팁 3: 금요일 온보딩 테스트가 통과되지 않으면 DB는 자산이 아니라 부채다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 카드가 늘어나는데 검색이 더 어려워짐
- 근본 원인: 태그/카테고리 체계가 통제되지 않음
- 해결 방법: 승인 태그 사전 운영 + 신규 태그는 리뷰 승인 후 추가

2) 블로커: style ID를 써도 결과 톤이 들쭉날쭉
- 근본 원인: 카드에 금지 조건/사용 채널이 없음
- 해결 방법: 카드 필수 필드에 금지 조건과 채널 적합성 추가

3) 블로커: 저품질 카드가 쌓여 신뢰도 하락
- 근본 원인: 폐기 기준 없이 등록만 계속함
- 해결 방법: 월간 폐기 기준(재현성 점수, 사용 빈도, 전환 기여) 적용

4) 블로커: 신규 팀원이 DB를 어려워함
- 근본 원인: ID 호출 예시가 부족하고 학습 경로가 없음
- 해결 방법: 입문용 Top 5 style ID와 사용 예시 세트를 별도 제공
