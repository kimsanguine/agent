## 1) 목적
- 48시간 내 체감 가능한 성과를 만드는 Level1 Quick Win 4종을 표준화한다.
- 문서/댓글/티켓/상태공유의 반복 업무를 Claude.ai + MCP로 즉시 단축한다.
- Level2 진입 전에 최소 운영 신뢰도를 확보한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 고급 모듈(2.1)은 맥락 파일을 붙여 PRD 초안을 만들고 다중 관점 리뷰(Engineer/Executive/Researcher)를 거쳐 품질을 높인다.
- 원문 2.2는 데이터 기반 의사결정을 위해 문제 분석→임팩트 추정→실험 분석 3단 산출물을 요구한다.
- 원문 2.3은 병렬 리서치와 구조화된 전략 문서화를 통해 “선택과 포기”를 명확히 한다.
- Company Context 문서군은 모든 작업이 회사 목표/KPI와 연결되도록 기준선을 제공한다.
- 즉, Level1은 “대화형 답변”이 아니라 “근거 있는 문서 산출물”을 빠르게 만드는 단계다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview의 실전 흐름을 반영하면, Level1 Quick Win은 단순 생성이 아니라 **문서 게시→피드백 분류→티켓화→공유**의 연속 작업으로 운영해야 한다.
- PM은 Claude AI 시작 튜토리얼 패턴대로 맥락을 먼저 고정한 뒤 실행하면 PRD 품질과 팀 합의 속도를 동시에 높일 수 있다.
- Opus 4.6 튜토리얼 관점에서 댓글 분류·티켓 전환 같은 복합 작업은 계획형 지시와 출력 포맷 고정이 효과적이다.
- Claude Code + MCP 조합은 Notion/Linear/Slack 핸드오프를 줄여 리드타임 단축에 직접 기여한다.
- Chrome 튜토리얼 활용 시 외부 리서치 근거를 Quick Win 입력으로 주입해 PRD 근거 밀도를 빠르게 높일 수 있다.
- 금융 서비스 튜토리얼에서 강조하는 신뢰성 관점처럼, 상태보고에는 출처·리스크·다음 액션을 강제해 의사결정 품질을 보호한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, PM이 이번 주 과제 1건을 선택하고 Company Context에서 목표/KPI/제약 3가지를 추출해 입력 맥락으로 고정한다.
2. 월요일 오후, Claude.ai에서 PRD 초안을 생성하고 Notion MCP로 게시해 팀이 같은 문서를 기준으로 피드백할 수 있게 만든다.
3. 화요일 오전, Notion 댓글을 Blocker/Question/Idea로 자동 분류하고 각 항목에 응답 초안과 담당자 후보를 붙인다.
4. 화요일 오후, PRD 기능을 Epic/Story 단위로 분해해 Linear MCP로 생성하고 AC 기준을 Story마다 1~3개로 제한한다.
5. 수요일, Linear 상태와 KPI 변화를 모아 OMTM 중심 상태보고 초안을 만들고 Slack 채널에 자동 공유한다.
6. 목요일, 4개 Quick Win 결과를 표본 검수해 사실 오류/누락/중복을 수정하고 재실행 규칙을 문서화한다.
7. 금요일 오전, 리드타임/재작업률/오류율을 비교해 가장 병목인 단계 1개를 식별한다.
8. 금요일 오후, 병목 개선 액션을 다음 주 운영 계획에 반영하고 Level2로 승격할 템플릿/명령어 후보를 확정한다.

### PM Use Case 확장 (Claude Code 기능 매핑)

#### 월~금 상세 실행표
| 요일 | Quick Win | 입력 준비물 | 산출물 | 검증 기준 | 복구 흐름 |
|---|---|---|---|---|---|
| 월 | PRD 게시 | 맥락/KPI/제약 | Notion PRD | 목표/범위/AC 완성 | 템플릿 보강 후 재생성 |
| 화 | 댓글 분류 | Notion 댓글 | 분류표+응답 초안 | 분류 정확도 90%+ | taxonomy 재정의 |
| 수 | PRD→티켓화 | 기능/의존성 | Linear Epic/Story | Story granularity 적정성 | 수동 분해 후 재동기화 |
| 목 | 상태 공유 | 티켓 상태+KPI | Slack 보고 | OMTM/리스크/다음 액션 포함 | 누락 항목 재수집 |
| 금 | 성과 리뷰 | 4개 실행 로그 | 주간 개선안 | 병목 1개 이상 식별 | 병목 단계만 집중 보완 |

#### 검증/복구 흐름
1. PRD가 generic하면 Company Context 입력을 추가하고 재생성한다.
2. 댓글 분류 정확도가 90% 미만이면 샘플 20건을 수동 라벨링해 taxonomy를 재고정한다.
3. 티켓이 과대/과소 분해되면 Story AC 개수를 1~3으로 제한하고 재생성한다.
4. 상태보고에서 OMTM 변화가 빠지면 보고를 반려하고 필수 3항목(변화/리스크/다음 액션)으로 재작성한다.
5. 금요일 리뷰에서 병목이 특정되지 않으면 리드타임 로그를 단계별로 쪼개 재분석한다.

### 실행 순서 (당일~2일)
1. **PRD 게시**
   - 입력: 회사 맥락 + 리서치 근거 + PRD 템플릿
   - 실행: Claude.ai 초안 → Notion MCP 게시
   - 산출물: Notion PRD URL
2. **댓글 분류/응답**
   - 입력: Notion 댓글
   - 실행: Blocker/질문/아이디어로 분류 + 응답 초안 생성
   - 산출물: 댓글 처리표
3. **PRD→Linear 티켓화**
   - 입력: PRD의 기능/AC/의존성
   - 실행: Epic 1개 + Story N개 생성
   - 산출물: Linear 이슈 링크 묶음
4. **주간 상태공유 초안**
   - 입력: Linear 상태 + KPI
   - 실행: OMTM 포함 상태보고 생성 후 Slack 공유
   - 산출물: Slack 상태보고 링크

### 실패 패턴
- 실패1: 입력 맥락 부족으로 PRD가 generic함
  - 대응: Company Context + KPI + 제약 조건을 강제 입력
- 실패2: 댓글 분류 기준 불일치
  - 대응: 분류 taxonomy(예: blocker/question/idea) 사전 고정
- 실패3: 티켓 granularity 과대/과소
  - 대응: Story당 완료 기준(AC 1~3개) 제한
- 실패4: 상태보고가 활동 나열로 끝남
  - 대응: OMTM 변화, 리스크, 다음 액션 3줄 필수

### 이번 주 완료형 실행 플랜
- **월~화:** Notion/Linear/Slack MCP 권한을 점검하고 PRD 게시→댓글 분류까지 Quick Win 1~2번을 당일 실행한다.
- **수~목:** PRD→Linear 티켓화와 OMTM 상태공유까지 Quick Win 3~4번을 완료하고 재실행 가능한 프롬프트를 고정한다.
- **금 검증:** 4개 Quick Win의 리드타임·오류·재작업률을 비교해 가장 병목인 단계를 식별한다.
- **Done 기준:** Quick Win 4종 결과 링크, 실행 로그, 실패 패턴 수정안이 1개 문서로 정리된다.
- **리스크·완화:** 권한 누락/필드 매핑 오류가 빈번하므로 실행 전 체크리스트(권한·필드·owner)를 선검증하고 실패 시 수동 백업 경로를 사용한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: MCP 권한은 연결됐지만 필드 매핑 오류 발생**
  - 증상: Notion/Linear 전송 시 필수 필드 누락으로 실패한다.
  - 해결: 실행 전 필수 필드 체크리스트(title/owner/status/due)를 자동 점검하고 누락 시 즉시 중단한다.
- **막힘 2: 댓글 분류는 되었지만 우선순위 기준이 모호함**
  - 증상: Blocker와 Question 경계가 불명확해 처리 순서가 흔들린다.
  - 해결: `영향도 x 긴급도` 2축 우선순위 룰을 도입해 분류 후 즉시 정렬한다.
- **막힘 3: 상태 공유가 활동 나열로 끝남**
  - 증상: 보고를 읽어도 다음 액션이 보이지 않는다.
  - 해결: OMTM 변화 수치, 리스크 1개, 다음 액션 3개를 강제 필드로 넣는다.
- **막힘 4: Quick Win을 했지만 Level2로 승격되지 않음**
  - 증상: 매주 같은 수작업을 반복한다.
  - 해결: 반복된 실행 문장을 Slash Command 후보로 분류해 CLAUDE.md 운영 자산으로 이관한다.
