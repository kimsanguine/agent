# 4.1 Vibe Coding - Setup

## 1) 목적
- Vibe Coding 실행 전, 프로젝트 파일/도구/협업 규칙을 정리해 **초기 시행착오를 최소화**한다.
- PM-LLM 협업의 기본 루프(Plan→Build→Iterate→Save→Go Live)를 환경 단계에서 체득한다.
- 4.2 계획 단계로 자연스럽게 넘어갈 수 있는 출발 상태를 만든다.

## 2) 원문 핵심 요약 (ccforpms)
1. 레슨 시작 명령은 `/start-4-1`이며, 약 10분 내 기본 준비를 마치는 구조다.
2. Module 4에는 신규 파일이 포함될 수 있어, 기존 파일이 있어도 **course files 재다운로드**가 필요하다.
3. Vibecoding 핵심은 직접 코드를 많이 치는 것이 아니라, 원하는 결과를 정확히 설명하고 AI가 구현하게 하는 방식이다.
4. 역할 분담은 명확하다: PM은 요구사항/피드백, Claude는 구현.
5. 이 협업 방식은 개발뿐 아니라 디자이너/에이전시/다른 AI 도구와의 협업에도 재사용 가능하다.
6. Next.js는 웹앱 제작과 Vercel 배포 연결성이 높아 실습 기본 스택으로 적합하다.
7. 설정 단계의 성공 기준은 “툴 설치 완료”보다, 이후 단계에서 바로 빌드 가능한 **작업 컨텍스트 확보**다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 설치 완료 체크보다, 4.2 인터뷰 기반 계획으로 곧바로 진입 가능한 상태를 목표로 둔다.
- **P2 맥락 우선**: 프로젝트 목적/타깃/완료조건을 초기부터 문서화해 품질 편차를 낮춘다.
- **P6 자동화 운영 기반**: 폴더 구조/템플릿/기본 명령을 표준화해 반복 작업 비용을 줄인다.
- **P8 Connected Hub**: 로컬 개발환경, 버전관리, 배포 경로를 한 번에 연결하는 허브 관점으로 셋업한다.
- **Level4 연결**: `/agent`가 활용할 수 있도록 PRD/TDD/Eval/Prototype 산출물 위치를 미리 정리한다.
- **P9 Words-to-Workflow**: 셋업 문서의 문장 하나하나가 이후 자동 실행 규칙(명령/폴더/검증)으로 작동하게 설계한다.

## 4) 실무 적용 시나리오
**상황: PM 2명과 디자이너 1명이 3일 스프린트를 시작**
1. 공통 폴더와 기본 템플릿을 준비해 누가 작업해도 같은 구조를 유지한다.
2. 팀이 합의한 요구사항 문장 포맷(목표/대상/성공지표)을 세팅한다.
3. 빌드와 미리보기 실행이 되는지 확인해 “환경 문제”를 초기에 제거한다.
4. 작업 시작 전 역할 분담(기획/구현/리뷰)을 명시한다.
5. 4.2로 넘어가 인터뷰 방식으로 실제 요구사항을 확정한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 개발 환경 설치, 코스 파일 재다운로드, 공통 폴더 구조/명령어를 팀 위키에 고정해 착수 준비를 끝낸다.
- **수~목 실행**: 샘플 실행과 로컬 빌드 확인을 완료하고, PM/구현/리뷰 역할 분담표를 확정한다.
- **금요일 검증**: 신규 팀원이 문서만 보고 30분 내 환경을 재현하는지 드라이런해 Setup 문서의 누락을 보완한다.
- **Done 기준**: Ready/Not Ready 판단표에서 모든 필수 항목이 Ready이며, 4.2 인터뷰 템플릿이 즉시 사용 가능한 상태다.
- **리스크/완화**: OS별 설치 차이로 막히는 경우 공통 명령을 최소화하고 OS별 예외 명령을 별도 블록으로 분리해 재시도 시간을 줄인다.
