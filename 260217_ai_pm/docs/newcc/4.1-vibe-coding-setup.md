# 4.1 Vibe Coding - Setup

**빠른 요약**
- 4.1 Vibe Coding - Setup 주제를 빠르게 이해하고 바로 써볼 수 있게 정리한 문서입니다.

**처음 보는 분 가이드**

## 1) 목적
- Vibe Coding 실행 전, 프로젝트 파일/도구/협업 규칙을 정리해 **초기 시행착오를 최소화**한다.
- PM-LLM 협업의 기본 루프(Plan→Build→Iterate→Save→Go Live)를 환경 단계에서 체득한다.
- 4.2 계획 단계로 자연스럽게 넘어갈 수 있는 출발 상태를 만든다.

## 2) 원문 핵심 요약 (ccforpms)
1. 레슨 시작 명령은 `/start-4-1`이며, 약 10분 내 기본 준비를 마치는 구조다.
2. Module 4에는 신규 파일이 포함될 수 있어, 기존 파일이 있어도 **course files 재다운로드**가 필요하다.
3. Vibecoding 핵심은 직접 코드를 많이 치는 것이 아니라, 원하는 결과를 정확히 설명하고 AI가 구현하게 하는 방식이다.
4. 역할 분담은 명확하다: PM은 요구사항/피드백, Claude는 구현.
5. 이 협업 방식은 개발뿐 아니라 디자이너/에이전시/다른 AI 도구와의 협업에도 재사용 가능하다.
6. Next.js는 웹앱 제작과 Vercel 배포 연결성이 높아 실습 기본 스택으로 적합하다.
7. 설정 단계의 성공 기준은 “툴 설치 완료”보다, 이후 단계에서 바로 빌드 가능한 **작업 컨텍스트 확보**다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **코스 파일 버전을 먼저 확인한다.**
   - 입력 예시: "Module 4 파일이 기존 폴더와 다를 수 있으니 최신본으로 교체".
   - 실행 액션: `claude-code-course` 폴더 기준으로 신규/변경 파일 목록을 체크한다.
   - 기대 출력: 팀원 전원이 동일한 파일 트리를 사용하는 상태.
2. **셋업 시작 명령으로 초기 가이드를 띄운다.**
   - 입력 예시: `/start-4-1`.
   - 실행 액션: Claude가 안내하는 설치/확인 항목을 순서대로 수행한다.
   - 기대 출력: 필수 항목 체크리스트가 `Ready`로 채워진다.
3. **런타임 버전 차이를 제거한다.**
   - 입력 예시: "Node 버전 불일치로 실행 오류".
   - 실행 액션: 팀 기준 Node 버전(예: 20.x)과 패키지 매니저를 문서에 고정한다.
   - 기대 출력: "내 컴퓨터에서만 안 됨" 유형 이슈가 크게 감소한다.
4. **기본 실행 경로를 검증한다.**
   - 입력 예시: "프로젝트 루트에서 dev server 실행".
   - 실행 액션: 의존성 설치 후 개발 서버를 실행하고 `localhost` 접속을 확인한다.
   - 기대 출력: 첫 화면 렌더링 + 콘솔 에러 0건.
5. **역할 분담 규칙을 세팅한다.**
   - 입력 예시: "PM은 요구사항/피드백, Claude는 구현".
   - 실행 액션: 요청 포맷(목표/변경/검증) 템플릿을 팀 위키에 저장한다.
   - 기대 출력: 지시문 품질이 일정해지고 재작업률이 줄어든다.
6. **4.2 진입용 입력값을 사전 준비한다.**
   - 입력 예시: "핵심 사용자, 문제 문장, 성공 지표 1개".
   - 실행 액션: 인터뷰 시작 전에 최소 입력 3종을 미리 작성한다.
   - 기대 출력: 4.2 인터뷰 시간이 단축되고 요구사항 누락이 감소한다.
7. **재현성 테스트로 Setup 완료를 판정한다.**
   - 입력 예시: "신규 팀원 1명이 문서만 보고 환경 재현".
   - 실행 액션: 온보딩 드라이런을 30분 타임박스로 수행한다.
   - 기대 출력: 실패 지점 로그 + Setup 문서 보완 항목이 도출된다.

### Claude Code 핵심 기능 연결 (official 반영)
- **프로젝트 온보딩 표준화**: Claude Code 실행 전 프로젝트 규칙(CLAUDE.md, 명령, 폴더 구조)을 읽어 초기 설정 오류를 줄인다.
- **터미널 기반 환경 검증**: 설치/실행/의존성 확인을 같은 세션에서 수행해 Setup 단계의 재현성을 확보한다.
- **계획 중심 작업 시작**: 바로 구현하지 않고 Setup 체크리스트를 먼저 잠가 4.2 진입 실패를 예방한다.
- **문맥 유지형 협업**: PM이 공유한 목표/제약/역할 분담을 이후 Build 단계까지 이어가 컨텍스트 손실을 줄인다.
- **GitHub/배포 전제 셋업**: 초기부터 저장소·배포 연동을 염두에 둔 구조를 잡아 4.4~4.5 전환 비용을 낮춘다.
- **반복 가능한 운영 템플릿**: 신규 팀원 온보딩 시 동일 프롬프트/검증 절차를 재사용할 수 있게 만든다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 설치 완료 체크보다, 4.2 인터뷰 기반 계획으로 곧바로 진입 가능한 상태를 목표로 둔다.
- **P2 맥락 우선**: 프로젝트 목적/타깃/완료조건을 초기부터 문서화해 품질 편차를 낮춘다.
- **P6 자동화 운영 기반**: 폴더 구조/템플릿/기본 명령을 표준화해 반복 작업 비용을 줄인다.
- **P8 Connected Hub**: 로컬 개발환경, 버전관리, 배포 경로를 한 번에 연결하는 허브 관점으로 셋업한다.
- **Level4 연결**: `/agent`가 활용할 수 있도록 PRD/TDD/Eval/Prototype 산출물 위치를 미리 정리한다.
- **P9 Words-to-Workflow**: 셋업 문서의 문장 하나하나가 이후 자동 실행 규칙(명령/폴더/검증)으로 작동하게 설계한다.

## 4) 실무 적용 시나리오
**상황: PM 2명과 디자이너 1명이 3일 스프린트를 시작**
1. 공통 폴더와 기본 템플릿을 준비해 누가 작업해도 같은 구조를 유지한다.
2. 팀이 합의한 요구사항 문장 포맷(목표/대상/성공지표)을 세팅한다.
3. 빌드와 미리보기 실행이 되는지 확인해 “환경 문제”를 초기에 제거한다.
4. 작업 시작 전 역할 분담(기획/구현/리뷰)을 명시한다.
5. 4.2로 넘어가 인터뷰 방식으로 실제 요구사항을 확정한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 개발 환경 설치, 코스 파일 재다운로드, 공통 폴더 구조/명령어를 팀 위키에 고정해 착수 준비를 끝낸다.
- **수~목 실행**: 샘플 실행과 로컬 빌드 확인을 완료하고, PM/구현/리뷰 역할 분담표를 확정한다.
- **금요일 검증**: 신규 팀원이 문서만 보고 30분 내 환경을 재현하는지 드라이런해 Setup 문서의 누락을 보완한다.
- **Done 기준**: Ready/Not Ready 판단표에서 모든 필수 항목이 Ready이며, 4.2 인터뷰 템플릿이 즉시 사용 가능한 상태다.
- **리스크/완화**: OS별 설치 차이로 막히는 경우 공통 명령을 최소화하고 OS별 예외 명령을 별도 블록으로 분리해 재시도 시간을 줄인다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: PM 2명, 디자이너 1명, 개발자 1명이 3일 스프린트에 들어가기 전 Setup을 표준화한다.

- **월요일**
  - Owner: PM(운영), 개발자(기술 확인)
  - Action: `/start-4-1` 실행, 환경 체크리스트 초안 작성, 코스 파일 재다운로드
  - Artifact: `setup-checklist-v1.md`, 버전 고정표(Node/패키지 매니저)
  - Check: 팀원 4명 중 4명 모두 동일 명령으로 실행 성공
- **화요일**
  - Owner: 개발자, PM
  - Action: 로컬 실행/빌드/미리보기 루틴 점검, 폴더 구조 통일
  - Artifact: 실행 로그 캡처, 공통 폴더 구조 다이어그램
  - Check: `localhost` 첫 화면 렌더링, 치명 에러 없이 동작
- **수요일**
  - Owner: PM
  - Action: 역할 분담(요구사항 작성/구현/리뷰)과 요청 템플릿 확정
  - Artifact: 팀 운영 규칙 문서, 요청 템플릿(`목표-변경-검증`)
  - Check: 샘플 요청 2건을 작성해 응답 품질 일관성 확인
- **목요일**
  - Owner: 신규 참여자(테스터), PM
  - Action: 문서 기반 온보딩 드라이런(30분), 막힘 포인트 기록
  - Artifact: 재현 실패 로그, Setup 문서 보완안
  - Check: 신규 참여자 단독으로 실행 성공 시 `Ready` 판정
- **금요일**
  - Owner: PM, 팀 리드
  - Action: 4.2 진입 전 최종 검토 회의(필수 입력값 3종 확인)
  - Artifact: `4.2-ready-report.md`, 오픈 이슈 목록
  - Check: 핵심 사용자/문제 문장/성공지표가 모두 확정되어 인터뷰 즉시 시작 가능

**리스크 트리거(발동 조건)**
- 화요일 종료 시점까지 팀원 30% 이상이 로컬 실행에 실패하거나, 실행 명령이 개인별로 다르면 Setup 실패 위험으로 판단한다.

**Rollback/Mitigation 흐름**
1. 개인 환경 최적화 시도를 중단하고 팀 표준 버전으로 강제 통일한다.
2. 실패한 머신은 "클린 재설치 경로"로 전환해 최소 실행 상태만 복구한다.
3. 문서에 OS별 예외 명령을 분리 추가하고, 공통 명령 블록은 그대로 유지한다.
4. 금요일 점검 범위를 줄여 4.2 입력값 준비를 우선 완료한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 동일 명령인데 팀원마다 실행 결과가 다름**
   - 해결: Node/패키지 매니저 버전을 표로 고정하고, 설치 경로를 절대경로 기준으로 맞춘다.
   - 확인: `node -v`, 패키지 매니저 버전 출력값을 Setup 문서에 캡처한다.
2. **막힘: 코스 파일 일부만 갱신되어 예제와 실제 구조가 다름**
   - 해결: Module 4 전체 파일을 다시 내려받고, 기존 파일과 비교해 누락 항목을 체크리스트로 정리한다.
   - 확인: "신규 파일 수/수정 파일 수"를 기록해 팀에 공지한다.
3. **막힘: 역할 분담이 모호해 PM 요청이 구현 지시로 변환되지 않음**
   - 해결: 요청 템플릿을 `목표-현상-기대동작-검증` 4블록으로 통일한다.
   - 확인: 샘플 요청 2건을 테스트해 답변 품질 편차를 점검한다.
4. **막힘: 신규 팀원 온보딩 시간이 1시간 이상 걸림**
   - 해결: 드라이런 로그에서 상위 3개 실패 지점을 문서 상단 FAQ로 승격한다.
   - 확인: 다음 드라이런에서 30분 내 완료율 80% 이상이면 통과.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)
- [ ] 이 문서의 목표를 **문제-행동-성과지표** 1문장으로 적었다.
- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.
- [ ] `CLAUDE.md`의 프로젝트 규칙(금지사항/품질게이트/보고형식)을 확인했다.
- [ ] `@` 또는 첨부 자료로 **맥락 입력(문서/데이터/정책) 3개 이상** 연결했다.
- [ ] 필요한 경우 **sub-agent/병렬 작업** 범위와 합성(synthesis) 책임자를 지정했다.
- [ ] **MCP/외부도구 연동 필요 여부**와 접근 권한 상태를 점검했다.
- [ ] 산출물에 **검증 기준(정확성/재현성/지표/테스트)** 을 명시했다.
