## 1) 목적
- `CLAUDE.md`를 프로젝트의 장기 기억 장치로 운영해 세션이 바뀌어도 일관된 품질을 유지한다.
- 글로벌/프로젝트/디렉터리/개인 로컬 메모리 계층을 구분해 충돌을 줄인다.

## 2) 원문 핵심 요약 (ccforpms)
- `CLAUDE.md`는 세션 시작 시 자동 로드되는 프로젝트 메모리 문서다.
- 해당 디렉터리 범위 내 대화에 지속 적용되며 git으로 팀 공유가 가능하다.
- 우선순위 규칙이 명확하다: 충돌 시 프롬프트보다 `CLAUDE.md` 규칙이 우선 적용된다.
- `#`으로 시작하는 규칙은 현재 세션 한정 임시 규칙이며, 검증 후 영구 규칙은 `CLAUDE.md`로 승격한다.
- 적합한 내용은 제품/회사 컨텍스트, 페르소나, 글쓰기 스타일, 용어 표준, ALWAYS/NEVER 불변 규칙이다.
- 부적합한 내용도 분명하다: 단기 스프린트 정보, 자주 바뀌는 목표, 민감정보(API 키).
- 계층 구조가 제공된다: `~/.claude/CLAUDE.md`(Global) → 프로젝트 루트 → 하위 디렉터리 → `CLAUDE.local.md`(개인).
- 운영 권장 기준은 50~200줄 유지, 분기별 점검, PR 리뷰 반영, 로컬 파일은 `.gitignore` 처리다.

### Claude Code 핵심 기능 연결 (official 반영)
- Project Memory는 Claude Code에서 가장 중요한 운영 장치 중 하나로, 세션이 바뀌어도 같은 품질을 재현하게 해주는 장기 컨텍스트 계층이다.
- multi-surface 환경에서도 규칙의 단일 소스는 `CLAUDE.md`이므로, 터미널 실행과 에디터 편집을 동일 기준으로 연결할 수 있다.
- PM 반복 업무 자동화 품질은 프롬프트보다 메모리 규칙의 일관성에 크게 좌우되며, 특히 ALWAYS/NEVER 규칙이 재작업 비용을 줄인다.
- MCP 연동 시에도 어떤 컨텍스트를 우선 적용할지 메모리 계층으로 정해두면 도구 간 충돌을 완화할 수 있다.
- sub-agent/agent team 결과 품질을 맞추려면 공통 출력 규칙과 용어 체계를 `CLAUDE.md`로 고정하는 것이 효과적이다.
- git/PR로 메모리 변경을 관리하면 규칙 추가·삭제·승격의 근거가 팀 자산으로 남는다.
- CLI composability(`#` 임시 규칙 → 효과 검증 → `CLAUDE.md` 승격) 루프를 운영하면 과도한 규칙 누적을 방지하면서 품질을 높일 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **메모리 계층 지도를 먼저 만든다.**
   - 실행 입력(프롬프트): `Global/Project/Directory/Local 메모리 계층을 우리 프로젝트 기준으로 표로 정리해줘.`
   - 기대 출력: 계층별 용도/적용 범위 표.
   - 체크포인트: 어디에 어떤 규칙을 둘지 팀 합의 가능 상태인지 확인.

2. **프로젝트 불변 규칙을 초안으로 만든다.**
   - 실행 입력(프롬프트): `프로젝트 CLAUDE.md에 넣을 ALWAYS/NEVER 규칙 10개를 제안해줘.`
   - 기대 출력: 규칙 후보 리스트.
   - 체크포인트: 단기 이슈가 아닌 장기 반복 규칙인지 확인.

3. **임시 규칙(`#`)으로 먼저 실험한다.**
   - 실행 입력(프롬프트): `# 이번 세션에서는 결과물 끝에 검증 기준 3개를 항상 추가해줘.`
   - 기대 출력: 세션 내 규칙 적용 결과.
   - 체크포인트: 실제 품질 개선이 체감되는지 확인.

4. **효과 검증 후 영구 규칙으로 승격한다.**
   - 실행 입력(프롬프트): `방금 임시 규칙을 CLAUDE.md 영구 규칙 문장으로 변환해줘.`
   - 기대 출력: CLAUDE.md 반영 가능한 문장.
   - 체크포인트: 지나치게 길거나 모호한 문장은 제거.

5. **부적합 정보를 분리한다.**
   - 실행 입력(프롬프트): `CLAUDE.md에 넣으면 안 되는 단기/민감 정보 예시를 분류해줘.`
   - 기대 출력: 제외 대상 목록.
   - 체크포인트: API 키, 임시 목표, 개인 민감정보가 제외됐는지 확인.

6. **로컬 개인 설정을 분리한다.**
   - 실행 입력(행동): `CLAUDE.local.md`에 개인 선호 설정만 기록.
   - 기대 출력: 팀 공유 규칙과 개인 규칙 분리.
   - 체크포인트: 로컬 파일이 git 추적 대상이 아닌지 확인.

7. **정기 점검 루틴을 캘린더에 등록한다.**
   - 실행 입력(프롬프트): `분기별 CLAUDE.md 점검 체크리스트를 만들어줘. 삭제/수정/승격 기준 포함.`
   - 기대 출력: 운영 점검표.
   - 체크포인트: 유지보수 주기와 책임자가 명확한지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 맥락 우선 + Level 2**: 문서 품질보다 컨텍스트 품질이 우선이라는 원칙을 메모리 계층으로 구현한다.
- **P4 속도-검증 균형 + Level 2**: 임시 규칙(`#`) 실험 후 효과 검증된 것만 영구화해 과적합을 방지한다.
- **P6 자동화 운영 기반 + Level 2**: 반복 지시를 메모리로 고정해 재설명 비용을 줄이고 운영 안정성을 높인다.
- **P8 Connected Hub + Level 2~3**: 글로벌/프로젝트/디렉터리 라우팅으로 다양한 업무 흐름을 단절 없이 연결한다.
- **P9 Words-to-Workflow + Level 4**: `CLAUDE.md` 자체가 실행 규칙으로 작동해 자연어 정책을 운영 코드처럼 다룬다.

## 4) 실무 적용 시나리오
PM 리더가 4개 스쿼드의 문서 품질 편차를 줄여야 한다. 글로벌 `CLAUDE.md`에 공통 원칙(용어, 금지 규칙, 보고 형식)을 두고, 스쿼드별 프로젝트 `CLAUDE.md`에 도메인 맥락을 분리한다. 분기마다 규칙을 리뷰해 불필요한 항목을 정리하고, 효과가 입증된 규칙만 표준으로 승격한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 현재 사용 중인 반복 지시를 수집해 Global/Project/Directory/Local 중 어느 계층에 둘지 분류합니다.
- **수~목 실행**: 프로젝트 `CLAUDE.md`에 ALWAYS/NEVER 규칙, 용어 표준, 출력 형식 규칙을 반영하고 실제 작업 2건으로 효과를 검증합니다.
- **금요일 검증**: 임시 규칙(`#`) 적용 결과를 리뷰해 유지/삭제/승격을 결정하고, 중복 규칙을 정리합니다.
- **Done 기준**: 메모리 계층 라우팅표가 완성되고, 팀원이 동일 요청에서 유사한 출력 품질을 재현할 수 있습니다.
- **리스크/완화**: 규칙이 과도하게 늘면 오히려 충돌이 생기므로, 50~200줄 원칙과 분기별 정리 의식을 운영 캘린더에 고정합니다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 시간 | 행동 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 월 | 10:00~11:00 | 반복 지시 수집 및 계층 분류 | 메모리 계층표 v1 | 규칙 10개 이상 분류 완료 |
| 화 | 15:00~16:00 | ALWAYS/NEVER 규칙 초안 작성 | CLAUDE.md 초안 | 모호한 표현 0건(측정 가능한 문장) |
| 수 | 11:00~12:00 | 임시 규칙(`#`) 실험 | 세션 실험 로그 | 실험 전후 품질 차이 기록 |
| 목 | 14:00~15:00 | 효과 있는 규칙 승격 | CLAUDE.md 개정본 | 승격 사유가 문서화됨 |
| 금 | 16:00~17:00 | 중복 규칙 정리 + 점검표 확정 | 분기 점검 체크리스트 | 규칙 길이/충돌 점검 완료 |

- **초보자 실행 팁 1**: 좋은 규칙은 "행동+검증"이 함께 있는 문장입니다.
- **초보자 실행 팁 2**: `CLAUDE.md`에는 변동성 큰 내용(스프린트 일정)을 넣지 않습니다.
- **초보자 실행 팁 3**: 개인 선호는 `CLAUDE.local.md`로 분리해 팀 규칙과 섞지 않습니다.
- **롤백/복구 액션**: 규칙 개정 후 출력 품질이 악화되면, 직전 안정 버전 `CLAUDE.md`로 즉시 롤백하고 신규 규칙은 임시 규칙(`#`) 단계로 되돌려 재검증합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 규칙이 많아져 서로 충돌함**
   - 해결: 분기 점검 시 중복 규칙 삭제, 상위 규칙만 남기고 하위 예시는 부록으로 이동.

2. **막힘: 임시 규칙을 검증 없이 영구화함**
   - 해결: 최소 2회 이상 실무 적용 후 성과 지표(재작업률/오류율)로 승격 여부 결정.

3. **막힘: 민감정보가 메모리에 섞임**
   - 해결: 보안 스캔 체크리스트를 추가해 API 키/개인정보를 작성 단계에서 차단.

4. **막힘: 팀원별 개인 설정이 섞여 결과 편차 발생**
   - 해결: 공유 규칙과 로컬 규칙 저장 위치를 분리하고 코드리뷰에서 경로를 점검.
