## 1) 목적
- `CLAUDE.md`를 프로젝트의 장기 기억 장치로 운영해 세션이 바뀌어도 일관된 품질을 유지한다.
- 글로벌/프로젝트/디렉터리/개인 로컬 메모리 계층을 구분해 충돌을 줄인다.

## 2) 원문 핵심 요약 (ccforpms)
- `CLAUDE.md`는 세션 시작 시 자동 로드되는 프로젝트 메모리 문서다.
- 해당 디렉터리 범위 내 대화에 지속 적용되며 git으로 팀 공유가 가능하다.
- 우선순위 규칙이 명확하다: 충돌 시 프롬프트보다 `CLAUDE.md` 규칙이 우선 적용된다.
- `#`으로 시작하는 규칙은 현재 세션 한정 임시 규칙이며, 검증 후 영구 규칙은 `CLAUDE.md`로 승격한다.
- 적합한 내용은 제품/회사 컨텍스트, 페르소나, 글쓰기 스타일, 용어 표준, ALWAYS/NEVER 불변 규칙이다.
- 부적합한 내용도 분명하다: 단기 스프린트 정보, 자주 바뀌는 목표, 민감정보(API 키).
- 계층 구조가 제공된다: `~/.claude/CLAUDE.md`(Global) → 프로젝트 루트 → 하위 디렉터리 → `CLAUDE.local.md`(개인).
- 운영 권장 기준은 50~200줄 유지, 분기별 점검, PR 리뷰 반영, 로컬 파일은 `.gitignore` 처리다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 맥락 우선 + Level 2**: 문서 품질보다 컨텍스트 품질이 우선이라는 원칙을 메모리 계층으로 구현한다.
- **P4 속도-검증 균형 + Level 2**: 임시 규칙(`#`) 실험 후 효과 검증된 것만 영구화해 과적합을 방지한다.
- **P6 자동화 운영 기반 + Level 2**: 반복 지시를 메모리로 고정해 재설명 비용을 줄이고 운영 안정성을 높인다.
- **P8 Connected Hub + Level 2~3**: 글로벌/프로젝트/디렉터리 라우팅으로 다양한 업무 흐름을 단절 없이 연결한다.
- **P9 Words-to-Workflow + Level 4**: `CLAUDE.md` 자체가 실행 규칙으로 작동해 자연어 정책을 운영 코드처럼 다룬다.

## 4) 실무 적용 시나리오
PM 리더가 4개 스쿼드의 문서 품질 편차를 줄여야 한다. 글로벌 `CLAUDE.md`에 공통 원칙(용어, 금지 규칙, 보고 형식)을 두고, 스쿼드별 프로젝트 `CLAUDE.md`에 도메인 맥락을 분리한다. 분기마다 규칙을 리뷰해 불필요한 항목을 정리하고, 효과가 입증된 규칙만 표준으로 승격한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 현재 사용 중인 반복 지시를 수집해 Global/Project/Directory/Local 중 어느 계층에 둘지 분류합니다.
- **수~목 실행**: 프로젝트 `CLAUDE.md`에 ALWAYS/NEVER 규칙, 용어 표준, 출력 형식 규칙을 반영하고 실제 작업 2건으로 효과를 검증합니다.
- **금요일 검증**: 임시 규칙(`#`) 적용 결과를 리뷰해 유지/삭제/승격을 결정하고, 중복 규칙을 정리합니다.
- **Done 기준**: 메모리 계층 라우팅표가 완성되고, 팀원이 동일 요청에서 유사한 출력 품질을 재현할 수 있습니다.
- **리스크/완화**: 규칙이 과도하게 늘면 오히려 충돌이 생기므로, 50~200줄 원칙과 분기별 정리 의식을 운영 캘린더에 고정합니다.

## 5) 실습/적용 체크리스트
- [ ] `/start-1-6` 실행 후 메모리 계층(Global/Project/Directory/Local)을 실제 경로로 정리했다.
- [ ] 프로젝트 `CLAUDE.md`에 불변 규칙(ALWAYS/NEVER)을 5개 이상 작성했다.
- [ ] 임시 규칙(`#`) 1개를 실험하고 영구화 여부를 평가했다.
- [ ] `CLAUDE.local.md`를 개인 설정으로 분리하고 `.gitignore` 반영을 확인했다.
- [ ] 분기 점검 체크리스트(삭제/수정/승격 기준)를 만들었다.
