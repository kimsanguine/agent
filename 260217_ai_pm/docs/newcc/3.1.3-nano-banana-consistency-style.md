## 1) 목적
- 시리즈형 이미지 제작에서 스타일 편차를 줄이고, 발표/제안/마케팅 자산의 **톤 일관성**을 확보한다.
- 재생성 난사를 줄이고, **선택안 정제(iteration)** 중심의 생산 체계를 정착시킨다.
- style·subject reference를 전략적으로 결합해 PM 산출물의 신뢰도를 높인다.

## 2) 원문 핵심 요약 (ccforpms)
1. Golden Rule: 처음부터 다시 생성하기보다 **기존 결과를 구체적으로 수정**하는 편이 품질/속도 모두 유리하다.
2. 태그 나열(tag soup)보다 자연어 브리프가 효과적이며, subject/setting/lighting/mood/textures/colors/composition을 명확히 써야 한다.
3. reference images는 목적별로 나눈다: **Style reference**(톤), **Subject reference**(인물/제품 동일성), **Composition reference**(배치).
4. subject 일관성은 보통 reference 3~5장일 때 안정성이 높다.
5. variants는 방향 탐색(2~3개), iteration은 선택안 완성으로 역할을 분리해야 한다.
6. grid 생성 시에는 크기(예: 3x3), 셀 내용, 공통 일관성 조건을 명시해야 품질이 올라간다.
7. 실패 패턴은 스타일 미스매치, 캐릭터 불일치, grid 불균일, 변형안 유사도 과다이며, 모두 지시 구체화로 완화 가능하다.
8. “meaningfully different creative interpretations” 같은 지시를 넣으면 variant 다양성을 높일 수 있다.

### Claude Code 핵심 기능 연결 (official 반영)
- **반복 개선 우선 워크플로(Overview)**: 재생성 난사보다 선택안 수정 중심으로 일관성과 생산성을 동시에 확보한다.
- **자연어 브리프 고도화(Tutorial: Getting Started)**: 태그 나열 대신 subject/lighting/mood/composition을 문장으로 명확히 정의한다.
- **정교한 차별화 지시(Tutorial: Claude Opus 4.6)**: variant 생성 시 “의미 있게 다른 해석”을 강제해 탐색 품질을 높인다.
- **참조자료 수집 보조(Tutorial: Claude for Chrome)**: 스타일 레퍼런스를 빠르게 모아 style/subject/composition 분리 운영에 활용한다.
- **협업 리뷰 체계(Tutorial: Claude Cowork)**: PM-디자이너가 공통 루브릭(색/질감/구도/동일성)으로 편차를 빠르게 정렬한다.
- **운영 규율 강화(Tutorial: Financial Services)**: 스타일 코어, 변경 승인, 버전 이력을 남겨 시리즈 자산의 추적 가능성을 높인다.
- **팀 재사용 템플릿화(Claude Code)**: variants→iteration 패턴을 표준 템플릿으로 고정해 신규 과제 진입 시간을 줄인다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 스타일 코어 문장 잠금
- 입력 아티팩트: 기존 우수 결과 3장, 브랜드 가이드
- 프롬프트 조각: "색감/조명/질감/구도를 포함한 스타일 코어 문장 2개를 만들고 최종 1개를 선택해줘."
- 기대 출력: `style-core-v1.md`
- 검증 체크포인트: 코어 문장에 추상어 대신 관찰 가능한 표현이 포함된다.

2) reference 세트 분리 구성
- 입력 아티팩트: style reference 2장, subject reference 3장, composition reference 2장
- 프롬프트 조각: "reference 목적을 style/subject/composition으로 분리하고 각 세트의 사용 금지 조건을 써줘."
- 기대 출력: `reference-set-map.md`
- 검증 체크포인트: 각 reference가 어떤 편차를 줄이는지 설명 가능하다.

3) variants 탐색(2~3개)
- 입력 아티팩트: 코어 문장 + reference 세트
- 프롬프트 조각: "의미 있게 다른 3가지 해석(creative interpretations)을 생성하되 스타일 코어는 유지해줘."
- 기대 출력: `outputs/style-variants-*.png`
- 검증 체크포인트: 결과가 유사 복제본이 아닌 방향성 차이를 가진다.

4) 선택안 iteration 집중
- 입력 아티팩트: variants 중 1순위 안
- 프롬프트 조각: "선택안에서 인물 동일성은 유지하고 배경 텍스처만 개선해줘."
- 기대 출력: `outputs/style-iter-*.png`
- 검증 체크포인트: iteration은 완성도 향상이며 방향 전환이 아니다.

5) 시리즈 5장 확장
- 입력 아티팩트: 슬라이드별 메시지 5개
- 프롬프트 조각: "스타일 코어를 고정한 채 장표별 메시지만 반영해 5장 시리즈를 만들어줘."
- 기대 출력: 5장 시리즈 세트
- 검증 체크포인트: 색/질감/조명 일관성 점수가 기준 이상이다.

6) 일관성 루브릭 점검
- 입력 아티팩트: 시리즈 5장, 평가 기준표
- 프롬프트 조각: "색감/질감/구도/인물 동일성 4축으로 점수화하고 편차 원인을 지적해줘."
- 기대 출력: `consistency-scorecard.md`
- 검증 체크포인트: 편차 큰 장표는 재작업 규칙에 따라 즉시 수정된다.

7) 운영 템플릿화
- 입력 아티팩트: 전체 로그, 수정 전후 사례
- 프롬프트 조각: "다음 프로젝트에서 재사용할 variants→iteration 템플릿을 만들어줘."
- 기대 출력: `consistency-workflow-template.md`
- 검증 체크포인트: 신규 과제에 1회 적용해 같은 품질을 재현한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 맥락 우선**: 스타일 일관성은 마법 키워드가 아니라 참조 품질과 사용 맥락 정의에서 나온다.
- **P4 속도-검증 균형**: variants로 빠르게 탐색하고 iteration으로 검증·수정해 낭비를 줄인다.
- **P5 한국 실무 적합성**: 캠페인/제안서/세일즈덱처럼 다장 구성물이 많은 국내 PM 환경에서 일관성 관리가 핵심이다.
- **P6 자동화 운영 기반**: 반복 지시 패턴을 템플릿화하면 팀 단위 제작 속도가 올라간다.
- **P7 Automation vs Augmentation**: 변형 생성은 자동화, 브랜드 톤 최종 판정은 PM/디자이너의 증강 판단으로 남긴다.
- **P9 Words-to-Workflow**: 자연어 브리프 품질이 곧 산출물 일관성 품질이라는 원칙을 실습으로 체득한다.

## 4) 실무 적용 시나리오
**상황: 동일 기능을 설명하는 5장 슬라이드 비주얼 제작**
1. 첫 장에서 스타일 기준(색감/질감/조명/카메라 톤)을 확정한다.
2. 나머지 4장은 subject와 메시지만 바꾸고 스타일 코어 문장은 고정한다.
3. 장별로 variants 2개씩 뽑아 전달력 높은 안을 고른다.
4. 선택안만 iteration으로 정리해 세트 전체의 균형을 맞춘다.
5. 최종본은 2K로 통일 출력해 문서/발표 채널에 재사용한다.

### PM Use Case 확장 (Claude Code 기능 매핑)
| PM 상황 | Claude Code 기능 | 실행 예시 | 검증 포인트 |
|---|---|---|---|
| 장표 시리즈 톤이 매번 달라짐 | 스타일 코어 고정 + 반복 개선 | `"색감/조명/질감/구도 고정 문장 1개로 잠그고 iteration만 수행"` | 시리즈 평균 일관성 점수 상승 |
| variants가 서로 비슷함 | 차별화 지시 강화 | `"3 variants를 meaningfully different 기준으로 생성"` | 방향성 중복률 감소 |
| 인물/제품 동일성이 깨짐 | reference 세트 분리 운영 | `"style/subject/composition reference를 분리 적용"` | subject 동일성 유지율 개선 |
| 리뷰가 감상 위주로 흐름 | 루브릭 기반 협업 리뷰 | `"색/질감/구도/동일성 4축 점수표 작성"` | 재작업 우선순위가 명확해짐 |
| 시간이 과도하게 소모됨 | 라운드 상한 정책 | `"variants 3개, iteration 2개 상한으로 운영 규칙 설정"` | 생산 시간 예측 가능성 향상 |

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 스타일 코어 문장(색감, 질감, 조명, 구도)을 확정하고 style/subject/composition reference 세트를 최소 1개 프로젝트 기준으로 정리한다.
- **수~목 실행**: 장표 5장 분량을 variants 2개씩 생성한 뒤 선택안만 iteration으로 정리해 세트 전체 톤을 맞춘다.
- **금요일 검증**: 팀 리뷰에서 일관성 루브릭(색, 질감, 인물 동일성, 레이아웃)을 점수화해 편차가 큰 장표를 재작업한다.
- **Done 기준**: 최소 5장 이상의 시리즈 자산이 같은 스타일 코어를 유지하고, 편차 수정 이력과 최종 선택 근거가 문서로 남아 있다.
- **리스크/완화**: style drift 위험은 고정 문장과 참조 이미지 세트를 잠그는 방식으로 완화하고, variant 과다 생성으로 시간 낭비 시 라운드별 생성 수 상한을 둔다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 액션 오너 | 실행 액션 | 산출물(Artifact) | Pass/Fail 체크 | KPI 미달 시 Fallback |
|---|---|---|---|---|---|
| 월 | PM + Designer | 스타일 코어 문장과 reference 세트 확정 | `style-core-v1.md`, `reference-set-map.md` | Pass: 코어 문장 1개로 합의 완료 | 코어 문장을 2문장→1문장으로 단순화 |
| 화 | PM | variants 3개 생성 후 방향성 선택 | `outputs/style-variants-*.png` | Pass: 방향성 차이 3개 확보 | 레퍼런스 구성을 바꿔 variants 재생성 |

- 운영 팁 1: 월요일 합의가 길어지면 화~목 일정이 무너진다. 코어 문장은 1개로 빠르게 잠근다.
- 운영 팁 2: 화요일 variants는 "좋다/나쁘다"보다 "어떤 방향인지"를 분류하는 용도로 사용한다.
- 운영 팁 3: 금요일 점수화는 감상평을 줄이고 팀 공통 언어를 만드는 핵심 단계다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 시리즈에서 스타일이 조금씩 흐트러짐(style drift)
- 근본 원인: 코어 문장과 reference가 매 라운드 바뀜
- 해결 방법: 코어 문장/핵심 reference 잠금 후 변경 승인 절차 적용

2) 블로커: variants가 서로 거의 비슷함
- 근본 원인: 다양성 지시(meaningfully different)가 없음
- 해결 방법: 변형 축(구도/카메라/질감)을 명시해 강제 분기

3) 블로커: 인물/제품 동일성이 깨짐
- 근본 원인: subject reference 수가 부족하거나 품질이 낮음
- 해결 방법: subject reference를 3~5장으로 보강하고 각도 다양성 확보

4) 블로커: 작업 시간이 과도하게 늘어남
- 근본 원인: variants와 iteration 경계가 없음
- 해결 방법: 라운드별 생성 수 상한(예: variants 3개, iteration 2개) 고정
