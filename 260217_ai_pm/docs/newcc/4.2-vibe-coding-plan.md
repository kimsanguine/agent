## 1) 목적
- 구현 전에 요구사항·로직·디자인 결정을 문서로 고정해, 빌드 단계의 재작업을 줄인다.
- PM의 핵심 역량(문제 정의, 규칙 설계, 우선순위 결정)을 코드 이전 단계에서 명확히 수행한다.
- PRD를 실제 개발 가능한 형태로 구체화해 **TDD/Eval/Prototype의 기반**을 만든다.

## 2) 원문 핵심 요약 (ccforpms)
1. 이 레슨은 약 20~25분이며, 구현 이전에 명확한 spec을 만드는 준비 단계다.
2. 시작 명령은 `/start-4-2`; Claude의 인터뷰 질문을 통해 요구사항을 구체화한다.
3. 모호한 지시보다 구체적 스펙이 결과 품질과 속도를 크게 개선한다.
4. 소프트웨어 로직은 if-then 규칙 집합이며, PM은 이를 자연어로 정의하면 된다.
5. 추천/퀴즈류 기능은 입력 → 점수화 → 최고 점수 결과 출력 구조로 설계할 수 있다.
6. 디자인 방향은 텍스트 설명만 하지 말고 HTML preview 비교로 빠르게 결정하는 방식이 유효하다.
7. 최종 의사결정(기능/로직/스타일/콘텐츠)은 `REQUIREMENTS.md`에 저장되어 4.3 구현 기준이 된다.
8. 계획 품질이 낮으면 4.3에서 반복 비용이 크게 증가한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **인터뷰 시작 전에 최소 입력을 준비한다.**
   - 입력 예시: "대상 사용자: 팀 리드 / 문제: 첫 프로젝트 설정이 복잡 / 목표: Activation +10pp".
   - 실행 액션: 사용자·문제·목표를 1줄씩 먼저 작성한다.
   - 기대 출력: 인터뷰 질문이 모호한 탐색이 아니라 결정 중심으로 진행된다.
2. **`/start-4-2`로 요구사항 인터뷰를 시작한다.**
   - 입력 예시: "핵심 기능은 온보딩 흐름 개선, 1차 릴리스 범위는 3화면".
   - 실행 액션: Claude 질문에 범위/제약/우선순위를 수치와 함께 답변한다.
   - 기대 출력: 기능 목록과 제외 범위가 동시에 정리된 초안.
3. **기능을 if-then 규칙으로 쪼갠다.**
   - 입력 예시: "필수 입력 누락이면 다음 단계 이동 불가".
   - 실행 액션: 정상/오류/경계 케이스를 각각 if-then으로 작성한다.
   - 기대 출력: 개발·QA가 동일하게 읽는 로직 규칙표.
4. **예외 처리와 실패 메시지를 먼저 잠근다.**
   - 입력 예시: "네트워크 실패 시 재시도 버튼 + 안내 문구".
   - 실행 액션: 실패 시 사용자 행동 경로를 UI 문구까지 포함해 정의한다.
   - 기대 출력: 구현 후 혼선 없이 재현 가능한 실패 처리 기준.
5. **디자인 방향을 Preview 비교로 결정한다.**
   - 입력 예시: "A안(정보 밀도 높음) vs B안(단순 단계형)".
   - 실행 액션: 2~3안 비교 후 선택 근거(완료율/인지부담)를 남긴다.
   - 기대 출력: 디자인 논쟁 시간을 줄이는 결정 로그.
6. **`REQUIREMENTS.md`를 팀 합의본으로 잠근다.**
   - 입력 예시: "필수 기능 5개, 제외 기능 4개, 수용 기준 6개".
   - 실행 액션: 승인자·승인일·재검토 조건을 문서 하단에 기재한다.
   - 기대 출력: 4.3에서 재질문 없이 바로 구현 가능한 기준 문서 v1.0.
7. **4.3 착수 전 사전 검증을 돌린다.**
   - 입력 예시: "개발자 1명에게 문서만 제공해 구현 계획 재현".
   - 실행 액션: 이해 불명확 문장을 찾아 문장 단위로 수정한다.
   - 기대 출력: 모호 문장 0건, 구현 착수 `Go` 판정.

### Claude Code 핵심 기능 연결 (official 반영)
- **요구사항 인터뷰를 실행 가능한 계획으로 변환**: PM의 답변을 기능/예외/검증 조건으로 구조화해 `REQUIREMENTS.md` 품질을 높인다.
- **Plan 모드 기반 사전 설계**: 큰 변경 전 계획을 먼저 검토해 구현 단계에서의 되돌림 비용을 줄인다.
- **자연어→코드 작업 단위 연결**: if-then 규칙을 실제 구현 단위로 연결할 수 있어 PM이 로직 중심으로 지시 가능하다.
- **반복 정제 루프 지원**: 질문-답변-수정 사이클을 짧게 돌려 모호 문장을 빠르게 제거한다.
- **검증 관점 내장**: 수용 기준(AC)과 실패 조건(FC)을 초기 문서에 넣어 4.3 테스트 관점까지 선행 확보한다.
- **협업 문맥 유지**: 계획 단계의 결정 근거를 Git/PR/배포 단계까지 이어 재설명 비용을 낮춘다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션**: PM은 코드 작성자가 아니라 의사결정 프레임 설계자다.
- **P4 속도-검증 균형**: 구현 전에 요구사항과 실패 조건을 확정해 추후 수정 비용을 줄인다.
- **P7 Automation vs Augmentation**: 구조화/초안 작성은 자동화, 우선순위·예외처리·정책은 PM이 판단한다.
- **Level4 직접 연결**: 이 단계 산출은 곧 **PRD → TDD → Eval → Prototype**의 입력값이 된다.
- **P9 Words-to-Workflow**: `REQUIREMENTS.md`는 단순 문서가 아니라 실행 파이프라인의 소스 코드 역할을 한다.
- v2.4 기준으로 계획 단계에서 OMTM/성공지표/검증 루브릭까지 연결해야 운영 가능한 제품이 된다.

## 4) 실무 적용 시나리오
**상황: 내부 생산성 앱 MVP 기획 회의 직후**
1. 회의 메모를 `문제-사용자-성과` 구조로 재정리한다.
2. `/start-4-2` 인터뷰로 핵심 기능과 비기능 제약을 확정한다.
3. 예외 케이스(잘못된 입력, 경계값, 실패 시 메시지)를 먼저 적는다.
4. HTML preview 2~3안을 비교해 디자인 의사결정을 빠르게 내린다.
5. `REQUIREMENTS.md`를 팀 합의본으로 잠그고 4.3 구현을 시작한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 인터뷰 기반 요구사항 수집으로 문제정의, 핵심 사용자, 성공지표를 확정하고 `REQUIREMENTS.md` 초안을 작성한다.
- **수~목 실행**: 기능/예외 로직을 if-then 규칙으로 정제하고, 디자인 preview 비교 결과를 반영해 합의본 v1.0을 잠근다.
- **금요일 검증**: 개발/디자인/PM 3자 리뷰로 모호 문장과 누락 예외를 제거하고 구현 착수 가능 여부를 최종 판정한다.
- **Done 기준**: 우선순위, 범위 제외 항목, 수용 기준, 검증 지표가 문서에 명시되어 4.3에서 재질문 없이 구현 가능한 상태다.
- **리스크/완화**: 이해관계자 요구가 충돌하면 “이번 주 MVP 필수/차주 후보” 2단 버킷으로 분리해 계획 지연을 막는다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 기획 회의 직후 산발적 요구사항을 5일 안에 구현 가능한 계획으로 정리해야 하는 PM.

- **월요일**
  - Owner: PM(리드)
  - Action: 회의 메모를 `문제-대상-성과` 3블록으로 재정리, 범위 초안 작성
  - Artifact: `planning-brief-v1.md`, 범위 후보 리스트
  - Check: 이해관계자 2명에게 읽혀 핵심 문제 인식이 일치하는지 확인
- **화요일**
  - Owner: PM, 기능 담당 개발자
  - Action: `/start-4-2` 인터뷰 진행, 기능·예외·제약을 수치 포함 문장으로 고정
  - Artifact: `REQUIREMENTS.md` 초안, if-then 규칙표
  - Check: 필수/제외 항목이 각각 최소 3개 이상 명시
- **수요일**
  - Owner: PM, 디자이너
  - Action: HTML preview 2~3안 비교, 사용자 완료율 관점으로 1안 선택
  - Artifact: 디자인 선택 로그, 비선택 사유 문서
  - Check: 선택 기준(완료시간/인지부담/오류율)이 숫자로 기록됨
- **목요일**
  - Owner: PM, QA
  - Action: 수용 기준(AC)과 실패 조건(FC) 정리, 테스트 관점 항목 보강
  - Artifact: `acceptance-criteria-v1.md`, 사전 테스트 시나리오
  - Check: 핵심 흐름별 AC/FC가 1:1로 매칭됨
- **금요일**
  - Owner: PM, 팀 리드
  - Action: 합의본 잠금(버전 태깅), 4.3 구현 착수 승인
  - Artifact: `REQUIREMENTS.md` v1.0(승인자/승인일 포함), 오픈 이슈 목록
  - Check: "재질문 없이 구현 가능" 리뷰 통과 시 `Go`

**리스크 트리거(발동 조건)**
- 수요일 오후까지 범위 제외 항목이 확정되지 않거나, 필수 기능 수가 초기 대비 30% 이상 증가하면 계획 붕괴 신호로 본다.

**Rollback/Mitigation 흐름**
1. 기능 추가 논의를 중단하고, 월요일 합의한 문제 문장으로 범위를 재정렬한다.
2. 필수 기능을 1개 사용자 흐름 중심으로 축소해 MVP 스코프를 다시 잠근다.
3. 제외된 항목은 `다음 스프린트 후보`로 이동하고 기대효과/비용만 기록한다.
4. 금요일 승인 전 20분 재검토 세션으로 문장 모호성만 집중 수정한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 요구사항 문장이 길고 모호해서 해석이 갈림**
   - 해결: 한 문장을 "행위 1개 + 조건 1개 + 결과 1개" 구조로 쪼갠다.
   - 확인: 문장당 동사 1개 원칙을 적용해 중복 동작을 제거한다.
2. **막힘: 예외 케이스가 뒤늦게 발견되어 4.3에서 재작업 발생**
   - 해결: 화요일에 실패 조건을 먼저 정의하고, 정상 케이스는 그 다음에 작성한다.
   - 확인: AC 문서에 최소 1:1로 대응하는 FC가 있는지 점검한다.
3. **막힘: 디자인 논의가 취향 싸움으로 길어짐**
   - 해결: preview 비교 기준을 "완료시간, 클릭수, 오류유발" 3개로 제한한다.
   - 확인: 선택 문서에 비선택 사유를 한 줄씩 남겨 재논쟁을 막는다.
4. **막힘: 승인권자가 마지막 날에 대규모 변경 요청**
   - 해결: 금요일 오전 이전에 `범위 변경 마감`을 공지하고, 신규 요청은 차주 후보로 분리한다.
   - 확인: 승인 문서에 변경 이력과 보류 사유를 함께 기록한다.
