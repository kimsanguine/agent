## 1) 목적
- `@` 파일 참조, 이미지 붙여넣기, output style 재사용을 통해 PM의 반복 커뮤니케이션 작업을 빠르게 자동화한다.
- 첫 실습에서 바로 재사용 가능한 결과물 템플릿을 만들어 이후 업무에 전이시킨다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 시작 명령은 `/start-1-3`, 권장 학습 시간은 약 30분이며 선행 모듈은 1.1·1.2다.
- 핵심 역량 5가지가 명시된다: 파일 읽기/쓰기, 콘텐츠 분석, 이미지 붙여넣기, 시각 분석, output style.
- `@` 표기는 파일 지정의 기본 규칙이며 단일/복수 파일과 폴더 단위 작업을 모두 처리할 수 있다.
- 자주 다루는 파일 포맷 범위가 넓다: `.md`, `.txt`, `.csv`, `.json`, `.pdf`, 이미지(`.png/.jpg/.gif`).
- 이미지 입력은 OS와 무관하게 터미널에서 `Ctrl+V`를 사용한다(Mac 포함).
- 동일 소스를 Slack/Email/Notion 등 채널별 톤과 깊이로 변환해 이해관계자 커뮤니케이션 효율을 높인다.
- output style을 만들어두면 Executive Briefing, User Story, Weekly Update 같은 문서를 반복 생성할 수 있다.
- 오류 대응 기본은 파일명 대소문자, 경로 정확성, 이미지 크기, 저장 위치 점검이다.

### Claude Code 핵심 기능 연결 (official 반영)
- First Tasks는 Claude Code의 multi-surface 가치(입력 즉시 파일/출력 확인)를 체감하는 모듈로, PM 반복 커뮤니케이션 자동화의 출발점이다.
- `@파일` 참조, 이미지 붙여넣기, 포맷 변환은 문서 작성 자동화의 핵심 컴포넌트이며 Slack/Email/Notion 같은 실무 채널에 바로 매핑된다.
- MCP를 연결하면 외부 도구 컨텍스트를 입력에 포함해 채널별 초안 정확도를 높일 수 있다.
- `CLAUDE.md`에 output style 규칙을 고정하면 팀 전체 메시지 톤/구조를 일관되게 유지할 수 있다.
- sub-agent/agent team으로 채널별 초안을 병렬 생성하고 PM이 최종 합성하면 속도와 품질을 동시에 확보할 수 있다.
- git/PR 흐름으로 초안 변경을 추적하면 커뮤니케이션 문서도 의사결정 자산으로 축적된다.
- CLI composability(`/start-1-3`, `@파일`, 채널 변환 지시, 검수 프롬프트)로 반복 업무를 표준 루틴화할 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **첫 태스크 모듈을 실행한다.**
   - 실행 입력(명령): `claude "/start-1-3"`
   - 기대 출력: 첫 태스크 실습 안내.
   - 체크포인트: 1.1/1.2 선행 완료 상태인지 확인.

2. **입력 파일 세트를 `@`로 지정한다.**
   - 실행 입력(프롬프트): `@meeting-notes.md @metrics.csv 읽고 이번 주 업데이트 핵심만 정리해줘.`
   - 기대 출력: 핵심 포인트 요약본.
   - 체크포인트: 참조 파일 누락 없이 모두 반영되는지 확인.

3. **채널별 초안을 병렬 생성한다.**
   - 실행 입력(프롬프트): `같은 내용을 Slack/Email/Notion 3가지 포맷으로 각각 작성해줘.`
   - 기대 출력: 채널별 톤이 다른 3종 문안.
   - 체크포인트: 길이, 톤, CTA가 채널 특성에 맞는지 확인.

4. **이미지 분석 입력을 추가한다.**
   - 실행 입력(행동): 스크린샷 붙여넣기(`Ctrl+V`) 후 `이 이미지에서 이슈 3개를 뽑아줘.`
   - 기대 출력: 시각 기반 이슈 목록.
   - 체크포인트: 텍스트 파일 인사이트와 충돌 여부 확인.

5. **output style을 생성한다.**
   - 실행 입력(프롬프트): `Weekly Update 스타일 템플릿을 만들어줘. 제목 규칙, 본문 구조, 마감 문구 포함.`
   - 기대 출력: 재사용 가능한 스타일 정의.
   - 체크포인트: 다음 작업에서도 동일 톤을 재현할 수 있는지 확인.

6. **검수 프롬프트로 품질을 보정한다.**
   - 실행 입력(프롬프트): `방금 만든 3개 문안에서 숫자/일정/담당자 누락 항목을 찾아 수정해줘.`
   - 기대 출력: 누락 보정본.
   - 체크포인트: 발행 전 필수 정보 3종(숫자·일정·담당자) 누락 0건.

7. **발행 직전 버전으로 확정한다.**
   - 실행 입력(프롬프트): `팀장 승인용 최종본으로 압축해줘. 리스크와 요청사항은 볼드 처리.`
   - 기대 출력: 즉시 공유 가능한 최종본.
   - 체크포인트: 승인자 관점에서 1분 내 핵심 파악이 가능한지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 1**: 첫 태스크를 파일 생성/변환 중심으로 설계해 즉시 체감 성과를 만든다.
- **P4 속도-검증 균형 + Level 1**: 결과물을 채널별로 변환한 뒤 비교 검토해 빠른 생산과 품질 점검을 동시에 수행한다.
- **P6 자동화 운영 기반 + Level 2**: output style 템플릿을 자산화해 반복 업무를 표준 프로세스로 전환한다.
- **P7 Automation vs Augmentation + Level 1**: 초안 생성은 자동화하되 최종 메시지 톤/우선순위 판단은 PM이 맡도록 분리한다.
- **P9 Words-to-Workflow + Level 2**: 자연어 지시(`@` + 변환 요청)가 곧바로 실무 산출물 생성 흐름으로 이어지게 한다.

## 4) 실무 적용 시나리오
주니어 PM이 주간 업데이트를 Slack, 이메일, 위키에 각각 올려야 하는 상황이다. 원본 회의 노트와 지표 파일을 `@`로 묶어 한 번에 입력하고, 채널별 출력 형식으로 3종 초안을 생성한다. 이후 팀장 검토 포인트(리스크/우선순위)만 수동 보정해 게시하면, 기존 대비 작성 시간을 절반 이하로 줄일 수 있다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 반복 커뮤니케이션 업무 1개(예: 주간 업데이트)를 선정하고 입력 파일 세트(`@회의노트`, `@지표`)를 표준 묶음으로 정의합니다.
- **수~목 실행**: 동일 입력으로 Slack/Email/Notion 버전 3종을 생성해 톤, 길이, 의사결정 정보 포함 여부를 비교합니다.
- **금요일 검증**: 실제 발행본 1건을 채널별로 배포하고 수신자 피드백(이해도·추가질문 건수)을 수집해 템플릿을 개선합니다.
- **Done 기준**: output style 1종 이상이 재사용 가능한 형태로 정리되고, 동일 작업의 작성 시간이 기준 대비 감소합니다.
- **리스크/완화**: 채널별 문맥 차이로 메시지가 부정확해질 수 있으므로, 발행 전 공통 검수 항목(숫자/일정/담당자)을 체크리스트화합니다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 시간 | 행동 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 월 | 10:00~11:00 | 반복 업무 1개 선정, 입력 파일 세트 확정 | 입력 세트 정의서 | 필수 입력 파일 누락 0건 |
| 화 | 14:00~15:00 | `@` 참조로 첫 요약 생성 | 요약 초안 v1 | 원문 대비 핵심 이슈 80% 이상 반영 |
| 수 | 11:00~12:00 | Slack/Email/Notion 3종 변환 | 채널별 초안 3개 | 채널별 길이·톤 기준 충족 |
| 목 | 16:00~17:00 | 이미지 분석 결과를 본문에 반영 | 보강본 v2 | 시각 이슈와 텍스트 이슈 충돌 0건 |
| 금 | 09:30~10:30 | 최종 발행 및 피드백 수집 | 발행본 + 피드백 로그 | 추가 질문 건수/재작업 건수 기록 완료 |

- **초보자 실행 팁 1**: 입력 파일명을 매번 다르게 쓰지 말고 템플릿 규칙으로 고정합니다.
- **초보자 실행 팁 2**: 채널 변환 후 "삭제된 핵심 정보"가 없는지 역검증합니다.
- **초보자 실행 팁 3**: output style은 문장 톤뿐 아니라 섹션 구조까지 포함해 정의합니다.
- **롤백/복구 액션**: 금요일 발행 후 오해 피드백이 2건 이상 발생하면, 수요일 생성본으로 즉시 롤백하고 검수 프롬프트(숫자/일정/담당자 누락 점검)를 추가한 뒤 재발행합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: `@` 참조했는데 일부 파일이 반영되지 않음**
   - 해결: 파일명 대소문자/경로를 점검하고, 파일 수가 많으면 2회로 나눠 입력.

2. **막힘: 채널 변환 시 핵심 메시지가 희석됨**
   - 해결: 변환 전에 `핵심 3문장`을 먼저 고정하고 모든 버전에 유지.

3. **막힘: 이미지 분석과 텍스트 분석 결론이 다름**
   - 해결: 충돌 항목을 별도 섹션으로 표기하고 PM이 최종 판단 근거를 기록.

4. **막힘: output style이 재사용될 때 일관성이 떨어짐**
   - 해결: 템플릿에 금지 표현/필수 표현을 명시해 스타일 드리프트를 방지.
