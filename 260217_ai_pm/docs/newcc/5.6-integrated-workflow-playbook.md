## 1) 목적
- Level1~4를 개별 문서가 아닌 **하나의 운영 플레이북**으로 통합한다.
- 아이디어→문서→티켓→자동화→프로토타입→운영 개선까지 단일 실행 루프를 제공한다.
- 팀이 상황별로 어떤 도구 조합을 선택해야 하는지 즉시 판단할 수 있게 한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 코스 전반은 학습보다 실행을 강조하며, 각 모듈이 실제 산출물 생성으로 끝나도록 설계되어 있다.
- 원문 2.1~2.3은 PRD/분석/전략의 고급 워크플로우를 제시하고, 다중 관점 검증을 통해 품질을 높인다.
- 원문 1.4/1.5는 병렬 에이전트와 역할 기반 서브에이전트를 활용한 분업 방식을 보여준다.
- 원문 4.x는 계획→구현→반복→GitHub→배포로 이어지는 end-to-end 실행 루프를 제공한다.
- company-context 문서군은 모든 실행이 KPI/전략/사용자 맥락에 연결되어야 함을 전제로 한다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview 기준으로 통합 플레이북의 핵심은 **한 단계 완료가 다음 단계 입력으로 자동 이어지는 구조**를 만드는 것이다.
- PM은 Claude AI- Claude Code- n8n- agenting을 도구별로 분리 관리하기보다 산출물 단위 핸드오프로 설계해야 병목이 줄어든다.
- Tutorials Hub 패턴처럼 단계별 완료 정의(DoD)를 명확히 해야 학습 문서가 운영 문서로 전환된다.
- Opus 4.6 튜토리얼 관점에서 복합 루프는 계획형 분해와 중간 검증 게이트가 필수이며, 이를 통해 재작업을 최소화할 수 있다.
- Chrome/Cowork 튜토리얼을 통해 외부 탐색과 파일 협업 입력을 확보하고, Claude Code에서 실행 가능한 작업으로 변환한다.
- 금융 서비스 튜토리얼의 신뢰성 모델을 반영해 공통 스키마(목표/근거/지표/owner/due)와 승인 로그를 플레이북 기본값으로 둔다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, 이번 주 핵심 과제를 1건 선택하고 문제 정의 문서를 Claude.ai에서 작성해 KPI와 함께 고정한다.
2. 월요일 오후, Claude Code에서 PRD 초안과 실험 설계를 만들고 `/status`로 진행 상태를 초기화한다.
3. 화요일, Level1 Quick Win을 적용해 PRD 게시→댓글 분류→티켓 생성까지 같은 날 연결한다.
4. 수요일, 반복되는 입력/분류/보고 단계를 n8n 플로우로 전환해 수동 개입 구간을 줄인다.
5. 목요일, `/agent` 기반으로 프로토타입과 Eval을 실행해 기능 완성도와 품질 리스크를 동시에 점검한다.
6. 금요일 오전, OMTM 리뷰에서 이번 주 변화와 실패 원인을 정리하고 다음 액션 3개를 확정한다.
7. 금요일 오후, 핸드오프 템플릿으로 문서/티켓/자동화/에이전트 결과를 연결해 차주 운영에 넘긴다.
8. 주간 종료 후, 개선 백로그를 업데이트하고 플레이북 버전을 올려 재현 가능한 팀 자산으로 축적한다.

### PM Use Case 확장 (Claude Code 기능 매핑)

| 단계 간 핸드오프 손실 최소화 | 공통 스키마/링크 관리 | 모든 단계 산출물에 목표/근거/지표/owner/due 필수 입력 | 핸드오프 실패율 |
| 운영 병목 조기 발견 | 주간 상태 리포트 + 게이트 검증 | 금요일 OMTM 리뷰에서 병목 2개 이상 식별 | 병목 해소 리드타임 |

| 플레이북 조직 자산화 | 버전 관리된 운영 문서 | 월간 플레이북 v1→v1.1 업데이트 및 적용 범위 기록 | 재사용률, 온보딩 기간 |

#### 월~금 상세 실행표
| 요일 | 루프 단계 | 담당 | 핵심 산출물 | 검증 기준 | 복구 절차 |
|---|---|---|---|---|---|
| 월 | 문제 정의 + 실행 설계 | PM | 문제정의 노트/PRD 초안 | KPI/제약 명시 | 입력 맥락 보강 |
| 화 | Level1 즉시 운영 | PM/운영 | 게시/댓글/티켓 링크 | 4개 연계 성공 | 수동 링크 복구 |
| 수 | Level3 자동화 확장 | 자동화 오너 | n8n 실행 로그 | 성공률 기준 충족 | 수동 처리 전환 |
| 목 | Level4 고급 실행 | Agent 오너 | prototype/eval 패키지 | Gate 통과율 | fail 항목 재작업 |
| 금 | Level5 성과 리뷰 | 리드/PM | 주간 운영 리포트 | 액션 3개 확정 | 백로그 재정렬 |

#### 검증/복구 흐름
1. 단계별 산출물에 공통 필드(목표/근거/지표/owner/due)가 빠지면 다음 단계로 넘기지 않는다.
2. 화요일 연계 실패(게시-티켓-공유)가 발생하면 수동 링크 복구 후 원인(권한/포맷/ID)을 로그로 남긴다.
3. 자동화 단계에서 오류율이 임계값을 넘으면 플로우를 비활성화하고 수동 운영으로 임시 전환한다.
4. 에이전트 단계의 Gate 미통과 항목은 신규 기능 개발보다 안정화 우선으로 재배치한다.
5. 금요일 리뷰에서 액션이 백로그에 반영되지 않으면 플레이북 완료로 인정하지 않는다.

### 통합 실행 순서 (표준 루프)
1. **문제 정의 (Claude.ai)**
   - 입력: company context + KPI
   - 산출물: 문제 정의 노트, 목표 지표
2. **실행 설계 (Claude Code / Level2)**
   - 입력: 문제 정의
   - 산출물: PRD 초안, 실험 설계, 명령어 실행 로그
3. **즉시 운영 (Level1)**
   - 실행: Notion 게시, 댓글 처리, Linear 티켓화, Slack 공유
   - 산출물: 실행 링크 세트
4. **자동화 확장 (Level3)**
   - 실행: n8n 스케줄/이벤트 플로우 연결
   - 산출물: 자동화 실행 로그, 실패 대응 기록
5. **고급 실행 (Level4)**
   - 실행: `/agent`로 prototype/eval/핸드오프
   - 산출물: 배포 후보 패키지
6. **성과 리뷰 (Level5)**
   - 실행: OMTM 변화 점검, 다음 주 우선순위 갱신
   - 산출물: 주간 운영 리포트

### 실패 패턴
- 실패1: 단계별 산출물 포맷이 달라 핸드오프 손실 발생
  - 대응: 공통 산출물 스키마(목표/근거/지표/owner/due) 강제
- 실패2: 도구만 바꾸고 운영 루프는 미변경
  - 대응: 도구 도입 KPI(리드타임/재작업률) 측정 의무화
- 실패3: 자동화 과신으로 검수 누락
  - 대응: 고위험 단계 human approval gate 유지
- 실패4: 회고 미실행으로 학습 축적 실패
  - 대응: 주간 회고를 `/today` 루틴에 고정 삽입

### 이번 주 완료형 실행 플랜
- **월~화:** 6단계 표준 루프의 공통 스키마(목표/근거/지표/owner/due)를 확정하고 팀에 배포한다.
- **수~목:** 실제 과제 1건을 선택해 Level1→Level4 구간을 축약 실행하며 단계별 핸드오프 누락을 점검한다.
- **금 검증:** OMTM 리뷰와 함께 루프 병목 2개 이상을 식별해 플레이북 개선 백로그로 등록한다.
- **Done 기준:** 통합 플레이북 v1, 핸드오프 템플릿 세트, 파일럿 실행 로그, 개선 백로그가 연결된다.
- **리스크·완화:** 단계 간 책임 공백이 생길 수 있으므로 각 핸드오프에 승인자와 승인 시점을 명시해 누락을 차단한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 단계는 정의됐지만 핸드오프 누락이 잦음**
  - 증상: 다음 단계 담당자가 이전 산출물을 찾지 못한다.
  - 해결: 핸드오프 템플릿에 링크 필드를 필수화하고 누락 시 자동 반려한다.
- **막힘 2: 자동화/에이전트를 도입했지만 리드타임이 안 줄어듦**
  - 증상: 도구 수는 늘었는데 운영 속도는 그대로다.
  - 해결: 도구 도입 KPI(리드타임, 재작업률, 누락률)를 주간으로 측정해 효과 없는 플로우를 정리한다.
- **막힘 3: 사람 승인 게이트가 모호해 책임 공백 발생**
  - 증상: 누가 최종 승인자인지 불분명해 배포가 지연된다.
  - 해결: 단계별 승인자 1명과 대체 승인자 1명을 고정하고 부재 시 대리 규칙을 명시한다.
- **막힘 4: 회고는 하지만 플레이북이 업데이트되지 않음**
  - 증상: 같은 실패가 반복된다.
  - 해결: 월간 플레이북 버전업 규칙(변경 항목/근거/적용일)을 강제해 학습을 운영 자산으로 축적한다.
