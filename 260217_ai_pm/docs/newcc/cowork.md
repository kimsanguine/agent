## 1) 목적
- Claude Cowork를 “채팅 도구”가 아닌 **파일 기반 실행 파트너**로 운영하는 기준을 제공한다.
- 비개발 PM이 대량 문서/리서치/정리 업무를 위임할 때 실패율을 낮춘다.
- 추후 `5.2`/`5.6` 통합 시, 남길 내용과 삭제할 내용을 명확히 분리한다.

## 2) 원문 핵심 요약 (ccforpms)
- Cowork는 Claude Code의 비개발자 친화 GUI 형태로, 폴더 내 파일을 직접 읽고 생성/편집할 수 있다.
- 일반 채팅과 다르게 `.xlsx`, `.pptx`, `.docx` 등 실제 업무 산출물을 바로 만들 수 있는 점이 핵심 가치다.
- 시작 조건은 macOS + Claude Desktop + Claude Pro이며, 초기에는 작업 전용 폴더 1개부터 연결하는 운영이 권장된다.
- 성과를 좌우하는 프레임은 Done Framework(완료 정의, 필수 맥락, 제약 조건)로 제시된다.
- 활용 영역은 파일 정리, 리서치 통합, 문서 생성, 데이터 추출, 배치 처리다.
- Sub-agents, Skills, Chrome 연계로 확장 가능하지만, 세션 메모리 부재·일부 커넥터 불안정 등 제약도 존재한다.
- 안전 수칙은 백업, 민감정보 분리, 삭제 금지 제약 명시가 기본이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview의 파일 중심 실행 원칙에 맞춰, Cowork 실습도 `입력 폴더-처리 규칙-출력 위치`를 명시한 뒤 실행해야 재현성이 올라간다.
- PM은 대화형 요청 대신 **완료 정의 + 금지사항 + 출력 포맷**을 한 번에 지정하면, Cowork 결과를 Claude Code 허브(5.2)로 넘기기 쉬워진다.
- Tutorials Hub의 실전 튜토리얼 흐름처럼, Cowork는 “탐색→정리→검수→공유” 루프로 운영할 때 팀 적용성이 높다.
- Opus 4.6 튜토리얼 관점에서 복합 문서 배치는 계획 단계와 실행 단계를 분리하면 품질 흔들림을 줄일 수 있다.
- Chrome 튜토리얼과 연계하면 웹 리서치 입력을 Cowork에서 구조화하고, Claude Code에서 후속 문서/티켓 액션으로 연결할 수 있다.
- 금융 서비스 튜토리얼의 안전 운영 기준을 반영해 민감 파일 분리, 원본 보존, 변경 이력 기록을 기본 정책으로 둔다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, PM이 작업 폴더를 `inbox/processed/outputs/reference` 4개로 나누고 파일 이동 규칙(원본 유지, 가공본만 outputs 저장)을 문서로 고정한다.
2. 같은 날, Claude Desktop에서 Cowork 대상 폴더 1개만 연결해 권한 범위를 최소화하고, 민감정보 폴더는 연결 대상에서 제외한다.
3. 화요일, Done Framework 프롬프트를 작성한다: `완료 정의(무엇이 끝인지) + 필수 맥락(분류 기준) + 제약(삭제 금지)` 3요소를 한 번에 입력한다.
4. 수요일, 30개 문서 배치를 실행하고 Cowork가 생성한 분류표와 요약표를 `outputs/`에 저장하게 한다.
5. 수요일 오후, PM이 결과 20%를 표본 검수해 오분류 유형을 `기준 모호/문맥 오해/형식 누락`으로 태깅한다.
6. 목요일, 오분류 원인별로 프롬프트를 보강해 2차 재실행하고 1차 대비 개선율(예: 오분류율 18%→9%)을 기록한다.
7. 금요일, 팀 공유본(최종 표 + 근거 문장)을 배포하고, 다음 배치부터 재사용할 프롬프트 버전(`v1.1`)을 잠근다.
8. 주간 회고에서 `처리량/정확도/재작업률` 3개 지표를 남겨 Level2 허브 문서에 연결한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: Cowork 사용 결과는 반드시 파일 산출물로 남기고, 링크와 버전까지 기록한다.
- **P2 Context-rich**: 요청 시 입력 데이터 범위/완료 조건/금지사항을 먼저 고정해 재작업률을 줄인다.
- **P7 Automation vs Augmentation**: 문서 포맷 변환·정리·초안 생성은 자동화, 최종 의사결정/대외 발신은 사람 검수로 분리한다.
- **P8 Connected Hub 정합성 검사**: Cowork 단독 운영은 임시로 허용하되, 장기적으로 `5.2(Claude Code Hub)`에 흡수 가능한지 평가한다.
- **NFR-010/011 관점**: 자동화 안정성과 관측성(실패 기록, 재실행 로그)이 없는 경우 확장 적용을 제한한다.

#### 검증/복구 흐름
1. 1차 배치 결과에서 오분류율이 10%를 넘으면 “배포 금지” 상태로 지정한다.
2. 오분류 샘플 6건 이상을 추출해 공통 원인을 태깅하고, 프롬프트 문장 단위로 수정한다.
3. 2차 재실행 후에도 기준 미달이면 자동화를 중지하고 해당 배치만 수동 분류로 전환한다.
4. 수동 분류 결과를 다시 Cowork 프롬프트 개선 데이터로 반영해 다음 주 정확도를 끌어올린다.
5. 주간 회고에 “재실행 횟수/복구 시간/최종 정확도”를 남겨 다음 사이클 진입 여부를 결정한다.

### 시나리오: 주간 리서치 정리 배치 작업
1. 입력 폴더를 `inbox/processed/outputs/reference`로 분리한다.
2. 시작 프롬프트에 완료 정의를 명확히 작성한다.
   - 예: “30개 문서를 카테고리 5개로 분류하고, 근거 문장 1개씩 포함한 요약표를 작성. 원본 삭제 금지.”
3. Cowork로 1차 분류/요약을 실행한다.
4. PM이 표본 20%를 검수해 오분류율을 기록한다.
5. 오분류율이 기준(예: 10%) 이하면 팀 공유본으로 배포, 초과하면 규칙 보강 후 재실행한다.

### 이번 주 완료형 실행 플랜
- **월~화:** `inbox/processed/outputs/reference` 폴더 구조를 팀 공용으로 만들고 Done Framework 프롬프트 템플릿 1종을 확정한다.
- **수~목:** 실제 리서치 30건을 대상으로 Cowork 분류/요약 배치를 2회 실행하고 오분류 유형을 태깅한다.
- **금 검증:** 표본 20% 검수 결과로 오분류율·누락률을 계산해 기준치(오분류율 10% 이하) 충족 여부를 판단한다.
- **Done 기준:** 템플릿 1종, 배치 로그 2회분, 검수 리포트 1종, 팀 공유본 링크가 모두 연결된다.
- **리스크·완화:** 커넥터 오류나 폴더 권한 이슈가 나면 수동 업로드 fallback 경로를 즉시 사용하고 재실행 시간을 로그에 남긴다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 프롬프트가 추상적이라 결과가 들쭉날쭉함**
  - 증상: 같은 입력인데 배치마다 분류 기준이 달라진다.
  - 해결: 분류 taxonomy와 출력 형식을 표로 고정하고, 예시 2개(정답/오답)를 프롬프트에 포함한다.
- **막힘 2: 원본 파일이 덮어써져서 복구가 어려움**
  - 증상: 결과물 생성 과정에서 원본이 변경된다.
  - 해결: `원본 읽기 전용 + outputs만 쓰기 허용` 정책을 운영 규칙으로 강제한다.
- **막힘 3: 팀 검수 기준이 달라 품질 논쟁이 길어짐**
  - 증상: 같은 결과를 두고 사람마다 pass/fail 판단이 다르다.
  - 해결: 검수 체크리스트를 5개 항목으로 고정하고, 4/5 이상만 통과시키는 합의 규칙을 둔다.
- **막힘 4: 커넥터/권한 오류로 배치가 자주 중단됨**
  - 증상: 중간 실패 후 재시작 지점이 불명확하다.
  - 해결: 실패 시점 로그를 남기고 `마지막 성공 파일 다음부터 재개`하는 재실행 규칙을 문서화한다.

## 통합/삭제 판단 기준
- **통합 권고 조건**
  - 최근 8주 기준 Cowork 단독 문서 참조율 < 20%
  - 동일 내용이 `5.2-level2-claude-code-hub.md`와 70% 이상 중복
  - Cowork 고유 실습 없이도 `5.6` 플레이북으로 업무 수행 가능
- **독립 유지 조건**
