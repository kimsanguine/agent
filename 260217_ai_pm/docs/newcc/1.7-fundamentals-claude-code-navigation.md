## 1) 목적
- 작업 성격에 맞춰 입력 모드와 분석 강도를 선택해, Claude Code 세션 운영 효율을 높인다.
- 속도 옵션과 안전 옵션의 트레이드오프를 이해해 실무 리스크를 통제한다.

## 2) 원문 핵심 요약 (ccforpms)
- 내비게이션 핵심 축은 3가지다: 입력 모드 선택, 분석 깊이 제어, 실행 속도 최적화.
- 입력 모드는 `Edit`, `Auto-Accept`, `Plan`이며 `Shift+Tab`으로 순환 전환한다.
- `Edit`는 diff 확인 후 승인하는 방식으로 중요 파일/학습 단계/안전성 우선 작업에 적합하다.
- `Auto-Accept`는 승인 없이 즉시 반영되어 단순·저위험 작업에서 속도가 빠르다.
- `Plan`은 실행 전 todo를 먼저 수립해 3단계 이상 복합 작업(조사→종합→산출물)에 강하다.
- 분석 강도 키워드가 있다: `think about`, `think harder about`, `ultrathink about`.
- 권한 스킵 실행 옵션 `claude --dangerously-skip-permissions`은 속도를 높이지만 리스크가 크다.
- 반복 사용 시 alias 예시(`cc="claude --dangerously-skip-permissions"`)가 가능하다.
- 실행 중 `Esc`로 즉시 중단 가능하며, 중단 시점까지 생성된 파일은 유지된다.
- 모듈 시작 명령은 `/start-1-7`, 완료 후 다음 단계는 `/start-2-1`이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Navigation 모듈은 Claude Code의 multi-surface 실행을 안전하게 운용하기 위한 제어판 역할로, 모드 선택이 곧 품질/속도 전략이 된다.
- PM 반복 업무 자동화에서는 Auto-Accept로 처리할 저위험 작업과 Edit/Plan으로 검증할 고위험 작업을 분리해야 운영 리스크를 줄일 수 있다.
- MCP 기반 외부 연동 작업은 영향 범위가 크므로 Plan + 강화된 검증 루프를 기본으로 두는 것이 안전하다.
- `CLAUDE.md`에 모드 선택 기준(업무 유형별 기본 모드/think 강도/승인 정책)을 고정하면 팀 운영 일관성이 높아진다.
- sub-agent/agent team 병렬 실행 전에도 모드·중단·복구 규칙을 합의해야 병렬 속도 이점이 품질 저하로 이어지지 않는다.
- git/PR 흐름과 결합하면 Edit 모드의 diff 검토 결과를 그대로 리뷰 근거로 활용할 수 있다.
- CLI composability(모드 전환, think 강도, 중단, 재실행) 숙련도가 올라갈수록 같은 작업을 더 예측 가능하게 운영할 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **작업 성격을 먼저 분류한다.**
   - 실행 입력(프롬프트): `지금 할 작업을 고위험/중위험/저위험으로 분류하고 추천 모드를 제안해줘.`
   - 기대 출력: 위험도별 모드 추천표(Edit/Plan/Auto-Accept).
   - 체크포인트: 핵심 문서 작업이 Auto-Accept로 분류되지 않았는지 확인.

2. **Plan 모드로 복합 작업 구조를 설계한다.**
   - 실행 입력(행동): `Shift+Tab`으로 Plan 모드 전환 후 `3단계 실행계획 만들어줘.`
   - 기대 출력: 조사→작성→검증의 todo 구조.
   - 체크포인트: 단계별 완료 기준이 포함되는지 확인.

3. **Edit 모드에서 고위험 수정 작업을 실행한다.**
   - 실행 입력(행동): `Shift+Tab`으로 Edit 모드 전환 후 핵심 파일 수정 요청.
   - 기대 출력: diff 확인 기반 수정 반영.
   - 체크포인트: 승인 전 변경 범위를 읽고 의도와 일치하는지 점검.

4. **Auto-Accept로 저위험 반복 작업을 처리한다.**
   - 실행 입력(행동): 단순 포맷 정리 작업을 Auto-Accept에서 실행.
   - 기대 출력: 빠른 대량 수정.
   - 체크포인트: 저위험 파일에만 적용됐는지 확인.

5. **think 강도를 작업 난도에 맞춘다.**
   - 실행 입력(프롬프트): `think harder about 이 요청의 리스크를 먼저 점검해줘.`
   - 기대 출력: 리스크 중심 분석.
   - 체크포인트: 일반 작업과 심화 분석 작업을 구분해 시간 낭비를 줄였는지 확인.

6. **중단/복구 절차를 실습한다.**
   - 실행 입력(행동): 실행 중 `Esc`로 중단 후 생성 파일 점검.
   - 기대 출력: 중단 시점 산출물 상태 파악.
   - 체크포인트: 중단 후 복구 기준(삭제/유지/수정)을 문서화했는지 확인.

7. **팀 공통 운영 기준으로 고정한다.**
   - 실행 입력(프롬프트): `작업 유형별 기본 모드와 think 강도, 승인 정책을 1페이지 표준안으로 정리해줘.`
   - 기대 출력: 팀 운영 가이드 초안.
   - 체크포인트: 신입도 표만 보고 모드를 선택할 수 있는지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P4 속도-검증 균형 + Level 2**: 모드 선택 기준(복잡도·리스크)을 명시해 빠른 실행과 검증 품질을 동시에 관리한다.
- **P1 실행 우선 + Level 2**: Plan/Edit/Auto-Accept를 상황별로 즉시 적용해 작업 정체를 줄인다.
- **P7 Automation vs Augmentation + Level 2**: 자동 승인(Automation)을 쓸 때는 PM 검증 포인트(Augmentation)를 별도 확보한다.
- **P3 PM 오케스트레이션 + Level 2**: PM이 모드·강도·중단 타이밍을 설계해 실행 흐름을 직접 통제한다.

## 4) 실무 적용 시나리오
시니어 PM이 분기 전략 문서와 주간 운영 리포트를 동시에 처리해야 한다. 전략 문서는 `Plan + think harder`로 구조를 먼저 세우고, 반복 포맷 변환은 `Auto-Accept`로 빠르게 처리한다. 마지막 통합 검토는 `Edit` 모드에서 diff를 확인해 리스크를 줄인다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 주요 업무 3종(전략 문서, 주간 보고, 단순 정리)에 대해 기본 모드(Edit/Plan/Auto-Accept)와 think 강도를 사전 매핑합니다.
- **수~목 실행**: 동일 작업을 두 가지 모드로 각각 수행해 소요시간, 수정 횟수, 품질 차이를 비교 기록합니다.
- **금요일 검증**: 권한 스킵 옵션 사용 로그를 리뷰해 허용 범위(저위험 작업)와 금지 범위(핵심 문서/배포 파일)를 확정합니다.
- **Done 기준**: 팀 공통 모드 선택표와 중단/복구 절차가 문서화되고, 다음 주 작업에서 동일 기준으로 운영 가능합니다.
- **리스크/완화**: 속도 우선 사용이 누적되면 품질 저하가 생길 수 있으므로, 고위험 작업은 Edit 모드 강제와 최종 diff 검토를 필수화합니다.

### PM Use Case 확장 (Claude Code 기능 매핑)
| PM 상황 | Claude Code 기능 | 실행 예시 | 검증 포인트 |
|---|---|---|---|
| 전략 문서 수정에서 실수 리스크가 큼 | Edit 모드 + diff 검토 | 핵심 문서는 Edit 모드로만 수정 | 승인 전후 변경점 추적 가능 |
| 반복 정리 작업이 많아 속도가 느림 | Auto-Accept 저위험 자동화 | 포맷 정리/제목 통일 작업 자동 반영 | 품질 저하 없이 처리 시간 단축 |
| 복합 과제가 중간에 길을 잃음 | Plan 모드 todo 구조화 | 조사→작성→검증 3단계 계획 후 실행 | 단계 누락/재작업 감소 |
| 난도 높은 판단 작업의 분석 깊이가 부족 | think harder / ultrathink | 고난도 요청에 강화된 분석 지시 | 근거 품질/대안 수 증가 |
| 실행 중 이상 징후 발생 | 즉시 중단(`Esc`) + 복구 루틴 | 중단 후 변경 파일 점검표 실행 | 불완전 산출물 확산 방지 |

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 시간 | 행동 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 월 | 09:30~10:30 | 업무 3종 위험도 분류 + 모드 매핑 | 모드 선택표 v1 | 고위험 작업 Edit/Plan 배치 완료 |
| 화 | 14:00~15:00 | Plan 모드로 복합 작업 설계 | 실행 todo 문서 | 단계별 완료 조건 명시 |
| 수 | 10:00~11:00 | Edit 모드로 핵심 문서 수정 | diff 검토 로그 | 승인 전/후 변경점 추적 가능 |
| 목 | 16:00~17:00 | Auto-Accept로 저위험 정리 작업 | 처리 결과 로그 | 처리 속도 향상 + 품질 저하 없음 |
| 금 | 11:00~12:00 | think 강도/중단 복구 실험 | 운영 정책 초안 | 중단 후 복구 절차 1회 재현 성공 |

- **초보자 실행 팁 1**: 모드를 먼저 정하고 작업을 시작하면 재작업이 크게 줄어듭니다.
- **초보자 실행 팁 2**: `think harder`는 모든 작업이 아니라 고난도 분석에만 씁니다.
- **초보자 실행 팁 3**: Auto-Accept는 저위험 파일에서만 사용하도록 체크리스트로 통제합니다.
- **롤백/복구 액션**: Auto-Accept 실행 후 예상치 못한 변경이 발견되면 즉시 중단(`Esc`)하고, 직전 백업본 또는 diff 기준 안정 상태로 롤백한 뒤 Edit 모드에서 재적용합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 모드 전환 없이 습관적으로 같은 방식만 사용**
   - 해결: 작업 시작 체크리스트에 "모드 선택" 항목을 첫 줄에 고정.

2. **막힘: `think harder` 남용으로 작업 속도 저하**
   - 해결: 고난도 의사결정 작업에서만 사용하고, 단순 정리는 기본 강도로 제한.

3. **막힘: Auto-Accept 사용 후 변경 추적 불가**
   - 해결: 저위험 작업이라도 결과 요약 로그를 남기고 핵심 파일은 Edit 모드로 전환.

4. **막힘: 중단(`Esc`) 후 어떤 파일이 바뀌었는지 모름**
   - 해결: 중단 즉시 변경 파일 목록을 점검하는 복구 루틴(파일 점검표)을 표준화.
