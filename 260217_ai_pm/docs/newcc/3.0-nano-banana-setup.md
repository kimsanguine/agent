# 3.0 Nano Banana - Setup

**빠른 요약**

## 1) 목적
- Nano Banana 트랙을 **5~10분 안에 바로 실행 가능한 상태**로 만든다.
- 이미지 생성 실습을 단발성 체험이 아니라, PM 실무에서 재사용 가능한 **시각 자산 파이프라인**으로 시작한다.
- 이후 3.1/3.2 모듈에서 막히지 않도록 계정·결제·폴더·명령어·로그 기준을 선제 정렬한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 준비물은 명확하다: **Claude Code + Google 계정 + Google Cloud billing 결제수단**이 필수다.
2. 학습 시작은 지정 폴더에서 `/start-3-1-1` 실행으로 진행되며, **폴더 위치가 틀리면 명령이 동작하지 않는다**.
3. 기본 작업 루트는 `~/Documents/claude-code-pm-course`이며, 자동 다운로드 실패 시 zip 수동 다운로드/압축해제로 대체 가능하다.
4. 실습 모델은 **Gemini 3 Pro Preview (Nano Banana Pro)** 기반이며, 단순 생성이 아니라 배치 생성·파일 관리·반복 개선까지 포함한다.
5. 비용 가이드는 이미지당 약 **$0.10**, 모듈 전체는 대략 **$5 이하** 수준으로 안내된다.
6. Mac에서 `curl/unzip` 계열 이슈가 나면 `xcode-select --install`이 우선 대응이고, Windows는 수동 다운로드가 안전한 대안이다.
7. 이 단계의 진짜 목표는 “첫 이미지 1장”이 아니라, 3.1(기초/스타일) → 3.2(PM 시각화)로 이어지는 **지속 가능한 생성 운영 루프**를 세팅하는 것이다.

### Claude Code 핵심 기능 연결 (official 반영)
- **실행형 온보딩(Overview)**: 문서 읽기보다 “명령 실행→결과 확인” 흐름으로 첫 성공 시간을 단축한다.
- **초기 세팅 프롬프트 구조화(Tutorial: Getting Started)**: 계정/결제/경로/키/출력 폴더를 체크리스트화해 설정 누락을 줄인다.
- **품질 높은 지시문 작성(Tutorial: Claude Opus 4.6)**: 모호한 설치 안내 대신 실패 분기(인증/경로/권한)별 대응 문장을 명확히 작성한다.
- **브라우저 보조(Tutorial: Claude for Chrome)**: 결제·문서 페이지를 빠르게 확인하고 설정 절차를 팀 가이드에 반영한다.
- **협업 온보딩(Tutorial: Claude Cowork)**: 숙련자/신규자가 같은 체크리스트를 기준으로 실행해 온보딩 편차를 줄인다.
- **운영 통제 관점(Tutorial: Financial Services)**: 키 관리, 접근권한, 실행 로그 보관을 기본 운영 규칙으로 내장한다.
- **실습 중심 반복 워크플로(Claude Code)**: `/start-3-1-1` 전 리허설을 통해 3.1/3.2 모듈 진입 실패율을 낮춘다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 환경 체크리스트 선확인
- 입력 아티팩트: `onboarding-checklist.md`, OS 정보, 계정 상태
- 프롬프트 조각: "Nano Banana 시작 전에 계정/결제/로컬경로/필수 툴 상태를 Yes/No 표로 만들어줘."
- 기대 출력: 준비 상태 표(계정, billing, 경로, CLI)
- 검증 체크포인트: No 항목이 0개가 아니면 생성 실습을 시작하지 않는다.

2) 프로젝트 루트와 폴더 구조 생성
- 입력 아티팩트: 팀 표준 폴더 규칙
- 프롬프트 조각: "`assets/`, `prompts/`, `outputs/`, `logs/` 4개 폴더를 기준으로 초보자용 운영 규칙을 작성해줘."
- 기대 출력: `folder-convention.md`
- 검증 체크포인트: 파일명 규칙(날짜-시나리오-버전)이 명시되어 있다.

3) API key 및 billing 연결 검증
- 입력 아티팩트: `.env.example`, 계정 결제 화면
- 프롬프트 조각: "키 형식/결제 활성화/권한 범위를 점검하는 3단계 확인 절차를 정리해줘."
- 기대 출력: `auth-billing-check.md`
- 검증 체크포인트: 키 형식(`AIza...`)과 결제 활성 여부를 모두 확인했다.

4) `/start-3-1-1` 실행 리허설
- 입력 아티팩트: 로컬 CLI 환경, 루트 경로
- 프롬프트 조각: "명령 실패 시 경로 문제인지 인증 문제인지 1차 진단 흐름을 만들어줘."
- 기대 출력: 실행/오류 분기 다이어그램(텍스트)
- 검증 체크포인트: 경로 오류와 인증 오류를 2분 내 구분할 수 있다.

5) 샘플 생성 3건 수행
- 입력 아티팩트: 테스트 프롬프트 3개
- 프롬프트 조각: "동일 주제의 난이도 다른 프롬프트 3개를 제안하고 결과 비교 기준을 표로 줘."
- 기대 출력: 샘플 이미지 3장 + 비교 메모
- 검증 체크포인트: 생성 시간, 품질, 재시도 횟수가 로그에 기록된다.

6) 장애 대응 플레이북 작성
- 입력 아티팩트: 실패 케이스(키오류, 경로오류, 설치오류)
- 프롬프트 조각: "빈도 높은 오류 5개를 재현 조건/원인/복구 단계로 정리해줘."
- 기대 출력: `troubleshooting-playbook.md`
- 검증 체크포인트: 신규 팀원이 문서만 보고 10분 내 복구 가능하다.

7) 다음 모듈 진입 준비 잠금
- 입력 아티팩트: 실행 로그, 샘플 결과, 장애 플레이북
- 프롬프트 조각: "3.1.1 이동 전에 필수 완료 조건(Go/No-Go) 체크리스트를 작성해줘."
- 기대 출력: `go-no-go-3.1.1.md`
- 검증 체크포인트: Go 조건 5개를 모두 충족해야 다음 단계로 이동한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 설치 설명보다 “첫 실행 성공”을 완료 기준으로 둔다.
- **P2 맥락 우선**: 계정/결제/폴더/명령어/출력경로를 함께 세팅해 품질 편차를 줄인다.
- **P6 자동화 운영 기반**: 다운로드-실행-출력-리뷰를 표준 절차로 고정해 반복 비용을 낮춘다.
- **P8 Connected Hub**: Claude Code(실행)와 생성 모델(결과)을 분리하지 않고 하나의 작업 루프로 연결한다.
- **P9 Words-to-Workflow**: `/start-3-1-1` 같은 자연어 명령을 실제 학습·생성 워크플로 트리거로 사용한다.
- 3.x 트랙은 Optional이지만, v2.4 관점에서 **콘텐츠/마케팅 자산 생산성**을 끌어올리는 실전 레버로 활용한다.

## 4) 실무 적용 시나리오
**상황: 신규 기능 소개용 시각 자료를 2일 안에 준비해야 하는 PM**
1. 오전 1시간 내 Nano Banana 환경을 세팅하고 첫 실행까지 완료한다.
2. 팀 공용 폴더 구조(`assets/`, `prompts/`, `outputs/`, `logs/`)를 생성한다.
3. 실습용 샘플 3장을 생성해 생성 속도·품질·비용을 확인한다.
4. 실패 케이스(인증 오류, 경로 오류, 압축 오류) 대응 규칙을 팀 노션에 기록한다.
5. 다음 모듈(3.1.1)로 이동해 실제 프로젝트 맥락 이미지를 만들 준비를 끝낸다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 로컬 환경에서 Nano Banana 실행 경로와 `.env`를 점검하고, `/start-3-1-1` 진입까지 팀 공통 온보딩 절차를 확정한다.
- **수~목 실행**: 샘플 생성 3건을 수행해 속도/품질/비용을 기록하고, 자동 설치 실패 시 수동 설치 플로우를 실제로 리허설한다.
- **금요일 검증**: 신규 팀원 1명이 가이드를 보고 20분 내 첫 생성에 성공하는지 온보딩 테스트를 실행한다.
- **Done 기준**: 설치-실행-출력-장애대응까지 한 번에 따라할 수 있는 운영 가이드와 로그 템플릿이 준비되어 있고 첫 생성 기록이 남아 있다.
- **리스크/완화**: 결제 미연결·키 오류·경로 혼선 리스크는 사전 점검표(계정/키/경로/권한)로 막고, 플랫폼별 설치 차이는 OS별 분기 가이드로 완화한다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 명령은 실행되는데 결과 파일이 안 보임
- 근본 원인: 작업 경로와 출력 경로를 혼동
- 해결 방법: 실행 전/후 현재 경로와 `outputs/` 절대 경로를 로그에 의무 기록

2) 블로커: API key는 넣었는데 인증 오류 반복
- 근본 원인: billing 미활성 또는 반영 지연
- 해결 방법: 결제 상태 확인 후 10~15분 대기, 세션 재시작 후 재시도

3) 블로커: 자동 다운로드가 실패해 시작 자체가 지연
- 근본 원인: OS/권한/압축 툴 환경 차이
- 해결 방법: 수동 zip 다운로드 표준 절차를 기본 fallback으로 문서화

4) 블로커: 팀마다 폴더 규칙이 달라 결과가 섞임
- 근본 원인: 공통 네이밍 규칙 부재
- 해결 방법: 날짜-시나리오-버전 규칙을 강제하고 예외 사용 금지
