## 1) 목적
- PM 데이터 분석을 Discovery→Impact Estimation→Experiment Analysis 3단계로 구조화해 의사결정 근거를 강화한다.
- 평균값 중심 해석 오류를 줄이고, 세그먼트·품질 지표까지 포함한 실행 결정을 내리게 한다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 시작 명령은 `/start-2-2`이며, 기능 개발 전후 판단을 데이터로 연결하는 흐름이 핵심이다.
- 3단계 표준 산출물이 명확하다: `problem-analysis.md`, `impact-estimate.md`/`roi-scenarios.md`, `experiment-readout.md`.
- Impact 계산식이 제시된다: `Impact = Users Affected × Current Action Rate × Expected Lift × Value per Action`.
- 추정 시 rollout 비율, 현재 전환율, 보수적 lift, LTV 기반 가치 등 현실 변수를 함께 반영해야 한다.
- ROI는 단일 수치가 아니라 pessimistic/realistic/optimistic 3개 시나리오로 제시한다.
- 실험 평가는 topline만 보지 않고 `p-value`, `95% CI`, 세그먼트(company_size 등), 품질 지표(retention), 선행 지표(template usage/invite rate)를 함께 확인한다.
- Claude Code는 CSV/TSV/JSON 직접 분석, 대용량 처리, 통계 계산, 정량·정성 교차분석에 유리하다.
- 핵심 인사이트는 “전체 평균이 약해도 특정 세그먼트에서 강하면 부분 rollout 전략이 가능하다”는 점이다.

### Claude Code 핵심 기능 연결 (official 반영)
- **다양한 데이터 입력 통합(Overview)**: CSV/TSV/JSON/문서 메모를 한 워크플로에서 다뤄 Discovery 분석의 준비 시간을 단축한다.
- **명확한 분석 지시문 설계(Tutorial: Getting Started)**: `분석 질문 → 기간/세그먼트 → 출력 포맷` 순으로 프롬프트를 고정하면 결과 재현성이 높아진다.
- **고난도 수치 추론(Tutorial: Claude Opus 4.6)**: Impact 공식, ROI 3시나리오, 민감도 비교 같은 다단 계산을 빠르게 검토할 수 있다.
- **반복 검증 루프(Claude Code 워크플로)**: `품질 점검 → 가정 수정 → 재계산` 루프를 짧게 돌려 잘못된 가정 고착을 막는다.
- **브라우저 기반 근거 수집(Tutorial: Claude for Chrome)**: 벤치마크 리포트/공개 지표를 빠르게 가져와 내부 데이터 해석 맥락을 보강한다.
- **멀티 관점 코워킹(Tutorial: Claude Cowork)**: PM, Analyst, Finance 관점을 분리해 권고안(Ship/Iterate/Kill)의 균형을 맞춘다.
- **감사 가능성 강화(Tutorial: Financial Services)**: 가정값 출처·버전·승인 이력을 남겨 의사결정 추적성을 확보한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 분석 질문과 범위 고정
- 입력 아티팩트: `funnel-events-raw.csv`, `signup-cohort.csv`, `survey-freeform.xlsx`, `metric-dictionary.md`
- 프롬프트 조각: "이번 분석의 핵심 질문을 1문장으로 정의하고, 필요 데이터 컬럼/기간/세그먼트를 표로 정리해줘."
- 기대 출력: `analysis-plan.md`(질문, 기간, 세그먼트, 필요한 컬럼)
- 검증 체크포인트: 질문마다 측정 컬럼이 1:1로 연결되어 있다.

2) 데이터 품질 진단
- 입력 아티팩트: 원본 CSV/TSV, 이벤트 스키마 문서
- 프롬프트 조각: "결측률, 중복 이벤트, 타임존 불일치, 이상치를 점검해 우선 수정 순위를 제안해줘."
- 기대 출력: `data-quality-report.md`
- 검증 체크포인트: 결측/중복/이상치가 분리 표기되고 수정 방식(삭제/보간/유지)이 명시된다.

3) Discovery 분석 실행
- 입력 아티팩트: 정제 데이터(`clean-funnel.csv`, `clean-survey.csv`)
- 프롬프트 조각: "전체 평균과 세그먼트(company_size, 신규/기존)를 분리해 drop-off 구간과 원인 가설을 도출해줘."
- 기대 출력: `problem-analysis.md` 초안
- 검증 체크포인트: Topline 결과와 세그먼트 결과가 각각 독립 섹션으로 존재한다.

4) Impact Estimation 계산
- 입력 아티팩트: `active-users-by-segment.csv`, `value-per-action.md`, `baseline-conversion.csv`
- 프롬프트 조각: "Impact = Users Affected × Current Action Rate × Expected Lift × Value per Action 공식을 보수/기준/낙관 가정으로 계산해줘."
- 기대 출력: `impact-estimate.md`(가정표 + 계산표)
- 검증 체크포인트: 모든 숫자에 단위(명, %, 원/달러)가 붙어 있다.

5) ROI 시나리오 작성
- 입력 아티팩트: `engineering-cost-estimate.md`, `ops-cost.csv`
- 프롬프트 조각: "pessimistic/realistic/optimistic 3시나리오로 ROI와 break-even 시점을 계산해줘."
- 기대 출력: `roi-scenarios.md`
- 검증 체크포인트: 각 시나리오에 발동 조건(예: 전환율 +x% 미만 시 보수안 채택)이 포함된다.

6) Experiment Readout 심화
- 입력 아티팩트: `ab-test-result.csv`, `retention-d7.csv`, `leading-indicators.csv`
- 프롬프트 조각: "p-value, 95% CI, 세그먼트별 uplift, retention 영향, leading indicator 변화를 함께 해석해줘."
- 기대 출력: `experiment-readout.md`
- 검증 체크포인트: '유의하지 않음' 결과일 때도 Ship 금지/추가 실험 조건이 명시된다.

7) 의사결정 권고안 확정
- 입력 아티팩트: 위 1~6단계 산출물 전체
- 프롬프트 조각: "Ship/Iterate/Kill 중 1안을 권고하고, rollout 범위·오너·다음 실험 2개를 제안해줘."
- 기대 출력: `decision-recommendation.md`
- 검증 체크포인트: 권고안에 실행 책임자와 실행 날짜가 빠지지 않는다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 1**: 분석 결과를 문장 요약으로 끝내지 않고 표준 산출물 3종으로 남겨 실행 연결성을 확보한다.
- **P4 속도-검증 균형 + Level 1**: 빠른 탐색 후 통계 유의성/세그먼트/품질 지표로 검증 깊이를 보강한다.
- **P6 자동화 운영 기반 + Level 3**: 반복 분석 템플릿과 시나리오 프레임을 자동화 파이프라인에 재사용 가능하게 만든다.
- **P7 Automation vs Augmentation + Level 1**: 데이터 집계/초안 생성은 자동화하되, rollout 범위와 Go/No-Go 판단은 PM이 담당한다.
- **P8 Connected Hub + Level 1~3**: 분석 결과를 상태보고·우선순위·자동 리포팅으로 연동할 수 있는 구조로 정리한다.

## 4) 실무 적용 시나리오
시니어 PM이 신규 온보딩 실험 결과를 경영진에 보고해야 한다. 먼저 퍼널/설문 데이터를 결합해 문제 지점을 정리하고, ROI 3시나리오로 기대효과를 제시한다. 실험 결과는 전체 평균과 함께 기업 고객 세그먼트를 분리해 해석하고, 최종 권고안을 “부분 롤아웃 + 추가 실험”으로 제안한다.

### PM Use Case 확장 (Claude Code 기능 매핑)
| PM 상황 | Claude Code 기능 | 실행 예시 | 검증 포인트 |
|---|---|---|---|
| 퍼널 하락 원인 파악이 지연됨 | 파일 기반 데이터 분석 + 요약 | `"funnel-events.csv와 cohort.csv를 결합해 drop-off 상위 3구간을 요약"` | 원인 구간마다 수치 근거가 붙음 |
| Impact 계산이 팀마다 다름 | 구조화 계산 프롬프트 | `"Impact=Users×Rate×Lift×Value 공식을 보수/기준/낙관으로 계산"` | 단위(명/%/원) 누락 없이 계산됨 |
| 실험 결과 해석이 topline 중심 | 세그먼트/품질 지표 동시 해석 | `"topline + company_size + retention + leading indicator를 함께 해석"` | Go/No-Go 기준이 세그먼트별로 명시됨 |
| 임원 보고용 한 줄 권고안 필요 | 결정형 리드아웃 생성 | `"Ship/Iterate/Kill 중 1안을 권고하고 이유 3개와 다음 실험 2개 제안"` | 권고안에 오너·기한이 포함됨 |
| 추정치 신뢰도에 의문 제기 | 근거 출처 추적성 강화 | `"가정값별 출처 링크/버전/갱신일 표 작성"` | 검토 회의에서 숫자 재논쟁 감소 |

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `/start-2-2` 실행 후 퍼널/행동로그/설문 데이터를 정리해 Discovery 분석을 먼저 끝내고 `problem-analysis.md` 초안을 작성한다.
- **수~목 실행**: Impact 계산식 입력값(사용자수·현재 행동률·예상 lift·행동 가치)을 실제 숫자로 채워 `impact-estimate.md`와 `roi-scenarios.md`를 완성한다.
- **금요일 검증**: 실험 결과를 topline과 세그먼트(company size, 신규/기존 등)로 나눠 리뷰하고 Ship/Iterate/Kill 권고안을 1안으로 확정한다.
- **Done 기준**: `problem-analysis.md`, `impact-estimate.md`/`roi-scenarios.md`, `experiment-readout.md` 3종 문서에 가정값, 통계 해석, 실행 권고가 모두 명시되어 있다.
- **리스크/완화**: 표본 편향과 과도한 낙관 추정 위험은 보수 시나리오를 기본값으로 두고, 데이터 결측 구간은 추정치/실측치를 분리 표기해 완화한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 액션 오너 | 실행 액션 | 산출물(Artifact) | Pass/Fail 체크 | KPI 미달 시 Fallback |
|---|---|---|---|---|---|
| 월 | PM + Analyst | 분석 질문/세그먼트/기간 정의, 데이터 요청 | `analysis-request.md` | Pass: 질문 1개, 세그먼트 2개 이상 명시 | 질문을 1개 핵심 퍼널 단계로 축소 |
| 화 | Data Analyst | 데이터 정제 및 품질 리포트 작성 | `data-quality-report.md` | Pass: 결측률/중복률/이상치 기준이 숫자로 명시 | 고품질 컬럼만으로 축소 분석 재시도 |
| 수 | PM + Analyst | Discovery + Impact 계산 실행 | `problem-analysis.md`, `impact-estimate.md` | Pass: 문제구간과 영향 추정이 연결됨 | 영향 계산을 보수 가정 기준으로 재산정 |
| 목 | PM + Finance/Eng | ROI 시나리오 및 구현비용 검증 | `roi-scenarios.md` | Pass: 3시나리오 모두 비용/효과/회수기간 포함 | 실험 범위를 세그먼트 파일럿으로 축소 |
| 금 | PM(의사결정) | Experiment readout 및 Ship/Iterate/Kill 결정 | `experiment-readout.md`, `decision-recommendation.md` | Pass: 권고안+오너+다음 실험 일정 포함 | KPI 미달 시 추가 실험 가설 2개 즉시 생성 |

- 운영 팁 1: 월요일 요청서가 모호하면 수요일부터는 분석이 아니라 해석 싸움이 된다.
- 운영 팁 2: 목요일 ROI 검증에는 반드시 비용 오너(재무/개발)를 참여시켜 숫자 신뢰도를 확보한다.
- 운영 팁 3: 금요일 권고안은 "무엇을 안 할지"까지 포함해야 실행 집중도가 올라간다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 평균값은 좋아졌는데 현장 체감이 없음
- 근본 원인: 세그먼트 분해 없이 topline만 보고 판단
- 해결 방법: 핵심 세그먼트 2개를 고정하고 결과를 별도 표로 분리

2) 블로커: ROI가 계속 바뀌어 의사결정이 지연
- 근본 원인: 가정값 출처가 문서화되지 않음
- 해결 방법: 가정값마다 출처 링크/버전/갱신일을 함께 기록

3) 블로커: 실험 결과가 유의하지 않아 다음 행동이 멈춤
- 근본 원인: 유의성 실패 시 의사결정 규칙이 사전에 없음
- 해결 방법: 사전 룰로 "유의성 미달 시 Iterate + 표본 확장" 정책 고정

4) 블로커: 데이터 품질 이슈로 분석 신뢰도 하락
- 근본 원인: 이벤트 정의가 팀마다 다름
- 해결 방법: metric dictionary를 단일 소스로 고정하고 용어 변경 시 승인 절차 적용
