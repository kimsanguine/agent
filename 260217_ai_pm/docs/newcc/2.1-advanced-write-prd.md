## 1) 목적
- AI를 PRD 대필기가 아닌 사고 파트너로 활용해, 빠르면서도 검증 가능한 PRD 초안을 만든다.
- 맥락 수집→질문 기반 정제→다중 관점 리뷰를 표준화해 고위험 문서의 재작업률을 낮춘다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 시작은 `/start-2-1`이며, 목표는 PRD를 처음부터 혼자 쓰기보다 AI와 함께 사고를 구조화하는 것이다.
- `@-mentions`로 회사 문서, 사용자 리서치, PRD 템플릿, 방법론 문서를 함께 연결할수록 결과 구체성이 올라간다.
- Socratic questioning을 활용해 문제 정의, 해법 타당성, 성공 지표, 제약, 전략 적합성을 3~5개 질문으로 깊게 검증한다.
- 동일 문제에 대해 2~3개 전략안을 병렬 생성하면 첫 아이디어 고착을 줄이고 선택 품질을 높일 수 있다.
- sub-agent 다중 관점 리뷰가 핵심이다: Engineer(기술 리스크), Executive(사업 우선순위), User Researcher(사용자 적합성).
- 결과가 generic할 때는 컨텍스트 파일을 늘리고, 리뷰가 얕을 때는 agent 역할과 검토 범위를 더 구체화해야 한다.
- 최종 초안은 버전명을 체계화해 저장/공유하고 이후 반복 개선 루프에 투입한다.
- 고품질 PRD는 “AI가 대신 작성”보다 “PM이 질문과 비교 기준을 설계”할 때 나온다는 점이 강조된다.

### Claude Code 핵심 기능 연결 (official 반영)
- **대규모 컨텍스트 결합(Overview)**: PRD 작성 전 리서치 노트, VOC, 기존 정책 문서를 한 세션에 연결해 문제 정의 누락을 줄인다.
- **구조화 프롬프팅(Tutorial: Getting Started)**: `역할(PM) → 목표(의사결정 문서) → 제약(일정/리소스) → 출력형식(PRD 섹션)` 순으로 지시하면 초안 품질이 안정된다.
- **고난도 추론(Tutorial: Claude Opus 4.6)**: 대안별 trade-off, 리스크, 성공 지표를 동시에 비교해 “왜 이 안을 선택했는가”를 문서화하기 좋다.
- **반복 편집 루프(Claude Code 워크플로)**: `초안 생성 → 반론 질문 5개 → 수정본`을 짧은 사이클로 돌려 재작업 비용을 줄인다.
- **병렬 코워킹(Tutorial: Claude Cowork)**: Engineer/Executive/User Researcher 관점을 분리해 검토하고 PM이 최종 통합한다.
- **웹 맥락 수집(Tutorial: Claude for Chrome)**: 경쟁사 릴리즈 노트/헬프센터를 빠르게 수집해 PRD의 근거 링크 품질을 높인다.
- **규제·감사 대응(Tutorial: Financial Services)**: 의사결정 근거, 승인 로그, 변경 이력을 남기는 문서 습관을 PRD 템플릿에 내장한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 문제 프레이밍 입력 묶기
- 입력 아티팩트: `voc-raw.csv`, `interview-notes.md`, `onboarding-funnel.csv`, `기존-prd-v3.md`
- 프롬프트 조각: "아래 4개 자료를 기준으로 핵심 문제를 '누가-언제-왜 실패하는지' 3문장으로 요약해줘."
- 기대 출력: 사용자 세그먼트별 문제 정의 초안 1페이지
- 검증 체크포인트: 문제 문장마다 정량 근거(로그/퍼널) 1개 이상이 연결되어 있다.

2) 성공 지표와 실패 기준 고정
- 입력 아티팩트: `north-star-metrics.md`, `quarterly-goals.md`
- 프롬프트 조각: "이번 PRD의 OMTM 1개와 Guardrail 지표 2개를 제안하고, 지표가 나빠졌을 때 중단 조건까지 써줘."
- 기대 출력: 목표 지표 표(지표명, 현재값, 목표값, 측정주기)
- 검증 체크포인트: 목표값이 최근 3개월 평균 대비 개선폭으로 계산 가능하다.

3) Socratic 질문으로 가정 압축
- 입력 아티팩트: 위 1~2단계 결과, `policy-constraints.md`
- 프롬프트 조각: "문제정의/해법/지표/제약/타이밍 관점에서 반박 질문 5개를 만들고, 각 질문에 필요한 증거를 지정해줘."
- 기대 출력: 질문-증거 매핑표
- 검증 체크포인트: 질문 5개 중 최소 3개가 '확인 가능한 데이터'를 요구한다.

4) 전략 대안 3안 병렬 생성
- 입력 아티팩트: `engineering-capacity-q2.md`, `sales-feedback.md`
- 프롬프트 조각: "동일 문제를 해결하는 전략안 A/B/C를 제시하고, 범위·개발비용·예상효과를 표로 비교해줘."
- 기대 출력: 대안 비교표(A/B/C)
- 검증 체크포인트: 최소 1개 안은 명시적으로 포기(Out) 가능한 안으로 제시된다.

5) PRD 본문 v0 작성(필수 섹션 강제)
- 입력 아티팩트: 팀 PRD 템플릿, 1~4단계 산출물
- 프롬프트 조각: "아래 템플릿을 그대로 유지해 v0를 작성하되, 각 섹션마다 의사결정 근거를 한 줄씩 포함해줘."
- 기대 출력: `prd-onboarding-v0.md`
- 검증 체크포인트: Problem/Goal/Scope/AC/Out of Scope/Risk 섹션이 모두 채워져 있다.

6) 다중 관점 리뷰 실행
- 입력 아티팩트: `prd-onboarding-v0.md`
- 프롬프트 조각: "Engineer, Executive, User Researcher 관점으로 각각 치명 리스크 3개와 수정 제안 3개를 작성해줘."
- 기대 출력: 역할별 리뷰 코멘트 묶음
- 검증 체크포인트: 역할별 코멘트가 중복이 아니라 서로 다른 판단 기준을 가진다.

7) v1 확정 및 승인 패키지 구성
- 입력 아티팩트: 리뷰 코멘트, 회의 메모
- 프롬프트 조각: "코멘트 반영 전/후 변경사항을 change log로 정리하고, 승인 회의용 1페이지 요약을 만들어줘."
- 기대 출력: `prd-onboarding-v1.md`, `decision-onepager.md`
- 검증 체크포인트: 변경 로그에 '왜 바꿨는지' 이유가 1줄 이상 기록되어 있다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 맥락 우선 + Level 1~2**: `@`로 입력 컨텍스트를 충분히 연결해야 PRD 품질이 올라간다는 원문 원칙을 그대로 적용한다.
- **P3 PM 오케스트레이션 + Level 2**: PM이 질문 설계·대안 비교·리뷰 조율을 맡아 최종 의사결정 책임을 유지한다.
- **P4 속도-검증 균형 + Level 1**: 빠른 초안 생성 후 다중 관점 리뷰와 AC 점검으로 품질 게이트를 건다.
- **P7 Automation vs Augmentation + Level 1**: 초안 생성은 자동화, 최종 우선순위/범위 확정은 PM 직접 수행으로 역할을 분리한다.
- **P8 Connected Hub + Level 1~2**: PRD를 끝 문서로 두지 않고 Notion 게시/Linear 티켓화로 이어지는 실행 체인으로 확장한다.
- **P9 Words-to-Workflow + Level 4**: 질문·프롬프트·리뷰 지시문이 실제 실행 산출물(PRD 버전)로 직결되는 구조를 만든다.

## 4) 실무 적용 시나리오
시니어 PM이 신규 온보딩 기능 PRD를 2일 안에 제출해야 한다. 먼저 리서치/기존 정책/템플릿을 `@`로 묶어 초안을 만들고, Socratic 질문으로 리스크 가정을 점검한다. 이후 Engineer·Executive·User Researcher 리뷰를 병렬로 받아 상충 지점을 정리하고, 최종안은 AC와 KPI를 포함해 버전 태깅 후 공유한다.

|---|---|---|---|
| 신규 기능 PRD 킥오프 | 대화형 스캐폴딩 + 컨텍스트 통합 | `"다음 자료를 바탕으로 PRD Problem/Goal/Scope 초안을 작성: @voc @funnel @policy"` | 섹션 누락 없이 v0가 1회 생성됨 |
| 대안 전략 비교가 어려움 | 고급 추론(대안 병렬 비교) | `"전략안 A/B/C를 비용·리스크·학습속도로 비교표 작성"` | 최소 1개 포기안(Out)이 명시됨 |

| 추후 감사/회고 대비 | 변경 이력/근거 기록 습관 | `"v0→v1 변경 로그를 이유 중심으로 표 작성"` | 변경 항목마다 근거 링크 1개 이상 |

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `/start-2-1`로 시작해 리서치, 기존 PRD, 정책, VOC 자료를 `@`로 4종 이상 연결하고 문제 정의/성공지표 초안(v0)을 작성한다.
- **수~목 실행**: Socratic 질문 5개로 핵심 가정을 검증하고 전략 대안 2~3개를 병렬 생성한 뒤 Engineer/Executive/User Researcher 리뷰를 반영해 v1을 만든다.
- **금요일 검증**: 문서 워크스루 1회를 진행해 AC, KPI, 제외 범위, 리스크 대응의 실행 가능성을 점검하고 최종본(vFinal)을 확정한다.
- **Done 기준**: PRD에 문제 정의, 목표 지표, AC, Out of Scope, 리스크/완화, 출시 후 검증 계획이 모두 포함되고 버전 이력까지 정리되어 있다.
- **리스크/완화**: 문서가 추상적으로 흐르면 사용자 원문 인용 3개와 로그 지표를 강제 포함하고, 리뷰 충돌 시 고객가치/매출영향/구현복잡도 기준표로 결론을 낸다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 액션 오너 | 실행 액션 | 산출물(Artifact) | Pass/Fail 체크 | KPI 미달 시 Fallback |
|---|---|---|---|---|---|
| 월 | PM(오너) | 문제정의 워크숍 45분 진행, VOC와 퍼널 로그를 한 문서로 통합 | `problem-brief-v0.md` | Pass: 사용자/상황/실패행동 3요소가 모두 명시됨, Fail: 불만 나열만 있음 | 인터뷰 원문 5개를 추가 인용하고 문제 문장을 다시 작성 |
| 화 | PM + Data Analyst | 목표 지표/가드레일 지표 확정, 기준선 계산 | `metric-baseline-sheet.csv` | Pass: 기준선·목표값·측정주기 3칸이 채워짐 | 지표를 행동지표 1개로 축소해 재설계 |
| 수 | PM + Engineer | 범위/의존성/기술 리스크 동기화 회의 | `scope-risk-map.md` | Pass: Must/Should/Could와 기술 리스크가 연결됨 | 릴리스 범위를 1단계(핵심 플로우)로 축소 |
| 목 | PM + User Researcher | AC 검증 시나리오 작성, 사용자 성공경로 점검 | `acceptance-scenarios.md` | Pass: AC마다 테스트 방법과 성공 기준이 있음 | AC를 사용자 행동 기준으로 재작성(모호 표현 제거) |
| 금 | PM(최종 책임) | 의사결정 회의 및 vFinal 잠금 | `prd-vFinal.md`, `decision-log.md` | Pass: 승인/보류 항목과 근거가 모두 기록됨 | 승인 보류 시 48시간 내 수정 범위만 재오픈 |

- 운영 팁 1: 월~수에는 문장 품질보다 근거 연결(데이터/원문) 품질을 우선한다.
- 운영 팁 2: 목요일 AC 리뷰에서 "누가 테스트할지"를 지정하지 않으면 금요일 승인률이 급격히 떨어진다.
- 운영 팁 3: 금요일 의사결정 로그는 다음 스프린트 분쟁 비용을 줄이는 핵심 증거다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: PRD가 예쁜 문장인데 실행 불가
- 근본 원인: 문제 정의와 AC 사이에 측정 가능한 연결이 없음
- 해결 방법: AC 항목마다 "측정 이벤트/로그 키/판단 임계값" 3종을 강제 입력

2) 블로커: 리뷰 의견이 서로 충돌해 결론이 안 남
- 근본 원인: 판단 기준(고객가치, 매출, 구현복잡도)이 사전에 합의되지 않음
- 해결 방법: 리뷰 시작 전 3축 우선순위 표를 고정하고, 코멘트를 해당 축에 매핑

3) 블로커: 범위가 계속 커져 일정이 밀림
- 근본 원인: Out of Scope를 문서에 적었지만 회의에서 지속적으로 예외 승인
- 해결 방법: Out 항목 재오픈 시 "추가 근거 + 일정 영향" 2개 없으면 자동 반려 규칙 적용

4) 블로커: KPI가 맞지 않아 승인 직전 다시 갈아엎음
- 근본 원인: 목표값이 근거 없는 희망치로 설정됨
- 해결 방법: 최근 3개월 중앙값 기준으로 목표를 재산정하고, 낙관/기준/보수 3안 병기
