## 1) 목적
- Claude Code 작업을 시각적으로 검증할 수 있는 워크스페이스를 구성해, 변경 실수를 줄이고 리뷰 속도를 높인다.
- 파일 구조와 숨김 폴더를 명확히 파악해 이후 에이전트/메모리 모듈의 기반을 만든다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 목표는 터미널 단독이 아니라 `터미널 + 에디터` split-screen 환경을 만드는 것이다.
- 권장 소요 시간은 약 15~20분이며, 시작 명령은 `/start-1-2`다.
- Nimbalyst는 WYSIWYG 마크다운, 파일 diff, `.claude/` 가시성, 세션 관리, Claude Code 통합 측면에서 추천된다.
- Obsidian은 문서 중심 작업에 강하지만 `.claude/` 숨김 폴더를 직접 표시하지 않아 Finder/Explorer 보조 확인이 필요하다.
- VS Code/Cursor는 코드 편집과 hidden folder 확인에 강점이 있지만 PM 문서 작업만 할 때는 상대적으로 복잡할 수 있다.
- `.claude/`에는 agents, commands, settings가 위치하며 도구별 접근 방식 차이를 이해해야 한다.
- 실습 검증은 Claude에게 `test-visualization.md` 생성을 요청하고 에디터에서 실시간 반영 여부를 확인하는 방식이다.
- 파일이 보이지 않을 때 가장 흔한 원인은 프로젝트 루트가 아닌 상위 폴더를 연 경우다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 2**: 환경 구성 즉시 파일 생성 테스트를 수행하게 해 “설정 완료”를 결과로 검증한다.
- **P2 맥락 우선 + Level 2**: 루트/숨김 폴더 가시성을 확보해 에이전트/명령/메모리 맥락 누락을 방지한다.
- **P4 속도-검증 균형 + Level 2**: split-screen으로 생성 즉시 확인 루프를 만들어 수정-검증 리드타임을 줄인다.
- **P8 Connected Hub + Level 2**: 에디터 선택은 취향이 아니라 운영 목적(문서·코드·숨김폴더 가시성) 기준으로 결정한다.

## 4) 실무 적용 시나리오
주니어 PM이 회의 요약 자동화 파일을 생성했는데 저장 경로가 계속 어긋나는 상황이다. 1.2 가이드대로 split-screen을 구성하고, 루트 폴더와 `.claude/` 표시 여부를 먼저 점검한다. 이후 테스트 파일 생성으로 경로 문제를 해결하고, 실제 회의 요약 산출물도 같은 검증 루틴으로 안정화한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 팀 표준 작업 환경(터미널+에디터 split-screen)과 루트 폴더 오픈 규칙을 통일하고, 도구별 설정값을 스크린샷으로 기록합니다.
- **수~목 실행**: 테스트 파일 생성/수정/삭제 사이클을 3회 반복해 경로 오류, 숨김 폴더 미표시, 저장 지연 이슈를 점검합니다.
- **금요일 검증**: 실제 업무 파일(회의 요약, PRD 초안) 1건을 동일 루틴으로 처리해 실무 재현성을 확인합니다.
- **Done 기준**: 팀원 전원이 `.claude/` 가시성 확인, 테스트 파일 실시간 반영 검증, 경로 오류 대응 절차 숙지를 완료합니다.
- **리스크/완화**: 에디터별 설정 차이로 혼선이 생길 수 있으므로, 필수 설정만 담은 최소 공통 프로필과 예시 화면을 함께 제공합니다.
