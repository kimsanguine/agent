## 1) 목적
- Claude Code 작업을 시각적으로 검증할 수 있는 워크스페이스를 구성해, 변경 실수를 줄이고 리뷰 속도를 높인다.
- 파일 구조와 숨김 폴더를 명확히 파악해 이후 에이전트/메모리 모듈의 기반을 만든다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 목표는 터미널 단독이 아니라 `터미널 + 에디터` split-screen 환경을 만드는 것이다.
- 권장 소요 시간은 약 15~20분이며, 시작 명령은 `/start-1-2`다.
- Nimbalyst는 WYSIWYG 마크다운, 파일 diff, `.claude/` 가시성, 세션 관리, Claude Code 통합 측면에서 추천된다.
- Obsidian은 문서 중심 작업에 강하지만 `.claude/` 숨김 폴더를 직접 표시하지 않아 Finder/Explorer 보조 확인이 필요하다.
- VS Code/Cursor는 코드 편집과 hidden folder 확인에 강점이 있지만 PM 문서 작업만 할 때는 상대적으로 복잡할 수 있다.
- `.claude/`에는 agents, commands, settings가 위치하며 도구별 접근 방식 차이를 이해해야 한다.
- 실습 검증은 Claude에게 `test-visualization.md` 생성을 요청하고 에디터에서 실시간 반영 여부를 확인하는 방식이다.
- 파일이 보이지 않을 때 가장 흔한 원인은 프로젝트 루트가 아닌 상위 폴더를 연 경우다.

### Claude Code 핵심 기능 연결 (official 반영)
- 파일 시각화 모듈은 Claude Code의 multi-surface 강점을 가장 직접적으로 체감하는 구간으로, 터미널 실행과 에디터 반영을 동시에 확인해야 한다.
- PM 반복 작업 자동화에서도 `결과가 실제 파일로 남는지`가 핵심이므로, 생성-수정-삭제 검증 루프를 운영 표준으로 삼는 것이 중요하다.
- MCP를 쓸 경우에도 최종 산출물은 로컬 프로젝트 파일로 귀결되므로, 루트/숨김 폴더 가시성 확보가 연동 품질의 출발점이다.
- `CLAUDE.md`에 폴더 규칙과 파일명 규칙을 고정하면 팀원별 저장 위치 편차를 크게 줄일 수 있다.
- sub-agent/agent team 결과를 합칠 때도 동일 경로 규칙이 있어야 병렬 산출물 충돌 없이 통합이 가능하다.
- git/PR 흐름에서 diff 검토가 가능한 상태를 만들려면 에디터 파일 트리 가시성과 변경 확인 루틴이 선행되어야 한다.
- CLI composability(`@파일`, 생성 요청, 수정 요청, 검증 요청)를 파일 트리와 함께 반복하면 실무 재현성이 높아진다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **시각화 실습 모듈로 진입한다.**
   - 실행 입력(명령): `claude "/start-1-2"`
   - 기대 출력: 워크스페이스 구성 안내.
   - 체크포인트: 터미널 단독이 아닌 split-screen 준비를 시작했는지 확인.

2. **에디터와 터미널을 동시 배치한다.**
   - 실행 입력(행동): 좌측 터미널, 우측 에디터(또는 반대)로 고정.
   - 기대 출력: 명령 실행과 파일 반영을 동시에 보는 작업 화면.
   - 체크포인트: 화면 전환 없이 수정 결과를 즉시 확인 가능한지 검증.

3. **루트 경로와 숨김 폴더를 점검한다.**
   - 실행 입력(프롬프트): `현재 프로젝트 루트와 .claude 가시성 점검 체크리스트를 만들어줘.`
   - 기대 출력: 루트 경로 확인 항목 + 숨김 폴더 확인 항목.
   - 체크포인트: `.claude/agents`, `.claude/commands` 접근 가능 여부 확인.

4. **테스트 파일 생성으로 동기 반영을 검증한다.**
   - 실행 입력(프롬프트): `test-visualization.md 파일을 생성하고 3줄 내용을 작성해줘.`
   - 기대 출력: 파일 생성 + 내용 반영.
   - 체크포인트: 에디터에서 즉시 파일이 나타나는지 확인.

5. **수정/삭제까지 한 사이클을 완료한다.**
   - 실행 입력(프롬프트): `test-visualization.md에 검증 문구 2줄 추가하고 마지막 줄을 삭제해줘.`
   - 기대 출력: diff가 보이는 수정 결과.
   - 체크포인트: 생성-수정-삭제의 전체 라이프사이클 재현.

6. **도구별 운영 기준을 문서화한다.**
   - 실행 입력(프롬프트): `Nimbalyst/Obsidian/VS Code 선택 기준을 PM 문서작업 기준으로 비교표로 만들어줘.`
   - 기대 출력: 도구 선택 매트릭스.
   - 체크포인트: 숨김 폴더 가시성/문서 편집성/학습난이도 항목 포함.

7. **실무 파일로 확장 테스트한다.**
   - 실행 입력(프롬프트): `실제 회의요약 파일 1개를 같은 루틴으로 생성하고 품질 점검 항목 5개를 붙여줘.`
   - 기대 출력: 업무용 샘플 문서.
   - 체크포인트: 테스트 환경에서 실무 환경으로 전환 가능한지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 2**: 환경 구성 즉시 파일 생성 테스트를 수행하게 해 “설정 완료”를 결과로 검증한다.
- **P2 맥락 우선 + Level 2**: 루트/숨김 폴더 가시성을 확보해 에이전트/명령/메모리 맥락 누락을 방지한다.
- **P4 속도-검증 균형 + Level 2**: split-screen으로 생성 즉시 확인 루프를 만들어 수정-검증 리드타임을 줄인다.
- **P8 Connected Hub + Level 2**: 에디터 선택은 취향이 아니라 운영 목적(문서·코드·숨김폴더 가시성) 기준으로 결정한다.

## 4) 실무 적용 시나리오
주니어 PM이 회의 요약 자동화 파일을 생성했는데 저장 경로가 계속 어긋나는 상황이다. 1.2 가이드대로 split-screen을 구성하고, 루트 폴더와 `.claude/` 표시 여부를 먼저 점검한다. 이후 테스트 파일 생성으로 경로 문제를 해결하고, 실제 회의 요약 산출물도 같은 검증 루틴으로 안정화한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 팀 표준 작업 환경(터미널+에디터 split-screen)과 루트 폴더 오픈 규칙을 통일하고, 도구별 설정값을 스크린샷으로 기록합니다.
- **수~목 실행**: 테스트 파일 생성/수정/삭제 사이클을 3회 반복해 경로 오류, 숨김 폴더 미표시, 저장 지연 이슈를 점검합니다.
- **금요일 검증**: 실제 업무 파일(회의 요약, PRD 초안) 1건을 동일 루틴으로 처리해 실무 재현성을 확인합니다.
- **Done 기준**: 팀원 전원이 `.claude/` 가시성 확인, 테스트 파일 실시간 반영 검증, 경로 오류 대응 절차 숙지를 완료합니다.
- **리스크/완화**: 에디터별 설정 차이로 혼선이 생길 수 있으므로, 필수 설정만 담은 최소 공통 프로필과 예시 화면을 함께 제공합니다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
1. **월요일: 화면/도구 기준 맞추기**
   - 시간: 10:00~11:00
   - 행동: 터미널+에디터 split-screen 설정, 프로젝트 루트 오픈.
   - 산출물: 개인 환경 설정 캡처.
   - 검증 기준: 루트 경로와 숨김 폴더 가시성 확인 완료.

2. **화요일: 테스트 파일 사이클 1차**
   - 시간: 14:00~15:00
   - 행동: `test-visualization.md` 생성→수정→삭제.
   - 산출물: 작업 로그 1부.
   - 검증 기준: 실시간 반영 지연 5초 이내, 경로 오류 0건.

3. **수요일: 테스트 파일 사이클 2차(실패 재현 포함)**
   - 시간: 11:00~12:00
   - 행동: 의도적으로 잘못된 경로에서 실행 후 복구.
   - 산출물: 실패/복구 체크리스트.
   - 검증 기준: 잘못된 경로 감지 후 10분 내 정상 복귀.

4. **목요일: 실제 업무 파일 적용**
   - 시간: 16:00~17:00
   - 행동: 회의 요약 파일 생성 및 리뷰.
   - 산출물: 회의 요약 초안 1부.
   - 검증 기준: 파일 생성 위치, 제목 규칙, 필수 섹션 누락 0건.

5. **금요일: 팀 합동 검증**
   - 시간: 15:00~16:00
   - 행동: 팀원 간 화면 공유로 설정 비교 및 표준안 확정.
   - 산출물: 팀 표준 설정 가이드 v1.
   - 검증 기준: 신규 입사자가 문서만 보고 30분 내 환경 재현 가능.

- **롤백/복구 액션**: 목요일 실무 파일 작성에서 경로 충돌이 반복되면, 개인 커스텀 설정을 임시 비활성화하고 팀 표준 프로필로 되돌린 뒤 테스트 파일 사이클을 다시 1회 수행합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 파일이 생성됐는데 에디터에 바로 안 보임**
   - 해결: 루트 폴더 오픈 상태를 먼저 확인하고, 필요 시 파일 트리 새로고침 및 경로 재확인.

2. **막힘: `.claude` 폴더가 보이지 않음**
   - 해결: 에디터의 숨김 파일 표시 옵션 활성화 또는 Finder/Explorer 보조 확인.

3. **막힘: 팀원마다 도구 설정이 달라 협업 혼선 발생**
   - 해결: 필수 설정만 포함한 최소 공통 프로필을 배포하고 예외 설정은 부록으로 분리.

4. **막힘: 테스트는 되는데 실무 문서에서 위치 오류 재발**
   - 해결: 실무 파일에도 동일한 파일명/경로 규칙을 강제하고 저장 전 체크리스트 적용.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)

- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.
- [ ] `CLAUDE.md`의 프로젝트 규칙(금지사항/품질게이트/보고형식)을 확인했다.
- [ ] `@` 또는 첨부 자료로 **맥락 입력(문서/데이터/정책) 3개 이상** 연결했다.
- [ ] 필요한 경우 **sub-agent/병렬 작업** 범위와 합성(synthesis) 책임자를 지정했다.
- [ ] **MCP/외부도구 연동 필요 여부**와 접근 권한 상태를 점검했다.
- [ ] 산출물에 **검증 기준(정확성/재현성/지표/테스트)** 을 명시했다.
