## 1) 목적
- 제품 개요를 기능 설명서가 아니라 **우선순위·투자·실험 판단 문서**로 전환한다.
- 제품 철학(async-first, context-rich, speed-first)을 로드맵 및 KPI에 직접 연결한다.
- PRD 작성 시 “왜 이 기능을 지금 하는가”를 정량/정성으로 증명할 수 있게 한다.

## 2) 원문 핵심 요약 (ccforpms)
- TaskFlow 포지셔닝은 Asana의 범용성과 Linear의 사용감을 결합한 원격 협업 특화 PM SaaS다.
- 핵심 가치 제안은 “업무 단위(task)에 맥락을 결합해 회의 의존도를 낮추는 async 협업”이다.
- 제품 철학 5축은 async-first, context-first, speed-first, opinionated but flexible, design quality다.
- 핵심 기능군은 Task 관리, 프로젝트 뷰, 협업, 알림, 연동, 리포팅/분석으로 구성된다.
- 차별 기능으로 Context cards, AI 댓글 요약, 의사결정 표시, smart batching, two-way sync, 예측 분석이 제시된다.
- 핵심 성과 지표는 WAT(Weekly Active Teams), Activation(45→60), Retention(65→75), Time-to-Value(45분→15분), Viral Coefficient(1.2→1.5+)다.
- 단기 로드맵은 모바일, SSO/권한/audit log, 온보딩 개선이며, 중기 이후 API v2·time tracking·AI 고도화로 확장한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **제품 가치 제안 1문장을 먼저 확정한다.**
   - 입력 예시: "TaskFlow는 async 협업에서 맥락 손실을 줄여 의사결정 속도를 높인다.".
   - 실행 액션: 문제·대상·차별점이 포함된 문장으로 고정한다.
   - 기대 출력: 기능 논의가 가치 제안과 일관되게 정렬된다.
2. **핵심 KPI 기준선을 최신화한다.**
   - 입력 예시: Activation 45%, TTV 45분, Retention 65%.
   - 실행 액션: 현재값/목표값/격차를 표로 정리한다.
   - 기대 출력: 기능 우선순위 판단의 수치 기준 확보.
3. **후보 기능 3개를 같은 포맷으로 비교한다.**
   - 입력 예시: A 첫 프로젝트 자동생성, B 역할별 템플릿, C 협업 초대 유도.
   - 실행 액션: 임팩트/비용/리스크를 동일 양식으로 기록한다.
   - 기대 출력: 선호 기반 논쟁보다 근거 기반 선택 가능.
4. **예상 임팩트를 시나리오별로 계산한다.**
   - 입력 예시: 보수/기준/공격 3시나리오의 Activation uplift 추정.
   - 실행 액션: 지표 영향과 측정 기간을 함께 명시한다.
   - 기대 출력: 파일럿 성공 기준이 명확해짐.
5. **상위 1개 기능을 PRD로 변환한다.**
   - 입력 예시: "온보딩 단계 축소로 TTV 45→25분".
   - 실행 액션: 범위, 수용 기준, 제외 항목을 문서화한다.
   - 기대 출력: 개발 착수 가능한 PRD 초안.
6. **실험 단위를 티켓으로 분해해 실행한다.**
   - 입력 예시: Epic 1개, Story 3개, 측정 이벤트 정의.
   - 실행 액션: 기능 구현과 계측 작업을 분리 배치한다.
   - 기대 출력: 출시 후 효과 측정 누락 방지.
7. **2주 파일럿 후 유지/확대/중단을 판정한다.**
   - 입력 예시: Activation +6pp면 확대, +2pp 이하면 중단.
   - 실행 액션: KPI 결과와 운영 비용을 함께 평가한다.
   - 기대 출력: 로드맵 반영 여부를 데이터 기반으로 확정.

### Claude Code 핵심 기능 연결 (official 반영)
- **제품 가설을 실행 단위로 분해**: 가치 제안과 KPI를 기능/실험 단위로 바로 전환해 실행 속도를 높인다.
- **Plan 모드 기반 로드맵 정렬**: 후보 기능을 임팩트/비용 관점으로 구조화해 분기 우선순위를 명확히 한다.
- **자연어 PRD 초안 가속**: PM이 "왜 지금 필요한가"를 설명하면 PRD 골격을 빠르게 만든다.
- **문맥 유지형 제품 의사결정**: 제품 철학(async/context/speed)을 구현/검증 단계까지 유지해 일관성을 확보한다.
- **실험-계측 연결 강화**: 기능 티켓과 측정 이벤트를 함께 설계해 출시 후 판단 공백을 줄인다.
- **협업 도구 연계 운영**: 문서, 이슈, 상태 공유를 연결해 제품 결정의 실행 추적성을 높인다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 기능 아이디어는 설명으로 끝내지 않고 “PRD→티켓→실험 계획”까지 한 세트로 산출한다.
- **P4 속도-검증 균형**: 빠른 초안 생성 후 AC/TC/Traceability로 검증하는 2단 구조를 표준으로 삼는다.
- **P8 Connected Hub**: 제품 문서(Claude/Notion)·이슈(Linear)·상태 공유(Slack)·자동 리마인드(n8n)를 하나로 묶는다.
- **FR-016~FR-020 반영**: Level1 즉시 워크플로우, Level2 허브, OMTM 대시보드를 제품 의사결정 루틴에 연결한다.
- **Words-to-Workflow(P9)**: “기능 설명 문장”을 실행 명령(` /prd`, `/status`, `/agent`)으로 바로 변환할 수 있어야 한다.

## 4) 실무 적용 시나리오
### 시나리오: 온보딩 개선 기능 우선순위 결정
1. 제품 지표에서 Activation 45%와 Time-to-Value 45분을 문제 정의로 확정한다.
2. `2.1-advanced-write-prd` 방식으로 온보딩 개선안 3개를 동시에 생성한다.
   - A안: 첫 프로젝트 자동 생성
   - B안: 역할별 템플릿 추천
   - C안: 첫 협업 초대 유도
3. `2.2-advanced-analyze-data` 방식으로 예상 lift를 시나리오(보수/기준/공격)로 계산한다.
4. Linear Epic/Story 생성 시 각 안의 KPI 영향도(Activation, TTV)를 명시한다.
5. 2주 파일럿 후 유지/확대/중단 결정을 KPI 기준으로 확정한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 제품 가치 제안 1문장과 핵심 KPI(Activation/Retention/TTV/Viral) 기준선을 최신화해 의사결정 출발점을 고정한다.
- **수~목 실행**: 기능 후보 2~3개를 동일 포맷(가설/임팩트/비용/리스크)으로 비교하고 파일럿 우선순위를 확정한다.
- **금요일 검증**: 파일럿 대상 기능의 수용 기준과 측정 방식이 준비되었는지 리뷰하고 출시/보류를 결정한다.
- **Done 기준**: KPI-기능 매핑, 우선순위 결정 근거, 파일럿 측정 계획이 문서에 연결되어 즉시 실행 가능한 상태다.
- **리스크/완화**: 기능 선호 논쟁이 길어지면 사전 정의한 KPI 가중치표로 판단하고, 동점 항목은 1주 파일럿으로 실측 후 결정한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 온보딩 개선 후보가 많은데 개발 리소스는 제한적인 주간, PM이 제품 문맥으로 우선순위를 확정해야 한다.

- **월요일**
  - Owner: PM, 데이터 분석가
  - Action: 제품 가치 제안과 KPI 기준선 최신화
  - Artifact: 가치 제안 카드, KPI 기준선 표
  - Check: 팀이 동일한 North Star/보조지표를 참조하는지 확인
- **화요일**
  - Owner: PM
  - Action: 기능 후보 3~4개를 동일 포맷으로 정리(가설/임팩트/비용/리스크)
  - Artifact: 후보 비교 시트, 우선순위 점수
  - Check: 각 후보가 최소 1개 KPI와 연결되는지 점검
- **수요일**
  - Owner: PM, 개발 리드, 디자이너
  - Action: 상위 1~2개 후보를 PRD로 전환, 파일럿 범위 결정
  - Artifact: PRD v1.0, 파일럿 범위 문서
  - Check: 구현 범위와 제외 범위가 명확히 분리됐는지 확인
- **목요일**
  - Owner: 개발자, PM
  - Action: Epic/Story 분해, 계측 이벤트 설계
  - Artifact: 티켓 세트, 이벤트 트래킹 정의서
  - Check: 출시 후 KPI 측정 가능 여부 사전 검증
- **금요일**
  - Owner: PM, 리더십
  - Action: 파일럿 승인 회의(Go/No-Go), 차주 실행 계획 확정
  - Artifact: 승인 로그, 차주 백로그
  - Check: 성공/중단 기준이 문서에 숫자로 명시됨

**리스크 트리거(발동 조건)**
- 수요일까지 후보 기능의 KPI 연결 비율이 70% 미만이거나, 구현 비용 추정 오차가 2배 이상 발생하면 우선순위 왜곡 위험으로 판단한다.

**Rollback/Mitigation 흐름**
1. 기능 후보를 다시 "KPI 직접 영향" 기준으로 1차 필터링한다.
2. 점수 동률 항목은 1주 미니 파일럿으로 전환해 실측 데이터를 우선 확보한다.
3. 비용이 과도한 항목은 범위를 축소해 핵심 가설만 검증한다.
4. 금요일에는 최종 1개 파일럿만 확정해 실행 집중도를 높인다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 기능은 많지만 어떤 KPI를 개선하는지 연결이 안 됨**
   - 해결: 기능 카드에 "직접 영향 KPI 1개"를 필수 입력으로 강제한다.
   - 확인: KPI가 비어 있는 카드는 우선순위 후보에서 제외한다.
2. **막힘: 팀이 좋아하는 기능과 실제 임팩트가 충돌**
   - 해결: KPI 가중치표(예: Activation 40, Retention 30, TTV 20, Viral 10)로 정량 판단한다.
   - 확인: 최종 선택 기능의 점수가 문서에 남아야 한다.
3. **막힘: 파일럿은 했지만 계측 이벤트가 없어 성과 판단 불가**
   - 해결: 기능 티켓과 계측 티켓을 분리해 동시 완료 조건으로 관리한다.
   - 확인: 배포 전 이벤트 수집 테스트를 반드시 수행한다.
4. **막힘: 중단 기준이 없어 낮은 성과 기능이 계속 유지됨**
   - 해결: 파일럿 시작 시 확대/유지/중단 임계값을 숫자로 고정한다.
   - 확인: 금요일 리뷰에서 임계값 기준으로 자동 판정되는지 점검한다.
