## 1) 목적
- Level1/2에서 사람이 반복 처리하던 루틴을 n8n으로 자동화해 운영 부하를 줄인다.
- 스케줄/이벤트 기반 실행으로 리드타임 변동과 누락 리스크를 낮춘다.
- 실패 대응(재시도/알림/수동 fallback)을 포함한 운영 신뢰성을 확보한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 2.2는 반복 데이터 분석을 템플릿화해 속도와 일관성을 높일 수 있음을 강조한다.
- 원문 2.3은 경쟁/전략 입력을 병렬로 수집하고 구조화하는 운영 방식을 제시한다.
- 원문 1.4의 병렬 에이전트 패턴은 자동화 설계에서 fan-out 처리와 유사한 사고를 제공한다.
- 원문 4.5는 배포 이후에도 반복 개선 루프가 중요함을 보여주며, 운영 자동화 필요성을 뒷받침한다.
- 즉, Level3의 본질은 “한 번 잘 만들고 끝”이 아니라 “반복 실행 가능한 파이프라인”이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview의 실행 원칙을 적용하면 n8n 자동화도 **입력 스키마-처리 규칙-검증 기준**이 명확해야 운영 안정성이 확보된다.
- PM은 Claude Code에서 플로우 설계 문서와 런북을 먼저 고정한 뒤 n8n 구현으로 넘어가야 재시도/복구 품질이 올라간다.
- Tutorials Hub 패턴대로 자동화 결과는 반드시 사람 의사결정 액션(owner/due)으로 연결되어야 성과가 축적된다.
- Opus 4.6 튜토리얼 관점에서는 복합 분류/요약 파이프라인의 규칙 정밀도가 성공률에 직접 영향을 준다.
- Chrome 튜토리얼과 결합하면 외부 데이터 수집을 자동화 입력으로 삼고, Claude Code에서 결과 검증/요약을 표준화할 수 있다.
- 금융 서비스 튜토리얼의 운영 안정성 관점을 반영해 실패 로그, 알림 경보, 수동 fallback을 필수 운영 요소로 둔다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일, 팀이 반복 업무를 목록화하고 빈도/규칙성/영향도 점수로 자동화 우선순위 상위 3개를 선정한다.
2. 월요일 오후, 각 후보 플로우에 대해 트리거(언제), 처리(무엇), 출력(어디), 검증(어떻게)를 1페이지 설계서로 고정한다.
3. 화요일, Flow-01(뉴스 다이제스트)을 구현해 1일 1회 실행하고 Slack 전송까지 end-to-end 테스트한다.
4. 수요일, Flow-02(피드백 분류)를 구현해 샘플 데이터 50건으로 정확도/오류율을 측정한다.
5. 수요일 오후, 재시도 정책(지수 백오프), 실패 알림, 수동 fallback 절차를 런북에 명시한다.
6. 목요일, 실제 운영 데이터로 2일 연속 안정성 테스트를 돌려 성공률/SLA 위반 건수를 기록한다.
7. 금요일 오전, 실패 유형을 `트리거 누락/API 제한/분류 오류/전달 실패`로 분류해 개선 우선순위를 정한다.
8. 금요일 오후, 운영 승인 기준(성공률 95%+, 평균 처리시간 기준 충족)을 만족한 플로우만 활성 상태로 전환한다.

| 범위 | 문서 생성 중심 | 뉴스/피드백/CS/콘텐츠 파이프라인 통합 |

| 관측성 | 실행 여부 확인 수준 | 성공률/오류율/SLA 대시보드화 |

- **P6 자동화 운영 기반**: 사람이 판단해야 할 구간만 남기고 반복 처리 자동화.
- **P7**: 자동 분류는 허용하되, 고위험 답변/외부 발신은 human-in-the-loop 유지.

#### 월~금 상세 실행표
| 요일 | 실행 항목 | 주요 입력 | 산출물 | 검증 기준 | 복구 절차 |
|---|---|---|---|---|---|
| 월 | 자동화 후보 선정 | 업무 목록/빈도 데이터 | 우선순위 표 | 상위 2개 확정 | 점수 기준 재합의 |
| 화 | Flow-01 구축 | RSS/API 소스 | 일간 다이제스트 | 전달 성공률 | 수동 브리핑 전환 |
| 수 | Flow-02 구축 | 피드백 원문 | 분류 리포트 | 정확도 기준 | 룰 기반 fallback |
| 목 | 안정성 테스트 | 2일 운영 로그 | SLA 점검표 | 성공률 95%+ | 재시도/큐 조정 |
| 금 | 운영 승인/리뷰 | 실패 로그/지표 | 개선 백로그 | 반복 실패 감소 계획 | 비활성화 후 보완 |

#### 검증/복구 흐름
1. 트리거 누락이 발견되면 heartbeat 알림을 즉시 활성화하고 누락 구간을 수동 실행으로 보충한다.
2. API rate limit 오류는 지수 백오프(예: 1m→2m→4m)와 최대 재시도 횟수로 제어한다.
3. 분류 정확도 미달 시 샘플 라벨 30건 이상으로 룰셋을 재조정하고 재평가한다.
4. Slack/이메일 전달 실패가 2회 연속 발생하면 1차 채널을 비활성화하고 대체 채널로 우회 전송한다.
5. 금요일 승인 회의에서 기준 미달 플로우는 운영에서 제외하고 개선 완료 후 재심사한다.

### 실행 순서 (2~3주)
1. **Flow-01 뉴스 다이제스트**
   - 트리거: 매일 07:30
   - 처리: RSS/API 수집 → 요약 → 저장 → Slack 브리핑
   - 산출물: 일간 다이제스트
2. **Flow-02 피드백 분류**
   - 트리거: Form/Sheet 신규 행
   - 처리: 분류(버그/요청/기타) → 우선순위 부여 → DB 저장
   - 산출물: 주간 피드백 리포트
3. **Flow-03 이메일 CS triage**
   - 트리거: Gmail 라벨 수신
   - 처리: 긴급도 분류 → 담당자 알림 → 처리 상태 기록
   - 산출물: CS 처리 SLA 리포트
4. **Flow-04 콘텐츠 발행 보조(선택)**
   - 트리거: 주간 스케줄
   - 처리: 초안 생성 → 검수 요청 → 게시 큐 적재
   - 산출물: 게시 대기 목록

### 실패 패턴
- 실패1: 트리거 누락으로 실행 자체가 안 됨
  - 대응: heartbeat 체크 + 실행 누락 알림
- 실패2: 외부 API rate limit
  - 대응: 지수 백오프 + 재시도 상한 설정
- 실패3: 분류 정확도 저하
  - 대응: 샘플 라벨링 재학습 + 룰 기반 fallback
- 실패4: 자동화는 돌아가지만 운영팀이 결과를 안 봄
  - 대응: Slack 요약에 “필수 액션 3개” 자동 포함

### 이번 주 완료형 실행 플랜
- **월~화:** 자동화 후보를 점수화(빈도/규칙성/영향도)해 우선 플로우 2개를 선정하고 트리거/입출력 스키마를 확정한다.
- **수~목:** Flow-01(뉴스)과 Flow-02(피드백) 최소 버전을 배포하고 재시도·알림·fallback 정책을 적용한다.
- **금 검증:** 실행 성공률, 평균 처리시간, 오류율을 측정해 SLA 임계값 충족 여부를 점검한다.
- **Done 기준:** 활성 플로우 2개 이상, 런북 1종, 실패 알림 동작 확인, 주간 성과 리포트 초안이 준비된다.
- **리스크·완화:** 외부 API rate limit로 실패가 증가할 수 있으므로 지수 백오프와 큐잉 간격 조절을 기본값으로 설정한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 자동화 후보 선정이 감(感)에 의존함**
  - 증상: 중요한 플로우보다 만들기 쉬운 플로우를 먼저 선택한다.
  - 해결: 빈도/규칙성/영향도 점수 합계가 높은 순으로만 착수하는 규칙을 적용한다.
- **막힘 2: 오류는 잡히는데 운영팀이 늦게 인지함**
  - 증상: 실패 알림이 묻혀 복구가 지연된다.
  - 해결: 실패 등급(P1/P2/P3)을 나누고 P1은 즉시 멘션, P2는 일일 요약, P3는 주간 회고로 분리한다.
- **막힘 3: 재시도 무한 루프로 비용/시간 낭비**
  - 증상: 실패 작업이 끝없이 재시도된다.
  - 해결: 재시도 상한과 dead-letter 큐를 도입해 일정 횟수 초과 시 수동 처리로 전환한다.
- **막힘 4: 자동화 산출물이 의사결정으로 연결되지 않음**
  - 증상: 리포트는 생성되지만 아무도 액션을 취하지 않는다.
  - 해결: 모든 자동화 결과에 `owner + next action + due` 필드를 강제로 포함한다.
