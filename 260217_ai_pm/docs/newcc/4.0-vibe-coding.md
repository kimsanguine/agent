# 4.0 Vibe Coding

**빠른 요약**
- 4.0 Vibe Coding 주제를 빠르게 이해하고 바로 써볼 수 있게 정리한 문서입니다.

**이번 문서 목표**
- 읽고 나면 4.0 Vibe Coding 내용을 내 업무 흐름에 맞게 적용할 수 있습니다.

**처음 보는 분 가이드**
- 먼저 `## 1) 목적`과 `## 5) 실습/적용 체크리스트`만 읽고 시작하세요.

## 1) 목적
- PM이 아이디어를 설명하는 수준을 넘어, **기획→구현→검증→배포**까지 이어지는 실행 루프를 운영하도록 만든다.
- 코딩 전문성 유무와 무관하게, 자연어 기반 협업으로 실제 웹앱을 출시하는 흐름을 체득한다.
- Module 4 전체(4.1~4.5)의 공통 운영 원칙을 정렬한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 모듈 목표는 1~2시간 안에 웹앱을 만들고 공개 URL까지 배포하는 것이다.
2. 기본 흐름은 4.1 Setup → 4.2 Plan → 4.3 Build & Iterate → 4.4 GitHub → 4.5 Go Live 순서다.
3. 실습 결과물 예시는 personality quiz 앱처럼 로직·결과·공유 가능한 링크를 갖춘 제품 단위다.
4. 선행 학습으로 Modules 1~3이 권장되며, 미설치 상태라면 0.1 설치 단계를 먼저 수행해야 한다.
5. 자료 준비는 `claude-code-course` 폴더 기준이며 자동/수동 다운로드 경로가 모두 제공된다.
6. 핵심 학습 방식은 설명(의도) → 생성(구현) → 반복(수정)의 짧은 사이클 운영이다.
7. 완료 기준은 기능 구현만이 아니라 GitHub 버전관리와 Vercel 배포까지 포함된다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **문제와 목표를 10분 안에 고정한다.**
   - 입력 예시: "신규 팀원이 첫 주에 TaskFlow 핵심 기능을 못 찾아 이탈한다. Activation을 45%→50%로 올린다."
   - 실행 액션: 목표 사용자 1명, 개선 지표 1개, 마감일 1개를 한 문장으로 작성한다.
   - 기대 출력: 팀이 같은 문장을 복붙해 써도 의미가 안 바뀌는 목표 문장 1개.
2. **4.1 Setup으로 실행 환경을 잠근다.**
   - 입력 예시: `/start-4-1` 후 "Node 버전, 폴더 구조, 실행 명령" 확인 요청.
   - 실행 액션: 로컬 실행 명령과 코스 파일 버전을 팀 위키에 기록한다.
   - 기대 출력: 신규 참여자가 문서만 보고 30분 내 실행 가능한 상태(Ready 판정).
3. **4.2 Plan에서 요구사항을 문서로 확정한다.**
   - 입력 예시: `/start-4-2` + "핵심 흐름: 가입→첫 Task 생성→공유".
   - 실행 액션: 기능/예외/성공지표를 `REQUIREMENTS.md`에 잠근다.
   - 기대 출력: 구현 담당자가 재질문 없이 착수 가능한 요구사항 문서 v1.0.
4. **로직을 if-then 규칙으로 변환한다.**
   - 입력 예시: "입력값 누락이면 저장 비활성, 저장 성공이면 공유 CTA 노출".
   - 실행 액션: 정상/실패/경계값 3종 케이스를 규칙표로 만든다.
   - 기대 출력: QA가 그대로 테스트 케이스로 사용할 수 있는 규칙표 1개.
5. **4.3 Build & Iterate로 핵심 흐름만 먼저 구현한다.**
   - 입력 예시: `/start-4-3` + "가입 완료 후 첫 Task 생성 버튼이 2클릭 내 도달".
   - 실행 액션: dev server에서 핵심 시나리오 1개를 먼저 통과시킨다.
   - 기대 출력: `localhost` 기준 데모 가능한 프로토타입과 수정 요청 스크린샷 세트.
6. **4.4 GitHub로 변경 이력을 안전하게 저장한다.**
   - 입력 예시: `/start-4-4` + "핵심 흐름 동작, 오류 메시지 개선" 커밋 메시지.
   - 실행 액션: 작은 단위 커밋 2~3개로 의도와 검증 결과를 남긴다.
   - 기대 출력: 되돌리기 가능한 기준 커밋 + 리뷰 가능한 PR 본문.
7. **4.5 Go Live로 공개 URL 검증 루프를 연다.**
   - 입력 예시: `/start-4-5` + "베타 사용자 5명 공유".
   - 실행 액션: 배포 URL 발급 후 모바일/데스크탑 핵심 경로를 점검한다.
   - 기대 출력: 공개 URL, 초기 피드백 로그, 다음 주 개선 백로그 3개.

### Claude Code 핵심 기능 연결 (official 반영)
- **Plan 모드(계획 선행)**: 4.0의 `Setup→Plan→Build→GitHub→Go Live`를 바로 구현하지 않고, 먼저 실행 순서/산출물/리스크를 계획으로 잠가 재작업을 줄인다.
- **자연어 기반 코드 작업**: PM이 기능 의도를 자연어로 전달하면 Claude Code가 코드/설정 변경으로 연결해 “words ARE the code”를 실무 루프로 만든다.
- **통합 터미널 실행**: 빌드, 테스트, 린트, 실행 로그 확인을 한 흐름으로 처리해 4.3 단계의 반복 속도를 높인다.
- **Git 워크플로우 연계**: 변경 요약, 커밋, PR 생성까지 이어져 4.4 단계의 버전 관리 기준을 일관되게 유지한다.
- **프로젝트 규칙 준수(예: CLAUDE.md)**: 팀 운영 규칙과 문서 기준을 작업 중 계속 반영해 산출물 편차를 줄인다.
- **외부 도구 연결(MCP/CLI)**: GitHub·배포·협업 도구를 연결해 4.5 운영 단계까지 단절 없이 이어진다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **Dennis 원칙 직결**: “**words ARE the code**”를 PM 실행 문맥으로 해석하면, PM의 문장은 곧 제품 동작 명세가 된다.
- **P3 PM 오케스트레이션**: PM은 개발 대체자가 아니라 문제정의·우선순위·검증 루프 설계자다.
- **P4 속도-검증 균형**: 빠른 프로토타입 생성과 테스트/리뷰 게이트를 동시에 운영한다.
- **P8 Connected Hub**: 전략 대화(Claude)와 실행 작업(Claude Code), 운영 자동화(n8n)를 하나의 허브로 연결한다.
- **Level4 연결 명시**: `/agent` 기반으로 **PRD → TDD → Eval → Prototype** 파이프라인을 운영하고, 이후 배포 핸드오프로 확장한다.
- **P9 Words-to-Workflow**: 문서(요구사항), 명령(작업 지시), 결과(코드/테스트/배포)가 단절되지 않게 설계한다.
- **P10 Validation-First Discovery**: PM의 핵심 역할을 “무엇을 만들지 결정”뿐 아니라 “무엇이 작동하는지 빠르게 검증”으로 확장해, 가설 검증 리드타임을 단축한다.

## 4) 실무 적용 시나리오
**상황: 1주 내 내부 검증용 MVP를 만들어야 하는 PM**
1. 월요일: 문제/가설을 1페이지 PRD로 정리한다.
2. 화요일: `/agent`로 TDD 초안을 생성해 실패 조건부터 정의한다.
3. 수요일: 프로토타입을 구현하고 핵심 사용자 흐름만 먼저 연결한다.
4. 목요일: Eval 루브릭(사용성/정확도/오류율)으로 테스트한다.
5. 금요일: GitHub 정리 후 스테이징 배포, 이해관계자 피드백 수집.
6. 다음 주: n8n 기반 알림/리포트 자동화를 붙여 운영 루프로 전환.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 4.1~4.2를 완료하면서 문제정의 1페이지, 핵심 사용자 1종, OMTM 1개를 팀 합의본으로 고정한다.
- **수~목 실행**: 4.3~4.4를 진행해 핵심 사용자 흐름 1개를 동작시키고, 변경 이력을 커밋/PR 단위로 정리한다.
- **금요일 검증**: 스테이징 또는 배포 URL 기준으로 핵심 시나리오 5개를 점검하고 이해관계자 3명에게 피드백을 수집한다.
- **Done 기준**: PRD→TDD→Eval→Prototype 산출물 링크, GitHub 이력, 배포 URL, 다음 주 개선 백로그가 한 문서에 연결되어 있다.
- **리스크/완화**: 주중 요구사항이 계속 추가되면 금요일 오전부터 변경 동결을 적용하고 신규 요청은 다음 스프린트 후보로 분리한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 개발 경험이 많지 않은 PM이 5일 안에 내부 검증용 MVP를 완성해야 한다.

- **월요일**
  - Owner: PM(리드), 파트너 디자이너
  - Action: 문제정의 1페이지 작성(문제/대상/지표/마감), `Module 4` 실행 범위 확정
  - Artifact: `MVP-brief-v1.md`, 목표 지표 카드(Activation, 완료율)
  - Check: 이해관계자 2명에게 "문제 정의 문장"을 읽혀 같은 해석이 나오는지 확인
- **화요일**
  - Owner: PM, Claude Code 운영 담당
  - Action: `/start-4-2` 기반 요구사항 인터뷰 진행, 기능/예외/수용 기준 잠금
  - Artifact: `REQUIREMENTS.md` v1.0, if-then 규칙표 1장
  - Check: "재질문 없이 구현 가능한가" 기준으로 개발 담당 1명 사전 리뷰 통과
- **수요일**
  - Owner: 구현 담당, PM 리뷰어
  - Action: `/start-4-3`로 핵심 사용자 흐름 1개(가입→핵심 행동 완료) 구현
  - Artifact: 로컬 데모 URL(`localhost`), 스크린샷 피드백 로그
  - Check: 핵심 시나리오 5개 중 최소 3개 통과, 치명 오류 0건
- **목요일**
  - Owner: PM, QA
  - Action: Eval 루브릭(완료율/오류율/시간)로 회귀 점검, 2차 개선 반영
  - Artifact: `eval-report-day4.md`, 우선순위 버그 리스트
  - Check: 평균 완료시간 목표 대비 편차 ±20% 이내, blocker 버그 전부 닫힘
- **금요일**
  - Owner: PM, 배포 담당
  - Action: `/start-4-4`→`/start-4-5` 순서로 GitHub 정리 및 스테이징 배포
  - Artifact: PR 링크, 배포 URL, 이해관계자 피드백 문서
  - Check: 의사결정자 3명 데모 완료, 다음 주 백로그 3개 이상 확정

**리스크 트리거(발동 조건)**
- 수요일 15:00 기준 핵심 시나리오 통과율이 60% 미만이거나, 요구사항 외 기능 추가가 2건 이상 발생하면 일정 붕괴 위험으로 간주한다.

**Rollback/Mitigation 흐름**
1. 신규 기능 추가를 즉시 중단하고, "핵심 흐름 안정화" 상태로 스코프를 재잠금한다.
2. 마지막 Green 커밋으로 되돌린 뒤(배포 후보 브랜치 기준), 필수 수정만 재적용한다.
3. 금요일 배포 범위를 "핵심 시나리오 1개 + 오류 없는 데모"로 축소한다.
4. 미반영 항목은 다음 스프린트 `Could` 버킷으로 이동하고 재추정한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 목표 문장이 추상적이라 구현 결과가 매번 달라짐**
   - 해결: 목표 문장을 `대상 사용자 + 행동 + 목표 수치 + 시점` 4요소로 강제 작성한다.
   - 즉시 적용 템플릿: "[누가]가 [어떤 행동]을 [언제까지] [몇 %] 달성한다."
2. **막힘: 구현 도중 아이디어가 계속 추가되어 4.3~4.5가 지연됨**
   - 해결: 변경 요청 접수 시 `Must/Should/Could`로 즉시 분류하고, `Must` 외 항목은 금요일 이후로 이월한다.
   - 운영 규칙: 수요일 18:00 이후 신규 기능은 기본 차단, 버그/보안 이슈만 예외 허용.
3. **막힘: 피드백이 느낌 중심이라 수정 우선순위를 못 정함**
   - 해결: 스크린샷 1장당 "문제 위치/기대 동작/실패 조건" 3줄을 함께 기록한다.
   - 검증 방법: 수정 후 같은 스크린샷 번호로 before/after를 비교해 닫힘 여부를 명시한다.
4. **막힘: 배포 직전 문서·코드·결정 로그가 분리되어 인수인계가 어려움**
   - 해결: 금요일 오전에 `Release Note` 단일 문서를 만들어 PR, 테스트 결과, 배포 URL, 미해결 이슈를 링크로 통합한다.
   - 완료 기준: 새 담당자가 10분 내 현재 상태를 재현 설명할 수 있으면 통과.
