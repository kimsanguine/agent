## 1) 목적
- PM이 아이디어를 설명하는 수준을 넘어, **기획→구현→검증→배포**까지 이어지는 실행 루프를 운영하도록 만든다.
- 코딩 전문성 유무와 무관하게, 자연어 기반 협업으로 실제 웹앱을 출시하는 흐름을 체득한다.
- Module 4 전체(4.1~4.5)의 공통 운영 원칙을 정렬한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 모듈 목표는 1~2시간 안에 웹앱을 만들고 공개 URL까지 배포하는 것이다.
2. 기본 흐름은 4.1 Setup → 4.2 Plan → 4.3 Build & Iterate → 4.4 GitHub → 4.5 Go Live 순서다.
3. 실습 결과물 예시는 personality quiz 앱처럼 로직·결과·공유 가능한 링크를 갖춘 제품 단위다.
4. 선행 학습으로 Modules 1~3이 권장되며, 미설치 상태라면 0.1 설치 단계를 먼저 수행해야 한다.
5. 자료 준비는 `claude-code-course` 폴더 기준이며 자동/수동 다운로드 경로가 모두 제공된다.
6. 핵심 학습 방식은 설명(의도) → 생성(구현) → 반복(수정)의 짧은 사이클 운영이다.
7. 완료 기준은 기능 구현만이 아니라 GitHub 버전관리와 Vercel 배포까지 포함된다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **Dennis 원칙 직결**: “**words ARE the code**”를 PM 실행 문맥으로 해석하면, PM의 문장은 곧 제품 동작 명세가 된다.
- **P3 PM 오케스트레이션**: PM은 개발 대체자가 아니라 문제정의·우선순위·검증 루프 설계자다.
- **P4 속도-검증 균형**: 빠른 프로토타입 생성과 테스트/리뷰 게이트를 동시에 운영한다.
- **P8 Connected Hub**: 전략 대화(Claude)와 실행 작업(Claude Code), 운영 자동화(n8n)를 하나의 허브로 연결한다.
- **Level4 연결 명시**: `/agent` 기반으로 **PRD → TDD → Eval → Prototype** 파이프라인을 운영하고, 이후 배포 핸드오프로 확장한다.
- **P9 Words-to-Workflow**: 문서(요구사항), 명령(작업 지시), 결과(코드/테스트/배포)가 단절되지 않게 설계한다.
- **P10 Validation-First Discovery**: PM의 핵심 역할을 “무엇을 만들지 결정”뿐 아니라 “무엇이 작동하는지 빠르게 검증”으로 확장해, 가설 검증 리드타임을 단축한다.

## 4) 실무 적용 시나리오
**상황: 1주 내 내부 검증용 MVP를 만들어야 하는 PM**
1. 월요일: 문제/가설을 1페이지 PRD로 정리한다.
2. 화요일: `/agent`로 TDD 초안을 생성해 실패 조건부터 정의한다.
3. 수요일: 프로토타입을 구현하고 핵심 사용자 흐름만 먼저 연결한다.
4. 목요일: Eval 루브릭(사용성/정확도/오류율)으로 테스트한다.
5. 금요일: GitHub 정리 후 스테이징 배포, 이해관계자 피드백 수집.
6. 다음 주: n8n 기반 알림/리포트 자동화를 붙여 운영 루프로 전환.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 4.1~4.2를 완료하면서 문제정의 1페이지, 핵심 사용자 1종, OMTM 1개를 팀 합의본으로 고정한다.
- **수~목 실행**: 4.3~4.4를 진행해 핵심 사용자 흐름 1개를 동작시키고, 변경 이력을 커밋/PR 단위로 정리한다.
- **금요일 검증**: 스테이징 또는 배포 URL 기준으로 핵심 시나리오 5개를 점검하고 이해관계자 3명에게 피드백을 수집한다.
- **Done 기준**: PRD→TDD→Eval→Prototype 산출물 링크, GitHub 이력, 배포 URL, 다음 주 개선 백로그가 한 문서에 연결되어 있다.
- **리스크/완화**: 주중 요구사항이 계속 추가되면 금요일 오전부터 변경 동결을 적용하고 신규 요청은 다음 스프린트 후보로 분리한다.

## 5) 실습/적용 체크리스트
- [ ] Vibe Coding 전체 흐름(4.1~4.5)을 일정에 반영했다.
- [ ] “words ARE the code” 원칙에 맞춰 요구사항 문장을 구조화했다.
- [ ] PRD→TDD→Eval→Prototype 순서를 실제 작업 계획에 반영했다.
- [ ] 구현 완료 기준에 테스트/로그/문서 업데이트를 포함했다.
- [ ] GitHub와 배포까지 포함한 End-to-End 완료 조건을 명시했다.
- [ ] 다음 반복을 위한 개선 항목(기술/사용성/운영)을 분리했다.

## 6) 산출물
- Vibe Coding 운영 캔버스(역할/흐름/완료조건)
- PRD→TDD→Eval→Prototype 실행 계획
- 주간 구현/검증/배포 일정표
- 첫 릴리스 기준 정의서(Go/No-Go)
- 주간 실행 리포트(실행 항목/검증 결과/차주 액션 3개)
