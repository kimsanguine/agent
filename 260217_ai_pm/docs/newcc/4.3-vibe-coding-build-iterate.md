## 1) 목적
- 로컬 개발 환경에서 빠르게 구현하고, 피드백 기반 반복 개선으로 제품 완성도를 높인다.
- “한 번에 완벽” 대신 **짧은 빌드-검증 루프**를 운영해 실패 비용을 줄인다.
- 시각 피드백(스크린샷) 중심 협업으로 PM-개발 커뮤니케이션 손실을 최소화한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 레슨 권장 시간은 30~40분이며 시작 명령은 `/start-4-3`이다.
2. dev server는 로컬에서 지속 실행되어야 하고 종료는 `Ctrl+C`로 한다.
3. `localhost:3000`은 내 컴퓨터 3000 포트 서버를 의미하며, 브라우저에서 즉시 결과 확인이 가능하다.
4. Plan Mode(`Shift+Tab`)를 활용하면 큰 작업 실행 전 계획 검토/수정이 가능하다.
5. 핵심 생산성 포인트는 AI를 통한 빠른 반복 개선 사이클이다.
6. `Ctrl+V`로 스크린샷을 붙여 시각적으로 수정 요청하면 커뮤니케이션 정확도가 높아진다.
7. 흔한 오류는 Node 미설치, 포트 충돌, 빈 화면, 캐시 이슈이며 각각 표준 대응이 있다.
8. Build 단계 이후 4.4 GitHub로 넘어가 버전관리/백업을 연결한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **빌드 세션 목표를 한 줄로 고정한다.**
   - 입력 예시: "오늘 목표는 가입→첫 Task 생성 흐름을 오류 없이 통과".
   - 실행 액션: 세션 시작 전에 목표 시나리오 1개와 완료 기준을 적는다.
   - 기대 출력: 작업 중 우선순위 흔들림 없이 핵심 경로 집중.
2. **`/start-4-3`로 구현 루프를 시작한다.**
   - 입력 예시: "핵심 CTA를 상단 고정하고 완료 버튼 가시성 개선".
   - 실행 액션: 요구사항 문서를 근거로 최소 기능부터 구현한다.
   - 기대 출력: 로컬에서 즉시 확인 가능한 동작 변화.
3. **dev server 상태를 유지한 채 빠르게 확인한다.**
   - 입력 예시: "`localhost:3000`에서 페이지 전환/저장 동작 점검".
   - 실행 액션: 변경 직후 브라우저 새로고침으로 결과를 확인한다.
   - 기대 출력: 작은 수정을 짧은 주기로 검증하는 루프 형성.
4. **큰 변경 전 Plan Mode로 위험을 줄인다.**
   - 입력 예시: `Shift+Tab` 후 "폼 구조 전체 변경 전 영향 범위 점검".
   - 실행 액션: 영향 파일/회귀 위험/검증 항목을 먼저 점검한다.
   - 기대 출력: 대형 수정 후 빈 화면·연쇄 오류 발생 확률 감소.
5. **스크린샷 기반으로 수정 요청을 명확히 한다.**
   - 입력 예시: "이 버튼 간격이 모바일에서 겹침" + 스크린샷 붙여넣기.
   - 실행 액션: 위치·현상·기대 결과를 한 번에 전달한다.
   - 기대 출력: 왕복 커뮤니케이션 횟수 감소, 수정 정확도 상승.
6. **오류를 표준 절차로 즉시 처리한다.**
   - 입력 예시: "포트 충돌", "빈 화면", "캐시 잔존".
   - 실행 액션: 포트 정리→의존성 확인→캐시 무효화 순으로 대응한다.
   - 기대 출력: 원인 추측 시간을 줄이고 복구 리드타임 단축.
7. **GitHub 전달 전 변경 목록을 잠근다.**
   - 입력 예시: "핵심 흐름 통과, 잔여 개선 4건".
   - 실행 액션: 반영/미반영 항목을 분리해 4.4 인계 문서를 만든다.
   - 기대 출력: 커밋 경계가 명확한 전달 패키지.

### Claude Code 핵심 기능 연결 (official 반영)
- **로컬 실행 중심 반복 루프**: 변경 직후 실행/확인/수정을 빠르게 반복해 Build 단계 리드타임을 단축한다.
- **Plan 모드로 대형 변경 리스크 제어**: 구조 변경 전 영향 범위를 점검해 빈 화면·회귀 오류를 줄인다.
- **스크린샷·맥락 기반 피드백 처리**: 시각 증거를 포함한 요청으로 수정 정확도를 높이고 왕복 횟수를 줄인다.
- **자연어 지시를 코드 수정으로 연결**: PM이 의도와 사용자 가치를 설명하면 구현 변경으로 즉시 전환 가능하다.
- **오류 대응 표준화**: 포트/캐시/의존성 문제를 동일 절차로 처리해 팀 내 복구 편차를 줄인다.
- **GitHub 인계 친화적 변경 관리**: 반영/미반영 목록을 명확히 나눠 4.4 커밋 경계 품질을 높인다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 추상 계획보다 동작하는 화면을 먼저 만들고 빠르게 피드백 받는다.
- **P4 속도-검증 균형**: Plan Mode로 사전 검증하고, 실행 후 스크린샷 증거 기반으로 수정한다.
- **P6 자동화 운영 기반**: 반복 요청 패턴과 오류 대응 패턴을 템플릿화해 팀 생산성을 높인다.
- **P7 Automation vs Augmentation**: 코드 수정 자동화는 LLM이 담당, 우선순위·제품 판단은 PM이 담당한다.
- **Level4 연결**: 구현은 PRD/TDD를 입력으로 수행하고, Eval 결과를 다음 iteration 입력으로 되돌리는 구조를 만든다.
- **P9 Words-to-Workflow**: “버튼 위치 바꿔줘”가 아니라 “사용자 완료율을 높이기 위해 CTA를 상단 고정”처럼 의도까지 포함한 문장이 더 좋은 코드를 만든다.

## 4) 실무 적용 시나리오
**상황: 사용자 테스트 전날 프로토타입 개선 스프린트**
1. 오전: 핵심 흐름 1개(가입→핵심행동 완료)만 먼저 동작시키는 빌드를 수행한다.
2. 점심 전: QA 스크린샷 10장을 수집해 우선순위 버그를 분류한다.
3. 오후: Plan Mode로 수정 계획을 점검하고 항목별로 순차 반영한다.
4. 저녁: 완료율/오류율 관점에서 최소 검증을 수행한다.
5. 결과를 이슈화해 4.4 GitHub 정리 단계로 넘긴다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 핵심 사용자 흐름 1개를 우선 구현하고, 로컬에서 end-to-end 동작 여부를 최소 기준으로 통과시킨다.
- **수~목 실행**: 스크린샷 피드백 루프를 하루 2회 운영해 UX/오류 이슈를 우선순위별로 정리하고 즉시 반영한다.
- **금요일 검증**: 회귀 테스트 체크리스트로 주요 경로를 재검증하고, 릴리스 후보 변경 목록을 고정한다.
- **Done 기준**: 치명 버그 0건, 핵심 시나리오 성공률 90% 이상, GitHub에 올릴 변경 요약과 검증 증거가 준비되어 있다.
- **리스크/완화**: 수정 범위가 커져 일정이 밀리면 기능 추가를 중단하고 안정화 전용 타임박스를 확보해 품질 하락을 방지한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 사용자 테스트 하루 전, 프로토타입의 완성도를 빠르게 끌어올려야 하는 운영 PM.

- **월요일**
  - Owner: 구현 담당, PM
  - Action: 핵심 사용자 흐름 1개를 end-to-end로 연결하는 최소 구현 수행
  - Artifact: 동작 영상(또는 GIF), 초기 버그 리스트
  - Check: 핵심 경로 1회 완주 가능 여부 확인
- **화요일**
  - Owner: PM, QA
  - Action: 스크린샷 기반 피드백 1차 수집(레이아웃/문구/클릭 동선)
  - Artifact: 스크린샷 이슈보드(번호/중요도/담당자)
  - Check: blocker·major·minor 분류 완료
- **수요일**
  - Owner: 구현 담당
  - Action: Plan Mode로 영향 범위 확인 후 blocker/major 순서로 수정
  - Artifact: 수정 로그, 회귀 점검 체크리스트
  - Check: blocker 0건, major 50% 이상 해소
- **목요일**
  - Owner: PM, QA
  - Action: 핵심 시나리오 회귀 테스트 + 오류 재현 테스트
  - Artifact: `regression-report.md`, 잔여 이슈 목록
  - Check: 핵심 시나리오 성공률 90% 이상
- **금요일**
  - Owner: PM, 개발 리드
  - Action: GitHub 인계용 변경 패키지 작성(반영/미반영/리스크)
  - Artifact: 4.4 전달 문서, 릴리스 후보 변경 목록
  - Check: 커밋 단위 분리가 가능한 상태인지 최종 확인

**리스크 트리거(발동 조건)**
- 수요일 저녁까지 blocker가 1건 이상 남거나, 핵심 시나리오 성공률이 80% 미만이면 릴리스 위험 단계로 진입한다.

**Rollback/Mitigation 흐름**
1. 기능 개선 작업을 즉시 중단하고 안정화 전용 브랜치로 전환한다.
2. 마지막 안정 커밋 기준으로 회귀한 뒤 blocker 수정만 우선 반영한다.
3. 금요일 테스트 범위를 "핵심 사용자 흐름" 중심으로 축소해 품질 보증을 확보한다.
4. 미반영 개선은 다음 주 백로그로 이동하고 기대효과를 재추정한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: dev server는 켜지는데 화면이 빈 페이지로 보임**
   - 해결: 콘솔 에러 확인 후 import 경로/런타임 에러를 우선 수정하고 브라우저 캐시를 강력 새로고침한다.
   - 확인: 첫 렌더 요소가 DOM에 존재하는지 점검한다.
2. **막힘: 수정 후 다른 화면이 연쇄적으로 깨짐(회귀)**
   - 해결: 큰 변경 전 Plan Mode로 영향 파일을 목록화하고, 수정 후 동일 목록을 역순으로 점검한다.
   - 확인: 회귀 체크리스트에서 실패 항목이 0인지 확인한다.
3. **막힘: 피드백이 많아 우선순위 없이 작업이 산만해짐**
   - 해결: blocker→major→minor 3단계로만 분류하고, 하루 2회 배치 처리한다.
   - 확인: 당일 종료 시 blocker 처리율 100%를 기준으로 삼는다.
4. **막힘: 4.4로 넘길 때 변경 범위가 불명확해 커밋이 비대해짐**
   - 해결: 반영/미반영/보류 항목을 분리한 전달 문서를 먼저 만들고 커밋 단위를 나눈다.
   - 확인: PR 설명에서 각 커밋의 목적을 1줄로 설명 가능해야 한다.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)

- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.

- [ ] 결과를 **Git/문서 링크/로그**로 추적 가능하게 남겼다.
