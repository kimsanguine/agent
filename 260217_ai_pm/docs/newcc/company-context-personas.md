## 1) 목적
- 페르소나를 설명 자료가 아니라 **기능 우선순위와 출시 판단의 운영 기준**으로 사용한다.
- 구매 결정권자/추천자/운영 영향자 관점을 분리해 전략 충돌을 줄인다.
- Junior/Senior/Leader PM이 동일한 사용자 언어로 PRD를 작성하도록 표준화한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문은 TaskFlow 핵심 사용자 3명(Sarah, Mike, Alex)을 기준으로 제품 결정을 내리도록 설계한다.
- Sarah(Enterprise Admin)는 보안·컴플라이언스·비용 통제를 최우선으로 보고 SSO, audit log, advanced permissions를 필수로 요구한다.
- Mike(IC Engineer)는 빠른 UX, keyboard shortcut, GitHub 연동, rich markdown, 충분한 task context를 중시한다.
- Alex(Team Lead)는 팀 가시성, blocked issue 조기 감지, workload 균형, sprint report, 댓글 요약을 요구한다.
- 영향력 관점에서 Sarah는 구매 결정권, Mike는 채택 추천권, Alex는 운영 성공 영향력이 크다.
- 기능 예시에서 Dark mode는 Mike 가치가 높고 구현비용이 낮아 빠른 출시 대상으로 평가된다.
- Q1 우선순위는 Mobile → SSO & Enterprise features → Activation 개선 → Dark mode로 제시된다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **핵심 페르소나 3명의 최신 상태를 갱신한다.**
   - 입력 예시: Sarah(보안/컴플라이언스), Mike(속도/개발 UX), Alex(팀 운영 가시성).
   - 실행 액션: 목표·고통·필수 기능을 최근 피드백으로 업데이트한다.
   - 기대 출력: 오래된 가정이 제거된 페르소나 카드.
2. **백로그 항목을 페르소나 영향도로 점수화한다.**
   - 입력 예시: SSO(5/1/3), Dark mode(1/5/2), Workload alert(2/2/5).
   - 실행 액션: 구매/채택/운영 영향 3축을 분리해서 기록한다.
   - 기대 출력: 충돌 기능의 우선순위를 수치로 비교 가능.
3. **분기 목표와 맞는 우선순위 규칙을 고정한다.**
   - 입력 예시: "이번 분기 Enterprise 확장이 최우선".
   - 실행 액션: Sarah 블로커 기능을 1차 우선으로 잠근다.
   - 기대 출력: 전략과 실행 백로그의 정합성 확보.
4. **채택 리스크 완화용 저비용 항목을 병행 배치한다.**
   - 입력 예시: Dark mode, shortcut 개선.
   - 실행 액션: Mike 관점의 빠른 체감 개선을 병렬로 넣는다.
   - 기대 출력: 구매 우선 전략에서도 사용성 이탈을 줄일 수 있음.
5. **팀 리드 운영 효율 항목을 별도 트랙으로 관리한다.**
   - 입력 예시: blocked issue 감지, workload 균형 알림.
   - 실행 액션: Alex KPI(팀 생산성/리포트 정확도)로 따로 측정한다.
   - 기대 출력: 운영 영향 항목의 우선순위 누락 방지.
6. **페르소나별 KPI를 분리 리포팅한다.**
   - 입력 예시: 수주율(Sarah), 활성률(Mike), 팀 만족도(Alex).
   - 실행 액션: 주간 리포트에 페르소나 탭을 분리해 기록한다.
   - 기대 출력: 하나의 평균 지표에 가려진 문제를 조기 발견.
7. **충돌 조정 로그를 남겨 재논쟁을 줄인다.**
   - 입력 예시: "SSO 우선, Dark mode는 병행 소규모 배치".
   - 실행 액션: 결정 근거와 보류 사유를 함께 기록한다.
   - 기대 출력: 다음 주에도 같은 기준으로 빠르게 의사결정 가능.

### Claude Code 핵심 기능 연결 (official 반영)
- **페르소나 관점 요구사항 구조화**: 구매/채택/운영 영향 관점을 분리해 충돌을 명확히 드러낸다.
- **Plan 모드 기반 우선순위 조정**: 분기 목표에 맞는 페르소나 가중치를 먼저 고정해 실행 혼선을 줄인다.
- **자연어 의사결정 로그화**: 선택/보류 사유를 즉시 문서화해 다음 주 재논쟁을 예방한다.
- **문맥 유지형 백로그 운영**: 기능 항목이 어떤 페르소나 문제를 해결하는지 계속 추적 가능하게 만든다.
- **저비용 병행 전략 설계 지원**: 핵심 목표를 해치지 않는 보완 기능(예: UX 개선)을 병렬 배치한다.
- **주간 리뷰 연계**: 페르소나별 KPI 리뷰를 통해 우선순위를 주기적으로 재보정한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션**: PM은 페르소나별 니즈를 한 문서에 병합하지 않고, 충돌 지점을 분리해 의사결정을 설계한다.
- **P7 Automation vs Augmentation**: 인터뷰 요약/태깅은 자동화하고, 우선순위 충돌 조정은 PM이 직접 수행한다.
- **FR-021 정합**: 페르소나-경쟁-가설을 5축(Product/Pricing/Positioning/Performance/People) 분석과 연결한다.
- **Junior-first + 확장 구조**: Junior는 페르소나 카드 템플릿부터 시작, Senior/Leader는 조직 단위 우선순위 정책으로 확장한다.
- **Level1~2 연계**: 페르소나 기반 PRD를 Notion/Linear/Slack으로 동일 포맷 게시해 의사결정 손실을 줄인다.

## 4) 실무 적용 시나리오
### 시나리오: Enterprise 기능과 Developer UX 기능 충돌 조정
1. 기능 후보를 Sarah/Mike/Alex 영향도로 점수화한다.
   - 예: SSO(5/1/3), Dark mode(1/5/2), Team workload alert(2/2/5)
2. 매출 영향(수주), 채택 영향(사용), 운영 영향(팀 효율)을 각각 분리해서 평가한다.
3. 분기 목표가 Enterprise 확장일 경우 Sarah 관련 블로커(SSO/권한/audit)를 1차 우선순위로 고정한다.
4. 단, Mike 채택 리스크 완화를 위해 저비용 고효율 항목(Dark mode, shortcut 개선)을 병행 배치한다.
5. 출시 후 페르소나별 KPI(수주율/활성률/팀 리드 만족도)를 따로 측정해 재조정한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: Sarah/Mike/Alex 페르소나 카드의 목표·고통·필수 기능을 최신화하고 인터뷰/고객 피드백을 재분류한다.
- **수~목 실행**: 이번 주 백로그를 페르소나 영향도 점수로 재정렬하고 충돌 항목의 우선순위 규칙을 문서로 확정한다.
- **금요일 검증**: 페르소나별 KPI 추적 가능 여부를 점검하고, 점수 대비 실제 효과를 리뷰해 다음 주 조정안을 확정한다.
- **Done 기준**: 기능-페르소나 매핑표와 충돌 조정 로그가 최신 상태이며, 각 페르소나별 성공 지표가 측정 가능한 데이터 소스로 연결되어 있다.
- **리스크/완화**: 특정 페르소나 요구가 과대표집되면 최근 4주 데이터 기준 샘플 수를 공개하고 가중치 규칙으로 균형을 맞춘다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: Enterprise 요구와 Developer UX 요구가 동시에 강하게 들어와 우선순위 충돌이 발생한 주간.

- **월요일**
  - Owner: PM
  - Action: Sarah/Mike/Alex 카드 최신화(목표/고통/필수 기능)
  - Artifact: 페르소나 카드 v2, 데이터 근거 메모
  - Check: 최근 4주 피드백 반영 비율 확인
- **화요일**
  - Owner: PM, 데이터 담당
  - Action: 백로그 항목 영향도 점수화(구매/채택/운영)
  - Artifact: 기능-페르소나 매트릭스, 점수표
  - Check: 점수 산정 근거가 문서화되어 재현 가능한지 확인
- **수요일**
  - Owner: PM, 리더십
  - Action: 분기 목표 기준으로 우선순위 1차 확정(예: Enterprise 우선)
  - Artifact: 우선순위 결정 로그, 보류 항목 리스트
  - Check: 전략 목표와 상충하는 항목이 없는지 점검
- **목요일**
  - Owner: PM, 개발 리드
  - Action: 채택 리스크 완화용 저비용 개선 1~2건 병행 배치
  - Artifact: 병행 실행 티켓, 기대효과 메모
  - Check: 핵심 목표를 침해하지 않고 병행 가능한지 확인
- **금요일**
  - Owner: PM, 운영 담당
  - Action: 페르소나별 KPI 리포트 작성 및 차주 조정안 확정
  - Artifact: 페르소나 KPI 주간 리포트, 차주 계획
  - Check: 수주율/활성률/운영 만족도 3축 모두 보고 완료

**리스크 트리거(발동 조건)**
- 특정 페르소나 관련 요청이 전체 요청의 70% 이상을 차지하고, 나머지 페르소나 KPI가 2주 연속 하락하면 편향 위험으로 판단한다.

**Rollback/Mitigation 흐름**
1. 과대표집된 페르소나 요구를 일시 보류하고 균형 점검 세션을 연다.
2. 최근 4주 샘플 수 기준으로 가중치를 재계산한다.
3. 하락한 페르소나 KPI를 복구하는 저비용 항목을 즉시 추가한다.
4. 다음 주 우선순위는 "핵심 목표 + 균형 복구" 2트랙으로 재편한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 페르소나 설명은 있는데 실제 백로그와 연결이 안 됨**
   - 해결: 모든 백로그 항목에 주 대상 페르소나 태그 1개를 필수로 붙인다.
   - 확인: 태그 없는 항목은 계획 리뷰에서 제외한다.
2. **막힘: 영향도 점수 산정이 주관적이라 신뢰가 낮음**
   - 해결: 점수 근거를 "최근 데이터/고객 인터뷰/매출 영향" 중 하나로 강제한다.
   - 확인: 점수표에 근거 칼럼이 비어 있지 않은지 점검한다.
3. **막힘: Enterprise 우선 시 개발자 채택률이 급락**
   - 해결: 저비용 고효율 UX 개선(예: shortcut, dark mode)을 병행 편성한다.
   - 확인: Mike 관련 활성 KPI 하락 폭이 완화되는지 주간 추적한다.
4. **막힘: 결정 사유 누락으로 다음 주 우선순위가 다시 뒤집힘**
   - 해결: 결정 로그에 "왜 선택/왜 보류"를 한 줄씩 의무 기록한다.
   - 확인: 다음 회의에서 로그만으로 결정 재현이 가능해야 한다.
