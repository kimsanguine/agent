## 1) 목적
- AI를 전략 대행이 아닌 사고 파트너로 활용해, 선택과 포기 기준이 있는 제품 전략을 설계한다.
- Diagnosis→Guiding Policy→Coherent Actions 흐름으로 전략 문서와 실행 계획을 함께 완성한다.

## 2) 원문 핵심 요약 (ccforpms)
- 모듈 시작 명령은 `/start-2-3`, 목표는 전략 수립부터 문서화·발표 자료 전환까지 전체 흐름을 익히는 것이다.
- 전략은 목표/기능 나열이 아니라 “어디에 집중하고 무엇을 하지 않을지”를 정하는 선택과 트레이드오프다.
- 핵심 프레임워크로 Rumelt’s Strategy Kernel(`Diagnosis` → `Guiding Policy` → `Coherent Actions`)을 사용한다.
- AI 활용 지점이 구체적이다: 병렬 경쟁 리서치(WebSearch), 구조화, 반론 제시(`devil’s advocate`), 문서/슬라이드 합성.
- 사람(PM)의 책임 범위도 명확하다: 전략적 선택, 제약 반영, 반론 수용 여부의 최종 판단.
- 경쟁 조사 실습 대상 예시가 있다: Notion/Linear/Asana의 AI 기능, 가격, 포지셔닝, 화이트스페이스.
- 선택 축 5개를 명시적으로 결정한다: Focus/Breadth, Competitive Response, Business Model, Product Scope, Risk Tolerance.
- 산출물은 H1 전략 문서와 임원 발표 자료이며, 필요 시 `pptx skill`과 `python-pptx` 설치 대응이 안내된다.
- 전략 문서는 고정물이 아니라 분기별 가정 검증 및 업데이트 대상이라는 운영 원칙이 포함된다.

### Claude Code 핵심 기능 연결 (official 반영)
- **광범위 리서치 오케스트레이션(Overview)**: 경쟁/고객/내부 제약 정보를 한 흐름으로 연결해 전략 문서의 공백을 줄인다.
- **전략 프롬프트 템플릿화(Tutorial: Getting Started)**: `Diagnosis → Guiding Policy → Coherent Actions` 순서로 지시해 전략 초안 구조를 안정화한다.
- **심층 판단 보조(Tutorial: Claude Opus 4.6)**: 선택 축(Focus, Scope, Risk)별 장단점과 포기 비용을 동시에 비교하기 좋다.
- **브라우징 기반 시장 스캔(Tutorial: Claude for Chrome)**: 경쟁사 가격/메시지/포지셔닝 변화를 빠르게 수집해 분기 전략 업데이트 속도를 높인다.
- **협업형 반론 검토(Tutorial: Claude Cowork)**: 반론 역할을 분리해 취약 가정을 조기에 발견하고 정책 대안을 정교화한다.
- **규제 산업형 의사결정 로그(Tutorial: Financial Services)**: 전략 가정, 승인 조건, 리스크 대응을 감사 가능한 포맷으로 남긴다.
- **실행 전환 중심 워크플로(Claude Code)**: 전략 문장을 그대로 백로그·오너·마일스톤으로 연결해 실행 단절을 줄인다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 전략 질문과 제약 정의
- 입력 아티팩트: `annual-goals.md`, `budget-cap-2026.xlsx`, `org-capacity-q1q2.md`
- 프롬프트 조각: "이번 반기 전략 질문을 1문장으로 고정하고, 반드시 반영할 제약(인력/예산/시간)을 표로 정리해줘."
- 기대 출력: `strategy-question-brief.md`
- 검증 체크포인트: 질문 문장이 기능 목록이 아니라 선택 문제(집중/포기)로 표현된다.

2) 경쟁 리서치 병렬 수집
- 입력 아티팩트: 공개 릴리즈 노트, 가격 페이지, 제품 영상 링크
- 프롬프트 조각: "Notion/Linear/Asana의 AI 기능·가격·포지셔닝을 비교하고, 우리에게 의미 있는 화이트스페이스 3개를 제시해줘."
- 기대 출력: `competitive-map.md`
- 검증 체크포인트: 단순 기능 나열이 아니라 고객 세그먼트 관점의 틈새가 명시된다.

3) Diagnosis 작성
- 입력 아티팩트: 시장/고객/내부역량 데이터, 경쟁 비교표
- 프롬프트 조각: "Rumelt Diagnosis를 5문장 내로 작성하되, 무엇이 핵심 장애인지 1순위부터 제시해줘."
- 기대 출력: `strategy-kernel-diagnosis.md`
- 검증 체크포인트: 장애 요인이 모호한 외부 탓이 아니라 통제 가능한 항목으로 표현된다.

4) Guiding Policy 설계
- 입력 아티팩트: Diagnosis 초안, 리소스 제약표
- 프롬프트 조각: "Diagnosis를 해결하기 위한 Guiding Policy를 2안 제시하고, 선택/포기 기준을 함께 써줘."
- 기대 출력: 정책안 A/B 비교표
- 검증 체크포인트: 정책마다 '무엇을 하지 않을지'가 명시된다.

5) Coherent Actions로 분기 실행 전환
- 입력 아티팩트: 정책안 선택 결과, 팀 로드맵
- 프롬프트 조각: "Q1/Q2 실행 액션을 오너/마감/선행조건까지 포함해 작성해줘."
- 기대 출력: `q1-q2-actions.md`
- 검증 체크포인트: 모든 액션에 책임자와 일정이 존재한다.

6) Devil’s Advocate 반론 테스트
- 입력 아티팩트: 전략 초안 전체
- 프롬프트 조각: "이 전략이 실패할 수 있는 시나리오 5개와 조기 경보 지표를 제시해줘."
- 기대 출력: `strategy-risks-counterarguments.md`
- 검증 체크포인트: 각 리스크에 선제 대응 액션이 1개 이상 연결된다.

7) 경영진 커뮤니케이션 패키지 완성
- 입력 아티팩트: Kernel 문서, 리스크 문서, 실행 계획
- 프롬프트 조각: "임원 보고용 10슬라이드 내러티브를 만들고, 의사결정 요청사항 3개를 명확히 적어줘."
- 기대 출력: `exec-strategy-deck-outline.md`
- 검증 체크포인트: 발표 자료와 전략 문서의 숫자/우선순위가 일치한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션 + Level 2**: AI가 조사/대안을 보조하고 최종 전략 선택은 PM이 책임지는 구조를 고정한다.
- **P4 속도-검증 균형 + Level 1~2**: 빠른 리서치 후 devil’s advocate 검증으로 취약 가정을 조기에 드러낸다.
- **P5 한국 실무 적합성 + Level 2**: 글로벌 프레임워크(Rumelt)를 한국 PM의 자원 제약·조직 맥락에 맞게 재해석한다.
- **P8 Connected Hub + Level 1~3**: 전략 문서를 발표/상태보고/실험 백로그로 이어지게 설계해 실행 단절을 줄인다.
- **P9 Words-to-Workflow + Level 4**: 전략 문장과 선택 축이 곧바로 분기 액션/지표/리스크 관리 흐름으로 변환된다.

## 4) 실무 적용 시나리오
PM 리더가 상반기 전략을 재정의해야 하는 상황이다. 경쟁사 리서치를 병렬로 수행해 `Diagnosis`를 만들고, 5개 선택 축에서 팀 역량과 예산 제약을 반영해 우선순위를 확정한다. 이후 devil’s advocate 검증으로 약한 가정을 제거하고, Q1/Q2 실행 계획과 성공 지표를 경영진 보고 자료까지 연결한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `/start-2-3`로 착수해 경쟁사 3곳 이상을 병렬 리서치하고 Rumelt의 Diagnosis 초안을 데이터/사례 기반으로 작성한다.
- **수~목 실행**: Guiding Policy와 Coherent Actions를 연결하면서 5개 선택 축(Focus, Response, Business Model, Scope, Risk Tolerance)의 선택/포기 근거를 명문화한다.
- **금요일 검증**: devil’s advocate 리뷰 세션을 열어 취약 가정을 반박 테스트하고, Q1/Q2 실행안과 지표를 임원 공유본 1안으로 확정한다.
- **Done 기준**: 전략 문서에 선택/비선택 기준, 핵심 가정, 분기별 실행 액션, 성공 지표, 업데이트 주기가 모두 들어가고 발표 자료와 메시지가 일치한다.
- **리스크/완화**: 경쟁사 기능 나열로 흐를 위험은 “우리의 선택 기준” 섹션을 강제해 통제하고, 조직 저항은 단계적 적용안(파일럿 팀→전사)으로 완화한다.

- 운영 팁 1: 월요일에 전략 질문을 못 고정하면 그 주는 조사만 많고 선택은 남지 않는다.
- 운영 팁 2: 수요일에는 "우리가 안 할 것"을 반드시 문서에 먼저 잠가야 목요일 실행안이 현실화된다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 전략이 비전 문구로만 남고 실행이 안 됨
- 근본 원인: Coherent Actions가 오너/마감 없이 선언형 문장으로 작성됨
- 해결 방법: 모든 액션을 "오너-일정-선행조건" 3요소로 강제 변환

2) 블로커: 경쟁사 자료는 많은데 우리 선택이 모호
- 근본 원인: 비교 기준(타깃, 가격, 차별축)이 고정되지 않음
- 해결 방법: 리서치 시작 전에 5축 비교 프레임을 먼저 확정

3) 블로커: 반론 검토가 방어적 토론으로 끝남
- 근본 원인: 반론을 실패 시나리오가 아닌 개인 의견으로 취급
- 해결 방법: 반론마다 조기 경보 지표와 대응 플랜을 붙여 운영 이슈로 전환

4) 블로커: 임원 승인 후에도 팀이 움직이지 않음
- 근본 원인: 전략 문서와 분기 백로그 연결이 끊겨 있음
- 해결 방법: 승인 직후 24시간 내 백로그 티켓으로 분해하고 책임자 배정
