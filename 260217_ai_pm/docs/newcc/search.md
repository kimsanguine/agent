## 1) 목적
- 검색을 “찾기 기능”이 아니라 **근거 수집과 의사결정 검증 프로세스**로 표준화한다.
- 문서/코드/이슈 탐색 시간을 줄이고, 근거 없는 판단을 차단한다.
- 추후 `5.2`와 통합 가능한 최소 독립 범위를 유지한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 Search는 Pagefind 기반 client-side 검색으로 코스 페이지를 빠르게 탐색하는 용도다.
- 검색 정확도를 높이기 위해 phrase 검색, 복수 키워드 조합, 결과 섹션 확인을 권장한다.
- 원하는 결과가 없으면 키워드 변경(동의어 재검색) 후 구조 탐색을 병행하는 방식을 제시한다.
- 검색 실패 시 Company Context, Getting Started 등 상위 허브 페이지로 역추적하도록 안내한다.
- 핵심 메시지는 “검색창 1회 입력”이 아니라 “질문 분해 + 키워드 반복 조정 + 섹션 단위 확인”이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview의 핵심인 코드/문서 동시 탐색 방식을 반영하면, Search를 단순 키워드 입력이 아니라 **근거 수집 파이프라인**으로 운영할 수 있다.
- PM은 질문을 분해한 뒤 파일명 탐색·내용 탐색·맥락 문서 탐색을 분리해 실행하면 오탐과 미탐을 동시에 줄일 수 있다.
- Tutorials Hub의 시작 가이드처럼 검색 결과는 즉시 산출물(PRD 가설, 백로그, 리스크 로그)로 연결되어야 실무 가치가 생긴다.
- Opus 4.6 튜토리얼 관점에서 복합 탐색은 단계적 계획(질문 정의→1차 탐색→재탐색→검증)으로 운영할 때 품질이 안정된다.
- Chrome 튜토리얼과 결합하면 외부 웹 근거를 수집하고 Claude Code 문서 탐색 결과와 교차 검증하는 구조를 만들 수 있다.
- 금융 서비스 튜토리얼의 신뢰성 관점을 적용해, 검색 결과마다 출처·시간·근거 등급을 남겨 의사결정 감사성을 확보한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일, 해결하려는 문제를 `문제/원인/영향` 3문장으로 분해하고 각 문장마다 핵심 키워드 3개를 작성한다.
2. 화요일 오전, 문서 레이어(PRD/회고/전략)에서 1차 검색을 수행해 근거 후보를 10건 수집한다.
3. 화요일 오후, 데이터 레이어(대시보드/코호트/퍼널)에서 같은 질문을 재검색해 수치 근거 5건 이상을 확보한다.
4. 수요일, 운영 레이어(이슈/배포/장애 로그)에서 시점 기반 검색을 수행해 최근 변경 이벤트와의 상관관계를 확인한다.
5. 수요일 오후, 1차 결과를 `사실/가설/미확인`으로 분류하고 미확인 항목만 별도로 재검색 큐에 넣는다.
6. 목요일, 동의어/기간/세그먼트를 바꾼 2차 검색을 돌려 미탐지를 줄이고 근거 품질 점수를 재산정한다.
7. 금요일, 최종 근거를 PRD 가설 섹션에 반영하고 출처 경로/라인을 함께 명시해 검증 가능성을 확보한다.
8. 주간 회고에서 오탐 원인 상위 3개를 기록하고 다음 주 검색 템플릿을 업데이트한다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 Context-rich**: 검색 요청 전 문제 정의를 분해해 키워드 세트를 설계한다.
- **P4 속도-검증 균형**: 빠른 검색 후, 반드시 파일 경로/라인/출처를 남겨 검증 가능성을 확보한다.
- **P9 Words-to-Workflow**: 자연어 질문을 검색 쿼리 템플릿으로 표준화해 반복 사용한다.
- **FR-013 정합**: Global/Project 컨텍스트 라우팅 규칙에 맞춰 검색 레이어(문서/코드/운영 로그)를 분리한다.
- **Automation vs Augmentation**: 대량 검색/초기 분류는 자동화, 해석과 의사결정은 PM이 수행한다.

#### 월~금 상세 실행표
| 요일 | 실행 항목 | 구체 동작 | 결과물 | 검증 기준 | 복구 절차 |
|---|---|---|---|---|---|
| 월 | 질문 분해 | 문제/원인/영향 3축 작성 | 질문 분해 시트 | 질문 3개 완성 | 모호 문장 제거 후 재작성 |
| 화 | 1차 검색(문서+데이터) | PRD/회고 + 퍼널 검색 | 근거 로그 15건+ | 출처 명시율 100% | 출처 없는 항목 폐기 |
| 수 | 2차 검색(운영 로그) | 배포/이슈/장애 시점 매칭 | 상관관계 노트 | 시점 불일치 0건 | 기간 재설정 후 재검색 |
| 목 | 재검색/정제 | 동의어·세그먼트 변경 | 사실/가설 분류표 | 미확인 축소율 | 미확인 큐 재우선순위화 |
| 금 | PRD 반영/리뷰 | 가설 섹션 업데이트 | 근거 첨부 PRD | 리뷰 통과율 | 반려 사유 기반 재수정 |

#### 검증/복구 흐름
1. 근거 로그에서 출처 경로 또는 라인이 누락된 항목은 즉시 `검증 불가`로 분류한다.
2. `검증 불가` 항목은 의사결정 근거에서 제외하고 재검색 큐로 이동한다.
3. 재검색 2회 후에도 근거가 확보되지 않으면 “가설 유지”가 아닌 “미확인 보류”로 처리한다.
4. 금요일 PRD 리뷰에서 미확인 보류가 30%를 넘으면 가설 범위를 축소해 다음 주 실험량을 줄인다.
5. 반복 오탐 쿼리는 템플릿에서 제거하고 대체 쿼리를 등록해 같은 실패를 방지한다.

### 시나리오: 온보딩 전환 저하 원인 탐색
1. 질문을 3개로 분해한다.
   - 어디서 이탈하는가?
   - 누구에게서 크게 발생하는가?
   - 최근 변경과 상관관계가 있는가?
2. 검색 레이어를 분리한다.
   - 문서: PRD/실험노트/회고
   - 데이터: 퍼널 리포트/코호트
   - 운영: 이슈/배포 로그
3. 키워드 조합 템플릿으로 1차 검색 후, 근거를 경로와 함께 수집한다.
4. 근거 부족 항목은 재검색(동의어/기간/세그먼트 변경)한다.
5. 최종적으로 “근거 있음/가설”을 분리해 PRD 가설 섹션에 반영한다.

### 이번 주 완료형 실행 플랜
- **월~화:** 온보딩 전환 이슈를 문제/원인/영향 3축으로 분해하고 검색 쿼리 템플릿 10개를 만든다.
- **수~목:** 문서·데이터·운영 3개 레이어에서 1차 검색을 수행해 근거 로그를 최소 20건 적재한다.
- **금 검증:** 사실/가설/미확인 분류 정확도를 리뷰어 1인과 교차검토하고, 오탐 원인 상위 3개를 정리한다.
- **Done 기준:** 쿼리 라이브러리 1종, 근거 로그 20건+, 재검색 규칙표 1종, PRD 반영 항목 3건 이상이 확보된다.
- **리스크·완화:** 키워드 편향으로 미탐지가 발생할 수 있으므로 동의어 사전과 기간/세그먼트 재검색 규칙을 강제 적용한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 질문이 넓어 검색 결과가 과다함**
  - 증상: 수백 건 결과가 나오지만 결정에 쓸 근거가 없다.
  - 해결: 질문을 단일 결정 단위(예: “가입 1일차 이탈 원인”)로 축소하고 기간/세그먼트를 고정한다.
- **막힘 2: 키워드 편향으로 중요한 근거를 놓침**
  - 증상: 특정 표현만 찾고 동의어를 놓친다.
  - 해결: 검색 시작 전에 동의어 사전을 5개 이상 준비하고, 2차 검색 때 반드시 교차 적용한다.
- **막힘 3: 사실과 가설이 섞여 PRD 품질 저하**
  - 증상: 근거 없는 문장이 사실처럼 작성된다.
  - 해결: 모든 항목에 `F(사실)/H(가설)/U(미확인)` 라벨을 붙이고, F만 의사결정 근거로 사용한다.
- **막힘 4: 검색은 잘했지만 실행 액션이 없음**
  - 증상: 인사이트 문서가 백로그와 연결되지 않는다.
  - 해결: 금요일 리뷰 시 근거당 액션 1개를 필수 생성하고 owner/due를 즉시 할당한다.

## 통합/삭제 판단 기준
- **통합 권고 조건**
  - `5.2-level2-claude-code-hub.md` 내 탐색 절차와 중복률 ≥ 75%
  - 검색 전용 템플릿 사용률이 4주 평균 20% 미만
  - 별도 문서 없이도 팀 평균 검색 소요시간 변동이 ±10% 이내
- **독립 유지 조건**
  - 신규 팀원 온보딩에서 검색 템플릿 사용 시 정확도 향상(예: 오탐률 20%↓)이 확인됨
  - 비개발 직군이 검색 문서를 독립적으로 참조할 필요가 큼
- **삭제/흡수 의사결정 방식**
  - 월간 운영 리뷰 2회 연속 통합 권고 조건 충족 시 본 문서를 `5.2` 부록으로 흡수
