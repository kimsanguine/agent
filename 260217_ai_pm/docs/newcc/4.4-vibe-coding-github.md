## 1) 목적
- 로컬에서 만든 결과를 GitHub에 안전하게 저장해 **버전 추적·협업·배포 준비**를 동시에 완료한다.
- PM 관점에서 변경 의도와 검증 결과가 남는 협업 습관을 정착시킨다.
- 4.5 배포 단계로 넘어가기 전, 코드 자산의 신뢰 가능한 기준점을 만든다.

## 2) 원문 핵심 요약 (ccforpms)
1. 레슨 시작은 `/start-4-4`이며, 목표는 약 15분 내 private GitHub repository 업로드 완료다.
2. 버전 관리 핵심은 변경 이력 보존, 복구 가능성, 비교 가능성 확보에 있다.
3. Git은 로컬 버전관리 도구, GitHub는 원격 저장/공유 플랫폼으로 역할이 다르다.
4. 실무 핵심 용어는 repository/commit/push/pull이며, 특히 commit 품질이 협업 효율을 좌우한다.
5. Vercel 배포는 GitHub 연동을 전제로 하므로, 이 단계는 단순 백업이 아니라 배포 파이프라인의 시작점이다.
6. 인증은 `gh` CLI로 진행하며, 브라우저 device code 방식 로그인 절차를 거친다.
7. 오류 패턴은 `gh: command not found`, authentication failed, repo already exists, permission denied가 대표적이다.
8. 대부분 인증/권한 문제는 `gh auth login` 재실행과 계정 확인으로 해결 가능하다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **저장소 전략을 먼저 정한다.**
   - 입력 예시: "private repo로 시작, main은 배포 기준 브랜치".
   - 실행 액션: 브랜치 전략(main/develop/feature)을 팀 공통으로 고정한다.
   - 기대 출력: 병합 시 충돌·권한 혼선이 줄어든 운영 기준.
2. **`/start-4-4`로 GitHub 업로드 플로우를 시작한다.**
   - 입력 예시: "오늘 목표는 핵심 기능 변경분 원격 백업 완료".
   - 실행 액션: 레슨 가이드에 따라 인증/저장소 생성 순서를 진행한다.
   - 기대 출력: 원격 저장소 준비 완료 상태.
3. **`gh auth login`으로 계정 인증을 마친다.**
   - 입력 예시: 브라우저 device code 인증.
   - 실행 액션: 조직/개인 계정 중 실제 배포 대상 계정을 선택한다.
   - 기대 출력: push/pull/PR 작업이 가능한 인증 상태.
4. **작은 단위 커밋으로 변경 의도를 남긴다.**
   - 입력 예시: "온보딩 CTA 위치 수정", "오류 메시지 명확화".
   - 실행 액션: 기능 단위로 2~4개 커밋으로 분리한다.
   - 기대 출력: 되돌리기 쉬운 이력과 리뷰 가능한 변경 경계.
5. **PR 본문에 테스트 증거를 포함한다.**
   - 입력 예시: "배경/변경점/테스트/리스크" 템플릿.
   - 실행 액션: 스크린샷, 재현 절차, 영향 범위를 함께 기록한다.
   - 기대 출력: 리뷰 왕복 횟수 감소, 승인 리드타임 단축.
6. **오류 패턴을 즉시 분류해 복구한다.**
   - 입력 예시: `gh: command not found`, auth failed, permission denied.
   - 실행 액션: 설치 확인→재인증→계정 권한 점검 순으로 처리한다.
   - 기대 출력: 인증/권한 이슈의 평균 복구시간 단축.
7. **4.5 배포용 기준 커밋을 태깅한다.**
   - 입력 예시: "staging-ready" 혹은 릴리스 노트 링크.
   - 실행 액션: 배포 후보 커밋을 문서와 함께 고정한다.
   - 기대 출력: 누구나 동일 커밋에서 배포를 재현 가능한 상태.

### Claude Code 핵심 기능 연결 (official 반영)
- **Git 상태 가시화**: 변경 파일, diff, 커밋 이력을 빠르게 확인해 PM도 변경 범위를 정확히 파악할 수 있다.
- **작은 단위 커밋 운영 지원**: 기능 단위로 변경을 분리해 리뷰·롤백·회고가 쉬운 이력을 만든다.
- **PR 생성/정리 자동화**: 배경·변경·테스트·리스크를 템플릿화해 리뷰 품질을 일관되게 유지한다.
- **CLI 기반 GitHub 연동**: 인증/저장소/브랜치 작업을 동일 흐름에서 처리해 운영 마찰을 줄인다.
- **배포 전 기준점 고정**: 4.5에 넘길 커밋 기준을 명시해 배포 재현성과 책임 경계를 명확히 한다.
- **문맥 보존형 협업**: 계획(4.2)·구현(4.3)에서 나온 의도와 증거를 PR에 연결해 의사결정 추적성을 높인다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **Dennis 원칙 적용**: “**words ARE the code**”를 Git 흐름에 연결하면, 커밋 메시지/PR 설명은 단순 기록이 아니라 실행 의도 명세다.
- **P4 속도-검증 균형**: 작은 단위 커밋과 명확한 PR 설명으로 빠른 병합과 안전한 검증을 동시에 달성한다.
- **P6 자동화 운영 기반**: 템플릿화된 커밋/PR 규칙으로 리뷰 시간을 줄이고 릴리스 예측성을 높인다.
- **P8 Connected Hub**: Claude Code에서 생성한 결과를 GitHub로 연결해 배포·협업·회고까지 한 허브로 통합한다.
- **Level4 연결 명시**: `/agent`가 만든 PRD→TDD→Eval→Prototype 산출물을 브랜치/커밋 단위로 분리 저장해 실험 이력을 추적한다.
- **P9 Words-to-Workflow**: “무엇을 바꿨는가”보다 “왜 바꿨는가”를 남겨 다음 iteration의 품질을 높인다.

## 4) 실무 적용 시나리오
**상황: 프로토타입 리뷰 직후 핫픽스 + 기능 개선 동시 진행**
1. 핫픽스와 개선 작업을 서로 다른 브랜치로 분리한다.
2. 각 변경을 작은 커밋으로 나누고, 사용자 영향 중심 메시지를 남긴다.
3. PR 템플릿에 테스트 결과와 리스크를 명시해 리뷰 시간을 단축한다.
4. 리뷰 반영 후 main 브랜치에 병합하고 릴리스 노트를 업데이트한다.
5. 이후 4.5 단계에서 GitHub 연동 배포를 진행해 변경을 즉시 공개 검증한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 저장소 생성, 인증 점검, 브랜치 전략(main/develop/feature)과 커밋 메시지 규칙을 팀 공통으로 확정한다.
- **수~목 실행**: 핵심 변경을 작은 단위로 커밋하고 PR 템플릿(배경/변경/테스트/리스크)을 적용해 리뷰 속도를 높인다.
- **금요일 검증**: 병합 전 최종 PR 점검으로 충돌/권한/누락 테스트를 확인하고 배포 대상 커밋을 태깅한다.
- **Done 기준**: 이번 주 작업이 재현 가능한 커밋 이력으로 정리되고, 리뷰 로그와 병합 근거가 남아 4.5 배포를 바로 진행할 수 있다.
- **리스크/완화**: 커밋이 비대해지면 기능 단위로 재분할하고, 금요일 오전부터는 신규 기능 커밋을 금지해 병합 리스크를 낮춘다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 프로토타입 리뷰 후 핫픽스와 기능 개선이 동시에 들어온 주간, PM이 GitHub 이력을 안정적으로 운영해야 한다.

- **월요일**
  - Owner: PM, 개발 리드
  - Action: 브랜치 전략 확정(main/develop/hotfix), 커밋 메시지 규칙 공지
  - Artifact: 브랜치 운영 문서, 커밋 규칙 가이드
  - Check: 팀원 전원이 동일 규칙으로 첫 커밋 생성 가능
- **화요일**
  - Owner: 개발자, PM 리뷰어
  - Action: 핫픽스와 개선 작업을 브랜치 분리, 기능 단위 커밋 작성
  - Artifact: 기능별 커밋 세트, 변경 요약 메모
  - Check: 커밋당 목적 1개 원칙 준수 여부
- **수요일**
  - Owner: PM
  - Action: PR 템플릿 강제 적용(배경/변경/테스트/리스크), 리뷰 라운드 운영
  - Artifact: PR 2~3건, 테스트 증거(스크린샷/재현 절차)
  - Check: 리뷰어가 1회 읽기로 영향 범위를 파악 가능한지 확인
- **목요일**
  - Owner: 개발자, PM
  - Action: 리뷰 반영 후 재검증, 병합 후보 커밋 태깅
  - Artifact: `merge-ready` 라벨, 릴리스 후보 목록
  - Check: 충돌/권한/누락 테스트 항목 0건
- **금요일**
  - Owner: PM, 팀 리드
  - Action: 배포 대상 커밋 확정, 4.5 인계 문서 작성
  - Artifact: 배포 기준 SHA, 인계 체크리스트
  - Check: 동일 커밋으로 재배포 가능한지 드라이런 통과

**리스크 트리거(발동 조건)**
- 목요일 오후까지 PR 미승인 건이 3건 이상 누적되거나, 충돌 해결 시간이 2시간을 넘으면 병합 지연 위험으로 판단한다.

**Rollback/Mitigation 흐름**
1. 신규 기능 브랜치 병합을 중단하고 hotfix 우선 정책으로 전환한다.
2. 배포 필수 변경만 선별해 임시 릴리스 브랜치로 재구성한다.
3. 미승인 기능은 다음 주로 이월하고, 충돌 원인을 브랜치 규칙에 반영한다.
4. 금요일에는 배포 기준 커밋 1개만 확정해 불확실성을 제거한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: `gh` 명령이 동작하지 않아 인증부터 막힘**
   - 해결: CLI 설치 상태를 확인하고 `gh auth login`을 재실행해 올바른 계정을 선택한다.
   - 확인: 저장소 목록 조회가 가능한지 테스트한다.
2. **막힘: 커밋이 너무 커서 리뷰어가 핵심 변경을 파악 못함**
   - 해결: 기능 단위로 커밋을 재분할하고 커밋 메시지에 사용자 영향도를 명시한다.
   - 확인: 커밋 하나당 설명 1~2문장으로 목적 설명이 가능해야 한다.
3. **막힘: PR에 테스트 근거가 없어 승인 지연**
   - 해결: PR 본문에 재현 절차, 기대 결과, 스크린샷을 필수 항목으로 넣는다.
   - 확인: 리뷰어 질문이 "어떻게 검증했는가"에서 "어디를 개선할까"로 이동하면 통과.
4. **막힘: 권한 이슈로 원격 push가 실패**
   - 해결: 조직 권한과 원격 URL 계정을 교차 점검하고, 필요 시 저장소 권한 요청 절차를 즉시 수행한다.
   - 확인: 금요일 이전에 최소 1회 push/pull 드라이런 완료.
