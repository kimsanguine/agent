# 5.2 Level 2 - Claude Code Hub

## 1) 목적
- Claude Code를 팀의 실행 허브로 표준화해, 문서-코드-운영 작업을 같은 프로토콜로 처리한다.
- Level1 산출물을 재사용 가능한 운영 자산(CLAUDE.md, Slash Commands, 템플릿)으로 승격한다.
- 사람 의존적인 반복 대응을 줄이고, 팀 단위 재현성을 확보한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 1.7은 작업 성격에 따라 Edit/Auto-Accept/Plan 모드를 선택하는 운영법을 제시한다.
- 원문 1.4는 병렬 agent를 활용해 대량 리서치/문서 생성을 fan-out 후 synthesis하는 패턴을 권장한다.
- 원문 1.5는 반복 역할을 custom sub-agent로 고정해 리뷰 품질과 속도를 높인다.
- 원문 1.6/Project memory 맥락은 CLAUDE.md 기반 지속 컨텍스트 유지의 중요성을 강조한다.
- 원문 2.x는 PRD/분석/전략을 파일 산출물로 남기는 실행형 패턴을 반복한다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview에서 강조하는 운영 핵심은 **프로젝트 메모리(CLAUDE.md) + 모드 선택 + 파일 산출물 추적**의 조합이다.
- PM은 Level2 허브에서 Plan 모드를 복합 작업 기본값으로 두고, 단순 수정은 Edit 중심으로 분리하면 품질과 속도를 함께 관리할 수 있다.
- Tutorials Hub 흐름처럼 명령어는 “질문 단축키”가 아니라 재현 가능한 실행 프로토콜로 설계해야 팀 교대 시에도 성과가 유지된다.
- Opus 4.6 튜토리얼 관점에서 병렬 작업 후 synthesis 단계를 명시하면 중복 산출물과 충돌을 크게 줄일 수 있다.
- Cowork/Chrome 튜토리얼과 연계해 비개발 입력(리서치, 파일 정리)을 허브로 수렴시키면 전사 협업 접점을 넓힐 수 있다.
- 금융 서비스 튜토리얼의 규제형 운영 기준을 반영해, 허브 문서에는 승인자/근거/감사 로그 필드를 기본 포함한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, 프로젝트 루트에 CLAUDE.md 운영 원칙(목표, 금지사항, 품질 게이트, 보고 형식)을 최신화한다.
2. 월요일 오후, `tasks/`, `research/`, `reports/`, `templates/` 폴더를 만들고 산출물 저장 위치를 팀 규칙으로 고정한다.
3. 화요일, `/today`, `/prd`, `/status`, `/agent` 4개 명령을 최소 입력 예시와 함께 테스트해 실패 케이스를 기록한다.
4. 수요일, Researcher/Reviewer/Synthesizer 역할을 분리해 같은 주제를 병렬 처리하고 단일 통합본으로 합친다.
5. 수요일 오후, 복합 작업은 Plan 모드로 강제하고 단순 수정은 Edit 모드로 분리해 모드 오남용을 줄인다.
6. 목요일, 산출물 품질 게이트(목표/근거/지표/다음 액션) 누락 항목을 체크해 미통과 문서를 재작성한다.
7. 금요일 오전, 처리시간/재작업률/검수 누락률을 집계해 Hub 운영 baseline을 수치화한다.
8. 금요일 오후, 반복 성공한 명령과 템플릿을 버전 태그(v1.0→v1.1)로 관리해 다음 주 운영자 교대에도 재현되게 만든다.

#### 검증/복구 흐름
1. CLAUDE.md에 금지 규칙 또는 품질 게이트가 없으면 Hub 운영 시작을 보류한다.
2. 커맨드 테스트 실패가 10%를 넘으면 실패 유형(입력 포맷/권한/출력 누락)을 분리해 우선 수정한다.
3. 병렬 결과가 충돌하면 Synthesizer가 단일 정본을 만들고 나머지는 참고본으로 격리한다.
4. 품질 게이트 미통과 문서는 배포하지 않고 같은 날 내 재작성 후 재검수한다.
5. 금요일 baseline에서 개선 추세가 없으면 다음 주 목표를 축소해 안정화에 집중한다.

### 실행 순서 (1주)
1. **허브 초기화**
   - 프로젝트 CLAUDE.md에 작업 원칙, 금지사항, 산출물 규격 정의
   - 폴더: `tasks/`, `research/`, `reports/`, `templates/`
2. **명령어 표준화**
   - `/today`, `/prd`, `/status`, `/agent` 최소 4개 운영
   - 각 명령어 입력/출력 포맷 고정
3. **역할 분리 운영**
   - Researcher / Reviewer / Synthesizer sub-agent 정의
   - 병렬 실행 후 최종 통합 문서 1개로 귀결
4. **품질 게이트 도입**
   - 문서 산출물 필수 항목(목표/근거/지표/다음 액션)
   - 검수 미통과 시 재작성 규칙 적용

### 실패 패턴
- 실패1: CLAUDE.md가 선언문에 그치고 실행에 미반영
  - 대응: 명령어/템플릿이 참조하는 필수 변수로 연결
- 실패2: 병렬 결과가 통합되지 않음
  - 대응: synthesis 담당 role 고정, 단일 최종본 원칙
- 실패3: 모드 선택 오류로 품질 저하
  - 대응: 3단계 이상 복합 작업은 Plan 모드 강제
- 실패4: 요청/응답 포맷 불일치
  - 대응: 요청 템플릿과 보고 템플릿 버전 고정

### 이번 주 완료형 실행 플랜
- **월~화:** 프로젝트 CLAUDE.md와 폴더 표준(`tasks/research/reports/templates`)을 확정하고 기본 커맨드 4개를 연결한다.
- **수~목:** Researcher/Reviewer/Synthesizer 역할을 실제 문서 작업 1건에 적용해 병렬-통합 흐름을 검증한다.
- **금 검증:** 처리시간·재작업률·검수 누락률을 측정해 Hub 운영의 기준선(baseline)을 산출한다.
- **Done 기준:** CLAUDE.md 반영본, 커맨드 테스트 로그, sub-agent 정의서, 품질 게이트 적용 결과가 모두 남는다.
- **리스크·완화:** 명령 포맷 불일치로 실패할 수 있으므로 입력/출력 예시를 템플릿에 고정하고 버전 태그를 부여한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: CLAUDE.md가 실제 실행과 분리됨**
  - 증상: 문서엔 원칙이 있지만 커맨드/템플릿은 다른 방식으로 동작한다.
  - 해결: 커맨드 템플릿이 참조하는 필수 변수(목표, 지표, 제약)를 CLAUDE.md와 동일 키로 맞춘다.
- **막힘 2: 병렬 작업 후 정본이 여러 개 생김**
  - 증상: 어떤 문서가 최종본인지 팀이 헷갈린다.
  - 해결: 정본 파일명 규칙(`FINAL_YYYYMMDD`)과 승인자 1명을 고정해 단일 정본만 유지한다.
- **막힘 3: 모드 선택 실수로 품질/속도 동시 저하**
  - 증상: 단순 작업에 Plan 모드 남용, 복합 작업에 Edit 모드 사용.
  - 해결: 작업 복잡도 체크리스트(3단계 이상/의존성 존재/외부 공유 여부)로 모드 선택을 사전 결정한다.
- **막힘 4: 커맨드 버전 관리 부재**
  - 증상: 누가 어떤 버전을 쓰는지 몰라 재현이 안 된다.
  - 해결: 커맨드에 버전 태그와 변경 로그를 붙여 회귀 문제 발생 시 즉시 롤백한다.
