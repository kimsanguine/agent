## 1) 목적
- Claude Code를 PM 업무 자동화 도구가 아니라 `문제 정의-실행-검증` 운영 체계로 이해하게 한다.
- 입문 단계에서 “무엇을 자동화하고 무엇은 사람이 판단할지” 기준을 세워 초기 시행착오를 줄인다.

## 2) 원문 핵심 요약 (ccforpms)
- Claude Code는 로컬 파일/폴더를 직접 다루며 문서 작성, 분석, 정리를 대화 기반으로 실행한다.
- 핵심 기능 축이 구체적으로 제시된다: 파일 작업(`@`), 시각 분석, 병렬 에이전트, 커스텀 서브에이전트, `CLAUDE.md` 프로젝트 메모리.
- 사용 전제 조건이 명확하다: Claude Pro 또는 Max 구독, Mac/Windows/Linux 환경, 설치 약 15분.
- 학습 난이도 가이드가 제공된다: 총 10~12시간 학습으로 주당 5~10시간 업무 절감을 목표로 한다.
- 적용 영역은 PRD/로드맵, 고객 피드백 분석, 티켓 변환, 릴리스 노트, 이해관계자 커뮤니케이션, 경쟁/시장 리서치까지 확장된다.
- 일반 챗봇과의 차별점으로 지속 메모리, 실제 실행(파일/명령), 다문서 동시 처리, Git/MCP 연동이 강조된다.
- Monday.com PM의 Reddit 댓글 34,000건 분석 사례처럼 대규모 데이터에서도 PM 의사결정 속도를 높일 수 있음을 보여준다.
- 핵심 전환점은 “명령어 암기”가 아니라 “원하는 결과를 설계하고 오케스트레이션”하는 역량이라는 점이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code는 CLI 중심이지만 IDE/데스크톱과 함께 쓰는 multi-surface 작업 모델이므로, PM은 `대화-파일-검토`를 한 루프로 설계해야 한다.
- 초안 작성, 요약, 포맷 변환 같은 반복 업무는 자연어 자동화에 맡기고, 우선순위·승인·리스크 판단은 PM이 담당하는 분업이 기본이다.
- MCP 연동을 통해 내부 문서/협업 도구 맥락을 연결하면, 입문 단계부터 실무 데이터 기반으로 결과를 만들 수 있다.
- `CLAUDE.md` 규칙을 초기에 정하면 팀 공통 톤, 용어, 산출물 구조가 맞춰져 온보딩 재작업이 줄어든다.
- sub-agent/agent team을 쓰면 리서치·정리·검증을 병렬화할 수 있어 입문자도 오케스트레이션 관점을 빠르게 체득한다.
- git/PR 흐름과 연결하면 "대화 결과"가 아니라 "리뷰 가능한 변경"으로 PM 산출물을 관리할 수 있다.
- CLI composability(`@파일`, `/start-*`, 모드 전환)를 익히면 같은 입력으로도 목적별 출력을 안정적으로 재생산할 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **업무 목표를 한 문장으로 고정한다.**
   - 실행 입력(프롬프트): `이번 주 PM 업무에서 Claude Code로 줄이고 싶은 반복작업을 1문장으로 정리해줘.`
   - 기대 출력: 대상 업무 1개와 기대 효과(시간 절감/오류 감소)가 포함된 문장.
   - 체크포인트: "무엇을"과 "왜"가 동시에 적혀 있는지 확인.

2. **Automation/Augmentation/Manual 3분류를 만든다.**
   - 실행 입력(프롬프트): `내 업무를 자동화/증강/직접판단으로 분류하는 표를 만들어줘. 각 항목에 예시를 2개씩 넣어줘.`
   - 기대 출력: 3열 표 + 실제 PM 업무 예시.
   - 체크포인트: 직접판단 영역에 승인/우선순위 결정 업무가 포함되는지 확인.

3. **첫 입력 컨텍스트를 수집한다.**
   - 실행 입력(프롬프트): `기획 문서 초안에 필요한 입력 컨텍스트 체크리스트를 10개 항목으로 만들어줘.`
   - 기대 출력: 사용자 리서치, KPI, 경쟁정보, 내부 제약 등이 포함된 목록.
   - 체크포인트: 출처가 불명확한 항목이 없는지, 실제 접근 가능한 자료인지 확인.

4. **초안 생성과 검증 요청을 분리한다.**
   - 실행 입력(프롬프트): `위 컨텍스트를 기준으로 PRD 구조 초안을 작성해줘. 아직 결론은 내리지 말고 가설로 표시해줘.`
   - 기대 출력: 섹션 구조가 잡힌 초안 문서.
   - 체크포인트: 확정 정보와 가설 정보가 구분되어 있는지 확인.

5. **품질 검증 프롬프트를 별도로 실행한다.**
   - 실행 입력(프롬프트): `방금 초안에서 근거 없는 주장만 찾아서 수정 제안해줘. 숫자 기준이 없는 문장도 표시해줘.`
   - 기대 출력: 문제 문장 목록 + 대체 문장.
   - 체크포인트: 수정 후 "측정 가능한 문장" 비율이 높아졌는지 확인.

6. **이해관계자용 버전으로 변환한다.**
   - 실행 입력(프롬프트): `같은 내용을 임원 보고용 7줄 요약과 실무자 실행용 10줄 체크리스트로 각각 변환해줘.`
   - 기대 출력: 청중별 2개 버전 산출물.
   - 체크포인트: 임원용은 의사결정 중심, 실무용은 행동 중심으로 분리되었는지 확인.

7. **다음 주 재사용 루틴을 기록한다.**
   - 실행 입력(프롬프트): `이번 실행 과정을 다음 주에 반복할 수 있게 15분 단위 운영 루틴으로 정리해줘.`
   - 기대 출력: 재사용 가능한 미니 플레이북.
   - 체크포인트: 시작 조건, 완료 조건, 실패 시 복구 조건이 포함되는지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 1**: 소개 단계부터 실제 산출물 생성 가능 범위를 제시해 학습 목표를 결과물 중심으로 고정한다.
- **P2 맥락 우선 + Level 2**: 품질이 프롬프트 길이보다 컨텍스트 구조에서 나온다는 원칙을 입문에서 먼저 심는다.
- **P3 PM 오케스트레이션 + Level 2**: PM을 작성자가 아닌 판단/우선순위/검증 루프 설계자로 정의해 역할 혼선을 줄인다.
- **P7 Automation vs Augmentation + Level 1**: 자동화 가능한 반복 작업과 PM 직접 판단이 필요한 영역을 구분해 과신을 막는다.
- **P8 Connected Hub + Level 1~3**: 단일 툴 학습이 아니라 MCP·협업도구·자동화로 이어지는 연결 관점을 초기부터 반영한다.

## 4) 실무 적용 시나리오
신입 PM이 첫 주에 기능 기획 문서를 써야 하는 상황에서, 0.0 문서를 기준으로 업무를 3단계로 나눈다. 1) 문제 정의와 성공 지표를 먼저 정리하고, 2) Claude Code로 기초 초안을 만들고, 3) PM이 최종 우선순위와 리스크를 확정한다. 이때 자동화는 자료 정리와 초안 생성에만 쓰고, 출시 판단은 사람이 맡도록 분리한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 현재 PM 업무를 `자동화/증강/직접판단` 3영역으로 나누고, 이번 주에 시험할 대표 업무 1개를 선정합니다.
- **수~목 실행**: 선정한 업무에 대해 Claude Code로 초안 생성→PM 검토→수정 반영까지 1회전 실행해 실제 리드타임을 측정합니다.
- **금요일 검증**: 실행 전후 시간을 비교해 절감 효과를 수치화하고, 품질 이슈(오해·누락·재작업) 빈도를 리뷰합니다.
- **Done 기준**: 팀이 공감하는 자동화 경계(어디까지 자동, 어디서부터 사람 판단)가 문서화되고 다음 주 재실행 항목이 확정됩니다.
- **리스크/완화**: 도구 과신으로 판단 책임이 흐려질 수 있으므로, 최종 승인 권한과 검토 체크포인트(정확성·비즈니스 적합성)를 PM에게 고정합니다.

### PM Use Case 확장 (Claude Code 기능 매핑)
| PM 상황 | Claude Code 기능 | 실행 예시 | 검증 포인트 |
|---|---|---|---|
| 신입 PM이 자동화 경계를 못 잡는 상황 | Plan 모드 + 역할 분리 프롬프트 | `자동화/증강/직접판단 3분류 표를 만들어줘` | 사람 판단 영역(승인/우선순위) 누락 0건 |
| 동일 내용의 이해관계자 커뮤니케이션이 많은 상황 | output style + 다중 포맷 생성 | 임원 요약/실무 체크리스트 동시 생성 | 대상별 톤/길이 기준 충족 |
| 팀 문서 품질이 작성자마다 흔들림 | `CLAUDE.md` 공통 규칙 | 필수 섹션/검증 항목 규칙 고정 | 재작업률 감소, 리뷰 피드백 수 축소 |
| 리서치 입력이 많아 초기 정리가 지연 | `@파일` 참조 + agent 병렬 요약 | 인터뷰/지표 파일 병렬 요약 후 통합 | 핵심 인사이트 누락 여부 점검 |
| 작성 결과를 조직 자산으로 남기고 싶음 | git/PR 기반 문서 변경 관리 | 초안→리뷰→확정 PR 워크플로우 | 변경 이력/승인 근거 추적 가능 |

### 실무 적용 시나리오 (초보자용 상세 실행안)
1. **월요일 (기준선 확보)**
   - 시간: 09:30~10:30
   - 행동: 이번 주에 실험할 대표 업무 1개를 정하고, 기존 방식으로 처리 시간(분)을 측정합니다.
   - 산출물: "기준선 기록표"(기존 소요시간, 참여자, 오류 횟수).
   - 검증 기준: 기준선 데이터가 최소 1회 이상 측정되어 숫자로 기록됨.

2. **화요일 (첫 자동화 시도)**
   - 시간: 14:00~15:30
   - 행동: 동일 업무를 Claude Code로 수행해 초안을 생성합니다.
   - 산출물: 초안 1부 + 사용 프롬프트 로그.
   - 검증 기준: 결과물에 목적/대상/기한 3요소가 모두 포함됨.

3. **수요일 (증강 루프 적용)**
   - 시간: 10:00~11:30
   - 행동: 초안을 PM 관점으로 검토하고 누락/오해/리스크를 수동 보정합니다.
   - 산출물: 수정본 v2 + 수정 사유 목록.
   - 검증 기준: 수정 사유가 "스타일"이 아닌 "의사결정 정확성" 중심으로 기록됨.

4. **목요일 (이해관계자 전달 테스트)**
   - 시간: 16:00~17:00
   - 행동: 같은 내용을 임원용/실무용 두 버전으로 전달하고 질문을 수집합니다.
   - 산출물: 전달본 2종 + 수신자 질문 리스트.
   - 검증 기준: 추가 질문 수가 기존 대비 감소하거나 동일 시간 내 대응 완료.

5. **금요일 (성과 리뷰 + 다음 주 계획)**
   - 시간: 11:00~12:00
   - 행동: 주간 리뷰에서 시간 절감, 품질, 재작업률을 비교합니다.
   - 산출물: 주간 회고 1부 + 다음 주 개선 액션 3개.
   - 검증 기준: 개선 액션마다 담당자와 완료일이 지정됨.

- **롤백/복구 액션**: 목요일 전달 테스트에서 오해가 2건 이상 발생하면, 자동 생성본 사용을 즉시 중단하고 화요일 프롬프트를 "입력 컨텍스트 명시형"으로 재작성한 뒤 수요일 보정 루프를 재실행합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 자동화와 판단 업무가 섞여 품질이 흔들림**
   - 해결: 요청 문장을 `초안 생성`과 `최종 판단`으로 분리해 2턴으로 실행합니다.

2. **막힘: 초안은 빨리 나오는데 근거가 약함**
   - 해결: 입력 전에 컨텍스트 체크리스트를 강제하고, "근거 없는 문장 표시" 검증 프롬프트를 후속으로 고정합니다.

3. **막힘: 팀장이 결과를 신뢰하지 않음**
   - 해결: 기존 방식 대비 시간/오류/재작업 수치를 같은 표에서 비교해 보고합니다.

4. **막힘: 매주 같은 실수를 반복함**
   - 해결: 금요일 회고에 "재발 방지 규칙 1개"를 반드시 추가하고 다음 주 시작 전에 재확인합니다.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)

- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.
- [ ] `CLAUDE.md`의 프로젝트 규칙(금지사항/품질게이트/보고형식)을 확인했다.
- [ ] `@` 또는 첨부 자료로 **맥락 입력(문서/데이터/정책) 3개 이상** 연결했다.
- [ ] 필요한 경우 **sub-agent/병렬 작업** 범위와 합성(synthesis) 책임자를 지정했다.
- [ ] **MCP/외부도구 연동 필요 여부**와 접근 권한 상태를 점검했다.
- [ ] 산출물에 **검증 기준(정확성/재현성/지표/테스트)** 을 명시했다.
- [ ] 결과를 **Git/문서 링크/로그**로 추적 가능하게 남겼다.
- [ ] 다음 액션 3개를 **담당자·기한·검증포인트**와 함께 확정했다.

## 5) 실습/적용 체크리스트
- [ ] 내가 맡은 PM 업무를 자동화 대상/증강 대상/직접 수행 대상으로 3분류했다.
- [ ] 다음 2주 안에 만들 핵심 산출물 1개(PRD/분석/전략)를 선택했다.
- [ ] 해당 산출물에 필요한 입력 컨텍스트(회사 문서, 사용자 리서치, 지표)를 목록화했다.
- [ ] Claude Code 적용 후 절감할 시간 목표(주간 기준)를 숫자로 설정했다.
- [ ] 실행 결과를 검증할 기준(정확성, 재작업률, 리드타임)을 정의했다.

## 6) 산출물
