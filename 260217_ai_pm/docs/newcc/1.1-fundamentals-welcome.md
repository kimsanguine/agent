## 1) 목적
- TaskFlow 실습 환경과 PM 역할 정의를 명확히 해, 이후 모듈에서 같은 기준으로 실행/리뷰하도록 만든다.
- “문서 읽기”가 아니라 “Claude Code 안에서 결과를 만드는 학습 방식”을 작업 습관으로 고정한다.

## 2) 원문 핵심 요약 (ccforpms)
- 이 모듈의 핵심은 오리엔테이션이며, 시작 명령은 `claude` 실행 후 `/start-1-1`이다.
- 학습 구조는 두 축으로 구성된다: 인터랙티브 모듈(실행형) + 웹 Reference Guide(복습/참고).
- 가상 제품 맥락이 명확하다: 원격 협업용 SaaS `TaskFlow`, 학습자 역할은 Senior Product Manager.
- 사용자 페르소나 3종이 제시된다: Enterprise Admin(Sarah), IC Engineer(Mike), Team Lead(Alex).
- 기본 폴더 구조를 통해 학습 경로를 잡는다: `company-context/`, `lesson-modules/`, `your-work/`, `CLAUDE.md`.
- `.claude/` 숨김 폴더는 이후 모듈에서 다루며, 초반에는 핵심 컨텍스트 폴더 이해가 우선이다.
- 모듈당 권장 학습 시간은 15~30분으로, 순차 진행이 학습 효율이 가장 높다.
- 진행 중 `Read(...)`, `Bash(...)` 같은 로그는 내부 동작이며 학습자는 결과물 품질에 집중하면 된다.

### Claude Code 핵심 기능 연결 (official 반영)
- Welcome 단계에서도 Claude Code는 터미널 실행과 에디터 검토를 함께 쓰는 multi-surface 사용이 기본이며, PM은 이 화면 운영 습관부터 맞춰야 한다.
- 킥오프 아젠다/요약/정렬 같은 초기 업무는 CLI 자동화로 빠르게 만들고, 최종 의사결정 문장은 PM이 보정하는 분업이 효율적이다.
- MCP를 연결하면 페르소나 근거를 외부 협업 문서/이슈 시스템과 연결해 오리엔테이션 결과를 실무에 바로 이식할 수 있다.
- `CLAUDE.md`에 온보딩 공통 규칙(용어, 저장 위치, 출력 포맷)을 명시하면 1.1 이후 모듈 품질 편차를 줄일 수 있다.
- sub-agent/agent team을 이후 모듈에서 쓰기 위해, 1.1에서 역할 정의와 출력 템플릿을 먼저 고정해두는 것이 유리하다.
- git/PR 흐름으로 킥오프 산출물을 리뷰하면 페르소나 정의와 우선순위 기준을 팀 합의 가능한 문서로 축적할 수 있다.
- CLI composability(`/start-1-1`, `@파일`, 모드 전환)를 초기에 익히면 다음 모듈 전환 시 학습 마찰이 크게 줄어든다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **모듈 진입과 역할 선언을 동시에 수행한다.**
   - 실행 입력(명령): `claude "/start-1-1"`
   - 기대 출력: Welcome 안내 + 모듈 목표.
   - 체크포인트: 현재 본인 역할이 Senior PM 관점으로 이해되었는지 확인.

2. **TaskFlow 컨텍스트를 빠르게 읽어온다.**
   - 실행 입력(프롬프트): `TaskFlow 제품 맥락을 5줄로 요약해줘. 사용자 페르소나 3명 포함.`
   - 기대 출력: Sarah/Mike/Alex가 포함된 요약.
   - 체크포인트: 페르소나별 문제/목표가 구분되어 있는지 확인.

3. **폴더 구조 기반 작업 규칙을 만든다.**
   - 실행 입력(프롬프트): `company-context, lesson-modules, your-work를 언제 쓰는지 표로 정리해줘.`
   - 기대 출력: 폴더별 목적과 금지행동이 적힌 표.
   - 체크포인트: 산출물 저장 위치가 `your-work`로 명확히 고정되는지 확인.

4. **첫 의사결정 아젠다를 만든다.**
   - 실행 입력(프롬프트): `페르소나 충돌 상황에서 킥오프 회의 아젠다 6개를 작성해줘.`
   - 기대 출력: 의사결정형 아젠다 항목.
   - 체크포인트: 각 항목에 기대 결정사항이 명시되는지 확인.

5. **PM 판단 기준을 문서화한다.**
   - 실행 입력(프롬프트): `우선순위 판단 기준을 가치/리스크/난이도 3축으로 작성해줘.`
   - 기대 출력: 평가 기준표.
   - 체크포인트: 기준이 모호한 형용사 대신 점수/레벨로 표현되는지 확인.

6. **실행 결과를 팀 공유 버전으로 변환한다.**
   - 실행 입력(프롬프트): `지금까지 내용을 팀 온보딩 공지 10줄로 바꿔줘.`
   - 기대 출력: 즉시 전파 가능한 공지문.
   - 체크포인트: 다음 행동(누가/언제/무엇)을 읽는 사람이 바로 이해할 수 있는지 확인.

7. **다음 모듈 전환 조건을 점검한다.**
   - 실행 입력(프롬프트): `1.2로 넘어가기 전 확인해야 할 사전 체크 5개를 만들어줘.`
   - 기대 출력: 전환 체크리스트.
   - 체크포인트: 환경 준비(에디터, 경로, 산출물 저장규칙)가 모두 포함되는지 확인.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선 + Level 2**: 첫 모듈부터 실제 명령(`/start-1-1`) 실행을 요구해 이론 중심 학습을 차단한다.
- **P3 PM 오케스트레이션 + Level 2**: PM 역할을 ‘코드 작성자’가 아닌 ‘문맥/우선순위/검증 설계자’로 학습 초기에 정의한다.
- **P5 한국 실무 적합성 + Level 2**: 페르소나 기반 의사결정 연습을 한국 실무의 이해관계자 커뮤니케이션 흐름과 연결한다.
- **P9 Words-to-Workflow + Level 2**: 슬래시 명령 한 줄이 실제 학습 흐름과 산출물 생성으로 이어지는 경험을 제공한다.

## 4) 실무 적용 시나리오
주니어 PM이 기능 개선 킥오프를 준비하면서 이해관계자 요구가 충돌하는 상황이다. 1.1 모듈 방식대로 페르소나를 분리해 요구를 정리하고, `company-context/`를 근거로 우선순위를 정해 회의 아젠다를 만든다. 이후 `your-work/`에 결정 기록을 남겨 다음 회차 논의를 재사용 가능한 문맥으로 누적한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `/start-1-1`로 공통 온보딩을 끝내고 팀 기준 페르소나 3종(의사결정자/실사용자/운영자)을 실제 프로젝트 맥락에 맞게 재정의합니다.
- **수~목 실행**: 페르소나별 핵심 요구와 반대 요구를 표로 정리한 뒤, 우선순위 결정 원칙(가치·리스크·난이도)을 문서화합니다.
- **금요일 검증**: 킥오프 시뮬레이션 회의에서 페르소나 표를 사용해 의사결정 근거가 일관되게 설명되는지 검토합니다.
- **Done 기준**: `company-context` 기반 페르소나 카드와 우선순위 기준표가 완성되고, 첫 회의 아젠다에 실제 반영됩니다.
- **리스크/완화**: 페르소나가 추상적으로 남을 수 있으므로, 최근 고객 사례 2건 이상을 근거로 카드 내용을 구체화합니다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
| 요일 | 시간 | 행동 | 산출물 | 검증 기준 |
|---|---|---|---|---|
| 월 | 09:30~10:00 | `/start-1-1` 실행 및 모듈 목표 확인 | 시작 로그 | 모듈 목표 3개 이해 여부 점검 완료 |
| 월 | 16:00~17:00 | 페르소나 3종 재정의 워크숍 | 페르소나 카드 v1 | 각 카드에 목표/고충/성공지표 포함 |
| 화 | 11:00~12:00 | 폴더 운영 규칙 합의 | 폴더 사용 규칙표 | `your-work` 저장 규칙 명문화 |
| 수 | 14:00~15:00 | 우선순위 기준표 작성 | 가치·리스크·난이도 표 | 팀원 2명 이상 동일 기준 적용 가능 |
| 목 | 10:00~11:00 | 킥오프 아젠다 초안 작성 | 회의 아젠다 1부 | 항목별 기대 결정사항 포함 |
| 금 | 15:00~16:00 | 킥오프 리허설 + 피드백 | 리허설 회고 | 근거 없는 주장 0건, 후속 액션 3개 확정 |

- **초보자 실행 팁 1**: 페르소나 설명에는 반드시 실제 사례(최근 이슈) 1개를 붙입니다.
- **초보자 실행 팁 2**: 우선순위 토론 시 감각 표현("중요해 보임") 대신 근거 지표를 먼저 제시합니다.
- **초보자 실행 팁 3**: 아젠다는 질문형보다 결정형 문장으로 작성합니다.
- **롤백/복구 액션**: 금요일 리허설에서 의사결정 기준이 일치하지 않으면, 수요일에 만든 기준표 버전을 이전 안정 버전으로 되돌리고(백업본), 기준 정의 워크숍을 30분 추가로 재진행합니다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 페르소나가 추상적이라 회의에서 안 쓰임**
   - 해결: 카드마다 최근 30일 고객 사례 1건과 실제 발화 1문장을 필수 추가.

2. **막힘: 폴더 규칙이 없어 산출물이 흩어짐**
   - 해결: 파일명 규칙(날짜-주제-작성자)을 고정하고 주 1회 정리 루틴 운영.

3. **막힘: 우선순위 기준이 사람마다 달라 결론이 지연됨**
   - 해결: 가치/리스크/난이도 3축 점수표를 사전 배포하고 회의 중 실시간 채점.

4. **막힘: 킥오프 아젠다가 정보 나열형으로 끝남**
   - 해결: 아젠다 끝에 반드시 "오늘 확정할 결정" 항목을 추가해 결론 중심으로 전환.

### Claude Code 핵심개념 체크리스트 (표준 템플릿)

- [ ] 작업 시작 전에 **모드(Plan / Edit / Auto-Accept)** 선택 기준을 기록했다.
- [ ] `CLAUDE.md`의 프로젝트 규칙(금지사항/품질게이트/보고형식)을 확인했다.
- [ ] `@` 또는 첨부 자료로 **맥락 입력(문서/데이터/정책) 3개 이상** 연결했다.
- [ ] 필요한 경우 **sub-agent/병렬 작업** 범위와 합성(synthesis) 책임자를 지정했다.
- [ ] **MCP/외부도구 연동 필요 여부**와 접근 권한 상태를 점검했다.
- [ ] 산출물에 **검증 기준(정확성/재현성/지표/테스트)** 을 명시했다.
- [ ] 결과를 **Git/문서 링크/로그**로 추적 가능하게 남겼다.
- [ ] 다음 액션 3개를 **담당자·기한·검증포인트**와 함께 확정했다.
