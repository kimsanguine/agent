**빠른 요약**
- `docs/newcc`에서 반복되는 핵심 용어를 한 페이지에서 빠르게 확인하는 실무형 용어집입니다.

**이번 문서 목표**
- 초심자도 용어를 이해한 뒤 같은 주 안에 실제 업무 문장과 체크리스트에 적용할 수 있습니다.

**처음 보는 분 가이드**
- 먼저 `## 1) 목적`과 `## 3) 핵심 용어`의 Day 1~Day 5 미니 가이드를 읽으세요.
- 이후 각 모듈 문서로 돌아가 용어를 실제 지시문(예: OMTM, AC/TC, FR/NFR)으로 바꿔 적용하세요.

## 1) 목적
- `docs/newcc` 전반에서 반복되는 핵심 용어를 초심자도 빠르게 이해하도록 정리한다.
- 문서를 읽다 막히면 이 페이지에서 뜻을 확인하고 다시 실습으로 돌아가게 돕는다.

## 2) 빠른 사용법
- 처음에는 Level 1~4, P1~P10, OMTM/KPI, PRD/TDD/Eval만 먼저 읽어도 충분하다.
- 각 용어의 “실무에서 이렇게 씀” 예시를 함께 보면 적용이 쉬워진다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, 이번 주 과제 1개를 정하고 해당 과제에 필요한 용어 5개(Level, OMTM, AC/TC, FR/NFR, Validation Gap)를 먼저 선택한다.
2. 월요일 오후, 선택한 용어를 실제 문장으로 바꿔 `이번 주 운영 선언문` 5줄을 작성한다.
3. 화요일, PRD 초안에서 추상 표현(예: “좋은 경험”, “빠르게”)을 찾아 AC/TC와 NFR 숫자로 치환한다.
4. 수요일, 팀 미팅에서 OMTM 1개와 보조 KPI 3개를 제시하고 왜 이 조합이 이번 주 의사결정에 필요한지 설명한다.
5. 수요일 오후, 업무 항목을 `Automation vs Augmentation`으로 분류해 자동화 가능한 항목 2개를 backlog로 분리한다.
6. 목요일, Connected Hub 관점으로 문서/티켓/리포트 링크가 끊기지 않았는지 점검하고 누락 링크를 보완한다.
7. 금요일, 회고에서 Validation Gap(아이디어→검증 소요시간)을 숫자로 기록하고 다음 주 목표값을 합의한다.
8. 금요일 마감 전, 이번 주에 실제로 사용한 용어 문장 10개를 `용어 적용 로그`로 저장해 다음 주 템플릿으로 재사용한다.

### 용어-기능 매핑 (Claude Code 실전 연결)
| 용어 | Claude Code 공식 기능 연결 | PM 실무 적용 예시 | 검증 포인트 |
|---|---|---|---|
| Project Memory (CLAUDE.md) | 프로젝트 맥락을 지속 반영하는 운영 기준 | 팀 규칙/금지사항/보고 형식을 CLAUDE.md에 고정 | 동일 요청 재실행 시 결과 일관성 |
| Plan Mode | 복합 작업을 계획 후 실행으로 분리 | PRD 개편/문서 통합처럼 의존성이 큰 작업에 선적용 | 누락 단계/되돌림 감소 |
| Subagents / 병렬 실행 | 역할 분리 및 동시 처리 | Researcher/Reviewer/Synthesizer로 문서 품질 향상 | 처리시간 단축, 충돌 건수 |
| MCP 연동 | 외부 도구와의 컨텍스트 연결 | Notion/Linear/Slack에 산출물 핸드오프 자동화 | 링크 누락·전달 실패율 |
| Tutorials 기반 운영 루프 | 학습→적용→검증 사이클 | 주간 루틴에 튜토리얼 패턴을 표준 절차로 반영 | 재작업률, 온보딩 속도 |
| Chrome/Cowork 확장 | 외부 탐색 및 파일 협업 확장 | 웹 리서치-문서화-티켓화를 한 흐름으로 연결 | 출처 추적성, 산출물 전환율 |

### 초보자 프롬프트 템플릿 (복붙용)
- **템플릿 1: OMTM 확정**
  - `우리 팀의 이번 주 목표는 [목표]다. OMTM을 1개만 제안하고, 보조 KPI 3개를 인과 순서로 제시해줘. 각 KPI의 데이터 소스와 측정 주기도 함께 적어줘.`
- **템플릿 2: PRD 품질 보강(AC/TC)**
  - `아래 PRD 기능 목록을 보고 각 기능마다 AC 2개와 TC 2개를 작성해줘. 모든 AC는 측정 가능한 수치/시간/조건을 포함해야 해.`
- **템플릿 3: FR/NFR 분리**
  - `다음 요구사항을 FR과 NFR로 분리해 표로 만들어줘. NFR에는 성능/신뢰성/보안/운영성 항목을 빠짐없이 확인해줘.`
- **템플릿 4: Validation Gap 축소**
  - `현재 아이디어에서 첫 검증까지 걸리는 시간을 30% 줄이기 위한 실험 계획을 주간 단위로 제안해줘. 실험당 가설, 성공 기준, 실패 시 학습 항목을 포함해줘.`

### 안티패턴 vs 교정 문장
| 안티패턴 문장 | 왜 문제인지 | 교정 문장(실행형) |
|---|---|---|
| `사용자 경험을 개선한다.` | 측정 불가, 범위 불명확 | `신규 사용자의 첫 핵심 기능 완료율을 42%→55%로 올린다.` |
| `빠르게 개발해보자.` | 완료 기준 부재 | `금요일까지 MVP 1개를 배포하고 AC 5개 중 4개 이상 통과한다.` |
| `지표는 나중에 보자.` | 검증 지연 유발 | `OMTM=Activation, 보조 KPI=Time-to-Value/리드타임/오류율을 주간 추적한다.` |
| `자동화해도 될 것 같다.` | 경계 모호, 위험 통제 없음 | `반복 요약/분류는 자동화하고, 대외 발신 문안은 PM 승인 후 배포한다.` |
| `문서는 대충 정리해도 된다.` | 핸드오프 손실 | `모든 산출물에 목표/근거/지표/owner/due를 포함해 다음 단계로 전달한다.` |

## 3) 핵심 용어

### 이번 주 바로 쓰는 용어 적용 미니 가이드
- **Day 1 (월):** 팀 현재 수준을 `Level 1~4`로 한 줄 선언한다.
  - 예시 문구: `우리 팀은 현재 Level 1, 이번 주 목표는 Level 2 진입 준비 완료.`
- **Day 2 (화):** 이번 주 핵심 지표를 `OMTM 1개 + 보조 KPI 3개`로 고정한다.
  - 예시 문구: `OMTM=Activation, 보조 KPI=Time-to-Value/PRD→티켓 리드타임/자동화 성공률.`
- **Day 3 (수):** 작성 중인 PRD에 `AC/TC`와 `FR/NFR`를 최소 1개씩 추가한다.
  - 예시 문구: `AC: 신규 사용자는 3분 내 핵심 기능 1회 완료, NFR: 주요 화면 응답 2초 이하.`
- **Day 4 (목):** 업무를 `Automation vs Augmentation`으로 분류한다.
  - 예시 문구: `반복 요약/분류는 자동화, 우선순위 결정/대외 발신은 PM 최종 승인.`
- **Day 5 (금):** 회고에서 `Validation Gap`을 숫자로 기록한다.
  - 예시 문구: `아이디어→첫 검증까지 평균 9일 → 다음 주 목표 5일.`

### 현장에서 바로 쓰는 짧은 실행 문구
- `이 요청은 Level 1 Quick Win으로 오늘 안에 끝낸다.`
- `이 안건의 OMTM 영향이 없으면 우선순위를 낮춘다.`
- `PRD에 AC/TC 없으면 개발 전달 전 단계로 되돌린다.`
- `자동화 후보인지(반복·규칙성·빈도) 먼저 체크한다.`
- `가설은 문장으로, 검증은 KPI로 남긴다.`
- `Connected Hub에 링크가 없으면 완료로 보지 않는다.`

### A. Level 체계
| 용어 | 쉬운 정의 | 실무에서 이렇게 씀 |
|---|---|---|
| Level 1 | 바로 성과를 내는 빠른 적용 단계 | PRD 게시, 댓글 분류, 티켓 생성, 상태 공유를 48시간 내 실행 |
| Level 2 | Claude Code를 팀 실행 허브로 고정하는 단계 | `CLAUDE.md`, Slash Command, 템플릿을 표준 운영 자산으로 사용 |
| Level 3 | 반복 업무를 자동화로 넘기는 단계 | n8n으로 스케줄/이벤트 기반 리포트와 분류 자동 실행 |
| Level 4 | 에이전트 기반으로 고급 실행 체계를 만드는 단계 | `/agent`로 PRD→TDD→Eval→Prototype 루프 운영 |

### B. 원칙(P1~P10)
| 용어 | 쉬운 정의 | 실무에서 이렇게 씀 |
|---|---|---|
| P1 실행 우선 | 설명보다 먼저 작동하는 결과를 만든다 | 회의 후 문서 정리보다 먼저 동작 데모 1개 생성 |
| P2 맥락 우선 | 프롬프트 길이보다 입력 맥락 품질이 중요하다 | 회사 문서/지표/제약을 먼저 붙이고 작성 시작 |
| P3 PM 오케스트레이션 | PM은 직접 구현보다 흐름과 우선순위를 설계한다 | 역할 분리, 리뷰 루프, 합성 기준을 PM이 결정 |
| P4 속도-검증 균형 | 빨리 만들되 검증 게이트를 함께 둔다 | 초안은 빠르게, AC/TC와 리뷰로 품질 확인 |
| P5 한국 실무 적합성 | 실제 국내 팀 운영 환경에 맞게 적용한다 | 보고 체계, 의사결정 속도, 조직 문화에 맞춰 문서화 |
| P6 자동화 운영 기반 | 반복 작업을 재사용 가능한 시스템으로 전환한다 | 템플릿/자동화 플로우/체크리스트를 표준화 |
| P7 Automation vs Augmentation | 자동화와 사람 판단의 경계를 분명히 한다 | 초안 생성은 자동, 대외 발신 최종 결정은 사람 |
| P8 Connected Hub | 도구를 따로 쓰지 않고 하나의 흐름으로 연결한다 | Claude.ai- Claude Code - n8n - Slack/Linear 연동 |
| P9 Words-to-Workflow | 문장이 바로 실행 가능한 작업으로 이어져야 한다 | "요약해줘"가 아니라 "요약 후 티켓 3개 생성"으로 지시 |
| P10 Validation-First Discovery | 만들기 전에 검증 루프를 먼저 설계한다 | 가설·지표·실험 기준을 먼저 고정하고 구현 시작 |

### C. 운영/요구사항/지표
| 용어 | 쉬운 정의 | 실무에서 이렇게 씀 |
|---|---|---|
| OMTM | 이번 기간에 가장 중요한 단 하나의 핵심 지표 | 분기 OMTM을 Activation으로 두고 주간 리뷰 운영 |
| KPI | 성과를 숫자로 측정하는 지표 묶음 | Activation, Retention, 리드타임을 주간 대시보드로 확인 |
| AC/TC | AC는 완료 기준, TC는 테스트 케이스 | PRD의 각 기능마다 AC 1~3개와 TC를 같이 작성 |
| FR/NFR | FR은 기능 요구사항, NFR은 성능/보안/신뢰성 요구사항 | "로그인"은 FR, "응답 2초 이하"는 NFR |
| Validation Gap | 아이디어와 실제 검증 사이의 시간/품질 간극 | 가설은 많지만 실험이 늦어 의사결정이 밀리는 상태 |
| Connected Hub | 문서·실행·자동화를 끊기지 않게 잇는 운영 허브 | 작성-티켓화-공유-리포트를 한 루프에서 처리 |
| Words-to-Workflow | 자연어 지시를 실행 흐름으로 바꾸는 방식 | "주간 보고서 작성 후 Slack 공유"를 한 번에 실행 |
| Automation vs Augmentation | 전자동 처리와 사람 보조 처리의 구분 기준 | 반복 정리는 자동화, 우선순위 판단은 PM이 수행 |

### D. 산출물/실행 도구
| 용어 | 쉬운 정의 | 실무에서 이렇게 씀 |
|---|---|---|
| PRD | 제품 요구사항 문서 | 목표, 사용자 문제, 범위, AC, 리스크를 담아 합의 |
| TDD | 테스트 기준을 먼저 잡고 구현하는 접근 | 실패 조건을 먼저 정의하고 기능을 구현 |
| Eval | 결과 품질을 평가하는 기준/절차 | 정확도·완료율·오류율로 산출물 점검 |
| Prototype | 빠르게 검증하기 위한 초기 제품 버전 | 핵심 흐름만 구현해 사용자 반응 먼저 확인 |
| MCP | Claude가 외부 도구와 연결되는 표준 인터페이스 | Notion/Linear/Slack 연동으로 작업 자동 전환 |
| n8n | 노코드/로우코드 자동화 워크플로 도구 | 매일 아침 뉴스 요약 후 Slack 자동 발송 |

### 공식 튜토리얼 기반 추천 학습 순서
1. Claude AI 시작 튜토리얼로 기본 인터랙션과 문서화 습관을 먼저 익힌다.
2. Claude Code Overview로 파일 중심 실행, 계획 기반 작업, 팀 협업 원칙을 정렬한다.
3. Opus 4.6 튜토리얼로 고난도 분석/작성 작업의 품질을 끌어올린다.
4. Chrome 튜토리얼로 외부 리서치 입력을 확장하고, Cowork 튜토리얼로 파일 협업 시나리오를 보강한다.
5. 금융 서비스 튜토리얼처럼 신뢰성/감사성 요구가 큰 조직에서는 근거 로그·승인 기록·정책 준수를 기본으로 둔다.

| 요일 | 점검 질문 | 통과 기준 |
|---|---|---|
| 월 | 우리 팀 레벨(Level 1~4)을 한 줄로 선언했는가? | 선언문이 문서 첫 단락에 존재 |
| 화 | OMTM 1개와 보조 KPI 3개를 숫자로 정의했는가? | 지표 산식/목표값이 명시됨 |
| 수 | PRD에 AC/TC, FR/NFR가 함께 들어갔는가? | 각 기능마다 AC/TC 최소 1세트 |
| 목 | 자동화와 사람 승인 경계가 명시됐는가? | Automation/Augmentation 표 완성 |
| 금 | Validation Gap 수치와 다음 주 목표를 기록했는가? | 현재값/목표값/개선 액션 존재 |

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 용어를 알아도 문장으로 못 씀**
  - 증상: 회의에서 용어는 언급하지만 실행 문장으로 이어지지 않는다.
  - 해결: 용어마다 `목표 수치 + 기간 + owner`를 붙여 문장을 완성하는 규칙을 적용한다.
- **막힘 2: OMTM과 KPI가 분리되어 우선순위 혼선**
  - 증상: KPI가 많아졌는데 핵심 의사결정이 느려진다.
  - 해결: OMTM 영향이 낮은 KPI는 보조 보드로 분리하고 메인 회의에는 OMTM 중심 카드만 사용한다.
- **막힘 3: AC/TC 없이 PRD가 개발로 넘어감**
  - 증상: 개발/기획 해석 차이로 재작업이 늘어난다.
  - 해결: AC/TC 누락 문서는 전달 금지 규칙을 적용하고, 체크리스트 통과 후에만 티켓화한다.
- **막힘 4: 안티패턴 문장이 반복됨**
  - 증상: `개선한다`, `빠르게` 같은 모호 표현이 계속 등장한다.
  - 해결: 안티패턴 표를 리뷰 기준으로 사용해 모호 표현 발견 즉시 교정 문장으로 치환한다.
