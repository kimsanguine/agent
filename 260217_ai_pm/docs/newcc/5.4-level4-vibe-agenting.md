## 1) 목적
- `/agent` 기반 Vibe Agenting으로 PRD→TDD→Eval→Prototype→배포 핸드오프를 체계화한다.
- 단일 작업자 생산성을 넘어, 팀 단위 병렬 실행과 품질 게이트를 운영한다.
- 실험적 프로토타이핑을 운영 가능한 제품 후보로 전환한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 1.4는 병렬 agents로 대량 작업을 분산하고 synthesis로 통합하는 패턴을 제시한다.
- 원문 1.5는 custom sub-agent를 통해 역할 기반 리뷰/생성을 반복 가능한 시스템으로 만든다.
- 원문 2.1은 다중 관점 리뷰(기술/비즈니스/사용자)로 문서 품질을 높이는 방법을 보여준다.
- 원문 4.2~4.5는 계획→구현→반복→GitHub→배포까지 end-to-end 실행 루프를 실습한다.
- 즉, Level4는 “아이디어 생성”이 아니라 “역할 분업 + 검증 + 출시 연결”이 핵심이다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview의 병렬 실행/역할 분리 원칙은 Level4 agenting의 기본 운영 규격으로 직접 매핑된다.
- PM은 `/agent` 실행 전에 공통 입력(PRD, 비목표, 성공지표)을 고정하면 에이전트 간 해석 편차를 크게 줄일 수 있다.
- Tutorials Hub 흐름처럼 고급 작업은 생성 자체보다 평가(Eval)와 핸드오프까지 닫힌 루프로 끝내야 한다.
- Opus 4.6 튜토리얼 관점에서 다중 관점 검토(기술/사업/사용자)는 실패 비용이 큰 단계에서 특히 효과적이다.
- Cowork/Chrome 튜토리얼 입력을 활용하면 비개발 리서치 산출물을 agenting 파이프라인의 상위 입력으로 연결할 수 있다.
- 금융 서비스 튜토리얼의 통제 기준을 반영해 승인 게이트, 롤백 계획, 변경 기록을 포함한 안전한 출시 체계를 구성한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일, PM이 문제 정의/비목표/성공지표를 1페이지로 확정하고 `/agent` 실행 전 공통 입력 문서로 배포한다.
2. 화요일, 역할별 agent(Planner, Builder, Reviewer)를 지정하고 각 역할의 입력/출력/승인 기준을 표로 고정한다.
3. 수요일 오전, Planner가 PRD를 정제하고 Reviewer가 리스크/누락 항목을 사전 검증해 구현 오해를 줄인다.
4. 수요일 오후, Builder가 프로토타입을 생성하고 실행 가능한 데모 시나리오와 환경 요구사항을 README에 기록한다.
5. 목요일, Eval 기준으로 기능/품질/사용성 테스트를 수행해 pass/fail 결과를 수치로 남긴다.
6. 목요일 오후, fail 항목은 우선순위 순으로 수정하고 재평가해 Gate 통과 여부를 재판정한다.
7. 금요일 오전, 운영 핸드오프(배포/모니터링/롤백) 체크리스트를 채우고 누락 항목을 마감한다.
8. 금요일 오후, 다음 주 확대 적용 여부를 결정하고 재현 가능한 agenting 패키지(README/instructions/eval/prototype)를 버전화한다.

#### 월~금 상세 실행표
| 요일 | 단계 | 담당 | 핵심 산출물 | 검증 기준 | 복구 흐름 |
|---|---|---|---|---|---|
| 월 | Stage A(PRD 정제) | PM/Planner | 확정 PRD | 비목표/리스크 명시 | 요구 재정의 후 재합의 |
| 화 | Stage B(TDD/Eval) | Reviewer | 테스트/평가 기준 | pass/fail 수치화 | 기준 재설계 |
| 수 | Stage C(Prototype) | Builder | 실행 가능한 MVP | 데모 재현성 | 환경 설정 재정비 |
| 목 | Stage D(평가/수정) | Reviewer+Builder | 개선 이력 | fail 항목 감소율 | 우선순위 재조정 |
| 금 | Stage E(핸드오프) | 운영 오너 | 배포 체크리스트 | 모니터링/롤백 준비 | 배포 보류 + 보완 |

#### 검증/복구 흐름
1. PRD에서 비목표 또는 제약이 빠지면 구현 착수를 금지하고 요구사항 정제부터 다시 진행한다.
2. Eval 기준이 정성 문장만 있으면 테스트 신뢰도가 떨어지므로 수치 기준을 추가할 때까지 Gate를 열지 않는다.
3. 프로토타입이 로컬 특정 환경에서만 동작하면 README에 환경 의존성을 명시하고 컨테이너/스크립트로 재현성을 보강한다.
4. 평가 fail 항목이 30%를 넘으면 신규 기능 추가를 중단하고 안정화 스프린트로 전환한다.
5. 핸드오프 문서에 롤백 절차가 없으면 배포 승인하지 않고 운영 리스크를 먼저 해소한다.

### 실행 순서 (3~4주)
1. **Stage A: PRD 정제**
   - 입력: 문제 정의, KPI, 제약
   - 산출물: 확정 PRD + 비목표 + 리스크
2. **Stage B: TDD/Eval 설계**
   - 입력: PRD
   - 산출물: 테스트 시나리오, Eval 기준, 합격 조건
3. **Stage C: Prototype 생성**
   - 실행: `/agent`로 프로토타입 구현
   - 산출물: 실행 가능한 MVP + 데모 가이드
4. **Stage D: 평가 및 수정**
   - 실행: Eval 기준으로 품질 측정
   - 산출물: 개선 이력, 실패 케이스
5. **Stage E: 운영 핸드오프**
   - 실행: n8n 또는 운영 시스템으로 배포/알림 연결
   - 산출물: 배포 체크리스트 + 모니터링 링크

### 실패 패턴
- 실패1: 에이전트 역할 경계가 모호해 중복 작업 발생
  - 대응: 역할별 입력/출력/책임 범위를 문서화
- 실패2: 프로토타입은 있으나 평가 기준 부재
  - 대응: 구현 전 Eval 항목 선고정(TDD/Eval 선행)
- 실패3: 배포 전 운영 고려 누락
  - 대응: 핸드오프 체크리스트(로그/알림/롤백) 강제
- 실패4: 데모 중심으로 끝나고 재현성 없음
  - 대응: 실행 스크립트와 환경 요구사항을 README에 포함

### 이번 주 완료형 실행 플랜
- **월~화:** PRD·비목표·성공지표를 확정하고 agent 역할 정의서(입력/출력/책임)를 작성한다.
- **수~목:** `/agent`로 프로토타입 1건을 생성하고 Eval 기준으로 1차 품질 측정을 수행한다.
- **금 검증:** Gate 통과 여부를 판단해 배포 핸드오프 가능 상태(또는 보완 항목)를 결정한다.
- **Done 기준:** agenting 패키지(README/instructions/eval/prototype), Gate 기록, 핸드오프 체크리스트가 모두 완성된다.
- **리스크·완화:** 역할 경계 중복으로 비효율이 생길 수 있으므로 책임 매트릭스(RACI 유사)를 먼저 고정하고 중복 산출물을 금지한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 역할 정의는 했지만 책임 경계가 겹침**
  - 증상: Planner와 Builder가 같은 산출물을 중복 작성한다.
  - 해결: 역할별 최종 책임 산출물을 1개씩 고정하고 승인자만 교차 지정한다.
- **막힘 2: Eval이 사후 점검으로 밀려 품질 저하**
  - 증상: 구현이 끝난 뒤 기준을 맞추느라 재작업이 폭증한다.
  - 해결: 착수 전 Eval 기준 승인 없이는 Stage C로 넘어가지 않는 Gate를 적용한다.
- **막힘 3: 프로토타입 데모는 되는데 운영 전환 실패**
  - 증상: 모니터링/롤백/로그가 없어 배포를 못 한다.
  - 해결: Stage E 체크리스트에 운영 필수 항목을 강제하고 누락 시 자동 보류한다.
- **막힘 4: agenting 산출물이 재사용되지 않음**
  - 증상: 프로젝트마다 처음부터 다시 만든다.
  - 해결: 성공 패키지를 템플릿 저장소로 이관하고 버전/적용 범위를 메타데이터로 남긴다.
