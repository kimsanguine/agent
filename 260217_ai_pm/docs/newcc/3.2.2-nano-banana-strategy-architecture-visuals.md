## 1) 목적
- CTO/CEO/Board 등 이해관계자별 질문에 맞춰 전략·아키텍처를 **시각적으로 빠르게 합의**한다.
- 복잡한 설명을 다이어그램·매트릭스·로드맵으로 구조화해 의사결정 시간을 줄인다.
- 문서 중심 커뮤니케이션을 발표/회의 중심 커뮤니케이션으로 확장한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 이 모듈의 본질은 “복잡한 내용을 청중별 visual로 재구성”하는 데 있다.
2. 대표 산출물 3종은 system architecture diagram, 2x2 prioritization matrix, product roadmap이다.
3. architecture diagram은 components/data flow/labels/groupings/colors를 명시할수록 품질이 안정적이다.
4. prioritization matrix는 일반적으로 Impact × Effort 2x2를 사용하고, Quick Wins/Major Projects/Fill-ins/Avoid 구분이 실무적이다.
5. roadmap는 분기별 theme + deliverables로 구성해 단계적 진화를 보여줘야 한다.
6. 이해관계자별 초점이 다르다: CTO(통합/의존성), CEO(우선순위 trade-off), Board(타임라인/마일스톤).
7. 전달력 강화를 위해 visual당 핵심 메시지를 1개로 좁히고 요소 수를 5~10개 수준으로 관리하는 것이 좋다.
8. 겹침/혼잡/진화 흐름 불명확 문제는 prompt에 high-level, not overlapping, clear evolution 지시를 넣어 개선한다.

### Claude Code 핵심 기능 연결 (official 반영)
- **의사결정형 시각화(Overview)**: 전략 문장을 바로 architecture/matrix/roadmap으로 전환해 회의 의사결정 속도를 높인다.
- **구조적 요청 작성(Tutorial: Getting Started)**: 청중, 결정 질문, 시각 유형을 먼저 명시하면 결과물이 장식화되는 위험이 줄어든다.
- **복합 논리 정리(Tutorial: Claude Opus 4.6)**: 우선순위 trade-off, 의존성, 리스크를 동시에 비교해 시각 자료의 설득력을 높인다.
- **외부 정보 탐색(Tutorial: Claude for Chrome)**: 경쟁/시장 신호를 빠르게 수집해 로드맵 가정의 신선도를 유지한다.
- **협업 검토(Tutorial: Claude Cowork)**: CTO/CEO/Board 관점을 분리 검토해 같은 내용도 청중별 핵심 포인트를 정렬한다.
- **거버넌스형 기록(Tutorial: Financial Services)**: 시각 요소와 실행 항목 매핑을 남겨 발표 후 실행 단절을 줄인다.
- **반복 테스트(Claude Code)**: 30초 설명 테스트를 루프에 넣어 전달 실패를 조기에 수정한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 이해관계자별 의사결정 질문 수집
- 입력 아티팩트: CTO/CEO/Board 미팅 어젠다, 최근 회의 질문 로그
- 프롬프트 조각: "각 이해관계자가 이번 회의에서 실제로 결정해야 할 질문 1개씩만 추려줘."
- 기대 출력: `stakeholder-decision-questions.md`
- 검증 체크포인트: 질문이 정보 공유가 아니라 의사결정형 문장이다.

2) visual 종류 매핑
- 입력 아티팩트: 질문 목록
- 프롬프트 조각: "질문별로 architecture/matrix/roadmap 중 가장 적합한 visual을 매핑해줘."
- 기대 출력: `visual-selection-map.md`
- 검증 체크포인트: 한 질문당 핵심 visual 1개가 명확히 배정된다.

3) architecture diagram 초안 작성
- 입력 아티팩트: 시스템 컴포넌트 목록, 데이터 흐름 설명
- 프롬프트 조각: "컴포넌트 8개 이하, 데이터 흐름 화살표, 그룹 라벨을 포함한 명확한 아키텍처 다이어그램을 설계해줘."
- 기대 출력: `outputs/architecture-v1.png`
- 검증 체크포인트: 흐름 방향이 모호하지 않고 의존성이 읽힌다.

4) 2x2 prioritization matrix 생성
- 입력 아티팩트: 백로그 아이템, 예상 임팩트/노력 추정
- 프롬프트 조각: "Impact×Effort 2x2에 Top 12 아이템을 배치하고 Quick Wins를 먼저 표시해줘."
- 기대 출력: `outputs/prioritization-matrix-v1.png`
- 검증 체크포인트: 각 아이템이 왜 그 사분면인지 근거가 남아 있다.

5) 분기 로드맵 시각화
- 입력 아티팩트: Q1~Q3 테마, 마일스톤 초안
- 프롬프트 조각: "분기별 theme와 deliverable, 리스크 완화 마일스톤을 포함한 roadmap을 만들어줘."
- 기대 출력: `outputs/roadmap-v1.png`
- 검증 체크포인트: 분기 간 진화 흐름이 단계적으로 읽힌다.

6) 30초 설명 테스트
- 입력 아티팩트: visual 3종
- 프롬프트 조각: "각 visual을 30초 안에 설명하는 스크립트와 예상 질문 3개를 작성해줘."
- 기대 출력: `30sec-script.md`
- 검증 체크포인트: 테스트 참가자가 핵심 결정을 정확히 말할 수 있다.

7) 실행 계획 매핑 및 최종본 확정
- 입력 아티팩트: 피드백 반영본, 실행 백로그
- 프롬프트 조각: "visual의 각 요소를 실행 항목(오너/일정/의존성)으로 매핑해 최종 표를 만들어줘."
- 기대 출력: `visual-to-execution-map.md`
- 검증 체크포인트: visual과 백로그 사이 미연결 항목이 0개.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P3 PM 오케스트레이션**: PM은 “누가 어떤 결정을 내려야 하는가”에 맞춰 visual 포맷을 선택한다.
- **P4 속도-검증 균형**: 빠르게 시안을 만들고, 이해관계자 30초 설명 테스트로 즉시 검증한다.
- **P6 자동화 운영 기반**: 다이어그램 유형별 프롬프트 템플릿을 표준화해 회의 준비 시간을 단축한다.
- **P8 Connected Hub**: 전략 문서, 실행 백로그, 발표 자료를 같은 메시지 체계로 연결한다.
- **P9 Words-to-Workflow**: 자연어 전략 설명을 실행 가능한 시각 모델(아키/우선순위/로드맵)로 변환한다.
- v2.4의 OMTM·우선순위 정렬 원칙과 맞물려, 시각 자료는 “예쁜 그림”이 아니라 의사결정 인터페이스가 된다.

## 4) 실무 적용 시나리오
**상황: 동일 제품으로 3개 회의(기술/사업/이사회)를 연속 진행**
1. CTO 미팅: 시스템 구성과 데이터 흐름 중심 아키텍처 다이어그램 제시.
2. CEO 미팅: Impact×Effort 매트릭스로 분기 우선순위 선택지를 비교.
3. Board 미팅: 3개 분기 로드맵으로 리소스 투입과 기대 성과를 설명.
4. 각 미팅 피드백을 반영해 visual 버전을 업데이트하고, 공통 메시지 충돌을 제거.
5. 확정본을 팀 실행 계획(백로그/마일스톤)과 연결해 바로 실행으로 전환.

| 아키텍처 자료가 과밀함 | 정보 단순화 지시 | `"컴포넌트 8개 이하, 흐름 화살표 명확히"` | 30초 내 핵심 흐름 설명 가능 |

| 로드맵이 희망 일정처럼 보임 | 의존성/리스크 명시 | `"분기별 theme+deliverable+리스크 완화 마일스톤 포함"` | 실행 가능성 질문 감소 |
| 회의 후 실행으로 연결이 안 됨 | visual→백로그 매핑 | `"각 visual 요소를 오너/일정/의존성으로 변환"` | 미연결 항목 0건 달성 |

### 이번 주 완료형 실행 플랜
- **월~화 실행**: architecture diagram, 2x2 matrix, roadmap 3종의 초안을 만들고 각 visual의 핵심 메시지를 1문장으로 고정한다.
- **수~목 실행**: 이해관계자별 버전(CTO/CEO/Board)을 분리해 설명 포인트를 조정하고, 요소 수를 5~10개로 줄여 혼잡도를 개선한다.
- **금요일 검증**: 30초 설명 테스트를 각 이해관계자 관점으로 수행해 질문 발생 지점을 반영하고 최종본을 실행 계획과 매핑한다.
- **Done 기준**: 3종 visual이 이해관계자별로 즉시 발표 가능한 상태이며, 우선순위/의존성/마일스톤이 실행 백로그와 연결되어 있다.
- **리스크/완화**: 시각자료가 장식용으로 흐를 위험은 의사결정 질문(무엇을 결정해야 하는가)을 visual 상단에 명시해 완화하고, 메시지 충돌은 공통 내러티브 시트로 정렬한다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 아키텍처 다이어그램이 복잡해서 읽히지 않음
- 근본 원인: 컴포넌트/라벨 과다
- 해결 방법: 핵심 흐름 기준으로 1차 단순화 후 부가요소는 별도 보조 슬라이드로 분리

2) 블로커: 2x2 배치 기준이 사람마다 다름
- 근본 원인: Impact/Effort 스코어링 기준 미정의
- 해결 방법: 점수 기준표(1~5)와 산정 근거를 사전 합의

3) 블로커: 로드맵이 희망 일정처럼 보임
- 근본 원인: 의존성과 리스크 마일스톤 미표기
- 해결 방법: 각 분기별 선행조건/리스크 완화 마일스톤을 필수 항목화

4) 블로커: 회의 후 실행 연결이 끊김
- 근본 원인: visual과 백로그 담당자 연결 부재
- 해결 방법: visual 요소별 오너/마감일 매핑표를 회의 당일 확정
