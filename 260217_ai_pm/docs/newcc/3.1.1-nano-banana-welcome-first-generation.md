## 1) 목적
- Nano Banana 트랙의 첫 생성 경험을 통해, PM이 **이미지 생성 루프를 스스로 운영**할 수 있게 만든다.
- API key/billing/.env/출력 확인까지 한 번에 끝내서, 이후 모듈에서 품질 개선에만 집중하도록 한다.
- “좋은 이미지 한 장”이 아니라 **재현 가능한 첫 워크플로**를 확보한다.

## 2) 원문 핵심 요약 (ccforpms)
1. 본 레슨은 약 10분 분량이며, **Module 3.0 준비가 완료된 상태**를 전제로 한다.
2. 시작 명령은 `/start-3-1-1`이며, 여기서 Google AI Studio API key 발급 및 billing 활성화를 안내한다.
3. Gemini 3 Pro(Nano Banana)는 텍스트 생성뿐 아니라 **reference photo 변환, 텍스트 오버레이, 스타일/인물 일관성** 작업에 강하다.
4. 실제 생성은 `image_gen.py`를 통해 수행되고, Claude가 `generate()` 호출과 세션/파일 저장을 오케스트레이션한다.
5. 결과 파일은 `outputs/`에 저장되며, 일반적으로 **10~15초 내 생성**된다.
6. `.env` 설정은 자동/수동 모두 가능하고, 수동 시 `.env.example` 복사 후 `GEMINI_API_KEY=...`를 정확히 입력해야 한다.
7. 문제 해결 우선순위는 `.env` 존재 여부 → key 형식(`AIza...`) → billing 반영 지연 확인 → 세션 재시작 순서다.
8. API key는 민감정보이므로 공유/커밋 금지, 로컬 비밀관리 원칙을 지켜야 한다.

### Claude Code 핵심 기능 연결 (official 반영)
- **빠른 첫 성공 루프(Overview)**: 온보딩 직후 바로 생성을 실행해 “설정만 하다 끝나는” 실패를 줄인다.
- **프롬프트 기본기(Tutorial: Getting Started)**: 목적·대상·출력 조건을 짧게 고정하면 첫 생성 성공률이 높아진다.
- **정교한 개선 지시(Tutorial: Claude Opus 4.6)**: “전체 재생성” 대신 “대비만 개선” 같은 국소 수정 지시로 품질을 빠르게 끌어올린다.
- **브라우저 보조 조사(Tutorial: Claude for Chrome)**: 참고 레퍼런스를 빠르게 수집해 첫 결과의 메시지 적합도를 높인다.
- **코워킹형 리뷰(Tutorial: Claude Cowork)**: PM-디자이너 협업 리뷰를 통해 첫 주에 품질 기준을 팀 공통 언어로 맞춘다.
- **보안/거버넌스 내장(Tutorial: Financial Services)**: 키 비공개, 로그 보관, 권한 최소화로 생성 실습을 운영 가능한 프로세스로 전환한다.
- **재현성 중심 운영(Claude Code)**: 실행 순서·오류 복구·출력 규칙을 런북으로 고정해 신규 팀원 재현성을 확보한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 인증/결제 준비 상태 확인
- 입력 아티팩트: `.env.example`, billing 상태 캡처, 계정 권한 정보
- 프롬프트 조각: "첫 생성 전에 인증 실패를 막기 위한 5개 체크 항목을 우선순위로 정리해줘."
- 기대 출력: `first-gen-readiness.md`
- 검증 체크포인트: 체크 항목 중 key 형식, billing, 세션 재시작 포함 여부 확인.

2) 첫 프롬프트 3종 설계
- 입력 아티팩트: 기능 설명 문장, 대상 사용자 페르소나
- 프롬프트 조각: "동일 기능을 설명하는 초보/중간/고급 난이도 프롬프트를 각 1개씩 작성해줘."
- 기대 출력: `prompt-pack-v0.md`
- 검증 체크포인트: 프롬프트마다 목적(무엇을 검증하는지)이 한 줄로 명시된다.

3) `/start-3-1-1` 실행 및 첫 생성
- 입력 아티팩트: 준비된 프롬프트 3개
- 프롬프트 조각: "첫 생성은 구도 안정성을 우선하고, 텍스트 오버레이 공간을 상단 20% 남겨줘."
- 기대 출력: `outputs/first-gen-*.png`
- 검증 체크포인트: 10~15초 내 결과 생성 여부와 저장 위치 확인.

4) 결과 비교 및 선택 기준 적용
- 입력 아티팩트: 생성 결과 3장, 팀 선택 기준표
- 프롬프트 조각: "전달력/브랜드 적합성/재사용성 3기준으로 점수화하고 1순위 안을 추천해줘."
- 기대 출력: `first-gen-review.md`
- 검증 체크포인트: 선택 이유가 감상이 아니라 기준 점수로 제시된다.

5) 개선 생성 1회(iteration)
- 입력 아티팩트: 1순위 안 + 수정 요청 문장
- 프롬프트 조각: "기존 이미지를 유지하고 배경 대비만 높여 CTA 가독성을 개선해줘."
- 기대 출력: `outputs/first-gen-v2.png`
- 검증 체크포인트: 한 번에 변수 1~2개만 변경되었는지 확인.

6) 오류 복구 리허설
- 입력 아티팩트: 의도적 키 오타/세션 만료 상황
- 프롬프트 조각: "오류 발생 시 진단 순서를 key→billing→세션→네트워크 순으로 안내해줘."
- 기대 출력: `error-recovery-log.md`
- 검증 체크포인트: 복구 시간을 10분 이내로 유지.

7) 재현성 패키지 저장
- 입력 아티팩트: 최종 프롬프트, 결과 이미지, 리뷰 메모
- 프롬프트 조각: "다음 팀원이 동일 결과를 재현할 수 있도록 실행 순서와 금지사항을 요약해줘."
- 기대 출력: `first-gen-runbook.md`
- 검증 체크포인트: 다른 팀원이 동일 흐름으로 1회 재생성 성공.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 이 레슨의 완료 기준은 “첫 이미지 생성 성공 + 재생성 1회”로 둔다.
- **P2 맥락 우선**: 프롬프트 품질 이전에 계정/결제/키/세션/출력경로 맥락을 먼저 안정화한다.
- **P4 속도-검증 균형**: 빠르게 생성하되, `outputs/` 결과 확인과 실패 원인 로그를 동시에 남긴다.
- **P6 자동화 운영 기반**: key 설정, 생성, 저장, 재시도를 절차화해 팀 온보딩 시간을 단축한다.
- **P7 Automation vs Augmentation**: 생성 자체는 자동화(Automation), 최종 선택과 메시지 적합성 판단은 PM이 수행(Augmentation).
- **P9 Words-to-Workflow**: “이런 이미지를 만들어줘”라는 자연어가 실제 파일 산출(`outputs/*.png`)로 이어지는 첫 사례를 만든다.

## 4) 실무 적용 시나리오
**상황: 기능 소개 덱 제작을 시작한 PM의 Day 1 세팅**
1. 오전에 `/start-3-1-1` 실행으로 key/billing/.env 세팅을 끝낸다.
2. 기능 설명용 이미지를 자연어로 3개 생성해 결과 차이를 확인한다.
3. `outputs/` 파일명을 `날짜-시나리오-버전` 규칙으로 정리한다.
4. 팀원과 첫 결과를 리뷰해 “다음 프롬프트 수정 기준 3개”를 확정한다.
5. 동일 세션에서 1회 개선 생성 후 3.1.2로 넘어가 변수 제어를 시작한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: `/start-3-1-1` 온보딩을 완료하고 API key, billing, `.env` 설정을 끝내 첫 생성 성공까지 진행한다.
- **수~목 실행**: 동일 주제로 프롬프트 3개를 만들어 결과 차이를 비교하고, 파일명 규칙(날짜-시나리오-버전)으로 `outputs/`를 정리한다.
- **금요일 검증**: 실패 복구 리허설(키 오타, billing 지연, 세션 재시작)을 수행해 팀원이 체크리스트만으로 복구 가능한지 검증한다.
- **Done 기준**: 첫 생성 결과 3장 이상, 설정 점검 로그, 실패 대응 체크리스트가 모두 준비되어 다음 모듈(3.1.2)로 무중단 전환 가능하다.
- **리스크/완화**: 인증 오류 반복 리스크는 점검 순서를 고정(키 형식→billing→세션)하고, 키 유출 리스크는 `.env` 미커밋 규칙과 권한 최소화로 완화한다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 첫 생성이 너무 늦어 실습 집중력이 깨짐
- 근본 원인: 과도한 프롬프트 복잡도 또는 네트워크 부하
- 해결 방법: 프롬프트를 핵심 3요소(주체/행동/분위기)로 축소 후 재시도

2) 블로커: 결과는 나오지만 메시지가 약함
- 근본 원인: 이미지 목적(무엇을 전달할지) 미정의
- 해결 방법: 생성 전 "핵심 메시지 1문장"을 먼저 고정

3) 블로커: 초보자 재현 테스트에서 반복 실패
- 근본 원인: 문서가 숙련자 관점으로 작성되어 오류 분기가 없음
- 해결 방법: 오류 코드별 분기표와 스크린샷 기반 단계 가이드를 추가

4) 블로커: API key 관리 사고 우려
- 근본 원인: 키 공유 방식/보관 위치 표준 부재
- 해결 방법: 개인 키 원칙 + `.env` 미커밋 + 월 1회 키 롤링 정책 적용

## 5) 실습/적용 체크리스트
