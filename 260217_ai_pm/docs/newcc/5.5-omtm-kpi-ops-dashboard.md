## 1) 목적
- OMTM(One Metric That Matters) 중심으로 팀 실행을 정렬하고, 활동 나열형 보고를 제거한다.
- Level1~4 산출물을 동일 KPI 체계로 연결해 운영 의사결정을 빠르게 만든다.
- 자동화/에이전트 도입 효과를 수치로 검증한다.

## 2) 원문 핵심 요약 (ccforpms)
- 원문 company-context는 Activation/Retention/Time-to-Value/Viral Coefficient 같은 핵심 지표를 의사결정 중심에 둔다.
- 원문 2.2는 임팩트 추정과 실험 해석에서 단일값이 아닌 시나리오 기반 판단을 권장한다.
- 원문 2.3은 전략 문서가 실행 메트릭과 연결되어야 한다는 점을 강조한다.
- 원문 4.x는 계획-구현-배포 루프를 반복하며 결과를 검증하는 실전 패턴을 제공한다.
- 즉, KPI 대시보드는 별도 리포트가 아니라 실행 루프의 일부여야 한다.

### Claude Code 핵심 기능 연결 (official 반영)
- Claude Code Overview 원칙에 따라 KPI 운영도 파일/작업 단위로 추적해야 하며, 리포트는 실행 액션으로 이어져야 한다.
- PM은 OMTM 1개를 중심으로 보조 KPI를 연결하고, Claude Code에서 주간 리뷰 템플릿을 고정해 보고 품질을 표준화할 수 있다.
- Tutorials Hub의 실전 적용 패턴처럼, 지표는 “측정”에서 끝나지 않고 우선순위 재배열까지 이어질 때 효과가 난다.
- Opus 4.6 튜토리얼 관점에서 원인 가설 분석은 다중 시나리오 비교 방식이 의사결정 품질을 높인다.
- Chrome/Cowork 튜토리얼 기반 입력 확장을 활용하면 정성 신호(시장/고객 반응)를 KPI 해석에 보강할 수 있다.
- 금융 서비스 튜토리얼의 통제 관점을 반영해 지표 정의/산식/출처를 단일 원본으로 관리하면 감사성과 일관성을 확보할 수 있다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. 월요일 오전, 분기 OMTM 1개를 확정하고 이번 주 변화 목표(예: +3pt)를 수치로 명시한다.
2. 월요일 오후, 보조 KPI 4개(Time-to-Value, Retention, 리드타임, 자동화 성공률)를 OMTM과의 인과 흐름으로 연결한다.
3. 화요일, 각 KPI의 산식/데이터 소스/수집 주기/owner를 지표 카드로 만들고 팀 합의를 받는다.
4. 수요일, n8n으로 자동 집계를 연결해 매주 정해진 시각에 Slack으로 요약 리포트를 발송한다.
5. 목요일, 임계값 경보 테스트를 수행해 이상치 발생 시 실제로 담당자에게 알림이 가는지 검증한다.
6. 금요일 오전, 주간 리뷰에서 OMTM 변화와 원인 가설 2개를 확인하고 즉시 실행 액션 3개를 확정한다.
7. 금요일 오후, 확정된 액션을 백로그에 등록하고 다음 주 우선순위를 KPI 영향도 기준으로 재배열한다.
8. 월말 회고에서 지표 개선이 사용자 체감성과 연결되었는지 교차 점검해 지표 체계를 보정한다.

#### 검증/복구 흐름
1. OMTM이 복수로 제안되면 의사결정이 분산되므로 1개만 남기고 나머지는 보조 KPI로 내린다.
2. 지표 산식 불일치가 발견되면 해당 지표 카드를 잠시 비활성화하고 정의 합의 후 재개한다.
3. 자동 집계 실패 시 그 주는 수동 집계로 전환하되, 실패 원인(권한/스키마/지연)을 당일 기록한다.
4. 경보 오탐이 많으면 임계값을 완화하지 말고 데이터 품질(누락/지연)을 먼저 점검한다.
5. 금요일 리뷰에서 액션이 3개 미만이면 회의를 종료하지 않고 owner/due가 채워질 때까지 보완한다.

### 시나리오: 주간 운영 리뷰
1. 분기 OMTM 1개를 확정한다.
   - 예: Activation Rate
2. 보조지표 4개를 묶는다.
   - Time-to-Value, Retention, PRD→티켓 리드타임, 자동화 성공률
3. 매주 월요일 n8n이 데이터 수집/집계 후 Slack에 자동 공유한다.
4. 리뷰 미팅에서 다음 항목을 고정 순서로 점검한다.
   - OMTM 변화
   - 원인 가설 2개
   - 즉시 실행 액션 3개
5. 금요일에 실행 결과를 업데이트하고 다음 주 우선순위를 재배치한다.

### 실패 패턴
- 실패1: 지표가 많아 핵심이 흐려짐
  - 대응: OMTM 1개 + 보조지표 3~5개 상한
- 실패2: 데이터는 있으나 액션이 없음
  - 대응: 모든 지표 카드에 “owner + next action + due” 필수
- 실패3: 수집 지연으로 의사결정 타이밍 상실
  - 대응: 자동 수집 실패 알림 + 수동 백업 폼
- 실패4: 수치 개선이 체감성과와 연결 안 됨
  - 대응: KPI와 사용자 영향(불편 감소/전환 개선) 동시 기록

### 이번 주 완료형 실행 플랜
- **월~화:** 분기 OMTM 1개와 보조지표 3~5개를 확정하고 지표 정의서(산식/주기/owner)를 작성한다.
- **수~목:** n8n 기반 자동 집계와 Slack 공유를 연결하고 경보 임계값 테스트를 1회 수행한다.
- **금 검증:** 주간 리뷰에서 OMTM 변화, 원인 가설 2개, 다음 액션 3개를 실제로 도출해 백로그에 반영한다.
- **Done 기준:** 대시보드 운영본, 임계값/경보 규칙, 주간 리뷰 노트, 백로그 반영 링크가 준비된다.
- **리스크·완화:** 지표 해석 불일치로 논쟁이 길어질 수 있으므로 지표 사전(정의/산식/제외 조건)을 리뷰 전에 합의한다.

### 자주 막히는 지점과 해결 가이드
- **막힘 1: 지표는 많은데 우선순위 결론이 안 남**
  - 증상: 리뷰 시간이 길어지고 실행 액션이 분산된다.
  - 해결: OMTM 영향도가 낮은 지표는 보조 보드로 이동하고 메인 보드는 5개 이내로 유지한다.
- **막힘 2: 지표 정의가 팀마다 달라 비교 불가**
  - 증상: 같은 KPI 수치가 보고서마다 다르다.
  - 해결: 지표 사전을 단일 원본으로 운영하고 변경 시 변경 이력/승인자를 남긴다.
- **막힘 3: 경보 피로로 알림을 무시함**
  - 증상: 경보가 자주 울려도 실제 대응이 없다.
  - 해결: 경보를 P1/P2/P3로 분리하고 P1만 즉시 대응, 나머지는 일괄 리뷰로 처리한다.
- **막힘 4: KPI 개선과 사용자 체감이 분리됨**
  - 증상: 숫자는 좋아졌지만 고객 불만이 줄지 않는다.
  - 해결: KPI 카드마다 사용자 영향 지표(불편 티켓, NPS 코멘트 등)를 함께 연결해 해석한다.
