## 1) 목적
- 출시 직전 필요한 마케팅 자산(App Store/소셜 광고/런칭 공지)을 **짧은 시간 안에 패키지로 완성**한다.
- 단일 이미지 제작이 아니라, 채널별 A/B 테스트 가능한 **운영형 자산 세트**를 구축한다.
- 브랜드 일관성과 전환 성능을 동시에 확보하는 제작 기준을 정립한다.

## 2) 원문 핵심 요약 (ccforpms)
1. Nano Banana 마지막 모듈로, 런칭 실무에서 가장 자주 쓰는 그래픽 3종을 다룬다.
2. App Store/Play Store는 규격이 달라서, 처음부터 플랫폼 기준 dimensions로 프롬프트를 작성해야 한다.
3. 스토어 자산은 device frame, 실제 UI, 사용 맥락, 간결한 배경이 핵심이다.
4. Social ad는 피드 주목도 확보가 중요하므로 강한 대비·명확한 가치 메시지·텍스트 여백 설계가 필요하다.
5. 광고 콘셉트는 문제→해결, 제품 중심, 사용자 맥락, 수치 강조, 후기형 등으로 목적에 맞게 선택한다.
6. Launch announcement는 product hero 중심 + 출시 이벤트 무드 + 브랜드 일관성을 동시에 반영해야 한다.
7. 실무에서는 variant를 여러 개 만들고 A/B 테스트 가능하게 구조화해야 성과 개선이 가능하다.
8. `Gemini 3 Pro + Claude Code + style library` 조합으로 속도와 품질을 누적 개선하는 운영이 권장된다.

### Claude Code 핵심 기능 연결 (official 반영)
- **출시 실행 중심 워크플로(Overview)**: 메시지 정의부터 채널별 자산 생성·검증까지 한 흐름으로 운영한다.
- **명확한 요청 구조(Tutorial: Getting Started)**: 채널, 타깃, KPI, CTA를 먼저 고정해 변형안의 실험 가치를 높인다.
- **고급 카피-비주얼 정합(Tutorial: Claude Opus 4.6)**: 메시지·톤·규격 조건을 동시에 반영해 런칭 자산의 완성도를 높인다.
- **웹 기반 레퍼런스 수집(Tutorial: Claude for Chrome)**: 경쟁 캠페인/스토어 사례를 빠르게 참고해 초기 실패를 줄인다.
- **협업형 런칭 준비(Tutorial: Claude Cowork)**: PM, 마케터, 브랜드 담당자가 동일 자산 세트를 기준으로 빠르게 합의한다.
- **통제 가능한 운영(Tutorial: Financial Services)**: 게시 전 체크리스트, 교체 임계값, 책임자 기록으로 대응 속도를 높인다.
- **반복 최적화 루프(Claude Code)**: 48시간 성과를 기준으로 카피/비주얼 교체 우선순위를 즉시 실행한다.

### ccforpms 단계별 실행 예시 (실전 재현)
1) 런칭 메시지와 KPI 고정
- 입력 아티팩트: `launch-brief.md`, 타깃 세그먼트, 기존 캠페인 성과
- 프롬프트 조각: "문제/해결/차별점 메시지를 각 1문장으로 확정하고 채널 KPI(CTR/CVR)를 매핑해줘."
- 기대 출력: `launch-message-kpi-map.md`
- 검증 체크포인트: 메시지마다 측정 KPI가 연결된다.

2) 채널 규격표 작성
- 입력 아티팩트: App Store/Play Store/Meta/LinkedIn 규격 문서
- 프롬프트 조각: "채널별 사이즈, 안전여백, 텍스트 제한을 한 표로 정리해줘."
- 기대 출력: `channel-spec-sheet.md`
- 검증 체크포인트: 규격 누락 없이 제작 전 체크 가능하다.

3) 스토어 자산 생성
- 입력 아티팩트: 제품 UI 캡처, 핵심 가치 문장
- 프롬프트 조각: "기기 프레임 + 실제 UI + 간결 배경으로 스토어 스크린샷 세트를 만들어줘."
- 기대 출력: `outputs/store-assets-*.png`
- 검증 체크포인트: 플랫폼 규격과 가독성 기준을 통과한다.

4) 소셜 광고 variants 제작
- 입력 아티팩트: 광고 콘셉트 3종, 카피 2종
- 프롬프트 조각: "3콘셉트 × 2카피로 최소 6개 변형안을 만들고 CTA 영역을 확보해줘."
- 기대 출력: `outputs/social-ads-*.png`
- 검증 체크포인트: 변형 간 차별점이 명확하고 A/B 실험 가능하다.

5) 런칭 공지 hero 제작
- 입력 아티팩트: 브랜드 키비주얼, 런칭 날짜/이벤트 정보
- 프롬프트 조각: "보도자료/뉴스레터/링크드인 공용으로 사용할 hero 이미지를 만들어줘."
- 기대 출력: `outputs/launch-hero-*.png`
- 검증 체크포인트: 멀티 채널 크롭 시 주요 요소가 손실되지 않는다.

6) 사전 리뷰 및 게시 준비
- 입력 아티팩트: 자산 패키지 전체, 리뷰 체크리스트
- 프롬프트 조각: "CTA 명확성, 브랜드 톤, 텍스트 가독성 기준으로 사전 리뷰표를 작성해줘."
- 기대 출력: `preflight-review.md`
- 검증 체크포인트: 게시 전 Fail 항목이 0개가 된다.

7) 48시간 성과 모니터링 및 교체 플랜
- 입력 아티팩트: 게시 후 초기 성과 데이터
- 프롬프트 조각: "48시간 내 KPI 미달 시 교체할 카피/비주얼 우선순위를 정해줘."
- 기대 출력: `launch-48h-optimization-plan.md`
- 검증 체크포인트: KPI 미달 조건과 교체 트리거가 수치로 정의된다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P1 실행 우선**: 설명서보다 “오늘 바로 배포 가능한 에셋팩”을 산출물 기준으로 둔다.
- **P2 맥락 우선**: 채널/타깃/CTA/규격 맥락을 프롬프트에 포함해 전환 가능성을 높인다.
- **P4 속도-검증 균형**: 빠른 초안 생성 후 A/B 실험 설계로 검증 루프를 닫는다.
- **P6 자동화 운영 기반**: 런칭 때마다 반복되는 제작을 템플릿화해 운영 부하를 낮춘다.
- **P8 Connected Hub**: 시각 자산 생성 → 채널 게시 → 성과 리뷰를 한 루프로 연결한다.
- **P9 Words-to-Workflow**: 자연어 캠페인 기획을 규격화된 채널 자산으로 즉시 변환한다.
- v2.4의 OMTM 관점에서 자산 평가는 미적 품질만이 아니라 CTR/전환/재방문 지표로 이어져야 한다.

## 4) 실무 적용 시나리오
**상황: 신규 서비스 오픈 D-2 마케팅 스프린트**
1. 핵심 메시지 3개(문제/해결/차별점)를 먼저 확정한다.
2. App Store용 feature graphic과 스크린샷 스타일을 생성한다.
3. 소셜 광고는 3콘셉트 × 2카피로 variant를 만든다.
4. 런칭 공지용 hero 이미지는 보도자료/뉴스레터/링크드인 공통 사용을 전제로 만든다.
5. style ID를 고정해 전체 자산의 톤을 통일한다.
6. 게시 후 48시간 CTR/클릭후전환률을 보고 다음 실험안을 즉시 업데이트한다.

| 게시 후 대응이 느림 | KPI 트리거 기반 최적화 | `"48시간 CTR/CVR 미달 시 교체 우선순위와 담당자 지정"` | 교체 실행 리드타임 단축 |

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 출시 메시지 3개(문제, 해결, 차별점)와 채널 규격(App Store, 소셜, 런칭 공지)을 확정하고 에셋 브리프를 작성한다.
- **수~목 실행**: 스토어/광고/공지 자산을 채널별로 생성하고 광고는 3콘셉트 × 2카피 이상으로 variant를 구성해 A/B 테스트안을 만든다.
- **금요일 검증**: 내부 사전 리뷰에서 CTA 명확성, 텍스트 가독성, 브랜드 톤 일관성을 점검해 게시 가능한 최종 패키지를 확정한다.
- **Done 기준**: 채널별 런칭 에셋 패키지와 A/B 테스트 설계안이 준비되고, 게시 후 48시간 성과 수집 계획까지 세팅되어 있다.
- **리스크/완화**: 규격 미스매치 리스크는 채널별 사이즈 체크리스트로 방지하고, 성과 부진 리스크는 첫 48시간 내 카피/비주얼 교체 기준을 사전에 정의해 완화한다.

- 운영 팁 1: 월요일 메시지 고정 없이 에셋 제작부터 시작하면 A/B 테스트가 의미를 잃는다.
- 운영 팁 2: 화요일 규격 검수는 반드시 배포 채널 담당자가 직접 확인한다.
- 운영 팁 3: 금요일 게시 전에는 교체 플랜을 먼저 잠가야 주말 대응 속도가 빨라진다.

### 자주 막히는 지점과 해결 가이드
1) 블로커: 채널 업로드에서 규격 오류 발생
- 근본 원인: 플랫폼별 사이즈/안전영역 차이 미반영
- 해결 방법: 채널별 규격표를 제작 시작 전 체크리스트로 의무 확인

2) 블로커: 광고 변형안이 많아도 성과 개선이 없음
- 근본 원인: 변형 축이 카피/비주얼로 명확히 분리되지 않음
- 해결 방법: 실험당 변수 1개 원칙(카피만 또는 비주얼만 변경) 적용

3) 블로커: 런칭 hero는 예쁜데 CTA가 안 보임
- 근본 원인: 텍스트 여백/대비 설계 부족
- 해결 방법: 생성 시 CTA 영역과 대비 기준을 프롬프트에 명시

4) 블로커: 게시 후 성과 부진 대응이 늦음
- 근본 원인: KPI 미달 임계값과 교체 책임자 미정
- 해결 방법: 48시간 임계값(CTR/CVR)과 교체 담당자/SLA를 사전 지정

## 5) 실습/적용 체크리스트
- [ ] 채널별 규격(App Store/소셜/공지)을 표로 정리했다.
- [ ] 각 채널 자산에 핵심 메시지 1개와 CTA를 연결했다.
- [ ] 광고 변형안(최소 3개)과 A/B 테스트 가설을 만들었다.
- [ ] 텍스트 오버레이 여백을 고려한 레이아웃으로 생성했다.
- [ ] style library 재사용으로 브랜드 톤을 통일했다.
- [ ] 게시 후 성과 지표(CTR, 전환, 반응률) 수집 계획을 수립했다.

## 6) 산출물
