## 2) 원문 핵심 요약 (ccforpms)
- TaskFlow는 코스 실습용 가상 회사이며, 원문 기준으로 2021년 설립·Series B·약 50명 조직 맥락을 갖는다.
- 핵심 비즈니스 지표는 가입자 50,000 / MAU 10,000 / DAU 2,500 / ARR $2.5M / MRR $250k / 유료 팀 500개다.
- 성장률은 사용자 MoM 15%, 매출 MoM 20%로 양호하지만, **Net Retention 65%**와 **Activation 45%**가 구조적 병목으로 제시된다.
- ICP는 Series A~C(50~500명) 원격/하이브리드 팀이며, 다음 확장 타깃은 Enterprise(500+)다.
- 2025 우선순위는 ① 모바일(iOS/Android) ② Enterprise readiness(SSO/권한/audit log) ③ 온보딩 개선으로 Activation 60% 달성이다.
- Senior PM의 핵심 KPI는 Activation, Time-to-Value, Viral Coefficient이며, 온보딩·핵심 task flow·알림·연동이 책임 범위다.

### ccforpms 단계별 실행 예시 (실전 재현)
1. **기준선 KPI를 먼저 잠근다.**
   - 입력 예시: "Activation 45%, Net Retention 65%, OMTM은 Activation".
   - 실행 액션: 데이터 소스와 계산식을 함께 명시해 수치 해석을 통일한다.
   - 기대 출력: 팀 간 숫자 논쟁 없이 같은 기준으로 의사결정 가능.
2. **목표 격차를 수치로 명확히 적는다.**
   - 입력 예시: "Activation 45→60(+15pp), Net Retention 65→75(+10pp)".
   - 실행 액션: 현재값/목표값/갭을 한 표로 정리한다.
   - 기대 출력: 우선순위 선정 기준이 명확해짐.
3. **갭 원인을 가설로 분해한다.**
   - 입력 예시: "온보딩 복잡도, 첫 가치 체감 지연, 협업 초대 전환 저조".
   - 실행 액션: 각 원인당 실험 가설 2개씩 도출한다.
   - 기대 출력: 실행 가능한 실험 백로그 초안.
4. **가설을 PRD 단위로 변환한다.**
   - 입력 예시: `/prd` 기반 "온보딩 단계 축소 실험".
   - 실행 액션: 문제·대상·성공지표·범위 제외를 문서화한다.
   - 기대 출력: 주간 실행 가능한 PRD 1~2건.
5. **실행 경로를 팀 도구와 연결한다.**
   - 입력 예시: Notion 게시→Linear 티켓→Slack 상태 공유.
   - 실행 액션: 동일 ID로 문서-티켓-상태보고를 연결한다.
   - 기대 출력: 진행 추적 누락 없는 운영 체계.
6. **주간 리뷰에서 숫자 기반으로 유지/중단을 판정한다.**
   - 입력 예시: "가설 A는 완료율 +8%, 가설 B는 변화 없음".
   - 실행 액션: 실험 결과를 OMTM 변화와 함께 기록한다.
   - 기대 출력: 다음 주 우선순위 재정렬 근거 확보.
7. **결정 로그를 남겨 재논쟁을 줄인다.**
   - 입력 예시: "이번 주는 온보딩 유지, Enterprise 기능은 차주".
   - 실행 액션: 결정/보류/폐기 사유를 표준 포맷으로 기록한다.
   - 기대 출력: 팀이 같은 맥락을 이어서 의사결정 가능.

### Claude Code 핵심 기능 연결 (official 반영)
- **컨텍스트 기반 작업 시작**: 회사 KPI·용어·우선순위를 먼저 반영해 PM 실행 문서의 기준선을 통일한다.
- **Plan 모드로 분기 실행 설계**: OMTM 중심으로 주간 과제를 구조화해 산발적 실행을 줄인다.
- **자연어 기반 전략→실행 변환**: "Activation 45→60" 같은 목표를 PRD/티켓 수준 작업으로 바로 연결한다.
- **문맥 유지형 협업 루프**: 개요 문서에서 정의한 기준을 Build/GitHub/배포 단계까지 일관되게 전달한다.
- **결정 로그 추적성 강화**: 변경 이유와 보류 사유를 남겨 다음 주 의사결정 리드타임을 단축한다.
- **도구 연계 운영(문서-이슈-상태)**: 문서와 실행 도구를 연결해 진행 추적 누락을 줄인다.

## 3) 이든 철학 반영 포인트 (v2.4)
- **P2 Context-rich**: 회사 개요 문서를 “읽는 자료”가 아니라 `/prd`, `/status`, `/today` 입력 컨텍스트로 강제한다.
- **P3 PM 오케스트레이션**: PM은 문서 작성자가 아니라 “문제 정의(Activation 45%) → 실행 우선순위(모바일/Enterprise/온보딩) → 검증 루프” 설계자다.
- **P8 Connected Hub**: Claude.ai(문제 구조화) + Claude Code(실행 문서화) + n8n(주간 자동 리포트)로 동일 지표를 연결한다.
- **FR-020 / M-014~M-017 정합**: OMTM 기반 운영, 리드타임 단축, 자동화 실행률(95% 목표)을 회사 맥락에서 시작한다.
- **Automation vs Augmentation(P7)**: 숫자 집계·분류는 자동화하고, 전략 선택(무엇을 포기할지)은 PM이 직접 수행한다.

## 4) 실무 적용 시나리오
### 시나리오: 분기 시작 주간 우선순위 정렬
1. `company-context-overview.md`에서 현재 지표와 목표 격차를 확정한다.
   - 예: Activation 45% → 60%(+15pp), Net Retention 65% → 75%(+10pp)
2. Claude.ai에서 “갭 원인 3개 + 각 원인별 실험 가설 2개”를 도출한다.
3. Claude Code에서 `/prd`로 각 가설을 PRD 초안으로 변환한다.
4. 5.1 Level1 워크플로우로 Notion 게시 → Linear 티켓화 → Slack 상태 공유를 당일 완료한다.
5. 주간 회고에서 OMTM 변화와 실험 리드타임을 기록하고, 다음 주 우선순위를 갱신한다.

### 이번 주 완료형 실행 플랜
- **월~화 실행**: 회사 기준선 KPI를 최신 데이터로 갱신하고, 이번 주 OMTM 1개와 목표 격차를 숫자로 확정한다.
- **수~목 실행**: 우선순위 3개(모바일/Enterprise/온보딩)에 대해 실행 과제와 담당자를 매핑하고 상태 리포트 템플릿을 배포한다.
- **금요일 검증**: OMTM 변동, 실험 진행률, 지연 원인을 리뷰해 다음 주 유지/중단/강화 결정을 내린다.
- **Done 기준**: KPI 기준선 표, 우선순위 매핑표, 주간 결정 로그가 같은 버전으로 정렬되어 팀이 동일 수치를 참조한다.
- **리스크/완화**: 팀별 지표 정의가 다르면 금요일 회의 전 공통 정의집(용어/계산식/데이터 소스)을 잠가 해석 차이를 제거한다.

### 실무 적용 시나리오 (초보자용 상세 실행안)
**대상 상황**: 분기 첫 주, PM이 회사 맥락 문서를 기반으로 실행 우선순위를 재정렬해야 한다.

- **월요일**
  - Owner: Senior PM, 데이터 담당
  - Action: KPI 기준선(Activation/Retention/ARR/OMTM) 최신화
  - Artifact: 기준선 표(현재값/목표값/갭), 데이터 소스 메모
  - Check: 팀별 보고 숫자와 기준선 표의 수치 일치
- **화요일**
  - Owner: PM
  - Action: 격차 원인 3개 정의, 원인별 실험 가설 도출
  - Artifact: 가설 백로그, 우선순위 점수표
  - Check: 각 가설이 KPI와 1:1 연결되는지 확인
- **수요일**
  - Owner: PM, 개발 리드
  - Action: 상위 가설 1~2개를 PRD로 변환, 범위/제외 항목 확정
  - Artifact: PRD 초안, 범위 제외 리스트
  - Check: 구현 가능성 검토(이번 주 착수 가능 여부)
- **목요일**
  - Owner: PMO, 운영 담당
  - Action: Notion-Linear-Slack에 실행 상태 동기화
  - Artifact: 연결 링크 목록, 상태 보고 템플릿
  - Check: 문서/티켓/상태보고 ID가 동일한지 점검
- **금요일**
  - Owner: Senior PM, 리더십
  - Action: 주간 리뷰에서 유지/중단/강화 결정
  - Artifact: 결정 로그, 차주 실행 계획
  - Check: OMTM 변화와 실험 리드타임이 보고서에 반영됨

**리스크 트리거(발동 조건)**
- 금요일 리뷰 시점에 동일 KPI 값이 팀별로 다르게 보고되거나, 가설-지표 연결이 50% 미만이면 컨텍스트 붕괴 위험으로 간주한다.

**Rollback/Mitigation 흐름**
1. 새 실험 착수를 잠시 중단하고 KPI 정의집을 우선 고정한다.
2. 데이터 소스를 단일 원천으로 통일하고 기존 보고서를 재계산한다.
3. 가설 백로그를 재점검해 KPI와 연결되지 않는 항목을 보류한다.
4. 다음 주 계획은 "지표 정합성 확보"를 최우선 목표로 재편한다.

### 자주 막히는 지점과 해결 가이드
1. **막힘: 같은 Activation인데 팀마다 계산 방식이 다름**
   - 해결: 분자/분모/기간을 명시한 계산식을 문서 상단에 고정한다.
   - 확인: 주간 보고서에서 동일 계산식 사용 여부를 체크한다.
2. **막힘: 우선순위 논의가 감각 중심으로 길어짐**
   - 해결: 목표 격차(갭)와 예상 임팩트 점수로만 1차 정렬하고, 감각 의견은 2차 참고로 제한한다.
   - 확인: 상위 3개 과제가 모두 KPI와 연결돼 있는지 점검한다.
3. **막힘: 문서는 있는데 티켓/상태 공유가 분리되어 추적이 안 됨**
   - 해결: 문서-티켓-상태보고에 공통 ID를 부여하고 링크를 상호 연결한다.
   - 확인: 어떤 티켓이 어떤 PRD를 근거로 하는지 30초 내 찾을 수 있어야 한다.
4. **막힘: 결정 사유가 남지 않아 다음 주에 같은 논쟁 반복**
   - 해결: 유지/중단/보류의 근거를 한 줄로라도 결정 로그에 강제 기록한다.
   - 확인: 차주 회의에서 이전 결정을 문서만으로 재현 설명 가능한지 검증한다.
