## 0. 오프닝 (00:00~03:00)

안녕하세요. 오늘 제목은 **협업의 기술, The Art of Collaboration**입니다.

먼저 한 문장으로 시작하겠습니다.

> "코드를 짜는 것과 제품을 만드는 것은 다릅니다."

AI가 코드를 대신 써주는 시대에, 개발자의 가치는 줄어드는 게 아니라 **이동**합니다.
어디로 이동하냐면,
- 무엇을 만들어야 하는지 판단하는 능력,
- 그리고 함께 일해서 결과를 끝까지 내는 능력으로 이동합니다.

오늘 발표는 협업 예절 이야기나 동기부여 강의가 아닙니다.
현업에서 실제로 팀이 어떻게 굴러가는지,
그리고 여러분이 지금 하고 있는 프로젝트 경험을 어떻게 **현업 언어**로 번역할지에 대한 이야기입니다.

오늘 60분 동안 드릴 핵심은 단 하나입니다.

**"여러분의 프로젝트를 과제가 아니라 제품으로 다루기 시작하는 순간, 커리어의 속도가 달라진다."**

전환하겠습니다. 왜 지금 협업 이야기를 하는지부터 보겠습니다.

## 1. 들어가며: 왜 지금 협업인가 (03:00~07:00)

여러분이 부트캠프를 마치고 현업에 들어가면 가장 먼저 만나는 벽은 기술이 아니라 사람입니다.

원문에서 제시한 숫자를 그대로 보겠습니다.
- 프로젝트 실패 원인의 86%는 기술 부족이 아니라 팀워크·커뮤니케이션 문제
- 잘못된 소통으로 인한 손실: 직원 1인당 연간 약 1,650만 원
- 전체 회의 중 72%가 비생산적
- 반대로 소통이 좋은 조직은 성과가 3.5배

이 숫자를 보면 명확합니다.
협업은 "있으면 좋은 것"이 아니라 **생산성 인프라**입니다.

여러분이 이미 겪은 문제들,
- Git 충돌,
- 역할 분담의 애매함,
- 기한 압박,
- 기여도 불균형,
이건 현업에서 없어지지 않습니다.
다만 현업 팀은 그걸 개인 노력으로 버티지 않고,
**시스템과 문화**로 처리합니다.

오늘은 그 시스템과 문화의 핵심을 10개 챕터 순서대로 보겠습니다.

## 2. Chapter 1 — 팀의 모양이 바뀌고 있다 (07:00~13:00)

질문부터 시작하겠습니다.
"현업 개발팀은 몇 명이 모여서 일하나요?"

많은 분이 10명, 20명, 30명을 상상합니다.
하지만 AI 네이티브 기업 사례를 보면 훨씬 작습니다.

### 2-1) Linear 사례
Linear는 직원 약 50명으로 20,000개 이상 기업에 서비스를 제공합니다.
놀라운 포인트는 전담 PM(Product Manager, 제품 관리자)이 거의 없다는 점입니다.
기능 단위로 디자이너 1명 + 엔지니어 2명이 붙고,
프로젝트가 끝나면 팀이 해체됩니다.
즉, 영구 조직보다 **프로젝트 단위 유동 조직**입니다.

### 2-2) Anthropic 사례
Anthropic은 연구자와 엔지니어를 강하게 분리하지 않습니다.
2~4명 소규모 팀이 문제를 해결하고,
전통적 프로덕트 관리보다 빠른 프로토타이핑을 중시합니다.

### 2-3) Vercel 사례
Vercel은 Design Engineer(디자인 엔지니어)라는 역할을 전면에 둡니다.
디자인과 프론트엔드를 분리하지 않고, 제품 전달 책임으로 통합합니다.

### 2-4) 조직 트렌드 정리
McKinsey가 말한 Agentic Organization(에이전틱 조직)은
2~5명의 인간이 50~100개의 AI 에이전트를 관리하는 구조입니다.
Marty Cagan도 AI 도구로 인해 8명 팀이 5~6명으로 축소될 가능성을 말합니다.

여기서 주니어가 불리해지느냐?
원문은 정반대라고 말합니다.
AWS CEO Matt Garman의 문장을 인용하겠습니다.

> "주니어를 뽑지 않겠다는 건 제가 들어본 것 중 가장 어리석은 말입니다."

GitHub 데이터에서도 주니어 생산성 향상 폭(21~40%)이 시니어(7~16%)보다 큽니다.

핵심 정리하겠습니다.
**팀이 작아질수록 한 사람이 맡는 영역은 넓어집니다.**
"나는 프론트엔드만"이 아니라,
UX 설계, API 연결, 배포까지 연결 가능한 사람이 필요해집니다.

여러분의 Supabase 연동, Vercel 배포 경험은 이미 이 방향에 맞아 있습니다.
면접에서는 이렇게 말하면 됩니다.

> "작고 자율적인 팀에서 제품 전체를 보며 일하고 싶습니다."

## 3. Chapter 2 — 스크럼은 죽었는가? (13:00~18:00)

원문의 문제 제기는 명확합니다.
많은 팀이 아직 스크럼을 쓰지만, Agile Theater(애자일 연극)가 늘고 있습니다.
형식은 있는데 실질이 없는 상태죠.

### 3-1) Shape Up
Shape Up은 질문을 바꿉니다.
"이 기능에 얼마나 걸리나요?"가 아니라
"이 기능에 얼마의 시간을 투자할 가치가 있나요?"

시간은 고정, 범위는 가변입니다.
6주 안에 안 끝날 것 같으면 범위를 줄입니다.
그리고 과감하게 말합니다.
"백로그가 없다."
선택받지 못한 아이디어는 버리고, 정말 중요하면 다시 떠오른다는 철학입니다.

여러분 프로젝트에 대입하면,
11페이지를 얕게 만드는 것보다 핵심 3페이지를 완성하는 전략이 더 강합니다.

### 3-2) Linear Method
Linear Method의 3원칙:
1) 모멘텀을 만들어라, 스프린트하지 마라
2) 바쁜 일에 No라고 하라
3) 80% 정보로 결정하고 전진하라

그리고 모든 이슈에 오너 1명을 둡니다.

### 3-3) Continuous Discovery
Teresa Torres의 접근은 유저 리서치를 이벤트가 아니라 습관으로 봅니다.
PM+디자이너+엔지니어 트리오가 매주 유저를 만나고,
기회를 발견하고, 솔루션을 테스트합니다.
원문 기준으로 채택률 35% 개선 데이터가 제시됩니다.

### 3-4) 실전 적용 3줄
- 범위를 줄여라
- 오너를 정해라
- 완벽한 계획보다 빠른 반복을 선택해라

## 4. Chapter 3 — AI 도구의 현실: 마법이 아니라 도구 (18:00~25:00)

이 파트는 과장도 비관도 아닌, **현실 균형**이 핵심입니다.

### 4-1) GitHub Copilot
- 누적 사용자 2,000만+
- 활발 사용자 코드의 46% 생성
- 속도 향상 최대 55%

하지만 동시에,
- AI 결과물 신뢰: 33%
- 주요 이슈 1.7배
- 보안 취약점 2.74배

즉, 생성 속도와 신뢰도는 별개입니다.

### 4-2) Cursor
18개월 만에 ARR 6,500억 원 규모.
Background Agent, BugBot, 이슈 연동 자동화 등으로
개인 생산성은 크게 올라갑니다.

### 4-3) v0
v0는 프로토타이핑 도구에서 프로덕션 워크플로로 확장됐습니다.
여기서 원문의 중요한 문장:
"우리가 하는 일의 90%는 기존 코드 수정이다."

주니어에게 매우 중요합니다.
현업은 빈 화면에서 새로 만드는 일보다,
기존 맥락을 읽고 안전하게 바꾸는 일이 훨씬 많습니다.

### 4-4) Claude + MCP
MCP(Model Context Protocol)는
AI와 외부 도구/데이터/서비스 연결을 표준화하는 흐름입니다.
디자인 컨텍스트와 코드 워크플로를 직접 연결하는 기반이 됩니다.

### 4-5) 숫자의 이면
원문에서 가장 중요한 역설:
- AI 많이 쓰는 팀은 완료 작업 21% 증가
- 그런데 PR 리뷰 시간 91% 증가, 컨텍스트 전환 47% 증가
- 일부 무작위 대조 실험에서는 작업 시간 19% 증가

정리하겠습니다.
**AI는 쓰는 시간을 줄여주지만, 검증하는 사고 비용은 줄여주지 않는다.**

### 4-6) 주니어를 위한 4원칙
1) 튜터로 쓰고 자동조종으로 쓰지 말 것
2) AI 코드를 주니어 동료 코드처럼 리뷰할 것
3) 주 2~3회 AI 없는 연습 시간 확보
4) 컨텍스트를 잘 주는 프롬프트 습관화

## 5. Chapter 4 — 바이브 코딩에서 에이전틱 엔지니어링으로 (25:00~30:00)

바이브 코딩은 빠르게 확산됐고, 빠르게 부작용도 드러났습니다.

핵심 구분은 Simon Willison의 문장으로 정리됩니다.

> LLM이 모든 코드를 썼더라도, 내가 리뷰·테스트·이해했다면 그건 바이브 코딩이 아니다.

즉, 차이는 타이핑 주체가 아니라 **책임 주체**입니다.

Karpathy도 이후에는 Agentic Engineering,
즉 인간이 에이전트를 오케스트레이션하고 감독하는 방향을 강조합니다.

원문의 핵심 문장으로 정리하겠습니다.

> "빠른 것은 AI의 일, 좋은 것은 여러분의 일."

실천 항목 3개:
- Accept All 이전에 이해
- 왜를 묻는 습관
- 에러를 스토리로 축적

## 6. Chapter 5 — 코딩 속도보다 프로덕트 씽킹 (30:00~35:00)

원문은 강하게 말합니다.
AI 우선 시대에 남는 차별화는 제품 판단력입니다.

코드의 상당 부분이 자동화되면,
개발자의 핵심 가치는 다음으로 이동합니다.
- 무엇을 만들지
- 왜 지금 만들지
- 어떤 순서로 만들지

프론트/백 경계도 실무에서는 점점 얇아집니다.
냠냠박스 예시처럼 서버 컴포넌트와 클라이언트 인터랙션을 함께 다루는 흐름이 일반화됩니다.

원문의 4개 핵심 역량:
1) 시스템 씽킹
2) UX 감각(a11y 포함)
3) 비즈니스 임팩트 인식
4) 연결 역량

결국 가장 가치 있는 엔지니어는
분절된 역할을 수행하는 사람이 아니라,
흐름을 연결해 결과를 만드는 사람입니다.

## 7. Chapter 6 — 하이퍼포밍 팀은 이렇게 일한다 (35:00~41:00)

이 챕터는 실무 운영 매뉴얼에 가깝습니다.

### 7-1) 도구 스택
- Slack
- Linear/GitHub Projects
- Notion
- Figma
- GitHub

핵심은 도구 자체가 아니라 운영 규칙입니다.

### 7-2) Async-First(비동기 우선)
원문의 4원칙을 그대로 사용하세요.
1) 상태 보고는 글로
2) 브레인스토밍은 미팅 전에
3) 실시간 시간은 관계/복잡한 의사결정에
4) 모든 결정에 DRI

### 7-3) Git 워크플로우
작은 팀은 GitHub Flow면 충분합니다.
그리고 이 4개를 지키면 품질이 올라갑니다.
- PR 작게
- 리뷰 24시간 내
- PR 템플릿 사용
- 의미 있는 커밋 메시지

### 7-4) RFC 문화
기술 결정을 문서로 남기면,
"왜 이렇게 했지?"를 나중에도 추적할 수 있습니다.
그리고 목소리 큰 사람 중심이 아니라,
근거 중심 토론이 가능합니다.

### 7-5) 디자이너-개발자 협업
원문의 실천 포인트:
- 공통 언어 만들기 (margin/padding 등)
- 디자인 크리틱에 개발자 초대
- 코드 리뷰에 디자이너 초대
- 설득보다 이해로 대화 시작

## 8. Chapter 7 — 기술 성과를 설득력 있게 발표하는 법 (41:00~45:00)

### 8-1) PSI

### 8-2) Working Backwards
기술 스택 소개로 시작하지 말고,

컨설팅 펌에서 쓰는 이유가 있습니다.

### 8-4) 미완성 발표 문장 교정
"못 만들었습니다" 대신
"MVP 우선순위로 전략적 선택을 했고, 다음 반복 로드맵이 있습니다."

면접관이 보는 건 완성도 100%보다
우선순위 판단 능력입니다.

## 9. Chapter 8 — README를 제품 소개서로 (45:00~49:00)

원문 메시지 그대로입니다.
README가 템플릿 수준이면 0.5초 만에 넘어갑니다.

README 6요소:
1) 한 줄 소개
2) 핵심 플로우 이미지/GIF
3) 기술 선택 이유
4) 내가 맡은 역할
5) 트러블슈팅
6) 회고/배운 점

특히 "전체적으로 도왔어요"는 약합니다.
기능 단위 책임 문장으로 바꿔야 합니다.
예: "레시피 작성 폼 인터랙션을 기획부터 구현까지 담당"

포트폴리오 차별화 3전략:
- 부트캠프 프로젝트 개인 확장
- 배포 링크 + 테스트 계정 제공

## 13. Q&A (57:00~60:00)

### Q1. AI 때문에 주니어 채용이 줄지 않나요?
원문 데이터상 주니어는 AI 도구 활용 향상 폭이 더 큽니다. 문제는 채용 축소가 아니라, 주니어에게 요구하는 역량이 "구현"에서 "판단+협업"으로 이동하는 것입니다.

### Q2. 스크럼은 이제 버려야 하나요?
원문 취지는 "버려라"가 아니라 "형식주의를 버려라"입니다. 어떤 프레임이든 범위·오너·리듬이 살아 있으면 동작합니다.

### Q3. AI 도구를 쓰면 왜 팀이 더 피곤해지기도 하나요?
작성 속도는 올라가도 리뷰·검증 비용이 남기 때문입니다. 따라서 팀은 품질 게이트와 리뷰 규칙을 같이 설계해야 합니다.

### Q4. 바로 내일부터 하나만 바꾼다면?
모든 작업에 오너 1명(DRI) 지정부터 시작하세요. 의사결정 지연이 가장 먼저 줄어듭니다.

### Q5. 포트폴리오에서 가장 먼저 고칠 곳은?
README 첫 화면입니다. "무엇을 만들었나"보다 "왜 그 선택을 했나"가 30초 안에 보이게 바꾸세요.
